1. 핵심 신호

신용 카드 요금 납부 VoC는 고객이 업무를 시작하고, 조건을 이해하고, 결과를 확인하고, 실패를 복구하는 전체 흐름을 정책 기준으로 재설계해야 함을 보여준다.

Core signals
01

가족 회선 납부 카드 일괄 변경 불가

" 가족 요금을 한 번에 처리하고 싶다 " → 번호별 개별 처리 강요로 인한 시간 · 노력 낭비에 대한 강한 불만 및 효율성 추구 욕구

02

납부 카드 정보 조회 · 변경 경로 미발견

" 어느 카드로 요금이 나가는지 명확히 알고 싶다 " → 납부 현황을 직접 눈으로 확인하고 싶은 정보 통제 욕구

03

타인 카드 등록 및 대리 납부 시 인증 절차 복잡

" 가족 카드 등록 시 동의 절차가 간편했으면 좋겠다 " → 가족 간 요금 관리의 편의성 및 신속한 처리 욕구

04

주말 · 야간 · 공휴일 업무 제한으로 납부 처리 불가

"잊어버리기 전에 지금 당장 처리하고 싶다 " → 시간 제약 없이 즉시 처리하고 싶은 즉시성 욕구

05

해지 후 잔여 요금 청구 및 납부 정보 불명확

" 해지했는데 왜 요금이 나오는지 명확히 알고 싶다 " → 청구 구조 이해 불가로 인한 불신 해소 욕구

6. 요구사항·정책서 전환 질문

VoC 분석 결과는 신용 카드 요금 납부 정책서에서 아래 질문에 답할 수 있는 기능과 정책 항목으로 전환되어야 한다.

Policy conversion
효율

반복 입력과 재문의를 어떻게 줄이는가?

반복 입력, 회선별 재처리, 동일 문의 재접수를 줄이도록 일괄 처리와 저장된 정보 재사용 기준을 설계해야 한다

진입점

고객은 어디에서 업무를 시작할 수 있는가?

통합채널의 홈, 마이, 요금, 상품 영역에서 같은 업무 진입점을 제공하고, 현재 상태에서 가능한 다음 행동을 명확히 보여줘야 한다

권한

누가 직접 처리하거나 대신 처리할 수 있는가?

본인, 가족, 법정대리인, 대리인의 처리 가능 범위와 인증, 동의, 위임 조건을 정책 기준으로 분리해야 한다

처리 가능 범위

언제 셀프 처리할 수 있고 언제 제한되는가?

평일, 야간, 주말, 공휴일, 청구 마감, 개통·해지 시점별 처리 가능 범위와 대체 행동을 정의해야 한다

사전 고지

무엇을 처리 전에 알려야 하는가?

요금, 혜택, 적용 시점, 제한 조건, 후속 영향을 업무 실행 전에 이해 가능한 문장과 비교 정보로 제공해야 한다

확인

처리 결과를 어떻게 확인시키는가?

처리 결과, 이력, 알림, 예상 청구 또는 사용 가능 상태를 고객이 직접 확인할 수 있어야 한다

상태

처리 상태를 어디까지 보여줘야 하는가?

신용 카드 요금 납부 업무의 접수, 진행, 완료, 실패, 취소 상태를 고객이 직접 확인하도록 정의해야 한다
