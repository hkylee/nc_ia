1. 개요

가. 범위

• 본 정책서는 통합채널에서 제공하는 이벤트, 미션, 기획전, 참여형 보상 프로그램의 탐색, 참여, 미션 수행, 결과 확인, 보상 수령, 운영 관리 기준을 정의한다.

• 이벤트 허브, 상세 핵심 정보, 참여 가능 여부, 입력 최소화, 단계형 참여, 참여 완료·실패, 참여 이력, 당첨·보상, 공유, 알림, 운영자 등록·검수·중지·리포트를 포함한다.

• 미션 프로그램은 출석형, 스탬프형, 친구초대형, 이용완료형, 구매연계형, 연속참여형, 팀형을 포함하며 진행률, 이어하기, 완료 인정, 보상 지급 기준을 포함한다.

• 쿠폰, 포인트, 멤버십 혜택의 상세 사용·정산 정책은 인접 정책서에서 상세화하고 본 문서에서는 이벤트/미션 보상 연결 기준만 다룬다.

• API 필드, DB 컬럼, 운영자 화면 상세, 배치 설계, 제휴 정산 상세는 제외하되 정책 판단에 필요한 상태, 조건, 제한, 이력 기준은 포함한다.

• 책임 경계는 통합채널 FO가 참여·미션 진행·결과 안내를 담당하고, 이벤트 연계 시스템이 참여 조건 판정·보상 상태 반영·외부 응답 연계를 담당하며, BSS는 고객·회선·구매·이용 결과 기준 정보를 제공하는 구조로 구분한다.

• 고객 표시 상태와 시스템 내부 상태는 분리한다.
고객에게는 참여 가능, 진행 중, 완료, 보상 대기, 지급 완료, 만료·사용불가처럼 다음 행동 기준을 보여주고, 내부적으로는 조건 판정, 보상 재고 확인, 지급 요청, 지급 실패, 회수·보정, 이력 저장 상태를 관리한다.

나. 설계 원칙

• 참여 가치 선명화: 고객이 참여 전 보상, 조건, 제한, 발표·지급 일정을 한눈에 이해할 수 있게 한다.

• 조건 사전 판정: 대상 여부, 남은 횟수, 필수 조건, 추가 인증, 보상 소진 여부를 행동 전에 안내한다.

• 미션 연속성 보장: 반복·시즌형 미션은 진행률, 누락일, 이어하기, 완료 인정 지연을 상태로 관리한다.

• 보상 신뢰성 확보: 당첨, 지급, 만료, 실패, 회수, 보정 기준을 명확히 하여 고객 혼선을 줄인다.

• 운영 통제 내재화: 등록, 검수, 승인, 배포, 중지, 롤백, 부정 참여 제재, 리포트 이력을 운영 기준으로 둔다.

• 인접 정책 경계 유지: 혜택·쿠폰·포인트·상품 사용 자체는 해당 정책서로 넘기고 이벤트/미션의 참여와 보상 연결만 정의한다.

4) 참여 자격·조건 판정 정책 (PG-EVT-ELIG-001)

• 대상 조건 (PI-EVT-ELIG-001-01)
- 참여 대상은 회원 상태, 회선 상태, 등급, 가입 상품, 구매 이력, 행동 완료, 지역·연령 제한 중 운영자가 지정한 조건 조합으로 판정한다.
- 판정에 사용한 조건과 미충족 조건은 고객 단위 이력에 저장한다.
• 횟수 제한 (PI-EVT-ELIG-001-02)
- 참여 횟수는 고객, 회선, 계정, 디바이스, 프로그램 단위 중 운영자가 지정한 기준으로 제한한다.
- 남은 횟수는 참여 전 표시한다.
• 불가 안내 (PI-EVT-ELIG-001-03)
- 대상 아님, 기간 종료, 조건 미충족, 횟수 초과, 인증 필요, 보상 소진은 표준 불가 사유로 안내한다.
- 고객이 취할 수 있는 다음 행동을 함께 제공한다.

5) 입력 최소화·동의 정책 (PG-EVT-CONSENT-001)

• 입력 최소화 (PI-EVT-CONSENT-001-01)
- 회원정보와 회선정보로 확인 가능한 항목은 자동 채움 또는 선택 방식으로 제공한다.
- 고객에게 중복 입력을 요구하지 않는다.
• 최종 동의 (PI-EVT-CONSENT-001-02)
- 참여 확정 전 개인정보 제공, 제휴 제공, 보상 지급, 유의사항 동의가 필요한 항목을 요약하고 최종 동의를 받는다.
• 동의 이력 (PI-EVT-CONSENT-001-03)
- 동의 문구 버전, 적용 시점, 동의 여부, 철회 여부는 프로그램과 고객 단위로 저장한다.

6) 참여·응모 접수 정책 (PG-EVT-PART-001)

• 접수 확정 (PI-EVT-PART-001-01)
- 필수 조건과 동의가 완료되면 참여 또는 응모 접수를 확정하고 참여 완료 상태로 저장한다.
• 단계형 진행 (PI-EVT-PART-001-02)
- 참여 과정이 2단계 이상이면 현재 단계, 진행률, 다음 단계, 중단 시 재개 가능 여부를 표시한다.
• 중복 방지 (PI-EVT-PART-001-03)
- 동일 고객·회선·디바이스의 동일 프로그램 중복 요청은 요청 키 기준 5초 이내 1건만 접수한다.
- 이미 접수된 응모는 추가 접수를 제한하고 기존 이력으로 안내한다.

7) 미션 진행·완료 인정 정책 (PG-EVT-MISSION-001)

• 미션 유형 (PI-EVT-MISSION-001-01)
- 미션은 출석형, 스탬프형, 실시간 반응형, 친구초대형, 이용완료형, 구매연계형, 연속참여형, 팀형으로 구분한다.
• 완료 인정 (PI-EVT-MISSION-001-02)
- 미션 완료는 지정 행동 이벤트, 구매·가입 결과, 친구 초대 성립, 팀 조건 달성 등 운영자가 등록한 기준이 충족될 때 인정한다.
- 외부 결과 반영이 지연되면 최대 24시간까지 인정 대기 상태로 두고 중복 수행을 제한한다.
• 인정 지연 (PI-EVT-MISSION-001-03)
- 이벤트 지연이나 외부 결과 지연으로 완료 인정이 늦어지면 완료 인정 대기 상태로 표시하고 예상 반영 시점을 안내한다.
• 일일 앵커 (PI-EVT-MISSION-001-04)
- 연속 방문 습관 형성 미션은 하루 1회 참여 가능 여부, 기준일 산정 시점, 누락일 처리, 다음 참여 가능 시간을 명확히 안내한다.
- 일일 앵커는 홈, 알림, 내 참여 이력에서 동일한 상태로 보여야 한다.
• 체류형 게이미피케이션 (PI-EVT-MISSION-001-05)
- 체류형 미션은 체류 시간, 탐색 완료, 콘텐츠 확인, 후속 행동 등 완료 인정 조건을 프로그램별로 등록한다.
- 백그라운드 체류, 반복 새로고침, 자동화 시도는 완료 인정에서 제외한다.
• 확정형 보상 목표 (PI-EVT-MISSION-001-06)
- 확정형 보상 목표 미션은 고객이 목표와 보상을 선택한 뒤 단계별 달성 조건, 남은 조건, 보상 확정 시점을 확인할 수 있어야 한다.
- 목표 선택 후 변경 가능 여부와 변경 제한 횟수는 프로그램별 정책으로 관리한다.

8) 미션 시즌·이어하기 정책 (PG-EVT-SEASON-001)

• 운영 주기 (PI-EVT-SEASON-001-01)
- 미션은 일간, 주간, 월간, 특정일 오픈, 시즌 단위로 운영할 수 있다.
- 주기별 시작일, 종료일, 남은 기간을 표시한다.
• 연속 참여 (PI-EVT-SEASON-001-02)
- 연속 참여 일수, 누락일, 다음 보상까지 남은 조건을 고객에게 표시한다.
- 누락일 복구 허용 여부는 프로그램별 정책으로 둔다.
• 이어하기 (PI-EVT-SEASON-001-03)
- 고객은 홈, 알림, 내 참여 이력, 혜택 보유 내역에서 진행 중인 미션으로 재진입할 수 있어야 한다.

9) 당첨·발표 정책 (PG-EVT-WINNER-001)

• 추첨 방식 (PI-EVT-WINNER-001-01)
- 당첨 방식은 자동 추첨, 수동 추첨, 즉시 당첨, 선착순, 조건 달성형 중 하나로 설정한다.
- 방식별 기준과 이력을 저장한다.
• 검증 기준 (PI-EVT-WINNER-001-02)
- 당첨 확정 전 중복 참여, 부정 참여, 대상 조건, 제외 조건, 보상 재고를 검증한다.
• 발표 안내 (PI-EVT-WINNER-001-03)
- 당첨 발표 일정, 발표 상태, 고객 알림, 미당첨 안내는 고객 이력과 상세 화면에서 확인 가능해야 한다.

10) 보상 지급·수령 정책 (PG-EVT-REWARD-001)

• 지급 기준 (PI-EVT-REWARD-001-01)
- 보상 지급 상태는 지급 예정, 지급 완료, 지급 실패, 회수, 만료, 사용불가로 구분한다.
- 상태 변경 시 고객 표시 상태, 보상함 저장 여부, 지급·회수 이력을 함께 갱신한다.
• 수령 행동 (PI-EVT-REWARD-001-02)
- 보상 지급 후 고객은 바로 받기, 저장, 즉시 사용, 관련 서비스 이동 중 가능한 행동을 선택할 수 있어야 한다.
• 만료·사용불가 (PI-EVT-REWARD-001-03)
- 보상 만료, 사용 조건 불일치, 제휴 장애, 회수 처리 시 사유, 해결 방법, 대체 행동 또는 CS 경로를 안내한다.
- 만료·사용불가 상태는 고객 표시 상태와 내부 보상 상태에 동시에 반영하고 이력을 저장한다.
• 랜덤 포인트 지급 (PI-EVT-REWARD-001-04)
- 이용내역 기반 랜덤 포인트는 대상 고객, 지급 확률 또는 구간, 1회 지급 한도, 일·월 지급 한도, 지급 제외 조건을 프로그램별로 관리한다.
- 지급 결과는 고객 이력과 운영 리포트에 함께 남긴다.
• 혜택 순번·랜덤 혜택 (PI-EVT-REWARD-001-05)
- 혜택 순번 또는 랜덤 혜택은 고객에게 지급 가능 순서, 남은 기회, 확정 시점, 미수령 시 만료 기준을 안내한다.
- 순번 변경이나 랜덤 혜택 확정 결과는 알림과 이력으로 확인 가능해야 한다.
• 결제 후 보상 루프 (PI-EVT-REWARD-001-06)
- 결제 완료 후 연계 이벤트 보상은 결제 성공, 취소 가능 기간, 보상 지급 가능 상태, 환불·취소 시 회수 기준을 확인한 뒤 지급한다.
- 결제 성공 후 보상 지급 실패가 발생하면 보정 대상 상태로 전환하고 재처리 이력을 저장한다.

11) 실패·중단·보정 정책 (PG-EVT-FAIL-001)

• 참여 실패 (PI-EVT-FAIL-001-01)
- 네트워크 또는 서버 오류로 실패하면 재시도, 나중에 하기, CS 전환 중 가능한 경로를 제공한다.
- 보존 가능한 입력값은 유지한다.
• 지급 실패 보정 (PI-EVT-FAIL-001-02)
- 보상 지급 실패, 미발급, 재고 소진, 연계 실패는 보정 대상으로 등록하고 원인, 보정 방식, 처리자를 이력으로 남긴다.
• 운영 이슈 (PI-EVT-FAIL-001-03)
- 발표 지연, 지급 지연, 제휴 장애, 긴급 중지 발생 시 일괄 공지와 대체 보상 기준을 즉시 반영할 수 있어야 한다.

12) 참여 이력·상태 노출 정책 (PG-EVT-HISTORY-001)

• 고객 이력 (PI-EVT-HISTORY-001-01)
- 고객은 참여 일시, 참여 방식, 조건 스냅샷, 진행 상태, 당첨·보상 상태를 이력 상세에서 확인할 수 있어야 한다.
• 상태 필터 (PI-EVT-HISTORY-001-02)
- 참여, 응모 완료, 추첨 대기, 당첨, 미당첨, 지급 예정, 지급 완료, 만료 상태로 필터링할 수 있어야 한다.
• 운영 이력 (PI-EVT-HISTORY-001-03)
- 운영자는 등록, 수정, 검수, 승인, 배포, 중지, 롤백, 보정, 회수, 수동 승인 이력을 조회할 수 있어야 한다.

13) 운영 등록·템플릿 정책 (PG-EVT-ADMIN-001)

• 기본 정보 (PI-EVT-ADMIN-001-01)
- 운영자는 프로그램 명칭, 기간, 상태, 유형, 상세 콘텐츠, 보상, 유의사항, FAQ를 등록한다.
• 유형 템플릿 (PI-EVT-ADMIN-001-02)
- 응모형, 즉시 당첨형, 출석형, 구매연계형, 친구초대형, 팀형은 참여 단계, 상태값, 보상 방식, 안내 문구를 템플릿으로 관리한다.
• 조건 변경 검증 (PI-EVT-ADMIN-001-03)
- 대상, 조건, 보상, 기간, 노출 우선순위 변경 시 영향 범위를 사전 검증하고 승인 전에는 고객에게 노출하지 않는다.
• 기능 모듈화 (PI-EVT-ADMIN-001-04)
- 혜택·프로모션 기능은 등록, 조건 판정, 보상, 알림, 리포트 모듈로 재사용 가능하게 관리한다.
- 동일 목적의 프로그램은 기존 모듈을 우선 사용하고, 신규 모듈은 중복 기능과 영향 범위를 검증한 뒤 등록한다.
• 파라미터 명명 규칙 (PI-EVT-ADMIN-001-05)
- 운영자가 등록하는 이벤트 파라미터는 프로그램 유형, 대상 조건, 보상 유형, 상태값을 식별할 수 있는 표준 명명 규칙을 따른다.
- 규칙을 벗어난 파라미터는 배포 전에 보정하고 변경 이력을 남긴다.
• 대상자 업로드 형식 (PI-EVT-ADMIN-001-06)
- 대상자 Excel 업로드는 고객 식별 기준, 참여 가능 여부, 보상 유형, 유효 기간, 제외 사유 컬럼을 포함한 표준 양식을 사용한다.
- 필수 컬럼 누락, 중복 고객, 형식 오류가 있으면 등록을 제한하고 오류 건수를 운영자에게 안내한다.

14) 검수·승인·배포·중지 정책 (PG-EVT-APPROVAL-001)

• 승인 흐름 (PI-EVT-APPROVAL-001-01)
- 운영 상태는 작성, 검수, 승인, 배포, 중지, 종료, 롤백으로 관리한다.
- 승인 전 프로그램은 고객에게 노출하지 않는다.
• 긴급 중지 (PI-EVT-APPROVAL-001-02)
- 법적 이슈, 지급 장애, 제휴 장애, 부정 참여 확산, 고객 피해 가능성이 있으면 운영자가 즉시 중지할 수 있다.
• 롤백 (PI-EVT-APPROVAL-001-03)
- 배포 후 오류가 발견되면 직전 승인 버전으로 롤백하고 롤백 사유와 영향 범위를 공지한다.
• 노출 정책 승인 (PI-EVT-APPROVAL-001-04)
- 이벤트 노출 채널, 위치, 상태값, 예약 시작·종료, 중지, 재개, 종료 후 후속 노출 규칙은 승인 후 적용한다.
- 노출 정책 변경은 고객 영향 범위와 충돌 이벤트를 확인한 뒤 배포한다.
• 한정 수량 운영 (PI-EVT-APPROVAL-001-05)
- 온라인 전용 특가와 한정 수량 이벤트는 수량 잠금, 소진 시점, 대기·실패 안내, 대체 혜택 제공 여부를 사전에 등록한다.
- 선착순 종료 후에는 추가 참여를 제한하고 종료 사유와 후속 행동을 안내한다.

15) 부정 참여·제재 정책 (PG-EVT-FRAUD-001)

• 탐지 기준 (PI-EVT-FRAUD-001-01)
- 중복 참여, 자동화 시도, 허위 초대, 비정상 IP·디바이스 패턴, 조건 우회 시도를 부정 참여 후보로 분류한다.
• 제재 기준 (PI-EVT-FRAUD-001-02)
- 부정 참여 의심 상태에서는 당첨과 보상 지급을 보류한다.
- 운영자 검증 후 제외, 회수, 복구, 수동 승인 중 하나로 처리한다.
• 이력 관리 (PI-EVT-FRAUD-001-03)
- 제재, 회수, 복구, 수동 승인 이력은 고객, 프로그램, 보상 단위로 저장하고 재검토 가능해야 한다.

16) 알림·공유·딥링크 정책 (PG-EVT-NOTI-001)

• 알림 유형 (PI-EVT-NOTI-001-01)
- 알림은 참여 완료, 미션 인정 대기, 당첨 발표, 지급 완료, 지급 실패, 만료 임박, 운영 중지 상태에서 발송한다.
