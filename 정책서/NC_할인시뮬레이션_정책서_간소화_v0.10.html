1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객이 현재 조건과 후보 조건을 바탕으로 예상 비용, 할인, 지원금, 혜택, 가입 가능성을 비교하고 다음 행동을 결정하는 기준을 정의한다.

• 대상 업무는 회선·요금제·단말·결합·구독·로밍·복지할인 조건 반영, 공통·별도·추가지원금 산정, 다중 시나리오 비교, 최대 혜택 제안, 가입 가능·승계 가능 판정, 결과 저장, 장바구니·주문서 전환 전 재검증이다.

• 대상 채널은 통합채널 앱·웹 FO와 운영자가 관리하는 시뮬레이션 BO이며, 할인 연계 시스템의 고객 조건·상품 관계·혜택 관계·지원금·가입 가능 판정 결과를 포함한다.

• 포함 범위는 예상 월 납부액, 일시 비용, 정상가, 할인 반영 후 금액, 공통지원금 합산 노출, 포함·제외 항목, 오차 가능성, 부적격 사유, 프론트 연동 문구, 삭제 제한, 상담 전환 기준이다.

• 제외 범위는 결제 승인, 청구 확정, 실제 원장 반영, 주문 이행, API 필드, DB 컬럼, 화면 상세 설계다.
단, 해당 항목이 고객 판단과 운영 삭제·배포 가능 여부에 영향을 주면 정책 항목으로 남긴다.

• 후속 산출물에서는 주문·계약, 결제 처리, 청구·납부, 상품 상세, 장바구니 정책서가 시뮬레이션 결과를 이어받아 실행·확정 기준을 상세화한다.

• 채널 업무 시스템은 할인 비교, 예상 청구액 표시, 조건 안내와 재계산 경로를 책임지고 할인 연계 시스템은 지원금·쿠폰·포인트·가입 유형·BSS 기준 정보를 연계해 산정 결과를 제공한다.

• 고객 표시 상태는 적용 가능, 조건 미충족, 예상 금액, 재계산 필요, 적용 제한으로 안내하고 내부 상태는 산식 버전, 지원금 정책, 쿠폰·포인트 잠금, BSS 반영 가능성으로 분리한다.

• 상세 요구사항 70건은 고객 할인 탐색, 시뮬레이션 계산, 지원금 운영, 문구 검증, 프론트 연동 기준으로 재구성해 유즈케이스·프로세스·기능·정책 중 하나 이상에 매핑했다.

• 중복 요청과 멱등 기준은 동일 고객·동일 조건·동일 산식 버전의 시뮬레이션 요청은 5분 동안 동일 결과 ID로 묶고, 조건·혜택·산식 버전이 바뀐 경우에만 새 결과로 산정하는 것이다.

• 통합 리스크는 할인 연계 시스템 장애, 산식 버전 불일치, 채널 간 캐시 차이, BSS 응답 지연으로 예상 금액이 달라지는 경우이며 이때 고객에게 예상값 지연 또는 재계산 필요 상태를 안내한다.

나. 설계 원칙

• 예상과 확정을 분리한다.
시뮬레이션 결과는 예상 기준으로 표시하고 실제 청구 확정 금액과 혼동하지 않게 한다.

• 고객 현재 조건을 우선한다.
로그인 고객의 보유 조건은 자동 반영하되, 고객이 직접 바꾼 조건은 해당 세션에서 우선 적용한다.

• 최대 혜택보다 조건 영향을 먼저 고지한다.
금액이 유리해도 약정, 해지, 승계 제한, 결합 영향이 있으면 사전 안내한다.

• 가능 여부를 사전에 판정한다.
고객이 주문이나 장바구니로 이동하기 전에 가입 가능, 승계 가능, 재가입 제한, 상품 관계 충돌을 확인한다.

• 계산 근거와 오차를 투명하게 표시한다.
포함·제외 항목, 추정치, 실제 청구 차이 가능성은 결과와 함께 제공한다.

• 운영 변경은 검수와 이력을 남긴다.
계산식, 마케팅프로그램, 안내 문구 변경은 사전 검수, 배포, 롤백, 모니터링 기준을 가진다.

• 지원금 트랙을 단일 기준으로 관리한다.
공통지원금 A/B 트랙, 별도지원금, 추가지원금은 가입유형과 할인방법에 따라 계산·노출·삭제 기준을 분리한다.

• 운영 등록값은 고객 노출 전 검증한다.
상품 코드, 노출 순서, 추천 여부, Bar 문구, 사용여부, 참조 데이터는 저장·삭제·배포 전 유효성을 확인한다.

- 시뮬레이션 결과는 예상 기준이며 실제 청구 확정 금액이 아니다.
- 고객 화면에는 예상 기준, 포함 항목, 제외 항목, 실제 청구 차이 가능성을 함께 고지한다.

2) 시뮬레이션 전제조건 정책 (PG-DSC-CND-001)

• 자동 반영 (PI-DSC-CND-001-001)
- 로그인 고객은 현재 요금제, 회선 수, 결합, 구독, 약정, 보유 할인 상태를 기본 조건으로 자동 반영한다.
- 고객이 직접 수정한 조건은 해당 세션의 계산에서 자동 반영값보다 우선한다.
• 다회선 선택 (PI-DSC-CND-001-002)
- 다회선 고객은 계산 대상 회선을 1개 이상 선택해야 한다.
- 회선별 조건이 다르면 변경 전/후 요금 정보와 혜택 차이는 선택한 회선 기준으로 계산한다.
• 신규 가입 (PI-DSC-CND-001-003)
- 신규 가입 시 변경 전 상품, 변경 전 마케팅프로그램, 변경 전 요금정보, 요금차액은 표시하지 않는다.
- 신규 기준 예상 월 납부액과 일시 비용만 제공한다.

3) 고객 설정 요소 정책 (PG-DSC-INPUT-001)

• 필수 조건 (PI-DSC-INPUT-001-001)
- 요금제, 상품 유형, 약정 기간, 할인 후보, 회선 기준은 필수 조건으로 관리한다.
- 필수 조건이 누락되면 계산을 실행하지 않고 누락 항목을 먼저 안내한다.
• 선택 조건 (PI-DSC-INPUT-001-002)
- 결합, 구독, 복지할인, 로밍, 단말보험, 가족 로밍 조건은 선택 조건으로 관리한다.
- 선택 조건은 고객이 추가하거나 제외할 수 있고 변경 즉시 재계산한다.

4) 현재 대비 비교 정책 (PG-DSC-COMPARE-001)

• 동일 단위 비교 (PI-DSC-COMPARE-001-001)
- 현재 대비 절감 금액, 데이터 차이, 추가 혜택, 주요 조건 변화는 월 기준과 일시 기준을 분리해 같은 단위로 표시한다.
- 금액, 기간, 혜택 단위가 다르면 비교 표에서 같은 행에 섞어 표시하지 않는다.
• 전환 근거 (PI-DSC-COMPARE-001-002)
- 추천안에는 절감액, 증가 비용, 혜택 증가, 약정 영향, 조건 제한 중 최소 2개 이상의 전환 근거를 함께 표시한다.
- 단순히 더 저렴하다는 문구만으로 추천하지 않는다.
• 다중 비교 (PI-DSC-COMPARE-001-003)
- 고객은 최소 2개, 최대 5개 시나리오를 나란히 비교할 수 있다.
- 5개를 초과하면 비교 후보 저장 후 기존 후보를 제거하거나 그룹을 나누도록 안내한다.

5) 예상 비용 산정·표시 정책 (PG-DSC-AMOUNT-001)

• 금액 구분 (PI-DSC-AMOUNT-001-001)
- 시뮬레이션 결과는 예상 월 납부액, 일시불 금액, 정상가, 할인 반영 후 총 비용을 분리해 표시한다.
- 즉시 발생 비용과 매월 반복 비용은 다른 라벨로 표시한다.
• 할인 0원 표시 (PI-DSC-AMOUNT-001-002)
- 할인 적용 후 실결제액이 0원인 경우 정상 이용 금액과 할인 후 결제 금액을 함께 표시한다.
- 고객이 이용 금액을 결제 금액으로 오인하지 않도록 “실 결제 0원” 라벨을 추가한다.
• 반환금 계산 (PI-DSC-AMOUNT-001-003)
- 할인 반환금 또는 약정 해지 영향은 예상 금액으로 표시하고 실제 청구 확정 전에는 확정 금액으로 표현하지 않는다.
- 반환금 산정 기준일과 포함 여부를 함께 고지한다.

6) 최대 혜택 우선 제안 정책 (PG-DSC-BEST-001)

• 최대 혜택 선적용 (PI-DSC-BEST-001-001)
- 적용 가능한 할인 조합이 여러 개이면 고객 부담 금액이 가장 낮은 조합을 우선 제안한다.
- 단, 약정 기간 증가나 해지 제한이 있으면 혜택 금액보다 조건 영향을 먼저 고지한다.
• 요금제 조건 일치 (PI-DSC-BEST-001-002)
- 최대 할인 요금제 추천은 고객이 선택한 요금제 또는 변경 후보 요금제 기준으로 계산한다.
- 고객의 현재 요금제와 다른 상위 요금제 할인은 “조건 변경 시 가능”으로 분리 표시한다.
• 고객 선택권 (PI-DSC-BEST-001-003)
- 최대 혜택 추천안이 자동 적용되어도 고객은 전체 마케팅프로그램 목록에서 다른 후보를 선택할 수 있다.
- 고객 선택 후에는 자동 추천값보다 고객 선택값을 우선 적용한다.
• 현재 요금제 구독가 (PI-DSC-BEST-001-004)
- 최대할인 구독가는 고객의 현재 요금제 또는 고객이 선택한 후보 요금제 기준으로 계산한다.
- 109 요금제 고객에게 129 요금제 조건의 할인 금액을 적용하는 경우에는 자동 적용하지 않고 요금제 변경 시 가능 조건으로 분리한다.
• 상품 구매 최대할인 구독가 (PI-DSC-BEST-001-005)
- 상품 구매를 위한 최대할인 구독가는 TVING 광고형 스탠다드, Wavve 광고형 스탠다드처럼 구독 후보별로 고객 현재 요금제 기준 할인액을 먼저 계산한다.
- 더 높은 요금제에서만 가능한 할인은 현재 요금제 할인으로 표시하지 않고 조건 변경 후보로 구분한다.

7) 할인 중복·배타 적용 정책 (PG-DSC-EXCL-001)

• 중복 기준 (PI-DSC-EXCL-001-001)
- 할인 간 중복 가능 여부는 허용, 배타, 조건부 허용으로 구분한다.
- 배타 할인은 동시에 적용하지 않고 제외 사유를 고객에게 표시한다.
• 적용 우선순위 (PI-DSC-EXCL-001-002)
- 중복 가능 할인은 고객 부담금 감소 효과가 큰 순서로 적용한다.
- 법정·복지성 할인처럼 우선 적용해야 하는 할인은 금액 효과와 무관하게 우선순위를 별도 관리한다.
• 충돌 예외 (PI-DSC-EXCL-001-003)
- 상품 운영자가 예외 허용 기간을 지정한 경우에만 배타 조건을 우회한다.
- 예외에는 시작일, 종료일, 승인자, 적용 대상 상품을 반드시 기록한다.

8) 마케팅프로그램 가입 가능 정책 (PG-DSC-MP-001)

• 운영상태 필터 (PI-DSC-MP-001-001)
- 가입 가능한 마케팅프로그램 목록은 운영상태인 상품만 제공한다.
- 중지, 종료, 예정 상태의 프로그램은 고객 선택 후보에서 제외하고 운영자 화면에서만 확인한다.
• 가입 가능 조회 (PI-DSC-MP-001-002)
- 고객 정보 기반으로 가입 가능한 마케팅프로그램을 조회하고 선택한 프로그램의 가입 가능 여부를 다시 확인한다.
- 가입 불가 시 불가 사유와 가능 시점을 표시한다.
• 프로그램 유형 (PI-DSC-MP-001-003)
- 요금제 프로그램, 결합 프로그램, 단말보험 상품은 Marketing Program의 가입 가능 정책을 참조한다.
- 참조 결과가 없으면 가입 가능으로 간주하지 않는다.

9) 약정·지원금·보험 비교 정책 (PG-DSC-SUPPORT-001)

• 공시지원금 (PI-DSC-SUPPORT-001-001)
- 단말 구매 시 공시지원금 금액, 선택약정 적용 금액, 약정 기간별 총 비용을 함께 비교한다.
- 공시지원금은 단말과 요금제 조합 기준으로만 표시한다.
• 선택약정 승계 (PI-DSC-SUPPORT-001-002)
- 선택약정 또는 T지원금 승계 가능 여부는 고객 상품 변경 이력과 재가입 제한 기준을 조회해 판정한다.
- 제한이 있으면 가능 일자와 제한 사유를 함께 안내한다.
• 보험·프로그램 (PI-DSC-SUPPORT-001-003)
- 단말보험과 요금제 프로그램은 가입 가능 여부 확인 후에만 업무처리 가능으로 표시한다.
- 확인 실패 시 가능 여부 미확정으로 표시하고 주문 전환을 제한한다.

10) 결합·다회선·구독 조합 정책 (PG-DSC-COMB-001)

• 결합 비교 (PI-DSC-COMB-001-001)
- 현재 결합과 변경 후보 결합의 할인 금액, 적용 조건, 가족 회선 영향, 해지·변경 제한을 비교한다.
- 결합 변경은 주문·계약 확정 전 재검증 대상이다.
• 다회선 조합 (PI-DSC-COMB-001-002)
- 다회선, 인터넷/IPTV, 구독 조합은 회선별 예상 비용과 전체 가구 또는 그룹 기준 예상 비용을 분리해 표시한다.
- 합산 금액에는 포함 회선을 명시한다.
• 상품 관계 (PI-DSC-COMB-001-003)
- 상품 간 동시 가입, 선가입, 선해지, 자동 처리 가능 여부는 관계 판정 결과를 기준으로 표시한다.
- 관계 데이터가 없으면 가능 여부 미확정으로 처리한다.

11) 복지·로밍 특화 할인 정책 (PG-DSC-SPECIAL-001)

• 복지할인 (PI-DSC-SPECIAL-001-001)
- 복지할인은 종류별 적용 조건과 예상 할인 금액을 비교한다.
- 증빙 또는 자격 확인이 필요한 경우 예상 할인으로만 표시하고 확정 적용은 후속 업무에서 확인한다.
• 로밍 비용 (PI-DSC-SPECIAL-001-002)
- 로밍 충전과 재가입 옵션은 예상 비용, 제공량, 유효 기간, 잔여량 처리 기준을 비교한다.
- 가족 로밍은 가족결합 상태를 기준으로 최저 비용 시나리오를 추천한다.
• 기기 구매 (PI-DSC-SPECIAL-001-003)
- 기기 구매 비용은 단말 가격, 공시지원금, 선택약정, 월 부담액, 일시 비용을 함께 표시한다.
- 단말 재고나 배송 가능성은 상품상세·장바구니 정책의 판정 결과를 참조한다.

12) 가입 가능·부적격 판정 정책 (PG-DSC-ELG-001)

• 가입 가능 (PI-DSC-ELG-001-001)
- 가입 가능은 고객 조건, 상품 운영상태, 상품 관계, 할인 관계, 청구 조건이 모두 충족된 경우에만 표시한다.
- 하나라도 확인 실패하면 가능으로 표시하지 않는다.
• 부적격 사유 (PI-DSC-ELG-001-002)
- 부적격 사유는 고객 상태, 상품 조건, 혜택 조건, 변경 이력, 운영상태, 연계 실패 중 하나로 분류한다.
- 고객에게는 이해 가능한 문구와 다음 행동을 함께 제공한다.
• 가능 시점 (PI-DSC-ELG-001-003)
- 가입 제한이 기간 조건이면 가능 시점 또는 재확인 기준일을 표시한다.
- 가능 시점을 산정할 수 없으면 상담 전환 또는 나중에 다시 시도 안내를 제공한다.
• 가입일자·청구조건 (PI-DSC-ELG-001-004)
- 가입가능여부 확인에는 Product Catalog의 가입정책, 업무 카탈로그 정책, 청구조건 후불 여부, 구매 후 D일 이내 가입일자 조건을 함께 반영한다.
- 조건 중 하나라도 확인되지 않으면 가입 가능으로 표시하지 않는다.
• 조회 시점 판정 (PI-DSC-ELG-001-005)
- 고객의 상품 가입 가능여부는 고객 조회 시점 실시간 판정을 원칙으로 한다.
- 성능 저하가 예상되는 상품군은 사전 필터링 결과를 먼저 제공하되 주문 또는 장바구니 전환 전에는 다시 실시간 판정한다.
• 고객 조회 시점 가입 가능여부 (PI-DSC-ELG-001-006)
- 고객의 상품 가입 가능여부를 고객 조회 시점에 제공할지, 사전 필터링으로 가입 가능한 상품만 먼저 제공할지는 상품군별 성능과 UX 기준으로 관리한다.
- 단, 사전 필터링을 사용해도 장바구니 또는 주문 전환 전에는 실시간 가입 가능여부를 다시 조회한다.

13) 변경 횟수·재가입 제한 정책 (PG-DSC-LIMIT-001)

• 제외 항목 (PI-DSC-DISCLOSE-001-002)
- 세금, 실제 청구 조정, 사용량 기반 과금, 후속 심사 결과처럼 시뮬레이션에 포함되지 않은 항목은 제외 항목으로 명시한다.
• 오차 고지 (PI-DSC-DISCLOSE-001-003)
- 추정 금액은 실제 청구 시 달라질 수 있음을 고지한다.
- 오차 가능성이 있는 결과는 확정형 표현을 금지하고 “예상” 또는 “참고” 라벨을 붙인다.

15) 주문·장바구니 전환 정책 (PG-DSC-HANDOFF-001)

• 전환 전 재검증 (PI-DSC-HANDOFF-001-001)
- 장바구니, 주문서, 상품 상세로 전환하기 전 계산 결과, 가입 가능 여부, 할인 적용 가능 여부를 다시 확인한다.
- 재검증 실패 시 전환하지 않고 사유를 안내한다.
• 전환 문맥 (PI-DSC-HANDOFF-001-002)
- 전환 문맥에는 입력 조건, 선택 할인, 예상 비용, 추천 사유, 부적격 사유, 계산 시각을 포함한다.
- 후속 화면은 이 문맥을 기준으로 재계산 여부를 판단한다.

16) 결과 저장·유효기간 정책 (PG-DSC-SAVE-001)

• 저장 기준 (PI-DSC-SAVE-001-001)
- 시뮬레이션 결과 저장 시 고객 조건, 계산 결과, 추천안, 제외 항목, 유효 기간을 함께 저장한다.
- 저장 결과는 24시간 동안 이어보기 후보로 제공한다.
• 만료 기준 (PI-DSC-SAVE-001-002)
- 저장 후 24시간이 지나거나 상품 가격, 혜택, 재고, 정책 조건이 변경되면 결과 만료로 전환한다.
- 만료 결과는 주문 전환에 사용할 수 없고 재계산을 요구한다.

17) 실패·부적격 대응 정책 (PG-DSC-FAIL-001)

• 계산 실패 (PI-DSC-FAIL-001-001)
- BSS 또는 연계 시스템 응답이 3초 안에 완료되지 않으면 지연 안내를 제공한다.
- 동일 계산이 2회 실패하면 상담 또는 조건 변경 경로를 제안한다.
• 미적용 안내 (PI-DSC-FAIL-001-002)
- 고객이 선택한 할인 또는 프로그램이 적용되지 않으면 미적용 사유와 대체 가능한 후보를 함께 표시한다.
- 사유 없는 미적용 표시는 금지한다.
• 본인인증 미완료 할인 미반영 (PI-DSC-FAIL-001-003)
- 본인인증이 완료되지 않아 결합·복지 할인이 반영되지 않으면 비교 결과에 미반영 사유를 표시한다.
- 고객에게는 인증 완료 후 재계산 가능 여부와 현재 결과가 예상 기준임을 함께 안내한다.

18) 상담·AI 지원 전환 정책 (PG-DSC-CONSULT-001)

• 상담 전환 (PI-DSC-CONSULT-001-001)
- 부적격 사유가 2개 이상 중첩되거나 고객이 같은 조건으로 2회 이상 실패하면 상담 전환을 제안한다.
- 상담 전환 시 입력 조건과 계산 결과를 상담 문맥으로 전달한다.
• AI 지원 (PI-DSC-CONSULT-001-002)
- 단순 조건 비교, 대체 후보 설명, 오차 안내는 AI 지원으로 먼저 제공할 수 있다.
- 고객 피해 가능성이 있는 확정 가입 가능 여부와 청구 확정 금액은 AI 단독 확정을 금지한다.

19) BSS·연계 판정 정책 (PG-DSC-BSS-001)

• 필수 회신 (PI-DSC-BSS-001-001)
- 할인 연계 시스템은 고객 조건, 상품 관계, 혜택 관계, 가입 가능 여부, 예상 비용, 지원금, 승계 가능 여부를 회신해야 한다.
- 필수 회신 누락 시 결과를 확정 표시하지 않는다.
• 응답 기준 (PI-DSC-BSS-001-002)
- 실시간 판정 응답 목표는 3초 이내로 둔다.
- 3초 초과 시 지연 안내를 표시하고 10초 초과 시 결과를 보류하며 재시도 또는 상담 전환을 제안한다.

20) 이력·감사·보관 정책 (PG-DSC-HISTORY-001)

• 고객 이력 (PI-DSC-HISTORY-001-001)
- 고객의 시뮬레이션 조건, 결과 저장, 주문 전환, 상담 전환 이력은 90일 동안 조회 가능하게 보관한다.
- 개인정보와 민감 조건은 마스킹 기준을 적용한다.
• 운영 이력 (PI-DSC-HISTORY-001-002)
- 계산식, 할인 규칙, 오차 문구, 마케팅프로그램 운영상태 변경은 변경 전후 값, 승인자, 배포 시각, 롤백 여부를 저장한다.

21) 운영 계산식·버전 관리 정책 (PG-DSC-OPS-001)

• 사전 검수 (PI-DSC-OPS-001-001)
- 계산식 또는 할인 규칙 변경은 배포 전 샘플 고객 조건 5개 이상으로 기존 결과와 신규 결과 차이를 검수한다.
- 예상 차이가 5%를 초과하면 승인자 확인 후 배포한다.
• 롤백 기준 (PI-DSC-OPS-001-002)
- 배포 후 계산 실패율이 5% 이상이거나 실제 청구 대비 오차 민원이 10건 이상 발생하면 이전 계산식으로 롤백하고 변경 이력을 남긴다.

22) 모니터링·품질 보정 정책 (PG-DSC-MON-001)

• 품질 지표 (PI-DSC-MON-001-001)
- 운영자는 계산 성공률, 평균 응답 시간, 부적격 사유 분포, 상담 전환율, 실제 청구 대비 차이를 일 단위로 모니터링한다.
• 보정 요청 (PI-DSC-MON-001-002)
- 특정 상품 또는 혜택에서 부적격 사유가 전일 대비 30% 이상 증가하면 관계 데이터와 운영상태를 확인하고 보정 요청을 생성한다.

23) 공통·별도·추가지원금 산정 정책 (PG-DSC-SUBSIDY-001)

• A/B 트랙 매핑 (PI-DSC-SUBSIDY-001-001)
- 공통지원금은 A only, A+B, B 추가 분기로 구분하고 할인방법 분기와 교차 매핑한다.
- 지원금 트랙과 할인방법이 충돌하면 고객 화면에는 적용하지 않고 운영 검수 대상으로 전환한다.
• 가입유형별 별도지원금 (PI-DSC-SUBSIDY-001-002)
- 별도지원금은 번호이동, 기기변경, 신규가입 가입유형별로 적용 범위를 분리한다.
- 가입유형이 확인되지 않으면 별도지원금을 예상 금액에 합산하지 않는다.
• 합산 노출 (PI-DSC-SUBSIDY-001-003)
- 지원금 총액은 공통지원금, 별도지원금, 추가지원금을 분리 노출한 뒤 합산 금액을 함께 표시한다.
- 추가지원금이 15% 기준으로 산정되면 산정 기준 금액과 적용 한도를 함께 고지한다.
• 변동 우선순위 (PI-DSC-SUBSIDY-001-004)
- 공통지원금 A만 변경, B만 추가, A+B 모두 변경이 동시에 발생하면 최신 배포 버전, 고객 선택 할인방법, 가입유형 적합성 순서로 우선순위를 적용한다.
- 우선순위가 결정되지 않으면 자동 적용하지 않는다.
• 월 납부금액 계산 (PI-DSC-SUBSIDY-001-005)
- 공통지원금 월 납부금액은 A only와 A+B 분기를 구분해 계산한다.
- A only는 공통지원금 A만 월 비용에서 차감하고, A+B는 공통지원금 A와 B를 합산한 뒤 별도·추가지원금 적용 가능 여부를 순차 판정한다.

24) 비교 화면 확장·선택값 유지 정책 (PG-DSC-CMP-UI-001)

• 비교 개수 확장 (PI-DSC-CMP-UI-001-001)
- 요금제 가격 비교는 기본 2개 이상 비교를 허용하고, 운영 설정에 따라 최대 비교 개수를 관리한다.
- 화면 가독성을 해치는 개수 초과 시 후보 저장 후 비교 대상을 교체하도록 안내한다.
• 선택값 유지 (PI-DSC-CMP-UI-001-002)
- 요금제 변경 후 구매조건 선택 페이지로 이동할 때 이전 선택값은 상품 유형, 가입유형, 할인방법이 동일한 경우에만 유지한다.
- 기준 중 하나라도 바뀌면 선택값을 초기화하고 재선택 사유를 안내한다.
• 연간 절감액 (PI-DSC-CMP-UI-001-003)
- 요금제 비교 화면은 월 납부금액 차액과 연간 절감 예상액을 함께 표시할 수 있다.
- 연간 절감액은 월 차액 12개월 환산값으로 표시하고 약정 기간·프로모션 종료일이 더 짧으면 해당 기간 기준으로 제한한다.

25) 추가지원금 운영 상태 정책 (PG-DSC-ADD-001)

• 조회 기준 유형 (PI-DSC-ADD-001-001)
- 추가지원금 조회는 요금제, 단말, 가입유형, 할인방법 중 운영자가 선택한 기준 유형으로 수행한다.
- 기준 유형이 선택되지 않으면 조회를 실행하지 않는다.
• 등록 모델 유형 (PI-DSC-ADD-001-002)
- 추가지원금 등록은 추가지원금 유형에 따라 저장 모델을 분리한다.
- 단말 기준, 요금제 기준, 요금약정할인 기준을 혼합 저장하지 않으며 다른 유형의 필수값을 요구하지 않는다.
• 절차 단계 상태 (PI-DSC-ADD-001-003)
- 추가지원금 등록은 임시저장, 검수요청, 승인완료, 배포완료, 반려, 삭제제한 상태로 관리한다.
- 승인완료 전에는 FO에 노출하지 않고 배포완료 이후 변경은 신규 버전으로 저장한다.
• 삭제 가능 상태 (PI-DSC-ADD-001-004)
- 추가지원금은 임시저장 또는 반려 상태에서만 즉시 삭제할 수 있다.
- 승인완료 또는 배포완료 상태는 사용여부와 참조 주문·시뮬레이션 이력을 확인한 뒤 삭제 대신 사용중지로 전환한다.
• 추가지원금 조회 기준 유형 (PI-DSC-ADD-001-005)
- 추가지원금 조회 기준 유형은 요금제 기준, 단말 기준, 가입유형 기준, 할인방법 기준으로 구분한다.
- 운영자는 기준 유형을 선택해야 조회할 수 있으며 선택한 기준과 맞지 않는 입력값은 조회 조건에서 제외한다.

26) 프론트 연동·문구 관리 정책 (PG-DSC-FRONT-001)

• 프론트 연동 (PI-DSC-FRONT-001-001)
- 시뮬레이션 BO의 상품 코드, 노출 순서, 추천 여부, Bar 문구는 FO 인터넷/IPTV 요금 계산기 상품선택 영역에 동일 기준으로 반영한다.
- FO 반영 결과가 BO 설정과 다르면 배포 완료로 보지 않는다.
• 상품 유형 노출 (PI-DSC-FRONT-001-002)
- 인터넷, IPTV, 셋톱박스 상품은 상품 유형별 최대 노출 개수와 중복 등록 여부를 검증한다.
- 동일 유형에서 추천 상품이 2개 이상이면 운영자가 우선순위를 지정해야 한다.
• Bar 문구 (PI-DSC-FRONT-001-003)
- Bar 문구는 상품별 1개만 저장하며, 공백만 입력하거나 운영 기준 길이를 초과하면 저장하지 않는다.
- 고객 오인을 만들 수 있는 확정 금액 표현은 예상·참고 라벨과 함께만 허용한다.
• 유의사항 문구 (PI-DSC-FRONT-001-004)
- 지원금 페이지 유의사항 문구는 운영자가 직접 등록·수정할 수 있으나, 배포 전 승인 이력과 적용 대상 상품군을 저장해야 한다.
- 문구 변경은 고객 노출 시점과 이전 문구를 함께 보관한다.
• 할인 혜택 변동 사전 안내 (PI-DSC-FRONT-001-005)
- 구독 상품 변경 전후 할인 ID가 달라져 할인 중단이 발생하면 운영 기준에 따라 FO에 사전 안내 정보를 반환한다.
- 할인 중단 안내에는 변경 대상 상품, 중단 할인명, 예상 영향, 대체 선택 가능 여부를 포함한다.

27) 참조 데이터 삭제 보호 정책 (PG-DSC-DELETE-001)

• 요금제 삭제 보호 (PI-DSC-DELETE-001-001)
- 요금제를 삭제할 때 해당 요금제를 참조하는 지원금, 프로모션, 시뮬레이션 결과, 주문 전환 문맥이 존재하면 삭제를 차단한다.
- 참조 목록을 제공하고 대체 요금제 지정 또는 사용중지 후 삭제하도록 한다.
• 사용여부 Y 차단 (PI-DSC-DELETE-001-002)
- 프로모션 또는 요금약정할인의 사용여부가 Y이면 삭제를 허용하지 않는다.
- 사용여부를 N으로 변경하고 배포 영향 검수와 이력 저장이 완료된 뒤 삭제 요청을 다시 받을 수 있다.
• 추가할인 삭제 확인 (PI-DSC-DELETE-001-003)
- 추가할인은 고객 노출, 계산식 참조, 배포완료 상태가 모두 해제된 경우에만 삭제한다.
- 삭제 전 운영자는 삭제 대상, 영향 상품, 복구 불가 여부를 확인해야 한다.
• 중복 조합 차단 (PI-DSC-DELETE-001-004)
- 요금제 코드와 요금약정할인 코드 조합이 이미 존재하면 신규 등록을 차단한다.
- 동일 조합을 수정해야 하는 경우 기존 항목의 신규 버전을 생성하고 이전 버전은 이력으로 보관한다.

28) 운영 입력 유효성 정책 (PG-DSC-VALID-001)

• 검색 태그 제한 (PI-DSC-VALID-001-001)
- 요금제 상세 검색 태그는 운영 기준의 최대 입력 개수와 개별 태그 길이를 초과할 수 없다.
- 초과 입력은 저장 전 차단하고 초과한 태그 위치를 안내한다.
• Bar 문구 제한 (PI-DSC-VALID-001-002)
- Bar 문구는 빈 값, 공백, 금칙어, 길이 초과를 저장하지 않는다.
- 문구가 고객 혜택을 확정적으로 오인시킬 수 있으면 검수 반려 상태로 전환한다.
• 클럽명 필수 (PI-DSC-VALID-001-003)
- 클럽 등록 저장 시 클럽명이 선택되지 않으면 저장을 허용하지 않는다.
- 클럽명은 추가지원금 유형과 연결되며 미선택 상태는 임시저장까지만 허용한다.
• 중복 조합 (PI-DSC-VALID-001-004)
- 동일 요금제 코드, 요금약정할인 코드, 가입유형, 지원금 유형 조합은 중복 등록하지 않는다.
- 중복 감지 시 기존 항목 수정 또는 신규 버전 생성 중 하나를 선택하게 한다.

최종 점검 기준

할인/시뮬레이션 정책서 제출 전 점검
1. 범위 정합성
할인/시뮬레이션이 결제 확정, 청구 확정, 주문 이행을 침범하지 않고 예상 비용·혜택 비교 범위로 정의되었는지 확인한다.
2. 고객 완결성
고객이 조건 설정, 비교, 선택, 가능 여부 확인, 저장·전환까지 셀프로 완료할 수 있는지 확인한다.
3. 비용 이해 가능성
월 납부액, 일시 비용, 정상가, 할인 후 금액, 포함·제외 항목이 같은 단위로 표시되는지 확인한다.
4. 할인 조합 정합성
최대 혜택 추천, 중복·배타 조건, 마케팅프로그램 운영상태, 특화 할인 조건이 정책으로 연결되는지 확인한다.
5. 가입 가능 판정
상품·혜택·약정·지원금·결합·재가입 제한이 BSS/연계 판정과 연결되는지 확인한다.
6. 상태 전이 정합성
조건 입력 전부터 결과 저장·만료·상담 전환까지 상태와 전이 이벤트가 누락 없이 연결되는지 확인한다.
7. 프로세스-기능-정책 연결성
모든 프로세스가 관련 기능과 관련 정책을 가지고 있고 정책 상세가 실제 판단 기준으로 작성되었는지 확인한다.
8. 오차·실패 대응
계산 실패, 응답 지연, 부적격, 실제 청구 차이 가능성, 상담 전환 기준이 정의되었는지 확인한다.
9. 운영 관리 가능성
계산식, 안내 문구, 마케팅프로그램, 관계 데이터 변경의 검수·배포·롤백·모니터링 기준이 있는지 확인한다.
10. 요구사항 반영
상세 요구사항의 비용 비교, 최대 혜택, 가입 가능 여부, 승계, 특화 할인, 저장·주문 전환 기준이 프로세스·기능·정책 중 하나 이상에 매핑되는지 확인한다.
11. 지원금 산정 정합성
공통지원금 A/B 트랙, 별도지원금 가입유형, 추가지원금 합산·변동 우선순위가 정책 상세에 연결되어 있는지 확인한다.
12. 운영 삭제 보호
요금제, 프로모션, 요금약정할인, 추가할인, 추가지원금 삭제 시 참조 데이터와 사용여부 Y 차단 기준이 있는지 확인한다.
13. 프론트 연동 검증
BO 설정값이 FO 상품선택 영역에 반영되기 전 상품 유형, 노출 순서, 추천 여부, Bar 문구, 유의사항 문구 검증 기준이 있는지 확인한다.
14. 요구사항 추적성
상세 요구사항 70건은 고객 할인 탐색, 비용 시뮬레이션, 가입 가능 판정, 지원금 운영, 프론트 연동, 운영 삭제 보호 기준으로 재구성해 유즈케이스·프로세스·기능·정책 중 하나 이상에 매핑했는지 확인한다.
