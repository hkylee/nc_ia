1. 개요

가. 범위

• 본 정책서는 통합채널의 공통 UX, 문구, 상태, CTA, 접근성, 오류·점검, 품질 Gate, 적응형 처리 기준을 정의한다.

• 대상 업무는 고객이 업무 시작 전 가치와 조건을 이해하고, 이전 맥락을 유지하며, 실패·제한·장애 상태에서 다음 행동을 확인하는 공통 흐름이다.

• 대상 채널은 통합채널 앱·웹 FO이며, 상담 전환, BSS 판정, 외부 앱·제휴·딥링크 복귀 결과가 고객 업무 완결에 영향을 주는 경우까지 포함한다.

• 고객 표시 상태와 시스템 내부 상태는 후속 행동 기준이 달라질 수 있으므로 상태명, 안내 문구, 이력 기준을 구분해 관리한다.

• 포함 범위는 공통 기준 등록, 검수, 승인, 배포, 모니터링, 회수, 재검수, 접근성 점검, 앱 버전·OTA 기준, 대기열·점검 모드, 통합 Admin 운영 이력 기준이다.

• 제외 범위는 개별 화면 UI 상세, API 필드, DB 컬럼, 개별 정책서의 업무별 세부 정책값, 상세 테스트 케이스, 인프라 구현 방식이다.

• 상세 요구사항 199건은 요구사항 Trace 기준으로 유즈케이스, 상태, 프로세스, 기능, 정책 또는 정책 항목 중 하나 이상에 매핑한다.

• 후속 산출물에서는 각 업무 정책서가 이 공통 기준을 바탕으로 업무별 허용 조건, 제한 조건, 예외, BSS 반영 기준을 상세화한다.

나. 설계 원칙

• Value First: 고객에게 로그인, 권한, 입력, 비용을 요구하기 전에 얻는 가치와 제한을 먼저 설명한다.

• Clear Action: 고객이 지금 할 수 있는 행동, 나중에 할 수 있는 행동, 상담으로 전환해야 하는 행동을 명확히 구분한다.

• Journey Continuity: 검색, 비교, 신청, 상담 전환, 외부 전환 후 복귀 사이에서 고객 맥락이 끊기지 않도록 상태와 이력을 연결한다.

• Accessible Quality Gate: 문구, CTA, 폼, 오류, 콘텐츠, 관리자 기능은 접근성·가독성·QA 기준을 충족한 경우에만 배포한다.

• Resilient Operation: 장애, 점검, 대기열, 과부하, 앱 버전 제한 상황에서도 고객이 가능한 업무와 대체 경로를 확인할 수 있게 한다.

• BSS Integrated Decision: 고객 상태와 업무 가능 여부에 영향을 주는 BSS·외부 연계 판정은 기능과 정책 기준에 포함한다.

2) 명확한 행동 기준 정책 (PG-COM-CTA-001)

• Primary CTA 단일화 (PI-COM-CTA-001)
- 고객이 즉시 완료해야 하는 핵심 행동은 Primary CTA 1개로 제한한다.
- 보조 행동은 Secondary CTA 또는 텍스트 링크로 분리하고 같은 강조 색상을 중복 사용하지 않는다.
• 위험 행동 재확인 (PI-COM-CTA-002)
- 해지, 취소, 회수, 개인정보 삭제, 요금 변경처럼 되돌리기 어렵거나 비용 영향이 있는 행동은 최종 확인을 1회 요구한다.
- 동일 업무에서 확인 모달을 2회 이상 중첩하지 않는다.
• 상담 전환 위치 (PI-COM-CTA-003)
- 고객 조치로 해결 가능한 오류는 셀프 복구 CTA를 우선 노출한다.
- 동일 오류가 2회 이상 반복되거나 운영 확인이 필요한 경우 상담 전환 CTA를 함께 제공한다.

3) 맥락 유지·복구 정책 (PG-COM-CTX-001)

• 맥락 저장 대상 (PI-COM-CTX-001)
- 검색어, 필터, 비교 대상, 입력값, 업무 단계, 선택 상품은 고객이 이어하기에 필요한 경우에만 저장한다.
- 민감정보, 인증번호, 결제 인증값은 맥락 저장 대상에서 제외한다.
• 맥락 유효 기간 (PI-COM-CTX-002)
- 일반 탐색 맥락은 7일, 주문·결제·변경 업무 맥락은 24시간, 인증 연계 맥락은 10분을 초과해 유지하지 않는다.
- 만료 시 재조회 또는 재입력을 요구한다.

4) 공통 문구·안내 정책 (PG-COM-MSG-001)

• 토스트 사용 (PI-COM-MSG-001)
- 토스트는 저장 완료, 복사 완료, 일시적 성공처럼 고객이 즉시 읽고 사라져도 되는 결과에만 사용한다.
- 실패, 비용 영향, 권한 제한, 법적 고지는 토스트 단독 안내를 제한한다.
• 팝업·모달 사용 (PI-COM-MSG-002)
- 고객 선택이 필요하거나 업무 진행을 멈춰야 하는 경우에만 팝업 또는 모달을 사용한다.
- 같은 업무에서 모달은 최대 1개만 열고 중첩 모달은 금지한다.
• 오류 문구 구조 (PI-COM-MSG-003)
- 오류 문구는 원인, 고객이 할 수 있는 행동, 재시도 또는 상담 전환 기준을 포함한다.
- 내부 오류 코드와 시스템명은 고객 문구에 직접 노출하지 않는다.
• Writing System 표준 (PI-COM-MSG-004)
- 운영 기능과 고객 화면에서 사용하는 용어, 상태값, 버튼명, 오류·안내 문구는 공통 문구와 용어 가이드를 기준으로 통일한다.
- 채널과 화면을 이동해도 같은 의미는 같은 표현으로 제공한다.
• 강요 없는 톤앤매너 (PI-COM-MSG-005)
- 고객 안내는 친근하고 기다려주는 톤을 사용한다.
- 과도한 판매 압박, 반복 팝업, 불안감 조성, 과장 표현은 금지한다.
- 동일 안내는 동일 세션에서 1회만 반복 노출하고 중요 알림은 고객이 나중에 다시 확인할 수 있게 한다.

5) 적응형 처리 정책 (PG-COM-ADP-001)

• 적응형 적용 대상 (PI-COM-ADP-001)
- 적응형 처리는 고객 이익, 업무 완료 가능성, 오류 감소 중 하나 이상이 확인되는 경우에만 적용한다.
- 광고성 노출이나 판매 압박 목적의 적응형은 공통 기준에서 제외한다.
• 기본값 자동 적용 (PI-COM-ADP-002)
- 고객이 최근 30일 안에 선택한 조건은 기본값 후보로 사용할 수 있다.
- 단, 비용, 약정, 개인정보 공개에 영향을 주는 값은 고객 확인 전 자동 확정하지 않는다.
• 3초 답변 기준 (PI-COM-ADP-003)
- 고객이 3초 안에 답하기 어려운 질문과 선택은 회사가 보유한 고객정보, 최근값, 이전 맥락, 기본값을 우선 재사용한다.
- 꼭 필요한 경우에만 추가 질문을 노출하고 질문 수와 입력 항목을 최소화한다.

6) 품질 Gate 정책 (PG-COM-GAT-001)

• 필수 Gate (PI-COM-GAT-001)
- 공통 기준은 고객 이해, 접근성, 정책 연결, BSS 조건, 오류 안내, 이력 저장 6개 Gate를 모두 통과해야 승인 완료 상태가 된다.
- 1개라도 미충족이면 검수 반려로 전환한다.
• 반려 우선순위 (PI-COM-GAT-002)
- 고객 피해, 비용 오표기, 권한 오류, 개인정보 노출 가능성은 P1로 분류한다.
- 문구 불명확, 접근성 누락, 중복 안내는 P2, 스타일 보정은 P3로 분류한다.
• 회수 기준 (PI-COM-GAT-003)
- 배포 후 고객 피해 가능성, 장애 확대, 정책 충돌, 법적 고지 누락이 확인되면 즉시 회수 필요 상태로 전환한다.
- 회수 처리 이력은 1년 이상 보관한다.
• 원칙 체크리스트 설계리뷰 (PI-COM-GAT-004)
- 설계 리뷰는 가치 우선, 명확한 행동, 노력·시간 최소화, 상황·맥락 유지, 개인정보 최소 수집, 접근성 기준을 체크리스트로 확인한다.
- 기준을 충족하지 못한 항목은 배포 전 보완 또는 예외 승인 대상으로 관리한다.

7) 운영 변경·이력 정책 (PG-COM-OPR-001)

• 변경 승인 (PI-COM-OPR-001)
- 공통 기준 변경은 등록자와 승인자를 분리한다.
- 단순 오탈자 수정은 운영자 승인 1명으로 처리할 수 있고, 고객 영향이 있는 변경은 품질 검수자 승인을 필수로 둔다.
• 버전 관리 (PI-COM-OPR-002)
- 공통 기준은 배포 단위별 버전을 부여한다.
- 버전에는 변경 전 값, 변경 후 값, 적용 채널, 영향 업무, 승인자, 배포 시점을 저장한다.

8) 권한·보안·개인정보 노출 정책 (PG-COM-SEC-001)

• 권한 요청 시점 (PI-COM-SEC-001)
- 권한 허용, 본인확인, 회원가입은 고객에게 제공할 가치와 필요한 이유를 먼저 안내한 뒤 요청한다.
- 고객이 거부해도 가능한 대체 경로가 있으면 대체 경로를 제공한다.
• 개인정보 최소 노출 (PI-COM-SEC-002)
- 고객 식별 정보는 마스킹을 기본으로 노출한다.
- 전화번호, 이메일, 주소, 생년월일은 업무 확인에 필요한 최소 자리만 표시한다.
• Privacy by Design 최소 수집 (PI-COM-SEC-003)
- 개인정보는 업무 완료에 필요한 최소 항목만 수집하고 수집 목적, 항목, 보관 기간, 제3자 제공 여부를 고객에게 명확히 안내한다.
- 중요한 개인정보 처리 기능은 재인증, 권한 관리, 접근 이력 기준을 함께 적용한다.
• 회선 오신청 방지 (PI-COM-SEC-004)
- 정지, 해지, 변경, 결제처럼 특정 회선에 영향을 주는 업무는 처리 전 회선명, 번호 일부, 상품명, 영향 내용을 재확인한다.
- 고객이 의도하지 않은 회선을 선택한 가능성이 있으면 최종 요청을 제한하고 다시 선택하게 한다.

9) 접근성·사용성 정책 (PG-COM-ACC-001)

• 이해 가능성 (PI-COM-ACC-001)
- 고객 안내는 고객이 사용하는 말로 작성한다.
- 내부 시스템명, 정책 약어, 개발 용어는 고객 문구에서 제외한다.
- 내부 용어가 불가피하게 포함되면 고객 행동 기준의 대체 용어 또는 설명 문구를 필수 제공한다.
• 조작 가능성 (PI-COM-ACC-002)
- 핵심 CTA는 모바일 기준 터치 영역 44픽셀 이상을 확보한다.
- 비활성 버튼은 비활성 사유와 활성화 조건을 함께 안내한다.
• 단순성 기준 (PI-COM-ACC-003)
- 신규 기능과 화면은 고객이 별도 학습이나 설명서 없이 이해·사용 가능해야 한다.
- 홈, 탭, 메뉴, 프로세스는 서비스 나열이 아니라 고객 목적과 다음 행동 기준으로 구성한다.
• 접근성 목표 (PI-COM-ACC-004)
- 핵심 과업 화면은 접근성 점검 점수 90점 이상을 목표로 관리한다.
- 홈, 검색, 비교, 가입, 결제, 혜택 사용, 상담 연결 화면은 배포 전 스크린리더, 키보드 조작, 명도, 터치 영역 기준을 확인한다.
• 폼·오류 접근성 (PI-COM-ACC-005)
- 입력 폼은 레이블, 필수 표시, 도움말, 오류 메시지, 오류 위치 이동을 함께 제공한다.
- 오류 발생 시 고객이 수정할 필드와 수정 방법을 바로 알 수 있어야 한다.
• 멀티미디어 대체 (PI-COM-ACC-006)
- 이미지, 배너, 동영상, 바코드, 혜택 카드는 대체텍스트 또는 설명 정보를 제공한다.
- 필수 대체 정보가 없는 콘텐츠는 배포 전 경고 또는 배포 제한 대상으로 분류한다.

10) BSS·연계 판정 정책 (PG-COM-INT-001)

• BSS 판정 범위 (PI-COM-INT-001)
- BSS는 고객 상태, 가입 상태, 요금·혜택 상태, 제한 여부처럼 정책 판단에 필요한 항목만 회신한다.
- 화면 표시용 상세 데이터는 별도 후속 산출물에서 정의한다.
• BSS 실패 처리 (PI-COM-INT-002)
- BSS 조회가 3초 안에 완료되지 않으면 지연 안내를 제공한다.
- 동일 조회가 2회 실패하면 상담 전환 또는 나중에 다시 시도 기준을 안내한다.

11) 실패·복구 처리 정책 (PG-COM-ERR-001)

• 재시도 기준 (PI-COM-ERR-001)
- 고객 입력 오류는 동일 세션에서 최대 3회까지 재시도를 허용한다.
- 인증 실패나 외부 연계 실패는 보안 기준과 기관 응답 기준을 우선 적용한다.
• 중복 요청 방지 (PI-COM-ERR-002)
- 같은 업무 요청이 5초 이내 반복되면 중복 요청으로 분류한다.
- 이미 접수된 요청은 추가 접수를 제한하고 기존 처리 상태를 안내한다.

12) 공통 템플릿·컴포넌트 정책 (PG-COM-TPL-001)

• 컴포넌트 표준 규격 (PI-COM-TPL-001)
- 전시 카드, 배너, 리스트, 필터, 정렬, 입력 폼, 오류 안내 컴포넌트는 필수 정보, 선택 정보, 접근성 속성, CTA 위치 기준을 표준 규격으로 관리한다.
- 주제별 정책서는 표준 규격을 벗어나는 예외 사유를 정책 항목으로 남긴다.
• 공통 카피 가이드 (PI-COM-TPL-002)
- 서비스 전반의 화면 문구는 고객이 이해하는 업무 용어를 우선 사용한다.
- 내부 조직명, 시스템 약어, 개발 용어는 고객 문구에서 제외하고 고객 행동 기준으로 재작성한다.
- 신규 공통 용어는 표준 용어 사전 등록 후 적용한다.
• 기능명 중심 메뉴 (PI-COM-TPL-003)
- 전체 메뉴와 공통 진입 구조는 서비스 브랜드명보다 고객이 수행하려는 기능명과 과업명 중심으로 분류한다.
- 동일 성격 업무는 하나의 구조에서 탐색되도록 하고 중복 메뉴는 운영 승인 없이 추가하지 않는다.
• 플로팅 CTA (PI-COM-TPL-004)
- 저장, 수정, 삭제, 목록, 다음 단계처럼 업무 완료에 직접 필요한 주요 버튼은 긴 본문에서도 고객이 찾을 수 있도록 고정 또는 반복 노출 기준을 둔다.
- 단, 위험 행동 CTA는 고정 노출 시에도 재확인 기준을 적용한다.
• 혜택 중심 카피 (PI-COM-TPL-005)
- 배너, 카드, 추천 영역의 문구는 모호한 홍보 문장보다 고객이 얻는 혜택, 대상 상품, 적용 조건, 종료 시점을 우선 표시한다.
- 혜택이 조건부이면 조건과 제외 대상을 같은 영역에서 안내한다.
• 단말 속성 표준 (PI-COM-TPL-006)
- 단말 속성 정보는 목록, 상세, 비교 화면에서 같은 항목명과 같은 단위로 표시한다.
- 색상, 용량, 요금, 재고, 배송 가능 여부처럼 구매 판단에 필요한 속성은 화면마다 누락되거나 다른 용어로 바뀌지 않아야 한다.

13) 성능·폴백 운영 정책 (PG-COM-PERF-001)

• 전시·상세 성능 기준 (PI-COM-PERF-001)
- 홈, 카테고리, 상세, 목록의 핵심 정보는 표준 성능 기준 안에서 우선 로드한다.
- 이미지 최적화, 캐시, 프리패치가 필요한 영역은 고객 행동 완료에 필요한 정보와 보조 정보를 구분해 적용한다.
• 폴백 페이지·안전모드 (PI-COM-PERF-002)
- 데이터 소스 오류, 렌더링 오류, 외부 연계 장애가 발생하면 기본 템플릿 또는 안전한 기본 콘텐츠로 폴백한다.
- 폴백 상태에서는 고객 피해 가능성이 있는 주문, 결제, 개인정보 변경 CTA를 제한한다.
• 캐시·반영 통제 (PI-COM-PERF-003)
- 공통 기준과 전시 콘텐츠는 즉시 반영, 예약 반영, TTL 반영을 구분한다.
- 고객 피해 가능성이 있는 오표기, 가격·혜택 오류, 링크 오류는 강제 무효화와 긴급 회수 대상으로 분류한다.
• 가용성 기준 (PI-COM-PERF-004)
- 핵심 서비스는 24시간 365일 이용 가능성을 기준으로 관리한다.
- 계획 점검은 사전 공지하고, 점검 중에도 영향 범위, 예상 복구 시간, 가능한 대체 경로를 제공한다.
• 버스트 트래픽 보호 (PI-COM-PERF-005)
- 이벤트, 주문, 인증, 결제처럼 트래픽 급증 가능성이 있는 업무는 대기열, 캐시, 기능 제한, 단계적 오픈 기준을 사전에 정의한다.
- 과부하 시 로그인, 내 정보, 고객센터, 주문 상태 조회를 우선 보호한다.

14) 실험·배포·롤백 정책 (PG-COM-EXP-001)

• 실험 포함·제외 규칙 (PI-COM-EXP-001)
- 실험 대상은 고객군, OS, 앱 버전, 회원 상태, 내부 계정 여부를 기준으로 포함·제외할 수 있다.
- 직원, 테스터, 장애 영향 고객군은 기본 제외 후보로 관리하고 예외 포함 시 승인 이력을 남긴다.
• 단계적 배포 (PI-COM-EXP-002)
- 신규 기능, 정책, 모델, 노출 로직은 사전 정의한 비율과 기간에 따라 단계적으로 확대한다.
- 각 단계는 오류율, 상담 전환율, 전환율, 고객 불만 지표가 가드레일 안에 있을 때만 다음 단계로 진행한다.
• 롤백 가드레일 (PI-COM-EXP-003)
- 고객 피해 가능성, 개인정보 노출, 결제·혜택 오표기, 장애율 급증, 주요 KPI 급락이 확인되면 즉시 이전 승인 버전으로 롤백한다.
- 롤백 후 원인, 영향 범위, 재배포 조건을 이력으로 저장한다.

15) 장애·점검·대기열 정책 (PG-COM-OUT-001)

• 점검 모드 안내 (PI-COM-OUT-001)
- 점검 또는 장애가 발생하면 진입 전 또는 처리 중에 영향 범위, 제한 업무, 예상 복구 시간, 문의 경로를 안내한다.
- 복구 시간이 확정되지 않은 경우에는 마지막 갱신 시각과 다음 갱신 예정 시각을 표시한다.
• 대기열 공정성 (PI-COM-OUT-002)
- 대기열 적용 시 고객에게 순번, 예상 대기, 진행 상태, 새로고침 시 영향, 이탈 후 재진입 기준을 안내한다.
- 동일 고객의 다중 접속은 하나의 대기 자격으로 묶어 공정성을 유지한다.
• 부분 장애 안내 (PI-COM-OUT-003)
- 일부 기능만 장애이거나 복구 중이면 정상 이용 가능한 업무와 제한 업무를 분리해 안내한다.
- 고객 피해가 예상되는 주문, 결제, 개인정보 변경 CTA는 복구 전까지 제한한다.
• 오프라인 안내 (PI-COM-OUT-004)
- 네트워크 불안정 또는 오프라인 상태에서는 현재 화면이 정상 동작하지 않는 이유와 재시도, 네트워크 설정 확인, 가능한 오프라인 확인 범위를 안내한다.

16) 앱·웹 실행 환경 정책 (PG-COM-APP-001)

• 최소 지원 버전 (PI-COM-APP-001)
- Android, iOS, 웹뷰의 최소 지원 버전은 운영 기준으로 관리한다.
- 기준 미만 환경은 업데이트, 웹 대체 경로, 상담 전환 중 하나를 안내하고 위험 업무 진입을 제한한다.
• 필수·선택 업데이트 (PI-COM-APP-002)
- 보안, 결제, 개인정보, 약관 변경에 영향이 있으면 필수 업데이트로 분류한다.
- 편의 개선은 선택 업데이트로 안내하고 고객이 계속 이용할 수 있는 범위를 함께 표시한다.
• OTA 롤백·킬스위치 (PI-COM-APP-003)
- OTA 자산 배포 후 오류율 증가, 화면 깨짐, 결제·주문 영향, 개인정보 노출 가능성이 확인되면 이전 자산 버전으로 롤백하거나 해당 기능을 킬스위치로 비활성화한다.
• 권한 재요청 (PI-COM-APP-004)
- 사진, 위치, 알림, 연락처 등 OS 권한은 기능 실행 시점에 목적과 혜택을 안내한 뒤 요청한다.
- 고객이 거부한 경우 재요청 조건과 설정 이동 경로를 제공한다.

17) 딥링크·외부 전환 정책 (PG-COM-LNK-001)

• 외부 전환 고지 (PI-COM-LNK-001)
- 외부 앱, 제휴 페이지, 스토어, 인앱브라우저로 이동하기 전 처리 주체, 전달 정보, 복귀 가능 여부, 세션 만료 영향을 안내한다.
• 복귀 맥락 복원 (PI-COM-LNK-002)
- 외부 전환 후 복귀하면 원래 목적 화면, 선택 상태, 입력값, 대상 ID를 가능한 범위에서 복원한다.
- 보안상 복원 불가 항목은 사유와 재입력 기준을 안내한다.
• 콜백 결과 반영 (PI-COM-LNK-003)
- 제휴 또는 외부 서비스의 성공, 실패, 취소 결과는 표준 상태로 변환해 고객에게 안내한다.
- 결과 수신 실패 시 마지막 확인 시각과 재확인 경로를 제공한다.

18) AI·콘텐츠 신뢰 정책 (PG-COM-AI-001)

• AI 근거 표시 (PI-COM-AI-001)
- AI가 요금, 정책, 혜택, 주문, 약관을 설명할 때는 근거 문서명, 데이터 기준 시점, 최신성, 불확실 여부를 함께 표시한다.
- 근거가 없거나 최신성이 확인되지 않으면 확정 답변을 금지하고 검색 결과, FAQ, 상담 전환으로 안내한다.
• 위험 업무 생성 제한 (PI-COM-AI-002)
- 요금, 약관, 위약금, 개인정보, 결제, 청구처럼 고객 피해 가능성이 있는 업무는 생성형 답변만으로 확정하지 않는다.
- 구조화 데이터, 룰, 승인된 문서 중 하나 이상으로 검증한 결과만 안내한다.
• 번역 친화 콘텐츠 (PI-COM-AI-003)
- 다국어 자동 번역 대상 문구는 짧은 문장, 명확한 주어, 표준 용어를 사용한다.
- 이미지 안의 핵심 정보는 텍스트로도 제공해 번역과 접근성 기준을 충족한다.
• GEO 메타데이터 (PI-COM-AI-004)
- 상품, 혜택, 요금, 정책, FAQ 콘텐츠는 AI 검색과 에이전트가 정확히 해석할 수 있도록 제목, 요약, 적용 조건, 최신 업데이트 시각, 출처 메타데이터를 관리한다.
• 외국인·다국어 지원 (PI-COM-AI-005)
- 외국인 고객이 주요 가입, 조회, 결제, 상담 업무를 이해할 수 있도록 다국어 자동 번역과 쉬운 표현 기준을 적용한다.
- 번역 대상 문구는 내부 약어와 중의적 표현을 줄이고 고객 행동 기준으로 작성한다.

20) 통합 IA·경험 동등성 정책 (PG-COM-IA-001)

• Parity 기준 (PI-COM-IA-001)
- 기존 서비스의 핵심 가치는 화면과 메뉴를 그대로 유지하는 기준이 아니라 고객 목적과 완료 결과가 유지되는 기준으로 판단한다.
- 기능 축소가 있으면 대체 경로와 변경 고지를 제공한다.
• 목적 기반 IA (PI-COM-IA-002)
- 메뉴와 홈 구조는 브랜드나 조직 기준보다 가입, 조회, 변경, 혜택 사용, 상담, 문제 해결 같은 고객 목적 기준으로 재그룹핑한다.
- 동일 목적의 중복 메뉴는 통합한다.
• 전역 탐색 (PI-COM-IA-003)
- 어떤 화면에서도 홈, 검색, 마이, 고객센터, 최근 과업으로 1~2회 액션 안에 이동할 수 있어야 한다.
- 고정 내비, 빠른 메뉴, 퀵액션은 채널 표준에 맞춰 제공한다.
• 현재 위치 인지 (PI-COM-IA-004)
- 고객이 현재 화면의 업무 맥락을 이해할 수 있도록 섹션 라벨, 상단 타이틀, 탭·필터 선택 상태, 경로 표시 기준을 정의한다.
• 셀프 처리 확대 (PI-COM-IA-005)
- 신청, 납부, 해지, 정지, 멤버십 카드 관리처럼 고객이 직접 완료할 수 있는 업무는 앱·웹 셀프 처리 경로를 우선 제공한다.
- 고객 조치로 복구되지 않는 실패가 2회 이상 반복되거나 복구 불가 상태이면 상담 전환을 제공한다.
• 퀵액션 제공 (PI-COM-IA-006)
- 멤버십 바코드, 구독 관리, 주문조회, 데이터 선물·리필, 쿠폰함처럼 반복 사용되는 업무는 홈, 마이, 검색 추천, 위젯, 단축 메뉴 중 하나 이상의 빠른 진입 경로를 제공한다.

최종 점검 기준

가이드라인/공통/품질/적응형 정책서 제출 전 점검
1. 범위 정합성
공통 기준이 34개 정책서 전체에 재사용되는 품질, 문구, 상태, CTA, 오류, 접근성, 적응형 기준으로 정의되었는지 확인한다.
특정 주제의 세부 정책값이나 화면 UI 상세가 공통 정책서에 과도하게 들어가지 않았는지 확인한다.
2. 공통 고객 경험 일관성
고객이 어떤 업무를 시작하더라도 가치, 필수 조건, 제한 사유, 다음 행동, 완료 결과를 같은 방식으로 이해할 수 있는지 확인한다.
문구와 CTA 기준이 고객 관점의 쉬운 표현과 접근성 기준을 따르는지 확인한다.
3. 장애·복구·대기열 대응
점검, 장애, 과부하, 대기열, 업데이트 필요 상태에서 고객이 가능한 업무와 대체 경로를 확인할 수 있는지 확인한다.
재시도, 상담 전환, 대체 안내, 안전모드 기준이 정책 항목으로 분리되어 있는지 확인한다.
4. BSS·외부 연계 포함성
업무 가능 여부에 영향을 주는 BSS 상태와 외부 연계 결과가 기능과 정책에 연결되어 있는지 확인한다.
연계 지연, 실패, 불일치 발생 시 고객 안내와 운영 확인 기준이 정의되어 있는지 확인한다.
5. 운영 품질 Gate
공통 기준 등록, 검수, 승인, 배포, 회수, 재검수 상태와 책임이 구분되어 있는지 확인한다.
품질 Gate 미충족 시 승인 완료나 배포 완료로 전환되지 않도록 정책이 정의되어 있는지 확인한다.
6. 프로세스-기능-정책 연결성
공통 유즈케이스에서 시작한 프로세스가 기능과 정책 ID로 끊기지 않고 연결되는지 확인한다.
기능 세부 구성은 짧은 하위 처리명으로 작성되어야 하며 정책값은 정책 상세에 분리되어야 한다.
7. 정책 구체성
정책 항목이 실제 값, 조건, 횟수, 시간, 상태, 예외, 고지, 이력 기준 중 하나 이상을 포함하는지 확인한다.
실행 기준 없는 미정 표현이나 책임 주체 없는 문장이 남지 않았는지 확인한다.
8. 요구사항·확산 가능성
상세 요구사항명과 상세 요구사항 설명에서 확인된 공통 기준이 프로세스, 기능, 정책 중 하나 이상에 매핑되었는지 확인한다.
PM-01에서 정의한 기준이 후속 33개 정책서 작성 지침으로 재사용 가능한지 확인한다.
