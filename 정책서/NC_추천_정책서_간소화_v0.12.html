1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객 상태와 맥락을 기반으로 상품, 혜택, 콘텐츠, 다음 행동을 추천하고 고객이 납득해 선택하거나 거절할 수 있도록 하는 추천 업무 기준을 정의한다.

• 포함 범위는 추천 정보 통합 제공, 추천 슬롯 표준화, 홈·목록·상세·MY·주문 완료 등 추천 위치, 개인화 추천, 비개인화 폴백, 상세 연관 추천, 추천 더보기 랜딩이다.

• 고객 제어 범위는 추천 근거 확인, 추천 결과 비교, 추천 실행 연결, 관심 없음, 숨김, 일정 기간 보지 않기, 개인화 설정 진입, 피드백 이력 확인을 포함한다.

• 운영 범위는 추천 슬롯 구성, 추천 유형·자산 기준, 비즈니스 룰, 제약 룰, 우선순위, 사전 검증, 시뮬레이션, 배포·회수·롤백, 성과 분석과 개선 승인 기준을 포함한다.

• BSS는 고객 상태, 보유 상품, 혜택 자격, 가입 가능 여부, 제한 조건, 실행 가능 여부의 기준 정보를 제공하고, 추천 연계 시스템은 해당 기준을 추천 노출과 CTA 연결 판정에 반영한다.

• 책임 경계는 통합채널 FO가 추천 노출·제어·실행 연결을 담당하고, 추천 연계 시스템이 후보 산정·룰 검증·폴백 생성을 담당하며, BSS는 고객 상태·보유 상품·실행 가능 여부의 기준 정보를 제공하는 구조로 구분한다.

• 추천 상태는 고객 표시 상태와 시스템 내부 상태를 구분해 관리하며, 고객에게는 추천 가능·보류·제한·숨김·실행 완료처럼 다음 행동 기준을 보여주고 내부적으로는 후보 산정, 룰 검증, BSS 기준 정보 반영, 이력 저장 상태를 별도로 관리한다.

• 제외 범위는 추천 모델 내부 알고리즘, 프롬프트 원문, 피처 가중치, API 필드, DB 컬럼, 화면 UI 상세, 개별 상품 가격·혜택 상세 정책값, 전시 관리·데이터 트래킹 전체 절차다.

나. 설계 원칙

• Decision Support: 추천은 판매 노출이 아니라 고객이 다음 행동을 결정할 수 있도록 이유, 조건, 영향, 대체 경로를 함께 제공한다.

• Explainable Personalization: 개인화 추천에는 고객이 이해 가능한 추천 근거와 활용 데이터 범주를 표시하고 민감한 내부 정보는 노출하지 않는다.

• Customer Control: 고객은 추천을 숨기거나 관심 없음으로 처리할 수 있고, 개인화 설정과 제어 이력을 확인할 수 있어야 한다.

• Safe Business Guardrail: 사업 우선순위와 수익성 룰은 고객 불이익 방지, 컴플라이언스, BSS 실행 가능성보다 앞설 수 없다.

• Recoverable Recommendation: 추천 실패, 추천 보류, 가입 불가, 근거 부족은 검색, 비교, 필터, 상담으로 복구 가능한 상태로 처리한다.

• Closed Quality Loop: 추천 노출부터 실행 완료와 부정 반응까지 측정하고 개선안 승인, 배포, 회수, 롤백 기준으로 운영 품질을 관리한다.

ST-REC-007실행 완료고객에게는 실행 완료 또는 연결 완료로 표시하고, 내부적으로는 실행 도메인 인계, 결과 회신 수신, 추천 이력 제외 기준을 구분한다.
실행 결과와 후속 확인 항목, 관련 추천 제외 또는 연계 추천 기준을 기록한다.
ST-REC-008추천 숨김고객이 특정 추천을 숨김, 관심 없음, 일정 기간 보지 않기로 처리한 상태다.
현재 지면에서 즉시 제외하고 차단시간과 피드백 사유를 이력으로 남긴다.
ST-REC-009추천 제한고객에게는 추천 제한 또는 대체 추천 필요 상태로 표시하고, 내부적으로는 룰 제한, BSS 기준 제한, 고객 보호 제한을 구분한다.
제한 사유와 제한 해소 조건 또는 상담 전환 기준을 안내한다.
ST-REC-010피드백 반영 대기고객 피드백이나 운영 분석 결과가 추천 기준에 반영되기 전 검토 중인 상태다.
피드백 유형, 발생 위치, 영향 범위를 분류하고 개선 후보로 등록한다.
ST-REC-011운영 검수 대기추천 슬롯, 룰, 자산, 예외 설정이 배포 전 검증 또는 승인 대기 중인 상태다.
대표 시나리오 검증, 충돌 확인, 고객 불이익 여부, 롤백 가능성을 점검한다.
ST-REC-012기준 배포 완료추천 슬롯, 룰, 자산, 폴백, 예외 기준이 승인되어 적용된 상태다.
적용 시점, 대상 범위, 변경자, 변경 사유, 성과 지표를 기록한다.
ST-REC-013회수·롤백성과 저하, 룰 위반, 고객 불만, 장애 영향으로 추천 기준을 회수하거나 이전 버전으로 되돌리는 상태다.
회수 사유와 영향 범위를 기록하고 안전한 폴백 기준을 적용한다.

2) 상태 전이 기준

현재 상태전이 이벤트다음 상태처리 기준 및 후속 처리
추천 미진입추천 영역 진입후보 산정 중고객이 홈, 목록, 상세, MY, 주문 완료, 혜택 사용 후 추천 영역에 진입하면 현재 화면 맥락을 유지해 후보 산정을 시작한다.
후보 산정 중고객 상태와 추천 룰 검증 통과추천 확인 가능고객 상태, 보유 상품, 동의, 제한 조건, 비즈니스 룰을 통과한 후보만 추천 확인 가능 상태로 전환한다.
후보 산정 중근거 부족 또는 룰 충돌 발생추천 보류개인화 신뢰도가 낮거나 고객 불이익 가능성, 룰 충돌, 필수 근거 누락이 있으면 추천을 보류한다.
추천 확인 가능추천 후보 비교 선택비교·대체 탐색 중고객이 비교 보기, 대체안, 관련 카테고리, 검색 전환을 선택하면 후보 비교 상태로 전환한다.
추천 확인 가능추천 실행 CTA 선택실행 연결 대기고객이 가입, 변경, 혜택 사용, 상담 연결 등 실행 CTA를 선택하면 실행 전 확인 상태로 전환한다.
실행 연결 대기실행 조건 최종 확인 통과실행 완료인증, 약관, 비용 영향, BSS 실행 가능 여부를 확인한 뒤 후속 업무로 연결되면 실행 완료로 전환한다.
실행 연결 대기실행 조건 미충족추천 제한가입 불가, 변경 제한, 혜택 사용 불가, 미납·정지 등 제한 조건이 있으면 제한 상태로 전환한다.
추천 확인 가능관심 없음 또는 숨김 선택추천 숨김고객이 관심 없음, 숨기기, 일정 기간 보지 않기를 선택하면 현재 지면에서 제외하고 차단시간을 적용한다.
추천 숨김피드백 사유 저장피드백 반영 대기고객 피드백 사유와 발생 위치를 저장하면 추천 개선 검토 대상으로 전환한다.
추천 보류검색 또는 상담 대체 경로 선택비교·대체 탐색 중추천이 보류되더라도 고객이 검색, 필터, 관련 카테고리, 상담 연결을 선택하면 흐름을 이어간다.
추천 미진입추천 슬롯과 룰 등록운영 검수 대기운영자가 슬롯, 노출 위치, 룰, 폴백, 예외 기준을 등록하면 검수 대기 상태로 둔다.
운영 검수 대기시뮬레이션과 충돌 검증 통과기준 배포 완료대표 시나리오, 고객 불이익, 룰 충돌, 폴백 적용을 통과하면 예약 또는 즉시 배포한다.
기준 배포 완료성과 저하 또는 위반 탐지회수·롤백숨김률 급증, 고객 불만, 룰 위반, 장애 영향이 확인되면 회수 또는 이전 버전을 적용한다.
피드백 반영 대기개선안 승인운영 검수 대기운영자가 고객 피드백과 성과 분석을 근거로 개선안을 승인하면 배포 전 검수 대상으로 전환한다.
운영 검수 대기자산 라벨과 CTA 검수 완료기준 배포 완료상품·콘텐츠 자산의 추천 라벨, CTA, 비교 연결, 비노출 조건이 검수되면 추천 자산 기준을 배포한다.
스크린 맥락 기반 추천 후보 감지고객이 혜택, 바코드, 쿠폰, 포인트, 상품 목록 등 특정 화면에 체류하고 후속 행동 후보가 있으면 현재 맥락을 유지해 후보 산정을 시작한다.
로그인·선택동의 Case 분기 완료로그인 상태, 선택동의, 고객 맥락, 오류 여부에 따라 개인화 또는 비개인화 노출 범위가 결정되면 추천 확인 가능 상태로 전환한다.
무료→유료 또는 크로스 도메인 추천 선택고객이 무료 혜택 확장, 구독·선물, 도메인 간 넛지, 요금제·구독 조합 추천을 선택하면 비용·조건·이동 맥락 확인 상태로 전환한다.

나. 정책 상세

1) 추천 범위·슬롯 정책 (PG-REC-SCOPE-001)

• 추천 슬롯 범위 (PI-REC-SCP-001)
- 추천 슬롯은 홈, 상품 목록, 상품 상세, MY, 주문 완료, 혜택 사용 후, 고객지원 전환 지점을 기본 범위로 둔다.
- 슬롯마다 슬롯 ID, 추천 목적, 최대 노출 개수, 더보기 연결 여부를 관리하고 목적이 없는 슬롯은 노출하지 않는다.
• 추천 유형 구분 (PI-REC-SCP-002)
- 추천 유형은 개인화 추천, 운영자 큐레이션, 비개인화 폴백, 컨시어지 Next Action으로 구분한다.
- 고객에게 표시되는 라벨은 이 네 가지 성격 중 하나를 반드시 드러낸다.
• 더보기 랜딩 기준 (PI-REC-SCP-003)
- 추천 더보기 랜딩은 동일 추천 목적의 후보를 3건 이상 제공할 수 있을 때 노출한다.
- 랜딩에는 추천 근거, 비교 진입, 검색 전환, 숨김·관심 없음 제어를 동일한 방식으로 제공한다.
• 추천 전시 유형·AI PICK 기준 (PI-REC-SCP-004)
- 추천 전시 유형은 카드, 리스트, 배너, AI PICK 캡션 중 하나로 구분한다.
- AI PICK 캡션은 고객 데이터 기반 추천 이유를 1문장으로 요약하되, 공식 상품 조건과 혼동되지 않도록 추천 보조 설명으로 표시한다.
• 라이프스타일·제휴처 추천 범위 (PI-REC-SCP-005)
- 라이프스타일, 오프라인 제휴처, 혜택 추천은 고객의 위치·생활권·혜택 사용 패턴이 확인되고 고객이 직접 사용할 수 있는 제휴처 또는 혜택으로 한정한다.
- 방문 가능성이 낮거나 이용 조건이 불명확한 후보는 일반 추천으로 승격하지 않는다.

2) 개인화 적용·폴백 정책 (PG-REC-PERSONAL-001)

• 개인화 적용 조건 (PI-REC-PRS-001)
- 개인화 추천은 고객 식별, 개인화 활용 동의, 최근 행동 또는 보유 관계 중 하나 이상의 근거, BSS 실행 가능 판정이 모두 충족될 때 적용한다.
• 비개인화 폴백 (PI-REC-PRS-002)
- 개인화 근거가 부족하거나 신뢰도가 낮으면 인기, 테마, 최근 많이 찾은 항목, 운영자 큐레이션 중 하나를 비개인화 폴백으로 제공한다.
- 폴백은 개인화처럼 보이지 않도록 라벨을 구분한다.
• 신뢰도 부족 처리 (PI-REC-PRS-003)
- 추천 후보의 근거가 2개 미만이거나 고객 제한 조건과 충돌하면 추천을 보류한다.
- 보류 시에는 검색, 필터, 비교, 상담 중 최소 2개 대체 경로를 제공한다.
• 무료→유료 전환 추천 (PI-REC-PRS-004)
- 무료 혜택 이용량이 기준 사용률 이상이고 유료 전환 시 추가 혜택, 예상 비용, 해지 가능 조건을 함께 안내할 수 있을 때만 유료 전환 추천을 노출한다.
- 고객이 비용을 부담하기 전 월 예상 부담과 해지·취소 조건을 같은 위계로 표시한다.
• 구독·선물 추천 기준 (PI-REC-PRS-005)
- 구독 중인 상품 기반 선물 추천은 보유 상품, 선물 가능 상태, 수신자 이용 가능 조건, 중복 구매 제한을 모두 충족할 때 노출한다.
- 조건을 충족하지 않으면 선물 추천 대신 상세 확인 또는 대체 혜택 추천으로 전환한다.
• 이탈 방지 혜택 추천 (PI-REC-PRS-006)
- 이탈 방지 혜택은 Customer 360 또는 고객 서비스 이용 행태 분석에서 해지 의향, 반복 실패, 장기 미사용, 상담 전환 위험이 확인된 경우에만 제안한다.
- 혜택은 기간, 대상, 중복 제공 제한, 고객 수락 여부를 이력으로 저장한다.

3) 추천 근거·투명성 정책 (PG-REC-REASON-001)

• 추천 사유 노출 (PI-REC-RSN-001)
- 추천 카드에는 최근 이용, 보유 상품, 조회 이력, 만료 임박, 진행 중 업무, 세그먼트 중 고객이 이해 가능한 사유를 최대 2개까지 노출한다.
• 근거 노출 제한 (PI-REC-RSN-002)
- 추천 근거는 민감정보 원문, 상담 상세, 결제 세부값을 직접 노출하지 않는다.
- 민감도가 높은 근거는 “최근 이용 패턴”, “보유 상품 상태”처럼 범주 단위로 표시한다.
• AI·개인화 라벨 (PI-REC-RSN-003)
- AI 생성 요약, 개인화 추천, 운영자 큐레이션은 서로 다른 라벨로 표시한다.
- 고객이 공식 상품 조건과 보조 추천 설명을 혼동하지 않도록 공식 조건은 상세 원천으로 연결한다.
• AI PICK 캡션 (PI-REC-RSN-004)
- AI PICK 캡션은 고객 맥락, 상품·혜택 핵심 가치, 추천 이유를 40자 안팎의 짧은 문장으로 제공한다.
- 민감정보 원문, 내부 점수, 추론 과정은 노출하지 않고 “최근 이용 패턴”, “관심 상품”, “생활권 혜택”처럼 범주 단위로 표현한다.
• 소셜 프루프 기준 (PI-REC-RSN-005)
- 소셜 프루프는 충분한 모수와 최신 집계 기준이 있을 때만 “유사 고객 인기”, “생활권에서 많이 사용”처럼 범주형 근거로 표시한다.
- 특정 개인을 식별할 수 있거나 과장된 순위 표현은 사용하지 않는다.

4) 추천 우선순위·충돌 정책 (PG-REC-PRIORITY-001)

• 우선순위 축 (PI-REC-PRI-001)
- 추천 우선순위는 컴플라이언스, 고객 불이익 방지, BSS 실행 가능성, 고객 적합도, 사업 우선순위, 최신성 순서로 적용한다.
• 충돌 처리 (PI-REC-PRI-002)
- 두 개 이상의 룰이 충돌하면 더 엄격한 제한 기준을 적용한다.
- 추천 후보가 모두 제한되면 추천 보류 상태로 전환하고 대체 탐색 경로를 제공한다.
• 고객 불이익 방지 (PI-REC-PRI-003)
- 현재 이용 상품 대비 월 부담이 증가하거나 약정·혜택 손실이 발생할 수 있는 추천은 예상 영향 안내와 고객의 실행 전 1회 확인을 거치기 전 실행 CTA를 제공하지 않는다.

5) 노출 빈도·피로도 정책 (PG-REC-LIMIT-001)

• 노출 빈도 제한 (PI-REC-LIM-001)
- 동일 추천 자산은 동일 고객에게 하루 3회, 7일 10회를 초과해 노출하지 않는다.
- 운영자가 더 낮은 제한값을 설정하면 운영자 설정을 우선한다.
• 차단시간 적용 (PI-REC-LIM-002)
- 고객이 관심 없음 또는 숨김을 선택한 추천 자산은 최소 30일 동안 같은 슬롯에서 재노출하지 않는다.
- “일정 기간 보지 않기”를 선택하면 고객 선택 기간을 우선한다.
• 중복 추천 제한 (PI-REC-LIM-003)
- 동일 화면에 같은 상품군 또는 동일 CTA를 가진 추천이 2개 이상 노출되면 우선순위가 낮은 후보를 대체 후보로 전환한다.

6) 추천 실행 연결 정책 (PG-REC-ACTION-001)

• CTA 연결 기준 (PI-REC-ACT-001)
- 추천 CTA는 상세 보기, 비교 보기, 가입·변경 시작, 혜택 사용, 상담 연결 중 하나로 연결한다.
- 실행 CTA는 고객이 영향과 제한을 확인할 수 있는 중간 확인 단계를 포함한다.
• 비교 연결 기준 (PI-REC-ACT-002)
- 요금, 혜택, 약정, 사용 조건이 달라지는 추천은 바로 실행보다 비교 보기를 우선 제공한다.
- 비교 항목이 2개 미만이면 상세 보기와 상담 연결을 대체 경로로 둔다.
• 실행 전 확인 (PI-REC-ACT-003)
- 가입, 변경, 혜택 사용, 상담 연결 전에는 고객 상태, 비용 영향, 약관·동의 필요 여부, BSS 실행 가능 여부를 확인하고 실패 시 실행을 보류한다.
• 크로스 도메인 넛지 (PI-REC-ACT-004)
- 통신·멤버십·구독·커머스 도메인 간 크로스셀 넛지는 고객이 현재 화면에서 수행 중인 과업과 직접 연결될 때만 제공한다.
- 다른 도메인 이동 시 추천 목적, 예상 혜택, 이동 후 유지되는 맥락을 1회 이상 함께 안내한다.
• 스크린 어웨어 Next Action (PI-REC-ACT-005)
- 스크린 어웨어 추천은 고객이 보고 있는 화면, 체류 시간, 탭·스크롤 이력, 상태 정보를 바탕으로 현재 화면에 이어지는 다음 행동을 1개 이상 제안한다.
- 단, 고객이 작업을 완료하기 전 과도한 판매성 추천으로 흐름을 방해하지 않는다.
• 컨시어지 카드 Next Action (PI-REC-ACT-006)
- 컨시어지 카드는 진행 중 주문, 만료 임박 혜택, 실패·미완료 업무, 고객지원 전환 맥락이 확인될 때만 노출한다.
- 카드에는 추천 사유, 실행 CTA, 대체 경로를 함께 제공하고 고객이 닫거나 숨기면 현재 세션에서 즉시 제외한다.
• 중복 요청·멱등 처리 (PI-REC-ACT-007)
- 동일 고객, 동일 추천 후보, 동일 실행 CTA가 동일 세션에서 2회 이상 중복 요청되면 하나의 요청 키로 묶어 멱등 처리한다.
- 이미 실행 연결 대기 또는 실행 완료 상태인 요청은 추가 실행을 차단하고 기존 처리 상태를 안내한다.

7) 고객 제어·피드백 정책 (PG-REC-CONTROL-001)

• 숨김 즉시 반영 (PI-REC-CTL-001)
- 관심 없음, 숨김, 일정 기간 보지 않기는 선택 즉시 현재 지면에서 제외한다.
- 제외 결과는 새로고침 또는 재진입 후에도 유지한다.
• 피드백 사유 수집 (PI-REC-CTL-002)
- 피드백 사유는 관심 없음, 이미 이용 중, 가격 부담, 관련 없음, 부정확함, 반복 노출의 6개 기본값으로 수집한다.
- 고객이 사유를 건너뛰어도 숨김 처리는 완료한다.
• 제어 이력 확인 (PI-REC-CTL-003)
- 고객은 설정 화면에서 최근 90일 내 숨김, 관심 없음, 피드백, 초기화 이력을 확인할 수 있다.
- 초기화는 고객 확인 후 적용하고 처리 이력을 남긴다.

8) 비즈니스 룰·제약 정책 (PG-REC-RULE-001)

• 수익성 가드레일 (PI-REC-RUL-001)
- 사업 룰은 추천 목적, 적용 슬롯, 대상 상품군, 우선순위, 시작·종료 일시를 포함해야 한다.
- 기간이 없는 예외 룰은 배포할 수 없다.
• 다운셀 제한 (PI-REC-RUL-002)
- 현재 이용 상품 대비 월 부담이 일정 기준 이상 낮아지는 추천은 고객 불이익이 없는 경우에만 노출한다.
- 예외가 필요하면 운영 승인과 노출 사유를 함께 저장한다.
• 세그먼트 룰 (PI-REC-RUL-003)
- 신규, 장기, 결합·가족, 고가치, 이탈위험, 외국인 등 세그먼트 룰은 추천 대상, 제외 대상, 우선순위, 폴백 기준을 분리해 관리한다.
• 캠페인 자동 생성 조건 (PI-REC-RUL-004)
- 캠페인 자동 생성 후보는 고객 세그먼트, 추천 목적, 대상 상품·혜택, 예상 효과, 제외 조건, 적용 기간이 모두 정의된 경우에만 운영 승인 대상으로 등록한다.
- 해지 의향 고객 방어 캠페인은 고객 불이익 방지와 수신 동의 기준을 우선 적용한다.
• 채널·오퍼링 조합 기준 (PI-REC-RUL-005)
- 채널과 오퍼링 조합 추천은 고객 접촉 빈도, 채널 선호, 실행 가능 상태, 보유 관계를 함께 판단한다.
- 최적 채널이 외부 또는 인접 서비스이면 고객에게 이동 사유와 완료 가능한 후속 행동을 함께 제시한다.

9) 사전 검증·보류 정책 (PG-REC-VALID-001)

• 사전 검증 기준 (PI-REC-VAL-001)
- 배포 전 검증은 대표 고객 상태 5종 이상, 추천 슬롯 3종 이상, 제한 룰 3종 이상을 조합해 추천 공백과 과도한 제한을 확인한다.
• 추천 보류 기준 (PI-REC-VAL-002)
- 룰 위반, 개인화 동의 부족, BSS 실행 불가, 고객 불이익 가능성, 근거 부족 중 하나가 확인되면 추천을 보류하고 보류 사유를 기록한다.
• 대체 경로 제공 (PI-REC-VAL-003)
- 추천 보류 또는 실패 시 검색, 필터 조정, 비교 보기, 상담 연결 중 최소 2개 경로를 제공한다.
- 고객이 입력한 조건과 화면 맥락은 대체 경로로 전달한다.

10) 슬롯 운영·배포 정책 (PG-REC-OPERATION-001)

• 등록 필수값 (PI-REC-OPR-001)
- 운영 등록 시 슬롯 ID, 추천 목적, 추천 유형, 대상 고객, 적용 기간, 폴백 기준, 책임 운영자, 검수자를 필수값으로 둔다.
• 검수·승인 기준 (PI-REC-OPR-002)
- 운영 검수는 고객 불이익, 개인정보 노출, 룰 충돌, 대체 경로, 롤백 가능성을 모두 통과해야 승인한다.
• 배포·회수 기준 (PI-REC-OPR-003)
- 배포 후 오류율, 숨김률, 상담 전환, 고객 불만, 룰 위반 중 사전 기준을 초과하면 즉시 회수하거나 직전 정상 버전으로 롤백한다.
• 개인화 컨텍스트 관리 (PI-REC-OPR-004)
- 개인화 컨텍스트는 카드, 코너, 배너 단위로 생성·수정·복제·삭제할 수 있으며 대상, 기간, 노출 위치, 우선순위를 필수값으로 관리한다.
- 컨텍스트 발행·회수·예약 발행은 변경자, 변경 사유, 적용 시점 이력을 남긴다.
• 개인화 항목·그룹 관리 (PI-REC-OPR-005)
- 개인화 항목과 항목 그룹은 추천 가능한 기능·콘텐츠 단위로 관리한다.
- 그룹 단위로 노출 조건, 정렬, 우선순위, 제외 조건을 설정하고, 항목이 삭제되면 연결된 추천 슬롯은 비개인화 폴백으로 전환한다.

11) 성과 측정·개선 정책 (PG-REC-METRIC-001)

• 전환 퍼널 (PI-REC-MET-001)
- 추천 성과는 노출, 클릭, 수용, 실행 시작, 실행 완료, 이탈, 상담 전환의 7단계로 측정한다.
- 각 단계는 슬롯 ID와 추천 유형 기준으로 집계한다.
• 부정 반응 지표 (PI-REC-MET-002)
- 숨김률, 관심 없음률, 부정확 신고, 반복 노출 불만, 추천 보류율은 추천 품질의 부정 지표로 관리한다.
- 부정 지표가 기준을 초과하면 개선 검토 상태로 전환한다.
• 실험·개선 기준 (PI-REC-MET-003)
- A/B 실험은 대상, 기간, 트래픽 비율, 성공 지표, 중단 기준을 등록한 뒤 시작한다.
- 실험 종료 후 개선안은 운영자 승인 전까지 배포하지 않는다.
• 추천 영향도 비교 (PI-REC-MET-004)
- 추천 영향도는 적용군과 미적용군 또는 대조군을 구분해 노출, 클릭, 수용, 실행, 완료, 이탈, 숨김, 상담 전환을 비교한다.
- 실험 기간, 대상, 트래픽 비율, 유의성 판단 기준은 결과 해석과 함께 저장한다.
• 캠페인 성과 분석 (PI-REC-MET-005)
- 캠페인 추천 성과는 대상 생성, 노출, 수락, 실행, 해지 방어, 전환 실패 사유를 단계별로 측정한다.
- 자동 생성 캠페인은 운영자 승인 여부와 실제 반영 결과를 함께 이력으로 남긴다.

12) 데이터 활용·고지 정책 (PG-REC-DATA-001)

• 활용 데이터·투명성 고지 (PI-REC-DAT-001)
- 개인화 추천에는 최근 메뉴, 조회·관심, 구매·혜택 이용, 문의 상태, 진행 중 업무 등 활용 데이터 범주를 고객에게 안내한다.
• 삭제·초기화 안내 (PI-REC-DAT-002)
- 고객은 설정 화면에서 추천 제어 이력의 삭제·초기화 방법을 확인할 수 있다.
- 추천 제어와 피드백 이력은 고객 확인 목적 기준 90일 이상 조회 가능하게 두고, 장기 분석 데이터는 비식별 기준으로 분리한다.
• 설정 소유권 (PI-REC-DAT-003)
- 개인화 설정의 실제 저장·철회·초기화 기능은 설정 또는 회원정보 관리 영역이 소유한다.
- 추천 화면은 안내와 진입 경로만 제공한다.
• 위치·소비패턴 활용 제한 (PI-REC-DAT-004)
- 위치, 생활권, 소비패턴, 혜택 사용 패턴은 고객 동의와 서비스 제공 목적이 확인된 경우에만 추천에 활용한다.
- 민감한 위치 원문이나 결제 세부값은 노출하지 않고 추천 근거에는 범주화된 설명만 사용한다.
• 소비 성향 온보딩 (PI-REC-DAT-005)
- 데이터가 부족한 고객에게는 소비 성향 테스트나 가벼운 선호 선택을 통해 추천 초기값을 수집할 수 있다.
- 온보딩은 선택 사항으로 제공하고, 입력하지 않아도 인기 혜택이나 비개인화 폴백으로 탐색을 계속할 수 있어야 한다.
• 고객 서비스 이용 행태 분석 (PI-REC-DAT-006)
- 고객 서비스 이용 행태 분석은 체류 시간, 특정 메뉴 탐색, 클릭, 스크롤, 이탈, 상담 전환 같은 행동 신호를 추천 Trigger로 분류한다.
- 분석 결과는 추천 목적과 직접 관련된 범위에서만 사용하고, 개인을 식별할 수 있는 원문 로그는 추천 근거로 노출하지 않는다.

13) BSS·연계 판정 정책 (PG-REC-BSS-001)

• 상태·보유 관계 판정 (PI-REC-BSS-001)
- 추천 연계 시스템은 가입 상태, 보유 상품, 미납·정지, 약정, 혜택 자격, 법정대리인 필요 여부를 BSS 기준 정보로 받아 추천 가능 판정에 반영한다.
- 보유 관계가 확인되지 않으면 개인화 추천보다 비개인화 폴백을 우선한다.
• 실행 가능 회신 (PI-REC-BSS-002)
- 추천 CTA 노출 전 BSS 실행 가능 여부가 확인되지 않으면 실행 CTA 대신 상세 확인, 비교, 상담 연결을 우선 제공한다.
• 연계 실패 처리 (PI-REC-BSS-003)
- BSS 또는 외부 연계 조회가 실패하면 개인화 추천을 보류하고 비개인화 폴백 또는 검색 전환을 제공한다.
- 실패 이력은 원인 구분과 함께 저장한다.

14) 실패·대체 경로 정책 (PG-REC-EXCEPTION-001)

• 무결과 처리 (PI-REC-EXC-001)
- 추천 후보가 없으면 고객에게 무결과 사유를 간단히 안내하고 검색어 조정, 필터 변경, 카테고리 이동, 상담 연결 중 1개 이상의 대체 경로를 제공한다.
• 부적합 안내와 대체 추천 (PI-REC-EXC-002)
- 추천 상품이 부적합하거나 가입 불가이면 불가 사유를 표시하고 같은 목적의 대체 상품 또는 조건 변경 경로를 제공한다.
- 대체 후보가 없으면 상담 연결을 제공한다.
• 추천 실패 복구 (PI-REC-EXC-003)
- 추천 생성 오류가 발생하면 현재 고객 입력과 화면 맥락을 유지한 채 재시도 1회, 폴백 1회, 상담 전환 순서로 복구한다.

15) 이력·감사 정책 (PG-REC-HISTORY-001)

• 노출 이력 (PI-REC-HIS-001)
- 추천 노출, 클릭, 비교 진입, 실행 시작, 실행 완료, 상담 전환은 슬롯 ID, 추천 유형, 고객 상태 범주, 발생 시각과 함께 저장한다.
• 피드백 이력 (PI-REC-HIS-002)
- 숨김, 관심 없음, 부정확 신고, 반복 노출 불만은 추천 자산, 슬롯, 사유, 차단시간, 반영 상태와 함께 저장한다.
• 룰 변경 감사 (PI-REC-HIS-003)
- 추천 룰의 생성, 수정, 삭제, 배포, 회수, 롤백은 변경자, 승인자, 변경 사유, 적용 범위, 적용 시점을 감사 이력으로 남긴다.

16) 세그먼트·비식별 고객 정책 (PG-REC-SEGMENT-001)

• 세그먼트 분기 (PI-REC-SEG-001)
- 세그먼트별 추천은 추천 대상, 제외 조건, 우선순위, 폴백 기준을 별도 관리한다.
- 세그먼트가 중복되면 고객 불이익 방지 기준을 우선한다.
• 비식별 추천 제한 (PI-REC-SEG-002)
- 비식별 고객에게는 검색어와 현재 화면 맥락 기반 추천만 제공한다.
- 보유 상품, 혜택 자격, 이력 기반 개인화는 로그인 후 동의와 식별이 확인된 뒤 적용한다.
• 로그인 후 연결 (PI-REC-SEG-003)
- 비식별 상태의 탐색 이력은 로그인 후 고객이 이어보기 또는 개인화 활용에 동의한 경우에만 식별 고객 프로필과 연결한다.
- 연결하지 않는 경우 비식별 이력은 개인화 추천에 사용하지 않는다.
• 로그인·선택동의 Case 분기 (PI-REC-SEG-004)
- 추천 노출은 로그인 여부와 선택동의 상태에 따라 식별 개인화, 제한 개인화, 비개인화, 오류 복구 Case로 분기한다.
- 선택동의가 없거나 오류가 발생한 경우에는 개인화처럼 보이는 문구를 사용하지 않고 일반 추천 또는 안내 상태로 전환한다.
• Cold Start 추천 기준 (PI-REC-SEG-005)
- 신규 고객이나 데이터 부족 고객은 인기 혜택, 기본 카테고리, 소비 성향 온보딩 결과, 현재 화면 맥락을 기준으로 추천한다.
- Cold Start 후보는 개인화 추천과 라벨을 구분하고 고객 제어 이력을 학습 신호로 반영한다.

최종 점검 기준

추천 정책서 제출 전 점검
1. 범위 정합성
추천이 전시 슬롯 설정이 아니라 추천 후보 이해, 실행 연결, 고객 제어, 운영 품질 기준으로 작성되었는지 확인한다.
2. 고객 완결성
고객이 추천 이유 확인, 비교, 실행, 숨김, 실패 복구까지 한 흐름으로 처리할 수 있는지 확인한다.
3. 유즈케이스 입자도
홈 추천, 상세 추천, 관심 없음 같은 절차 항목이 상위 유즈케이스가 아니라 프로세스·기능·정책으로 내려갔는지 확인한다.
4. 상태 전이 정합성
추천 가능, 보류, 제한, 숨김, 피드백 반영, 배포·회수 상태가 고객 후속 행동과 운영 처리 기준을 바꾸는지 확인한다.
5. 프로세스 연결성
각 고객·운영자 유즈케이스가 2개 이상의 의미 있는 프로세스로 분해되고 기능·정책에 연결되었는지 확인한다.
6. 정책 구체성
노출 빈도, 차단시간, 우선순위, 보류 기준, 검증 기준, 이력 저장 기준 등 실제 판단값이 포함되었는지 확인한다.
7. 고객 제어성
숨김, 관심 없음, 설정 진입, 초기화, 피드백 이력이 고객이 이해하고 실행할 수 있는 기준으로 작성되었는지 확인한다.
8. BSS 기준 정보 포함성
BSS가 제공하는 고객 상태, 보유 상품, 실행 가능 여부, 제한 조건 기준과 추천 연계 시스템의 반영 책임이 분리되어 있는지 확인한다.
9. 운영 품질 루프
추천 성과, 부정 반응, 룰 위반, 실험, 개선안 승인, 배포·회수·롤백이 운영 루프로 연결되었는지 확인한다.
10. 요구사항 반영 여부
상세 요구사항명과 상세 요구사항 설명 기준의 78건이 유즈케이스, 프로세스, 기능, 정책 중 하나 이상으로 재구성되었는지 확인한다.
