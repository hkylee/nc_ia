# IA-first 입력 규격 (작업자용)

AS-IS IA 문서와 정책서를 바탕으로 **IA를 먼저 통째로 입력**해서 화면·계층·흐름을 한 번에 생성하기 위한 작업 규격이다.
작업자는 이 문서의 컬럼·규칙대로 **스프레드시트 한 장(한 행 = 한 화면)**으로 채워서 전달하면 된다.

> ⚠️ **이 문서의 모든 예시 값은 샘플링(추론)이다.** §5 워크드 예제, UC 코드·카테고리, 화면 구성·흐름 등은 형식을 보여주기 위한 가상 데이터이며, **실제 작업은 참조 문서(정책서 + 4채널 AS-IS IA 문서)를 준수해 다시 잡아야 한다.** 컬럼·규칙(§2~§4)만 규격으로 고정이고, 내용 값은 참조 문서가 정답이다.

---

## 1. 멘탈 모델

- **홈에서 시작**하고, **각 모듈의 진입 페이지**가 그 모듈 트리의 시작점(depth1 뿌리)이 된다.
  - 예: 회원(MBR) 모듈은 `로그인`이 진입 페이지 → 거기서 회원가입·T-ID·소셜·계정찾기 등이 펼쳐짐.
- 계층은 트리(nested)가 아니라 **`depth1 > depth2 > depth3 …` 경로**로 표현한다. 작업은 depth 컬럼을 왼쪽부터 채우는 방식.
- 두 종류의 연결이 있다:
  - **depth = 포함관계** (메뉴 트리상 부모-자식) — 자동으로 엣지가 됨
  - **`from`(들어옴) = 화면 전환** — 이 화면으로 들어오는 출발 화면. depth로 안 잡히는 전환만 적는다.
  - **모듈 경계를 넘는 흐름은 `from`으로** 이어진다 (예: 로그인(MBR)의 from에 마이(MY)).

## 2. 컬럼 규격

| 컬럼 | 필수 | 값 | 설명 |
|---|---|---|---|
| `domain` | ✅ | `채널-모듈코드` | 화면이 **속한 모듈** 코드 (보이는 위치 아님). 채널 프리픽스 포함. 예: `앱-DSP`, `앱-MBR` |
| `uc_code` | ✅ | `US-…` | UC 코드(§2.2). 형식 `US-[도메인]-[카테고리]-[순번]`. 예: `US-MBR-LGN-001` |
| `uc_name` | ✅ | 텍스트 | UC명. 예: `회원가입` |
| `page_id` | ✅ | `NOVA-…` | **스크린 ID(= 우리 규격 `legacy_screen_id`, 가장 중요).** `NOVA-[도메인]-[타입]-[순번]-[변형]`(§2.1). 같은 페이지면 어디서든 같은 값 → 중복 방지·노드 식별 |
| `depth1` | ✅ | 텍스트 | 계층 1단계 (모듈 진입페이지명 또는 `홈`) |
| `depth2`,`depth3`,… | ⬜ | 텍스트 | 하위 계층. 필요한 만큼 컬럼 확장. **왼쪽부터 빈칸 없이** 채움 |
| `dev_type` | ⬜ | `PG\|PU\|CP\|AP\|BS` | 구현유형. 모르면 비움 |
| `description` | ⬜ | 텍스트 | 화면 설명·진입조건·정책 |
| `from` | ⬜ | 화면명(들) | 이 화면으로 **들어오는 출발 화면 이름**(들어옴). depth로 안 잡히는 전환만. 여러 개면 `;`로 구분 |
| `flow_type` | ⬜ | `기본`/`케이스 N`/`예외 N` | 플로우 구분(§3.1). 비우면 `기본`. 번호는 **같은 owner UC 내** 기준 |
| `owner_page` | ⬜ | `✅`/공백 | **원본 페이지 여부**(맞다/아니다). 같은 `page_id` 행들 중 마스터면 ✅. 단일 UC 페이지는 ✅ |
| `owner_uc` | ⬜ | `✅`/공백 | **원본 UC 여부**(맞다/아니다). 이 UC가 페이지의 소속(원본) UC면 ✅ |

> 순서는 별도 컬럼 없이 **시트의 행 순서 = UC 내 화면 순서**로 정한다. 원하는 노출 순서대로 행을 적어 내려가면 된다.

**dev_type 범례**: `PG`=페이지, `PU`=풀스크린모달, `BS`=바텀시트, `CP`=센터팝업, `AP`=액션팝업, `CT`=커스텀, `MD`=모달

### 2.1 스크린 ID(`page_id`) 채번 규칙

형식: **`NOVA-[도메인]-[개발타입]-[순번]-[변형]`** — 예: `NOVA-MBR-PU-005-0`, `NOVA-PRDD-PG-004-1`

| 세그먼트 | 값 | 설명 |
|---|---|---|
| prefix | `NOVA` | 고정 (legacy 파일엔 `NC-`도 존재) |
| 도메인 | `MBR`·`DSP`… | 모듈 코드(2~6 대문자). **채널 프리픽스 제외** — `domain`의 `앱-` 뺀 값 |
| 개발타입 | `PG·PU·BS·CP·AP·CT·MD` | = `dev_type` |
| 순번 | `001` | 3자리 zero-pad, **도메인+개발타입별**로 1부터 (※ 순번 스코프는 정책 확정 필요) |
| 변형 | `0` | 동일 화면 변형/상태, 기본 `0` |

- **수동 채번**(자동 생성 함수 없음). 라우팅 slug라 중복 불가 — 활성 문서 스코프 DB 트리거가 차단.
- 근거: `ScreenIdField` placeholder, `DEV_TYPES` 정규식, `/^NOVA-/i` 검증.

### 2.2 UC 코드 채번 규칙 *(⚠️ 추론 — 정책서로 확정 필요)*

형식(추론): **`US-[도메인]-[카테고리]-[순번]`** — 예: `US-MBR-LGN-001`

| 세그먼트 | 값 | 설명 |
|---|---|---|
| prefix | `US` | 고정 |
| 도메인 | `MBR`·`DSP`… | 모듈 코드(채널 프리픽스 제외) |
| 카테고리 | `LGN`·`JON`… | **프로세스 분류 — 정책서 정의값** (아래 예제 코드는 추론 placeholder) |
| 순번 | `001` | 3자리 zero-pad |

- 시스템 자동 생성은 카테고리 자리에 `NEW` 고정(`US-MBR-NEW-001`), 사용자 입력은 자유형(`^[A-Z0-9][A-Z0-9_-]*$`, 도메인 내 unique), 연습장은 `DRAFT-[YYYYMMDD]-[순번]`.
- 근거: `createEmptyUseCase`/`createUseCaseWithCode` ([web/app/usecase/actions.ts](web/app/usecase/actions.ts)), `/US-([A-Z]+)-/` 파싱.
- ⚠️ **카테고리 코드·순번 스코프는 정책서 확인 전까지 추론.** 예제의 `LGN·JON·SNS·FND·DMT·WDR` 등은 임시값.

## 3. 작성 규칙

1. **마지막(가장 깊은) depth 값 = 그 화면의 이름.** 별도 screen_name 컬럼 없음.
2. **부모 행 누락 금지.** `로그인 > 회원가입`을 쓰려면 `로그인`만 있는 행도 존재해야 한다.
3. **depth는 왼쪽부터 빈칸 없이.** depth1 비우고 depth2만 채우면 안 됨.
4. **`domain`은 화면이 속한 모듈 코드.** 홈에서 마이로 가도, 마이 행의 domain은 `앱-MY`. 경로(depth)는 모듈 경계를 넘어도 된다.
5. **화면 순서는 행 순서로** 정해진다. 같은 UC 안에서 원하는 노출 순서대로 행을 배치.
6. **한 페이지는 여러 UC에 속할 수 있다.** 같은 페이지면 모든 행에 같은 `page_id`(중복 방지). 마스터 행에 `owner_page=✅`, 원본 UC 행에 `owner_uc=✅`(같은 page_id당 각각 1개). 단일 UC 페이지는 둘 다 ✅. flow_type은 owner UC 기준.
7. **조건부 진입**(예: 휴면해제, 로그인 후 진입)은 `description`에 조건을 명시.
8. **전환은 `from`(들어옴) 한 방향만** 적는다. depth로 이미 그려지는 부모→자식은 생략하고, ①모듈 간 진입 ②형제 간 순서 ③복귀 같은 **depth 밖 전환만** 기재. (IA 문서의 들어옴을 통째로 붙여도 됨 — 중복은 import가 정리)
9. **`flow_type`은 §3.1 기준**으로 `기본`/`케이스 N`/`예외 N` 기입. 비우면 `기본`. 맵에서 분류별 다른 스타일로 렌더.

### 3.1 flow_type 분류 기준

| 값 | 의미 | 판단 기준 |
|---|---|---|
| `기본` | 주 성공 흐름(happy path) | 목표를 가장 직접적으로 달성하는 **단일 정상 경로**. 여정당 1세트 (비우면 기본) |
| `케이스 N` | 정상 범주의 **대안/분기** | 선택·조건에 따라 갈라지나 **모두 valid**한 흐름 (예: 로그인 수단 카카오/네이버/Apple) |
| `예외 N` | **오류·예외 처리** | 실패·미충족·비정상 상태 (예: 비번 오류, 휴면, 계정 분실) |

- **판단:** 정상 달성 → 단일 경로면 `기본`, 갈라지는 정상 대안이면 `케이스 N` / 막힘·실패 → `예외 N`
- **번호:** **같은 owner UC 내**에서 1부터. 같은 시나리오 화면들은 같은 번호 공유.
- **owner UC 기준:** 한 화면이 여러 UC에 중복돼도 **owner UC는 하나**라 flow_type이 모호하지 않다. 참조 행(다른 UC에서 끌어온 화면)은 flow_type을 비워 owner를 따른다.
- **UC를 시나리오 단위로:** 기본/케이스/예외가 한 UC에 모이려면 UC를 시나리오 크기로 잡아야 한다. 잘게 쪼개면 각 UC가 자기 `기본`만 갖게 되어 케이스/예외가 흩어진다.

## 4. 플로우 맵의 앵커 = 화면 이름

플로우 맵(노드·화살표)은 **화면 이름을 조인 키(join key)**로 그려진다.

| 역할 | 컬럼 | 맵에서 |
|---|---|---|
| 노드(점) | `page_id` (표시명 = 마지막 depth 값) | 같은 page_id = 같은 점 1개 |
| 계층 엣지 | `depth1 > depth2 > …` | 부모→자식 포함 선 |
| 전환 엣지 | `from` | 이동 화살표 (들어옴, 이름으로 매칭) |
| 흐름 스타일 | `flow_type` | 기본=실선 / 케이스=색 구분 / 예외=점선·흐림 등 구분 렌더 |
| 그룹/레인 | `domain`, `uc_code` | 모듈·UC 단위 묶음 (배치 기준) |
| 노드 정체성 | `page_id` | 같은 page_id = 같은 페이지(병합). 이름 아닌 ID로 식별 |
| 원본 표시 | `owner_page`,`owner_uc` | 마스터 페이지·원본 UC 체크(✅) — 병합 시 대표 인스턴스 |
| 배치 순서 | 행 순서 | 그룹 내 정렬 |

### ⚠️ 노드 식별은 page_id 기준
- **같은 `page_id` = 같은 페이지로 병합**(사방에서 화살표 수렴). page_id가 다르면 이름이 같아도 별개 노드 — 이름 충돌 걱정이 사라진다.
- **`from`은 화면 이름으로 매칭**하므로 같은 UC 안에선 이름이 유일해야 함. (page_id가 중복 페이지를 안전히 구분)
- `from`의 값은 반드시 실재하는 화면 — 오타·미존재는 끊긴 화살표(dangling).

### Import 전 검증 패스 (전달받으면 먼저 수행)
1. 부모 행 누락 / depth 좌측 공백
2. `from` 값 중 실재하지 않는 화면 이름 (dangling)
3. `page_id` 누락·불일치 (같은 페이지인데 다른 id / 다른 페이지인데 같은 id), 각 page_id에 `owner_page`·`owner_uc` ✅가 각각 1개인지
4. 채널-도메인 매핑, 정책서엔 있는데 IA에 없는 화면(또는 반대)

### 예제에 적용된 모델링 메모
- **`from`은 depth 밖 전환만** 채웠다. 대부분 행이 비어 있고(부모→자식은 depth가 그림), 모듈 간 진입(로그인 ← 마이)·형제 순서(정보 입력 ← 본인인증)·복귀(로그인 ← 가입 완료·아이디 찾기·비밀번호 재설정)만 기재됨.
- **flow_type**: owner UC 기준 번호. 소셜 로그인 UC의 3종은 정상 대안이라 `케이스 1·2·3`. 계정 찾기·휴면 해제는 각자 별도 UC라 그 안에선 `기본`(빈칸) — 로그인의 예외로 잡으려면 로그인 UC로 합쳐야 함.
- **본인인증(CP)**: 4개 행이 같은 `page_id=NOVA-MBR-CP-001-0` → **한 페이지**. 회원가입 행만 `owner_page=✅`·`owner_uc=✅`(마스터·원본 UC), 나머지 3개 UC 행은 둘 다 공백(참조). 맵·DB에선 점 1개.
- **회원탈퇴·휴면해제**는 로그인 아래 배치했으나 실제 진입이 마이라면 depth1을 `마이`로 조정.
