# PM-29 통합 알림 Topic Learning & Core Design

## 1. 작성 목표

- 통합 알림은 단순 발송 채널 관리가 아니라 고객이 알림을 통해 필요한 행동을 인지하고, 후속 업무를 바로 실행하거나 복구할 수 있게 하는 업무 허브로 정의한다.
- 고객은 알림함에서 행동 필요 알림, 참고성 알림, 필수·거래성 알림, 선택·마케팅 알림을 구분해 확인하고 수신 설정을 직접 제어할 수 있어야 한다.
- 운영자는 알림 유형, 템플릿, 발송 실패, 중복·빈도, 조용한 시간, 이력·품질 기준을 관리해 고객 피로도와 미수신 리스크를 함께 줄여야 한다.

## 2. 요구사항 근거

- 요구사항 DB 기준 `normalized_depth4=통합알림` 93건을 사용했다.
- 상세 요구사항명과 상세 요구사항 설명을 중심으로 알림함, 수신 설정, OS 권한, 필수·선택 구분, 우선순위, 중복·빈도, 실패 복구, 발송 이력 축으로 재구성했다.
- 가입, 약관, 결제, 혜택 문구는 알림의 트리거 또는 유형으로만 해석하고 별도 정책서 범위를 침범하지 않도록 제한했다.

## 3. Core Design Map

- 액터: 고객, 운영자, BSS, OS 권한 시스템, 연계 시스템, 채널 업무 시스템.
- 고객 유즈케이스: 통합 알림 확인, 통합 알림 수신 설정 실행, 통합 알림 후속 처리·복구 관리.
- 운영자 유즈케이스: 통합 알림 운영 기준 및 품질 관리.
- 보조 시스템 유즈케이스: 고객·업무 조건 판정, OS 권한 상태 제공, 연계 결과 제공, 상태·이력 처리.
- 핵심 프로세스: 알림함 진입 및 유형 확인, 행동 필요 여부 및 우선순위 분류, 알림 상세 및 컨텍스트 확인, 수신 유형 및 채널 선택, 필수·선택 알림 구분, OS 권한 및 연락처 확인, 중복·만료·무효 알림 정리, 발송 실패·수신 제한 복구.
- 핵심 정책: 수신 설정·채널 기준, 알림함·후속 연결, 우선순위·중복 제어, 발송 실패·복구, 필수·거래성 알림, 알림 운영 기준.

## 4. 품질 점검 결과

- 통합 알림 전용 semantic axis와 프로세스 템플릿을 보강한 뒤 생성했다.
- 인증기관으로 잡혔던 보조 액터를 OS 권한 시스템으로 정정하고, OS 푸시 권한·서비스 수신 동의·연락처 유효성의 책임 경계를 분리했다.
- 상태는 `업무 시작 전/처리 완료` 같은 범용 상태가 아니라 `알림 생성 전 → 알림 생성됨 → 발송 대기 → 발송 완료/수신 제한 → 대체 확인 필요/확인·처리 완료/만료·무효` 흐름으로 재정의했다.
- 프로세스 내부 연결값은 순수 ID로 정규화하고, 정책 목록 items는 정책 상세와 추적 가능한 `{id, name}` 구조로 보강했다.
- 요구사항 Trace는 상세 요구사항 93건 중 자동 매칭 93건, 수동 검토 필요 0건이다.
- Health Check는 98점, 필수 Gate 통과, 판정 `우수`이다.
- Schema Gate와 Stage Critical Gate는 모두 통과했다.

## 5. Agent 개선 인사이트

- 알림 주제에서 `가입`, `결제`, `혜택`, `보안`, `주문` 같은 단어가 나와도 해당 단어는 알림 유형 또는 행동 필요 알림의 트리거로 해석해야 한다.
- `잔고 부족`, `출금 예정`, `청구`, `요금` 같은 문구는 일반 알림이 아니라 필수·거래성 알림 축으로 분류해야 한다.
- 알림 후속 처리 프로세스는 실패를 단순 분류하는 문장으로 끝내지 말고 재시도 가능 여부, 대체 안내 경로, 상담 전환 기준을 함께 판정해야 한다.
