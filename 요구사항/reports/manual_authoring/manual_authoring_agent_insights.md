# Manual Authoring Agent Insights

## 인사이트 적용 등급
- [COMMON] 모든 정책서에 적용 가능한 계위, 연결, 품질, 렌더링, 추적성 규칙이다. AGENTS.md, Validator, Inspector, Mock 규칙으로 승격할 수 있다.
- [TOPIC] 특정 PM/주제에서 발견한 업무 경계 또는 판단축이다. 현재 주제, 요구사항, TK, 첨부자료가 같은 맥락일 때만 적용한다.
- [PATTERN] 다른 주제에도 유사하게 참고할 수 있는 작성 패턴이다. 그대로 복사하지 말고 현재 주제의 요구사항 언어로 재구성한다.
- [DO_NOT_GENERALIZE] 특정 주제 경계, 수치, 업무 이관 판단처럼 다른 정책서에 강제하면 오염되는 내용이다. 공통 prompt, mock 템플릿, validator로 승격하지 않는다.

기본 적용 기준:
- PM-XX 섹션은 명시가 없으면 [TOPIC]으로 본다.
- 상태 전이 `criteria` 필수, 복합 액터명 금지, 순수 ID 연결, 요구사항 원문 복사 금지, 직접 Inspector/Health Check 필수처럼 주제와 무관한 구조 문제만 [COMMON]으로 승격한다.
- 특정 주제에서 나온 업무 경계, 예를 들어 환불 산정 범위, 선물 발신/수신 구조, 청구·수납 상태 분리 등은 현재 주제 근거가 맞을 때만 적용한다.
- 인사이트를 적용하기 전 현재 주제의 상세 요구사항명/설명, TK, 첨부자료, 승인된 Blueprint와 충돌하지 않는지 확인한다.
- 주제별 인사이트를 근거 없이 일반화해 액터, 유즈케이스, 상태, 프로세스, 기능, 정책을 추가하지 않는다.

## 최상위 고정 규칙: 직접 Inspector / 직접 Health Check 필수
[COMMON]
- 자동 Inspector, 자동 Health Check, 테스트, Trace 점수는 보조 신호일 뿐이며 Codex 직접 판단을 대체하지 않는다.
- 정책서를 직접 작성하거나 보완한 뒤 다음 정책서로 넘어가기 전에는 반드시 Codex가 직접 Inspector 역할로 범위, 계위, 연결, 정책 구체성, 요구사항 재구성, 샘플 밀도, 렌더링 품질을 확인한다.
- 반드시 Codex가 직접 Health Check 역할로 10개 영역과 필수 Gate를 PASS/FAIL 판정한다.
- 자동 점수가 100점이어도 직접 Inspector 또는 직접 Health Check에서 P1/P2 문제가 있으면 완료 처리하지 않는다.
- 작업 기록에는 `direct_inspector`, `direct_health_check`, `auto_vs_manual_delta`, `remaining_p1_p2` 관점을 남긴다. 이 기록이 없으면 해당 문서는 완료로 보지 않는다.

## PM-18 주문 상태/사후 관리 작성 인사이트
- 주문 사후 관리는 상태 조회로 축소되기 쉬우므로 “상태 + 가능 행동 + 제한 사유 + 복구 경로”를 한 세트로 정책화해야 한다.
- 배송/교환/반품과 상품 연동/인증 요구사항이 함께 들어오므로, 액터를 배송사·제휴사 등으로 세분화하지 말고 BSS/연계 시스템으로 묶은 뒤 기능·정책에서 책임 경계를 분리하는 편이 안정적이다.
- 환불 금액 산정은 PM-19 범위로 넘기되, PM-18에서는 클레임 진행과 환불 예정 상태 표시까지는 다뤄야 요구사항 누락을 피할 수 있다.
- 주문 완료 후 온보딩, 바코드 이용권 첫 사용, 나의쇼핑 팝업/탭, 알림 신청·취소, 구매후기, 개통신청내역 통합처럼 화면성 요구로 보이는 항목도 PM-18에서는 “주문 이후 다음 행동과 상태 기반 안내 기준”으로 정책화해야 Trace 누락을 막을 수 있다.
- 시스템/BSS 유즈케이스가 `process_target=N`이어도 배송·클레임·제휴 권한·증빙·운영 보정처럼 고객 표시 상태나 권한·이력을 바꾸는 전이에는 `state_transitions.usecase_ids`에 함께 연결한다.
## PM-19 해지/환불/취소 작성 인사이트
- 해지/환불/취소는 “가능 여부 확인”이나 “요청 접수”를 유즈케이스로 두면 절차 단계로 과분화된다. 유즈케이스는 처리 경로 결정, 영향 확인, 실행 의사 확정, 예약·철회, 서비스 해지, 결과 확인처럼 고객 완결 목적 중심으로 잡아야 한다.
- 환불·위약금·혜택 회수·재가입 가능일은 하나의 영향 요약 카드로 묶되, 정책 상세에서는 산정 기준과 BSS/제휴 반영 기준을 분리해야 개발/QA가 테스트 조건을 만들 수 있다.
- 요구사항 Trace는 본문 품질과 별개로 상세 요구사항명 핵심어가 너무 멀어지면 자동 매칭이 약해진다. 정책 항목명은 요구사항 문구 복사가 아니라 판단축을 보존하는 짧은 명칭으로 잡는 편이 좋다.

## PM-18/PM-19 보완 인사이트
- 간소화 문서에는 보이지 않더라도 `process_details`와 `function_details`가 같은 문장으로 반복되면 Full 버전 확장 시 세부 장이 얕아진다. 프로세스명, 유즈케이스명, 관련 정책명을 반영해 진입 조건, 종료 조건, 처리 기준을 업무별로 다르게 작성한다.
- 기능의 세부 기능 구성은 단어를 잘라 나열하지 않고 `대상 식별`, `조건 판정`, `결과 구성`, `이력 저장`처럼 실제 하위 처리명으로 작성한다.
- Trace 커버리지는 정책 항목명을 요구사항 원문으로 복사해서 맞추지 않는다. 대신 누락된 판단축을 정책 항목으로 재구성하고, 정책 목록의 `items`에도 같은 항목명을 반영해 목록-상세 정합성을 유지한다.

## PM-20 나의 가입 정보 작성 인사이트
- 나의 가입 정보는 구독, 결합, 멤버십, 회선 변경, 증빙처럼 여러 인접 정책서 요구사항이 모이는 허브다. 실행 상세를 모두 끌어오면 범위가 흐려지므로 PM-20은 “현재 상태 표시, 가능 여부 판단, 다음 행동 연결, 기준 시점·이력”까지만 다루고 실제 실행 로직은 인접 정책서로 넘기는 경계가 필요하다.
- 범위가 넓은 허브형 문서는 유즈케이스를 항목별 조회 단위로 쪼개면 다시 프로세스가 1개씩 붙는 문제가 생긴다. 유즈케이스는 “통합 홈 확인”, “상세와 다음 행동 선택”, “혜택 만료 영향 확인”, “긴급·불일치 조치 선택”처럼 고객 완결 목적 중심으로 잡고, 항목별 차이는 프로세스·기능·정책에서 처리한다.
- Trace에서 누락되는 세부 요구사항은 대부분 “정책서 범위 밖 실행 업무”처럼 보이지만, 가입정보 관점의 표시·진입·권한 기준으로는 다뤄야 한다. 예를 들어 환불/취소/철회, 로밍 재신청, 보험 한도, 보호자 자녀 관리는 실행 상세가 아니라 가입정보에서의 가능 여부·진입 기준으로 보강한다.
- 주요 용어 장은 렌더러가 `용어 ID` 컬럼을 지원하므로 `TM-{업무코드}-NNN`을 spec 단계에서 반드시 부여한다. 용어명과 설명만 넣으면 HTML에서 ID가 빈 칸으로 보인다.
- `법정대리인/위임받은 대리인`처럼 복합 액터명을 만들지 않는다. 책임이 같으면 `대리 권한자`처럼 하나의 책임명으로 통합하고, 책임 차이는 권한 조건·정책 항목·상태 전이에서 분리한다.
- 자동 연장 설정처럼 인접 업무 실행 상세로 보이는 요구사항도 고객이 가입정보에서 직접 확인·변경할 수 있으면 별도 정책 항목과 기능 세부 구성으로 남겨야 한다.
- 화면 카드, 다음 행동, 상담 전환, 변경 가능 업무 구성을 바꾸는 시스템 유즈케이스는 `process_target=N`이어도 상태 전이 `usecase_ids`에 연결해야 한다.
## PM-21 회선 변경/관리 작성 인사이트
- 회선 변경/관리는 요구사항 수가 많고 정지, 해지, 분실, 계약자 변경, 해외·대리·사망자 처리까지 함께 들어온다. 유즈케이스는 개별 신청 버튼이나 절차 단계가 아니라 고객이 완결하려는 회선 관리 목적 단위로 묶고, 세부 차이는 프로세스와 정책 항목에서 분리하는 편이 안정적이다.
- 자동 Trace에서 누락된 항목은 대부분 본문에 개념은 있지만 “해지 총 부담금”, “채널 간 해지 기준”, “PUK”, “보상센터”, “임대 단말”처럼 세부 판단축 명칭이 정책 항목에 명확히 남지 않은 경우였다. 요구사항 문구를 그대로 복사하지 않되, 추적 가능한 판단축은 정책 항목명과 상세 기준에 명시해야 한다.
- 분실·도난은 단순 정지 업무가 아니라 고객 피해 방지, 후속 복구, 보험·임대폰·금융/로밍 차단 연계를 포함한다. 긴급 차단은 우선 처리하고 후속 서비스는 체크리스트와 상태 이력으로 연결해야 개발/QA 테스트 경계가 명확하다.
- 회선 관련 정책은 BSS 원장 반영 성공 전에는 고객 완료 상태로 표시하지 않는다는 원칙을 유지해야 한다. FO 표시 상태와 BSS 상태가 어긋나는 구간은 `처리 중`, `운영 검토 필요`, `반려·보완 필요` 상태로 분리하는 것이 좋다.
- 유즈케이스명에 `가능 여부 확인`, `후속 조치 확인`이 들어가면 절차 단계로 판정되기 쉽다. 회선 업무처럼 범위가 넓은 문서는 `업무 진입과 처리 경로 결정`, `신청 이력과 복구 행동 관리`처럼 고객 목적과 완료 상태가 드러나는 이름으로 잡는다.
- 정책 상세는 `policy_id` 아래 `items` 중첩 구조보다 정책 항목 단위 flat 구조가 Validator, Health Check, 추적표, 보완 루프에서 안정적이다.
- 정책 상세에서 `검토 필요`라는 표현은 금지 표현으로 잡힌다. 상태 표현이 필요하면 `운영 확인 필요`처럼 후속 주체와 완료 전 제한 기준이 드러나는 표현으로 쓴다.
## 상태 전이 렌더링 인사이트
[COMMON]
- 상태 전이표의 `처리 기준 및 후속 처리` 컬럼은 renderer가 `criteria` 필드를 읽는다. 작성 spec에서 `rule`만 채우면 검증을 통과해도 HTML에서는 빈 칸으로 보일 수 있으므로, 상태 전이 데이터는 반드시 `criteria`를 기준으로 작성한다.
- validator가 상태 전이 `criteria` 공란을 Critical Gate로 잡도록 보강했으므로, 다음 문서부터 상태 전이 기준 누락은 작성 단계에서 차단된다.

## PM-22 멤버십 카드 관리 작성 인사이트
- 요구사항 수가 적은 문서도 카드 상태, 제시 수단, 발급·재발급, 해지 유예, 대표 회선처럼 상태 전이가 많은 경우에는 유즈케이스를 절차별로 쪼개지 말고 고객 완결 목적별로 묶어야 한다.
- 멤버십 혜택과 카드 관리는 쉽게 섞인다. 혜택 상세 사용 조건은 인접 정책서로 두고, PM-22는 카드 사용 가능 상태와 카드 제시·복구 기준에 집중해야 범위가 안정된다.
- 해지 확정 후 복구 불가 삭제처럼 고객 영향이 큰 상태는 상태 코드, 전이 기준, 정책 상세에 모두 같은 용어로 남겨야 렌더링과 Health Check에서 누락으로 보이지 않는다.
- 시스템 유즈케이스가 `process_target=N`이어도 멤버십 자격 동기화, 카드 제시 결과 회신, 화면·후속 행동 구성처럼 고객 표시 상태와 복구 경로를 바꾸면 상태 전이 `usecase_ids`에 함께 연결한다.
- 시스템 액터도 `BSS/멤버십 연계 시스템`, `제휴처/외부 사용처`처럼 슬래시로 묶지 말고 `멤버십 연계 시스템`, `외부 사용처`처럼 책임명을 단순화한다. 세부 시스템 차이는 설명과 정책 항목에서 다룬다.

## PM-23 청구 및 수납 관리 작성 인사이트
- 청구·수납은 청구서 조회, 납부수단, 납부 실행, 미납 복구를 나누어 쓰면 문서가 흩어진다. 고객 입장에서는 금액 이해에서 수납 완료와 증빙 확인까지 하나의 여정이므로 유즈케이스를 과업 단위로 묶고 프로세스에서 분해하는 방식이 안정적이다.
- 결제 승인과 BSS 수납 반영은 반드시 분리해야 한다. 승인만 성공하고 원장 반영이 지연되는 상태를 별도 상태로 두지 않으면 고객에게 완료로 오안내될 수 있다.
- 통합 포인트는 독립 문서처럼 과하게 쓰기보다 청구·수납 문서에서는 자산 조회, 소멸 예정, 결제/납부 영향, 반영 지연 기준에 한정하는 편이 범위가 안정된다.


## 2026-05-09 PM-24 나의 데이터·통화 작성 인사이트

## PM-05 AI 검색 직접 품질 검수 인사이트

- direct_inspector: 복합 액터 0건, 상태 전이 기준 누락 0건, 프로세스 관련 기능·정책 누락 0건, 정책 그룹 items 문자열 0건을 확인했다.
- direct_health_check: 10개 영역과 필수 Gate를 직접 확인했고, 자동 Health Check 100점과 직접 판단 사이의 P1/P2 불일치는 없었다.

## PM-06 추천 직접 품질 검수 인사이트
- 추천 문서는 요구사항명이 바뀌거나 세부 요구사항이 추가되면 기존 Trace가 맞아 보여도 실제로는 판단축이 빠질 수 있다. `requirements_count_changed`가 있으면 최신 DB 기준으로 다시 조회하고, 자동 Trace `검토필요` 항목을 0건으로 낮출 때까지 정책 항목명과 기능 상세를 보강한다.

- direct_health_check: 10개 영역과 필수 Gate를 직접 확인했고, 자동 Health Check 100점과 직접 판단 사이의 P1/P2 불일치는 없었다.

## PM-11 상품상세/담기 직접 품질 검수 인사이트
- 상품상세/담기는 요구사항이 192건으로 확장되면서 운영자 화면 세부 요구가 많이 섞인다. 이를 화면/필드 단위로 쓰면 정책서가 BO 기능 명세가 되므로 입력 검증, 삭제 보호, 일괄 변경 부분 실패, 참조 데이터 확인, 변경 이력 같은 정책 판단축으로 묶는다.
- Product Catalog와 상품 라이프사이클 요구사항은 상품 상세의 표시 기준뿐 아니라 담기 가능성 판정과 상태 전이에 직접 연결해야 한다. 원장 기준과 채널 보조 문구가 충돌하면 원장을 우선한다.
- 후기·Q&A, 키워드·카테고리, 디바이스·액세서리, 외부채널 설정은 별도 액터로 쪼개지 않고 운영자 유즈케이스 아래 프로세스·기능·정책으로 내려야 액터 과분화가 재발하지 않는다.
- 담기 CTA는 담기/바로결제/구독하기/바로가입의 다음 단계가 서로 다르므로 정책 항목에서 고객이 이동할 목적지와 재검증 기준을 구분한다.
- 요구사항 수가 큰 문서는 자동 Trace 100%보다 수동 Health Check가 더 중요하다. 본문이 192개 요구사항명을 모두 노출하지 않아도, 11개 부모 요구축과 정책 판단축이 완결적으로 재구성되었는지 확인한다.

## PM-12 카트/장바구니 직접 품질 검수 인사이트
- 장바구니는 임시 저장함이 아니라 주문 전 구매안 검증 공간이다. 담기 후 동선, 탭별 처리 단위, 수량·금액 산정, 조합 검증, 모회선 인증, 만료·삭제가 모두 주문 전환 가능성을 바꾸므로 기능 설명이 아니라 정책 항목으로 남긴다.
- 요구사항 수가 바뀐 문서는 기존 Topic Learning과 Core Design의 요구사항 수를 반드시 갱신한다. PM-12처럼 27건 기준 초안에 43건 DB가 붙으면 자동 Trace가 통과해도 신규 요구사항 3~5건이 암시만 되고 빠질 수 있다.
- 액세서리 수량 조절, 함께쓰기 모회선 인증, 담기 후 쇼핑 계속/장바구니 이동처럼 화면 기능처럼 보이는 요구도 고객 후속 행동과 상태 전이를 바꾸면 프로세스·기능·정책에 모두 연결한다.
- 장바구니 문서는 “개인고객/그룹고객/로그인/비로그인”을 액터로 분리하지 않는다. 고객 상태와 권한 조건은 상태 전이와 정책 항목으로 내리고, 책임 주체는 고객, 운영자, 상담사, 채널 업무 시스템, BSS/연계 시스템 수준으로 유지한다.
- 자동 Trace가 100%가 되어도 관련 산출물 후보가 너무 넓게 잡히는 경우가 있다. 핵심 요구사항은 정책 항목명 또는 기능명에 판단축이 직접 드러나는지 Codex가 직접 확인한다.

## PM-13 할인/시뮬레이션 직접 품질 검수 인사이트
- 할인/시뮬레이션은 고객 비교 흐름과 BO 운영 요구가 섞여 있다. 공통지원금 트랙, 별도지원금 가입유형, 추가지원금 등록·삭제, 프론트 연동, Bar 문구 검증은 별도 고객 유즈케이스가 아니라 운영자 프로세스·기능·정책으로 내려야 계위가 유지된다.
- 지원금 요구사항은 단순 금액 표시가 아니다. 공통지원금 A/B 트랙, A only/A+B 월 납부금액 계산, 별도지원금 가입유형, 추가지원금 15% 합산, 변동 우선순위를 정책 항목으로 분리해야 개발/QA가 계산 케이스를 만들 수 있다.
- 요금제 비교 수 확장, 구매조건 선택값 유지, 연간 절감액 노출은 UI 요구처럼 보이지만 고객의 비교 판단을 바꾸므로 `비교 화면 확장·선택값 유지 정책` 같은 정책 그룹으로 남긴다.
- 삭제 관련 BO 요구는 “운영자가 삭제한다”가 아니라 참조 데이터 존재, 사용여부 Y, 배포 상태, 중복 코드 조합, 삭제 전 확인 절차를 기준으로 차단·허용·사용중지를 판정해야 한다.
- 자동 Trace가 0건 수동 검토로 끝나도 중간 신뢰도 항목이 남으면 직접 Inspector가 정책 항목명·기능명에 판단축이 드러나는지 확인한다. 필요한 경우 요구사항 원문 복사가 아니라 정책 판단 언어로 한 번 더 보강한다.

## PM-14 주문/계약/가입 직접 품질 검수 인사이트
- 주문/계약/가입은 요구사항 수가 326건으로 많고, 화면 통합·배송·인증·요금제·셀프동의·BSS 운영 요구가 한 문서에 섞인다. 자동 Trace가 약한 항목은 누락으로 단정하기보다 어떤 정책 판단축에 흡수되어야 하는지 먼저 재분류한다.
- 배송 조건 선확정, 설치/방문 일정, 수령인·연락처, 바로픽업·바로도착은 배송 상세 문서로 넘기더라도 주문 제출 가능성과 고객 후속 행동을 바꾸므로 주문 정책 항목에 최소 기준을 남긴다.
- 유선 주문서 Step, 셀프동의 URL Step, 액세서리 주문서 통합, 인앱 모바일웹 통합은 화면 개선 요구처럼 보여도 주문 Core 단계와 채널 통합 기준에 해당한다. 별도 유즈케이스로 쪼개지 말고 주문 Core 운영 정책과 진입/유형 정책으로 내려야 한다.
- 요금제 유지 가능, 요금제 변경 가능, 요금제 리스트 구성, 전체보기 팝업, 정렬 기준은 전시 정책으로만 보면 누락된다. 주문 단계에서 고객 선택 가능 여부와 최종 제출 가능성을 바꾸므로 운영 정책과 상품·가입 방식 기능에 연결한다.
- 번호이동 사전동의, 인증수단 default, 인증번호 오입력/재요청, SWING 개통 검토는 인증 기능 설명이 아니라 상태 전이와 제출 가능성을 바꾸는 정책 기준이다. 실패·미완료·초과·보류를 상태와 복구 경로에 함께 연결한다.
- 운영자 화면 요구는 본문에 화면 필드로 쓰지 않는다. 필수 확인 사항, 안내 팝업, T기프트 콘텐츠, 용어가이드, 엑셀 업로드 팝업, 삭제 연관데이터 검증은 운영 승인, 삭제 차단, 버전 이력, 영향 대상 표시 기준으로 재구성한다.

## PM-15 선물주문 직접 품질 검수 인사이트
- 선물주문은 발신 고객과 수신 고객이 서로 다른 책임을 갖는 양면 고객 업무다. 액터는 발신 고객, 수신 고객, 대리 권한자처럼 책임 주체 기준으로 두고, 선물 유형·고객 상태·회원 여부는 정책 조건으로 내린다.
- `선물 자격·상태 판정 회신` 같은 시스템 유즈케이스는 process_target=N이어도 상태 전이 판단에 반드시 연결해야 한다. 발신 가능 여부, 수신 자격, 수락 가능성은 고객 상태를 바꾸는 판정이므로 state_transitions.usecase_ids에서 빠지면 Stage Gate가 끊긴다.
- 선물 문서는 수락 완료 이후가 중요하다. 수락 후 선물함, 쿠폰함, 이용권함, 데이터 잔량, 단말·서비스 신청 화면 중 어디로 귀속되는지와 귀속 실패 시 등록 보류·재처리·상담 기준을 정책 항목으로 남긴다.
- `답례 선물하기`, `구독 중 상품의 차기 월 자동 적용`, `보유 혜택 전송`은 별도 유즈케이스로 과분화하지 않는다. 선물 수락 후 행동, 중복 보유 사전 안내, 전송 가능 혜택 범위의 정책 항목으로 내려야 전체 계위가 유지된다.
- 관련 정책 ID가 중복 저장되면 HTML은 정상처럼 보여도 Full 버전 확장, Health Check, 보완 루프에서 연결 신뢰도가 떨어진다. 프로세스의 related_policies는 순수 ID를 중복 없이 유지한다.

## PM-16 상품변경 직접 품질 검수 인사이트
- 상품변경은 요금제, 구독, 옵션, 일시정지, 예약, 무료충전, 패키지 교체 요구가 한 문서에 섞인다. 이를 세부 요구사항별 유즈케이스로 쪼개지 말고 `변경 가능성과 영향 비교`, `상품 변경 신청`, `예약 변경`, `구독·옵션 변경`, `진행 조회와 실패 복구` 같은 완결 과업으로 묶는다.
- `변경 조건과 영향 산정`처럼 시스템이 수행하는 판정 유즈케이스는 process_target=N이어도 상태 전이의 원인이 된다. 변경 가능, 변경 제한, 영향 확인 필요, 예약 충돌, 처리 실패처럼 고객 표시 상태를 바꾸는 전이에 usecase_ids로 반드시 연결한다.
- 대체 요금제 제안, 통합 변경 예상 절감액, 로밍 무료충전 신청 경로, 패키지 상품 교체 금액 비교처럼 요구사항명이 명확한 항목은 기능 설명에 암시하지 말고 정책 항목명과 판단 기준으로 직접 드러낸다.
- 상품변경 기능 상세는 반복 템플릿을 쓰기 쉽다. `변경 대상 조회`, `셀프 가능성 판정`, `영향 비교`, `예약 충돌 검증`, `구독·옵션 상태 조회`처럼 기능별 처리 역량과 산출물이 달라지도록 세부 기능 구성을 분리한다.
- 요구사항 건수가 업데이트되면 Topic Learning, 문서 히스토리, final_check, Trace Report 숫자를 모두 함께 갱신한다. JSON 본문만 최신화하고 HTML/Trace를 재생성하지 않으면 서비스 화면에서는 예전 기준처럼 보인다.

## PM-17 결제 직접 품질 검수 인사이트
- 결제는 고객 결제 흐름과 운영자 프로모션·제휴카드 관리 요구가 같이 들어온다. 제휴카드 등록, 약관, 노출상태, 삭제, 상품 매핑을 별도 고객 유즈케이스로 만들지 말고 `결제·혜택 기준 운영`의 프로세스·기능·정책으로 내려야 계위가 유지된다.
- 비로그인 쿠폰 대안, 제휴카드 탐색 위치, 약관 버전, 노출순서처럼 상세 요구사항명이 명확한 항목은 기능 설명에 암시하면 Trace와 직접 검수에서 빠진다. 요구사항 원문 복사가 아니라 정책 판단축 명칭으로 정책 항목에 남긴다.
- 결제 가능성·혜택 산정 같은 시스템 유즈케이스는 `process_target=N`이어도 금액 확인, 결제수단 선택, 혜택 조합, 결제 요청, 중복 차단 전이의 원인이 된다. 상태 전이 `usecase_ids`에 고객 유즈케이스와 함께 연결해야 BSS/시스템 판정이 문서 밖으로 빠지지 않는다.
- 요구사항 수가 85건에서 104건으로 바뀐 것처럼 데이터가 업데이트되면 Topic Learning, 문서 히스토리, final_check, requirements_mapping_summary, Trace Report, HTML/BPMN을 모두 재생성한다.

## PM-19 해지/환불/취소 직접 품질 검수 인사이트
- 해지/환불/취소는 “가능 여부”, “영향도”, “요청 접수” 같은 상위 정책명만으로는 상세 요구사항이 묻히기 쉽다. `당일가입 즉시취소`, `가입 익일부터 해지를 요청하는 경우 일반해지 처리`, `비회원 간편해지`, `알림연계 원터치해지`, `외부 제휴서비스 원터치 해지`, `즉시 해지 지원`처럼 고객 처리 경로를 바꾸는 요구사항은 정책 항목명 또는 기능 세부 구성에 직접 남긴다.
- 요금·위약금·결합 영향 시뮬레이션은 단순 사전 고지 문장으로 합치지 않는다. 예상 청구 금액, 잔여 기간·할인 반환금, 결합 할인 종료 영향은 각각 개발/QA 계산 케이스가 되므로 정책 항목을 분리한다.
- 신규가입/번호이동 당일 즉시취소와 가입 익일 일반해지는 같은 “해지 가능” 축 안에서도 고객 결과가 달라진다. 즉시취소, 일반해지, 위약금/반환금/혜택 회수, 재가입 제한의 전환 기준을 독립 정책 항목으로 검수한다.
- 시스템/BSS/결제/외부 제휴 판정 유즈케이스는 `process_target=N`이어도 고객 상태, 완료 표시, 복구 경로에 영향을 준다. 상태 전이 `usecase_ids`에 연결하고, 관련 기능·정책에도 순수 ID 기준으로 연결한다.
- 자동 Trace가 100%라도 직접 Inspector가 정책 항목명과 본문을 다시 본다. 특히 “해지 가능 정책”처럼 넓은 항목 아래에 세부 요구사항을 숨기면 다음 보완 루프와 Health Check에서 다시 누락될 수 있다.

## 2026-05-10 PM-23 청구 및 수납 관리 보완 인사이트

## 2026-05-10 PM-24 나의 데이터·통화 보완 인사이트
- 최신 요구사항 DB로 바뀐 뒤에는 기존 요구사항 수와 Trace 수가 맞아도 안심하지 않는다. PM-24는 69건 기준 초안에 70건 DB가 붙으면서 데이터 선물하기, 충전하기, 자동선물 조회, 셰어링 USIM, 경량 조회, 주요 사용 패턴 같은 상세 요구사항명이 기능·정책명에 직접 드러나지 않았다.
- 상세 요구사항명은 원문 복사 대상이 아니라 누락 방지 신호다. 핵심 명칭은 프로세스명, 기능명, 정책 항목명 중 하나에 정책 판단 언어로 재구성해 노출하고, 상세 요구사항 설명은 조건·예외·후속 처리 기준으로 풀어쓴다.
- `보호자/가족 대표`, `BSS/사용량 연계 시스템`, `외부 제휴/로밍 시스템`처럼 `/`로 묶은 액터명은 재발시키지 않는다. 액터는 단일 책임명으로 두고, 역할 차이는 유즈케이스 설명·기능·정책 조건으로 내린다.
- 시스템 유즈케이스가 process_target=N이어도 고객 표시 상태, 알림, 딥링크, BSS·외부 결과 반영에 영향을 주면 상태 전이 `usecase_ids`에 연결한다. 그렇지 않으면 작성은 통과해도 후속 Inspector가 시스템 판정 누락으로 판단한다.
- 정책 목록의 `items`는 항상 `{id, name}` 객체로 유지한다. 문자열 항목은 HTML에는 보일 수 있어도 Full 확장, Health Check, 보완 루프에서 정책 상세와 연결이 끊길 수 있다.

## 2026-05-10 PM-25 상품·서비스 혜택 이용/공유 보완 인사이트
- 혜택 문서에서 `미구독`, `로그아웃`, `빈 화면`, `퀵메뉴` 요구가 나오면 고객 상태를 액터나 별도 유즈케이스로 늘리지 않는다. 혜택 탐색 진입 프로세스, 첫 이용 가이드 기능, 빈 상태 대체 행동 정책으로 내려야 계위가 유지된다.
- `BSS/혜택 연계 시스템`처럼 묶인 시스템 액터는 `혜택 연계 시스템`처럼 단일 책임명으로 정리한다. BSS 원장, 제휴사 회신, 채널 화면 구성의 책임 차이는 상태 전이와 정책 항목에서 FO·BSS·외부 시스템 경계로 풀어쓴다.
- 기능 설명이 “혜택 상태, 고객 조건, 제휴 응답을 기준으로…”처럼 반복되면 세부 기능 구성도 함께 약해진다. 기능별 산출물(체크리스트, 퀵메뉴, 딥링크, 손실 가치, SLA, 오남용 차단)을 각각 다른 하위 처리명으로 작성한다.
- Health Check 90점대라도 `핵심 정책 질문`, `고객 표시 상태 vs 내부 상태`, `FO/BSS/외부 책임 경계`, `개발/QA 전환성`이 약하면 개요 원칙과 최종 점검 기준에 짧게 보강한다. 점수 보정이 아니라 실제 검토자가 확인할 판단축을 추가하는 방식이어야 한다.

## 2026-05-10 PM-26 통합 쿠폰/이용권함 보완 인사이트
- 요구사항 DB가 19건에서 31건으로 바뀐 문서는 기존 초안을 유지하지 말고 Topic Learning, Core Design, 상태, 프로세스, 기능, 정책, Trace를 모두 새 기준으로 재동기화한다.

- `BSS/혜택 원장`처럼 slash로 축약한 표현은 액터명뿐 아니라 본문에서도 혼란을 만든다. 문서 표현은 `BSS와 혜택 원장`, 액터명은 `혜택 원장 시스템`처럼 단일 책임명으로 정리한다.

- 쿠폰/이용권함은 조회함이 아니라 “혜택 자산의 상태·수령·사용·만료·복구·공유를 한곳에서 관리하는 업무”다. 상태와 정책 항목에는 고객 표시 상태, 원장 우선 기준, 제휴 실패 복구, 이력 보관 기준을 함께 남긴다.

## 2026-05-10 PM-27 회원 가입/탈퇴 보완 인사이트
- 기존 문서가 Health Check는 우수여도 Stage Critical Gate에서 시스템 유즈케이스 상태 전이 연결 누락을 잡을 수 있다. 인증기관, 회원 원장 시스템, 채널 업무 시스템처럼 process_target=N인 유즈케이스도 고객 상태·알림·이력·원장 반영을 바꾸면 state_transitions.usecase_ids에 연결한다.
- `BSS·회원 원장`처럼 `·`로 묶은 액터명은 단일 책임 주체 원칙을 흐린다. 액터명은 `회원 원장 시스템`으로 두고 BSS 판정, 원장 저장, 채널 상태 반영은 정책 항목에서 책임 경계를 설명한다.

- 정책 그룹 items가 문자열이면 정책 상세와의 계약이 약해진다. 모든 정책 목록 항목은 `{id, name}`으로 유지하고, 정책 상세의 policy_id와 1:1로 추적 가능해야 한다.
- Trace 리포트 생성기는 trace_matrix의 `item_id/item_name`도 읽어야 한다. 기존 `requirement_id/detail_name`만 보면 직접 작성 산출물의 매핑표를 일부 무시해 수동 검토 필요가 과다 발생한다.

## 2026-05-10 PM-28 회원정보 조회/변경 보완 인사이트
- 회원정보 조회/변경은 가입·약관 축으로 빨려 들어가기 쉽다. 프로필, 주소록, 연락처, 환불 계좌, 대표값, 관계·그룹, 증빙·복구처럼 회원정보 전용 판단축을 먼저 만들고 유즈케이스와 프로세스를 그 축에 맞춘다.
- `전자공문서 불러오기 제출 지원`처럼 보완 제출의 한 방식으로 보이는 요구도 고객 제출 경로, 접수 이력, 실패 안내를 바꾸면 기능명 또는 정책 항목명에 직접 드러나야 한다.
- 자동 Trace 100% 이전에 1건이라도 수동 검토가 남으면 “표현상 매칭 부족”으로 넘기지 않는다. 요구사항의 핵심 명사가 프로세스, 기능, 정책 항목 중 어디에 정책 판단 언어로 반영됐는지 직접 확인한다.
- 프로세스 related_functions/related_policies와 정책 그룹 items는 렌더링 결과가 아니라 내부 계약이다. HTML이 정상이어도 순수 ID와 `{id, name}` 구조가 아니면 Full 확장, Health Check, 보완 루프에서 재발한다.
- `BSS·연계`, `BSS/연계`처럼 책임 경계를 묶은 본문 표현은 단일 책임 주체 원칙과 충돌한다. 문장에서는 `BSS와 연계 시스템`, 액터명에서는 단일 책임명으로 풀어 쓴다.

## 2026-05-10 PM-29 통합 알림 보완 인사이트
- 통합 알림에서 `인증기관`은 과분화된 액터가 되기 쉽다. OS 푸시 권한, 설정 앱 이동, 권한 거부 결과처럼 단말 권한 책임은 `OS 권한 시스템`으로 두고, 본인확인·계정 인증은 다른 정책서 또는 관련 기능의 조건으로 내려야 한다.
- 알림 상태는 범용 `업무 시작 전/처리 가능/처리 완료`로 두면 정책 판단이 약해진다. `알림 생성 전`, `알림 생성됨`, `발송 대기`, `발송 완료`, `수신 제한`, `대체 확인 필요`, `확인·처리 완료`, `만료·무효`처럼 고객 표시와 후속 행동이 달라지는 상태로 작성한다.
- 수신 설정 요구사항은 단순 설정 화면이 아니라 서비스 수신 동의, OS 권한, 채널별 연락처, 필수·선택 알림, 빈도 제한, 조용한 시간, 대체 확인 경로의 조합 정책이다. 정책 상세에는 이 조합의 허용·제한 기준이 직접 드러나야 한다.
- 알림함 요구사항은 “목록 조회”로 축소하지 않는다. 행동 필요/참고성 분리, 카테고리 필터, 동일 사건 타임라인 병합, 읽음·처리 완료·무효 동기화까지 고객 후속 행동 기준으로 풀어 쓴다.
- 정책 상세가 요구사항명만 바꾼 반복 템플릿처럼 보이면 Health Check 점수가 높아도 직접 보완한다. PM-29는 RQCOV 정책 항목 28건을 요구사항별 알림 판단 기준으로 재작성했다.

## 2026-05-10 PM-30 통합 약관 보완 인사이트
- 통합 약관은 단순 약관 목록이나 체크박스 정책이 아니다. 약관 권리, 필수·선택 동의, 개인정보·제3자 고지, 철회·재동의, 버전·증적, 운영 승인·배포 기준을 하나의 고객 권리 경험으로 설계한다.
- 상태는 `업무 시작 전/처리 가능/처리 완료`처럼 범용으로 두지 않는다. `약관 확인 전`, `약관 열람 중`, `필수 동의 필요`, `동의 완료`, `선택 동의 거부·철회`, `재동의 필요`, `예외 동의 확인 중`, `증적 보관 완료`처럼 고객 표시와 후속 행동이 달라지는 상태로 둔다.
- 필수 동의와 선택 동의는 정책 항목에서 반드시 분리한다. 선택 동의 거부는 핵심 서비스 제한 사유가 아니며, 제한 범위는 개인화, 마케팅, 선택 알림, 부가 기능으로 한정해 안내한다.
- 운영 요구사항은 화면 어드민 기능으로 쓰지 않는다. 약관 등록·수정·승인·삭제 제한, 종료일 자동 전환, 참조 무결성, 접근 권한, 외부채널 일원화는 운영 정책 판단 기준으로 재구성한다.
- Health Check가 `필요 시` 같은 표현을 빈 정책 마커로 잡을 수 있다. 실제 의미가 명확하더라도 정책서 본문에서는 `원문 확인이 필요한 경우`처럼 조건과 행동이 드러나는 문장으로 쓴다.
