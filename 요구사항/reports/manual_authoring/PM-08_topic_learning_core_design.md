# PM-08 이벤트/미션 프로그램 Topic Learning & Core Design

## 1. 작성 목적
이벤트/미션 프로그램 정책서는 단순 프로모션 페이지나 배너 운영 기준이 아니라 고객이 이벤트와 미션을 발견하고, 참여 가능 여부를 이해하고, 참여·미션 수행·보상 수령·이력 확인까지 완료하는 참여형 성장 경험의 정책 기준을 정의한다.
운영 측면에서는 프로그램 등록, 조건·제한 설정, 검수·승인·배포, 당첨·보상, 부정 참여, 지급 실패 보정, 성과 리포트까지 하나의 운영 루프로 연결한다.

## 2. 요구사항 해석 기준
66개 상세 요구사항은 `상세 요구사항명`과 `상세 요구사항 설명`을 기준으로 읽었다.
테마/기획전, 이벤트/멤버십 혜택 허브, 상세 핵심 정보, 참여 가능 카드, 입력 최소화, 단계형 UX, 참여 완료·실패, 참여 이력, 당첨·보상, 공유, 운영자 등록·검수·중지, A/B, 라벨/UI, 노출 정책, AI 번역, 특가·한정 수량, 유형별 템플릿, 부정 참여, 리포트, 미션 허브·진행·이어하기·보정·팀 미션을 모두 고객 과업과 운영 프로세스로 재구성했다.
추가 요구사항인 기능 모듈화, 파라미터 명명 규칙, 대상자 Excel 업로드, 공통 알럿, 랜덤 포인트, 혜택 순번 알림, 결제 완료 후 보상 루프, 일일 앵커, 체류형 게이미피케이션, 확정형 보상 목표, 선착순 트래픽 안정화, 이벤트·쿠폰 현황 대시보드는 별도 유즈케이스로 과분화하지 않고 운영·보상·미션·리포트 정책으로 흡수했다.
요구사항 문구는 그대로 복사하지 않고 상태, 프로세스, 기능, 정책 판단 기준으로 전환했다.

## 3. Core Design Map
- 액터: 고객, 이벤트/미션 운영자, 마케팅 운영자, 채널 업무 시스템, 보상/혜택 시스템, 알림센터, BSS/연계 시스템.
- 핵심 유즈케이스: 고객의 탐색·참여 가능 판단, 이벤트 참여·응모 완료, 미션 진행·이어하기, 결과·보상·이력 확인, 운영자의 프로그램 기준 등록·검수·배포, 보상·부정 참여·정산 관리, 성과 리포트·최적화 운영.
- 상태 흐름: 진입 가능 -> 상세 확인 중 -> 참여 진행 중 -> 참여 완료/응모 완료 -> 추첨·검증 대기 -> 당첨/보상 지급 예정 -> 보상 지급 완료를 정상 흐름으로 둔다. 조건 미충족, 참여 실패, 지급 실패, 만료/사용불가, 부정 참여 의심, 운영 중지는 예외·제한 흐름으로 둔다.
- 프로세스: 고객 탐색, 조건 확인, 입력·동의, 참여/응모, 미션 수행, 보상 수령, 운영 등록·검수·배포, 보상·부정 참여 관리, 리포트·실험 최적화로 구성한다.
- 기능: 허브/목록, 상세정보, 조건 판정, 자동채움, 동의, 참여접수, 미션상태, 보상지급, 이력조회, 실패보정, 공유/딥링크, 운영 템플릿, 기능 모듈, 파라미터 검증, 대상자 업로드, 승인배포, 부정탐지, 리포트·대시보드, 알림, 실험/SEO, 선착순 안정화, 팀 미션을 처리 역량으로 정의한다.
- 정책: 프로그램 범위, 허브/목록, 상세정보, 참여자격, 동의/입력, 중복제한, 미션진행, 시즌/이어하기, 당첨, 보상, 결제 후 보상 루프, 실패보정, 이력, 운영등록, 승인배포, 노출 정책, 부정참여, 알림공유, 리포트, 노출최적화, 선착순 트래픽 안정화 기준을 둔다.

## 4. 작성 금지 및 경계
- 전시/관리 기능, 상품 목록, AI 검색, 데이터 트래킹 체계, 외부 쿠폰의 전체 절차를 대신 작성하지 않는다.
- 쿠폰·포인트·멤버십 혜택의 상세 사용 정책은 인접 정책서에 두고, 본 문서에서는 이벤트/미션 보상 지급·수령·만료 연결 기준만 작성한다.
- API 필드, DB 컬럼, 운영자 화면 상세, 지급 배치 설계는 제외한다.
- SEO/GEO, AI 번역, A/B 테스트는 콘텐츠 운영 절차가 아니라 이벤트/미션 참여 경험에 필요한 적용 기준과 승인·노출 기준으로만 작성한다.

## 5. Inspector 메모
- 유즈케이스가 상세 단계로 쪼개지면 프로세스 1:1 문제가 재발하므로 고객의 완결 업무 목표 기준으로 검수한다.
- 보상, 당첨, 미션 완료 인정, 부정 참여, 지급 실패 보정, 이력 기준이 빠지면 P1로 본다.
- 고객에게 보이는 조건·제한·다음 행동이 불명확하면 정책 구체성 부족으로 보완한다.
- 운영 보조 요구사항은 별도 유즈케이스로 만들지 않고 등록·검수·배포, 보상·당첨·부정 참여 관리, 성과 리포트·최적화 운영의 기능·정책으로 연결한다.
- 선착순·한정 수량·랜덤 보상은 고객 상태, 재고 잠금, 실패 복구, 알림, 이력까지 함께 확인한다.
