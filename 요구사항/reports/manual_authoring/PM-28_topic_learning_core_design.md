# PM-28 회원정보 조회/변경 Topic Learning & Core Design

## 1. 작성 목표

- 회원정보 조회/변경은 고객이 본인 정보, 연락처, 주소, 계좌, 관계 정보를 안전하게 확인하고 필요한 변경을 완료하는 업무로 정의한다.
- 단순 가입 정보 표시가 아니라 대표값·보조값·출처·기준 시점·마스킹·재인증·변경 이력까지 고객이 이해할 수 있게 구성한다.
- 변경 실패, 정보 불일치, 증빙 부족, 권한 부족 상황에서는 셀프 복구 또는 상담 전환 기준을 명확히 남긴다.

## 2. 요구사항 해석 기준

- 기준 자료는 `reports/evidence/requirements.db`의 `normalized_depth4=회원정보조회변경` 75건을 사용했다.
- 요구사항 ID보다 상세 요구사항명과 상세 요구사항 설명을 우선 해석했다.
- `조회`, `목록`, `용도`, `대표값`, `출처`는 조회 범위·정합화 축으로 분류했다.
- `변경`, `수정`, `등록`, `해지`, `삭제`, `대표 전환`은 회원정보 변경 축으로 분류했다.
- `대리`, `법정`, `그룹`, `가족`, `권한`, `관계`는 권한·관계 정보 관리 축으로 분류했다.
- `인증`, `본인확인`, `증빙`, `검증`은 검증·재인증 축으로 분류했다.
- `이력`, `복구`, `보정`, `불일치`, `실패`, `상담`은 변경 이력·복구 축으로 분류했다.

## 3. Core Design Map

| 계층 | 설계 결과 |
|---|---|
| 핵심 액터 | 고객, 운영자, BSS, 인증기관, 연계 시스템, 채널 업무 시스템 |
| 고객 유즈케이스 | 회원정보 통합 조회·이해, 회원정보 변경·검증 처리, 회원정보 정정·복구 관리 |
| 운영 유즈케이스 | 회원정보 운영 기준 및 품질 관리 |
| 시스템 유즈케이스 | 고객·업무 조건 판정, 인증 결과 제공, 연계 결과 제공, 상태·이력 처리 |
| 핵심 프로세스 | 조회 대상 확인, 권한·마스킹 판정, 대표값·출처 구성, 변경 대상 선택, 재인증·증빙 확인, 변경 접수, BSS 반영, 이력 저장, 정정·복구 |
| 핵심 정책 | 접근·권한, 고객 상태 조회, 대상 정보 노출, 입력값 검증, 가능 여부 검증, 인증·동의, BSS 연계 판정, 처리 결과·이력, 개인정보·로그 보호 |

## 4. 품질 보완 메모

- 최초 품질 점검에서 회원정보 주제가 `가입·신청`, `약관·동의` 같은 일반 축으로 분류되는 위험을 확인했다.
- 회원정보 전용 요구사항 축과 프로세스 템플릿을 추가해 가입/약관 축 유입을 차단했다.
- BPMN 업무 흐름도에서 프로세스 ID와 프로세스명이 한 라벨로 묶이며 말줄임표가 생기는 문제를 보완했다.
- 최종 산출물은 빈 표 셀, 금지 표현, 말줄임표, 잘못된 가입/약관 축이 없는지 확인했다.

## 5. 최종 직접 검수 및 보완 결과

- 최신 요구사항 DB 기준 75건을 기준으로 재검토했고, 상세 요구사항 Trace는 75/75건 자동 매칭으로 확인했다.
- `전자공문서 불러오기 제출 지원`은 보완 제출 기능과 정책에 암시만 되어 있어 프로세스, 기능, 정책 항목에 명시적으로 보강했다.
- 프로세스의 관련 기능·관련 정책은 내부 JSON에서 순수 ID만 보관하도록 정리했다.
- 정책 목록의 정책 항목은 문자열이 아니라 `{id, name}` 객체로 정리해 정책 상세와 추적 가능하게 맞췄다.
- 본문 표현 중 `BSS·연계 결과`처럼 책임 경계를 묶어 보일 수 있는 표현은 `BSS와 연계 결과`로 풀어 썼다.
- 최종 검증 결과 `validate_policy_spec`, `validate_stage_critical`, 빈 셀·금지 표현·잘못된 관련 유즈케이스 컬럼 점검을 모두 통과했다.
