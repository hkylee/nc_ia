# PM-01 / PM-02 샘플 비교 및 보완 기록

## 비교 기준

- 비교 샘플: `input/samples/NC_정책서_간소화_v1.0_확정본.html`
- 보조 비교 샘플: `input/samples/NC_AI검색_정책서_간소화_v1.5.html`
- 비교 관점: 장 구성, 최종 점검 기준 밀도, 정책 상세 문장 구조, 세부 기능 구성 표현, 요구사항 추적성

## 발견 사항

1. PM-01의 최종 점검 기준이 단문 목록 형태라 샘플 대비 제출 전 점검 기준의 판단력이 약했다.
2. PM-02는 스펙에 `{item, criteria}` 구조를 사용했지만 기존 HTML에는 dict 문자열이 그대로 노출되어 가독성과 신뢰성이 떨어졌다.
3. PM-01/PM-02 모두 기능의 세부 기능 구성은 이미 짧은 하위 처리명으로 보정되었고, 샘플의 기능 표 스타일과 부합했다.
4. PM-01/PM-02의 Evidence Gap은 단순 설명에 가까워 결정 주체, 기한, 정책 처리 방식이 샘플 수준보다 약했다.

## 반영 사항

1. PM-01 최종 점검 기준을 `item + criteria` 구조로 전환했다.
2. PM-01 최종 점검 기준을 범위 정합성, 공통 고객 경험, 맥락 유지, BSS·외부 연계, 운영 품질 Gate, 연결성, 정책 구체성, 요구사항 확산 가능성으로 보강했다.
3. PM-02 최종 점검 기준을 샘플처럼 각 항목별 판단 기준이 2문장 이상 드러나도록 보강했다.
4. PM-01/PM-02 Evidence Gap에 결정 주체, 결정 기한, 정책 처리 방식을 추가했다.
5. 렌더러가 구조화된 최종 점검 기준을 title/body로 표시하는지 테스트를 추가했다.

## Agent 개선 인사이트

- Final Check는 문자열 목록보다 `{item, criteria}` 구조가 샘플과 더 잘 맞고, 검수 기준 재사용성도 높다.
- 기능 세부 구성은 정책값이나 설명문이 아니라 짧은 하위 처리명으로 제한하는 현재 Gate를 유지해야 한다.
- Evidence Gap은 단순 미정 목록이 아니라 결정 주체, 결정 기한, 임시 정책 처리 방식까지 포함해야 후속 설계/QA가 움직일 수 있다.
