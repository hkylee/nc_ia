# TK Reference HTML 작성 지침

## 목적

TK PDF를 단순 변환하지 않고, 정책서 작성자가 과제의 전략적 의미와 정책 판단 축을 빠르게 이해할 수 있는 설명형 HTML로 다시 작성한다.

## 작성 원칙

- PDF의 과제 목적, 문제 정의, As-Is, 지향점, 기대 효과, KPI, 주요 프로세스와 기능은 빠짐없이 읽고 반영한다.
- 문장을 그대로 옮기기보다 고객 과업, 운영 판단, 정책서 전환 기준으로 재구성한다.
- 벤치마킹, 채널 방향성, VoC, IA 분석, AI 경험 검토 지식은 보강 근거로만 활용한다.
- 새 수치 목표나 PDF에 없는 확정 사실은 만들지 않는다.
- `원문`, `연관 문서`, `문서번호`, `원문 분량`, `추출 텍스트`, `문서 링크`, `연관 업무` 같은 작업 보조 표현은 화면에 노출하지 않는다.
- 문장은 마침표 뒤에 `<br/>`로 줄바꿈한다.

## 진행 단계

- 1차 작성은 12개 TK 문서를 모두 같은 품질 기준으로 설명형 HTML로 완성하는 것을 목표로 한다.
- 1차 작성 중에는 각 문서의 PDF 내용과 명확한 정책 판단 축을 우선 반영한다.
- 12개 문서가 모두 작성된 뒤 2차 보강 라운드를 수행한다.
- 2차 보강에서는 12개 문서를 함께 보며 벤치마킹, 채널 방향성, VoC, IA 분석, 정책서 작성 지식, 주제 간 의존성을 반영해 내용 밀도와 연결성을 보강한다.
- 2차 보강 전에는 개별 문서에 다른 TK 문서의 판단을 과도하게 선반영하지 않는다.

## 표준 구성

1. Hero
   - 과제명, 한 줄 정의, 작성자, 업데이트일, 4개 전략 기둥을 표시한다.
2. 전략 전환 맵
   - `As-Is 문제 → 전략 전환 → To-Be 결과` 3단 구조로 과제의 변화 방향을 설명한다.
   - 하단에는 고객, 핵심 처리 체계, 데이터/연계, 운영 관리가 어떻게 순환하는지 표시한다.
3. 과제 설명
   - 과제 범위와 목적을 고객 과업 중심으로 정리한다.
4. 문제 정의와 As-Is 문제점
   - PDF의 현 상태, Pain Point, Root Cause, Risk를 보존하되 전략 언어로 다듬는다.
   - As-Is는 카드형으로 구성하고 각 카드에 문제 유형 태그를 붙인다.
5. To-Be 전환 흐름
   - 단순 절차표가 아니라 고객 여정과 운영 피드백이 이어지는 흐름도로 작성한다.
6. 핵심 지향점·기대 효과·KPI
   - 표 대신 전략 성과 보드로 구성한다.
   - To-do, 기대 효과, 관찰 KPI를 같은 카드 안에서 연결한다.
7. 주요 프로세스 및 기능 설명
   - PDF의 프로세스를 3~5개 경험 영역으로 묶어 역할, 기대 결과, 주요 단계를 설명한다.
8. 핵심 기능 상세
   - 개별 기능 나열을 줄이고 `고객 이해/선택`, `실행/검증`, `연계/후속 처리`, `운영/데이터`처럼 정책서 작성에 유용한 기능 구조도로 재배치한다.
9. 정책서 작성 시 남겨야 할 판단 기준
   - 업무 판단, 예외, 고객 고지, 이력, 연계 실패, 운영 보정 기준을 정책 항목 후보로 정리한다.
10. 정책서로 전환할 때의 핵심 질문
    - 후속 정책서 작성자가 반드시 결정해야 하는 질문과 산출물을 연결한다.

## 시각화 기준

- 톤은 절제된 컨설팅 보고서 스타일을 유지한다.
- 표를 그대로 늘리지 말고 전략 맵, 여정 루프, 성과 보드, 기능 구조도로 바꾼다.
- 색상은 파랑, 청록, 앰버, 로즈 계열을 보조적으로 쓰되 한 가지 색에 치우치지 않는다.
- HTML/CSS는 독립 파일로 동작해야 하며 외부 이미지나 프레임워크에 의존하지 않는다.
- 모바일에서는 모든 그리드가 1열 또는 2열로 자연스럽게 접혀야 한다.

## 검증 기준

- `python3 scripts/build_tk_task_queue.py`
- `python3 -m py_compile scripts/build_tk_task_queue.py`
- `python3 -m py_compile src/web_app.py`
- `node --check web/app.js`
- `git diff --check`
- 삭제 요청된 보조 표현이 결과 HTML에 다시 포함되지 않았는지 확인한다.
- 생성된 HTML에 전략 전환 맵, To-Be 전환 흐름, 전략 성과 보드, 기능 구조도가 모두 포함되어 있는지 확인한다.
