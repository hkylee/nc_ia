# PM-10 멤버십 혜택/T 플러스포인트 Topic Learning & Core Design

## 1. 요구사항 해석
- PM-10은 멤버십 혜택, 등급, 제휴처 사용, 정기 프로모션, T 플러스포인트 적립·사용·소멸·이관·선물·통합·정정까지 포괄한다.
- 고객에게는 오늘 쓸 수 있는 가치, 등급과 혜택 상태, 포인트 자산, 바코드 사용수단, 오류 복구가 하나의 리워드 경험으로 연결되어야 한다.
- 운영자에게는 프로그램 편성, 등급 로직, 제휴처 매핑, 바코드 정책, 포인트 정산, 성과 개선을 같은 이력 체계로 관리하는 기준이 필요하다.
- 최신 상세 요구사항은 70건에서 81건으로 확장되었고, 내 혜택 한눈에 보기·쿠폰 일괄 다운로드·통합 바코드·0 Depth 호출·브랜드 Identity·소셜 티커·브랜드 친밀도·AI 절약 리포트·공유 지갑·포인트 사전 세팅이 추가 판단축이 되었다.

## 2. Core Design Map
- 고객 업무: 멤버십 리워드 탐색과 가치 이해 → 멤버십 혜택 사용 경험 완결 → 멤버십 등급과 혜택 상태 관리 → T 플러스포인트 자산 관리 → 멤버십 알림과 재방문 관리.
- 운영 업무: 혜택 프로그램 편성·배포 운영 → 등급·제휴처 혜택 정책 운영 → 바코드·사용수단 정책 운영 → 포인트 적립·회수·정산 운영 → 혜택 오류·정지·회수 예외 운영 → 리워드 성과·전환 최적화.
- 시스템 보조: 멤버십 엔진, T 플러스포인트 시스템, 제휴처 시스템, 알림센터, BSS/연계 시스템은 process_target=N으로 두고 고객·운영 프로세스의 기능·정책 기준에 반영했다.
- 새 요구사항은 신규 유즈케이스로 분산하지 않고 리워드 탐색, 현장 사용, 포인트 자산, 알림·리텐션, 운영 정책의 하위 프로세스·기능·정책 항목으로 내렸다.

## 3. 작성 판단
- 유즈케이스는 고객과 운영자의 완결 업무 단위로 묶고, 조건 확인·바코드 표시·정산 보류 같은 세부 판단은 프로세스와 기능으로 내렸다.
- 포인트는 혜택의 하위 수단이지만 적립·사용·소멸·선물·통합·이관은 고객 자산 관리 성격이 강해 별도 고객 유즈케이스로 분리했다.
- 외부 쿠폰·결제 할인·이벤트 미션 상세 정책은 경계로 남기고 멤버십 리워드 실행 기준만 작성했다.
- 통합 바코드와 포인트 사전 세팅은 결제 정책이 아니라 멤버십 현장 사용 경험의 정책 기준으로 반영했다.
- 브랜드 친밀도, 소셜 티커, AI 절약 리포트는 마케팅 문구가 아니라 고객에게 혜택 사용 이유와 다음 행동을 설명하는 리텐션 기준으로 제한했다.
- 공유 지갑은 단순 선물이 아니라 공동 혜택 자산의 대표자 권한, 동시 사용 차단, 실시간 동기화, 사용 이력 기준으로 작성했다.

## 4. Stage Inspection 요약
- Actor coverage: 고객, 멤버십 운영자, 마케팅 운영자, 시스템 액터가 모두 유즈케이스에 연결되었다.
- Process linkage: 모든 process_target=Y 유즈케이스는 2개 이상의 프로세스로 구성된다.
- Function linkage: 기능은 처리 단위로 작성했고 세부 기능 구성은 설명문이 아닌 하위 처리명으로 작성했다.
- Policy specificity: 등급, 포인트, 시간, 횟수, 상태, 예외, 이력, 정산, 알림 기준을 정책 항목에 포함했다.
- Requirement Trace: 최신 상세 요구사항 81건이 유즈케이스, 상태, 프로세스, 기능, 정책, 정책 항목 중 하나 이상에 연결되었다.
- Health Check: 고객 표시 상태와 내부 상태, 중복 요청·멱등, FO·BSS·제휴처 책임 경계, 통합 장애 영향 제한을 별도 정책 항목으로 보강했다.
