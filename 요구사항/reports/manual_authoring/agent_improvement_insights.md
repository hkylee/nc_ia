# Agent Improvement Insights

이 파일은 34개 정책서를 수동 작성하면서 발견한 Agent 개선 인사이트를 누적한다.

## 기록 원칙

- 정책서 작성 중 발견한 구조 문제, 반복 오류, 품질 저하 원인을 기록한다.
- 특정 주제에만 맞춘 하드코딩 제안은 기록하지 않는다.
- 개선 후보는 LLM 모드와 Mock 모드를 분리해서 영향 범위를 적는다.
- 실제 코드 변경은 별도 사용자 지시가 있을 때만 진행한다.

## 2026-05-08

### 1. 내부 연결값과 화면 표시값 분리 필요

- 발견 위치: 회원 가입·탈퇴 수동 초안 점검.
- 증상: `related_functions`, `related_policies`에 `ID + 명칭` 문자열이 같이 들어가면 자동 검증에서 실제 ID와 매칭되지 않는다.
- 영향: 프로세스-기능-정책 연결 검증, Health Check, Full 버전 확장, 최종 보완 라우팅이 끊길 수 있다.
- 개선 방향: 내부 JSON은 순수 ID만 저장하고, HTML 렌더링 단계에서 ID와 명칭을 조합해 표시한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성 프로토콜 전체.
- 상태: 지침 반영 완료. 코드 출력 가드 추가 여부는 별도 검토 필요.

### 2. 상태 전이표의 `관련 유즈케이스` 컬럼 추가 방지

- 발견 위치: 이전 mock/수동 생성물 점검.
- 증상: 샘플과 템플릿은 `전이 이벤트` 컬럼에 유즈케이스 또는 업무 사건을 쓰는 구조인데, 생성물이 별도 `관련 유즈케이스` 컬럼을 추가할 수 있다.
- 영향: 템플릿 이탈, 렌더링/검수 기준 혼선, 상태 전이 검증 로직 오판 가능성이 있다.
- 개선 방향: 상태 전이표에는 `현재 상태 / 전이 이벤트 / 다음 상태 / 처리 기준 및 후속 처리` 구조를 유지한다. 유즈케이스 연결은 내부 검증 매트릭스에서 관리한다.
- 적용 범위: State Agent, Renderer, Inspector, 수동 작성 프로토콜.
- 상태: 지침 반영 완료. 생성물 회귀 점검 필요.

### 3. 요구사항 trace 밀도 보강 필요

- 발견 위치: 회원 가입·탈퇴 수동 초안 점검.
- 증상: 상세 요구사항 162건을 검토했지만 trace matrix는 묶음 단위로만 남아 세부 요구사항명/설명 기준의 1차 추적성이 약하다.
- 영향: 작성자가 요구사항을 충분히 봤더라도 사용자가 커버리지 신뢰도를 판단하기 어렵다.
- 개선 방향: 정책서 본문에는 과도한 요구사항 매핑표를 넣지 않더라도, 별도 JSON/MD 리포트에는 상세 요구사항명/설명 기준 매핑을 남긴다.
- 적용 범위: 수동 작성, eval harness, Health Check, 요구사항 커버리지 리포트.
- 상태: `scripts/build_manual_requirement_trace.py`를 추가해 큐 작업 산출물에 별도 trace 리포트를 생성할 수 있게 했다.

### 4. 수동 작성도 Agent 루프를 지켜야 비교 가능

- 발견 위치: 수동 작성 속도와 품질 비교.
- 증상: 사람이 빠르게 전체 문서를 쓰면 Agent의 장별 작성, 검수, 보완 루프와 비교하기 어렵고 품질도 샘플 밀도에 못 미칠 수 있다.
- 영향: 수동 초안이 Agent 개선 기준 데이터로 쓰이기 어렵다.
- 개선 방향: 수동 작성도 `Topic Learning → Blueprint/Core Design → 장별 작성 → JSON Critical Gate → Stage Inspector → 보완 루프 → Final Inspector`를 따른다.
- 적용 범위: 수동 작성 프로토콜.
- 상태: 지침 반영 완료.

### 5. 프로세스 연결값 순수 ID 허용과 화면 표시 보강

- 발견 위치: PM-01 수동 작성 JSON Gate.
- 증상: 지침은 `related_functions`, `related_policies`를 순수 ID로 저장하도록 정리했지만 Validator는 아직 `ID + 명칭`을 요구하고 있었다.
- 영향: 올바른 내부 연결값으로 작성해도 Gate에서 실패하고, 반대로 화면 표시를 위해 문자열을 섞으면 downstream 라우팅이 약해질 수 있다.
- 개선 방향: Validator는 순수 ID를 허용하고, Renderer는 HTML에서 ID를 명칭과 조합해 표시한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성, Full 버전 확장.
- 상태: `src/validator.py`, `src/renderer.py`에 반영 완료. PM-01 산출물로 회귀 확인 완료.

### 6. 액터-유즈케이스 다이어그램 누락 점검 필요

- 발견 위치: PM-03 상품 목록 수동 작성 JSON Gate.
- 증상: 상품 기획자, 채널 업무 시스템처럼 보조 책임을 가진 액터를 추가했지만 최초 다이어그램에는 일부 유즈케이스 연결이 누락될 수 있었다.
- 영향: 본문에는 유즈케이스가 있어도 다이어그램과 연결 검증에서 누락처럼 보이고, Health Check나 후속 Full 확장 시 책임 주체 추적이 약해질 수 있다.
- 개선 방향: 액터 또는 유즈케이스를 추가한 뒤에는 `usecase_diagram.lines`가 모든 유즈케이스 ID를 최소 1회 이상 포함하는지 자동 점검한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성 프로토콜 전체.
- 상태: PM-03 산출물은 보완 완료. 코드 가드는 별도 검토 필요.

### 7. 정책서 경계가 겹치는 주제는 `must_not`과 trace matrix를 더 강하게 사용

- 발견 위치: PM-03 상품 목록 작성 중 PM-02 전시/관리 기능과의 경계 점검.
- 증상: 상품 목록은 카테고리, 필터, 노출, 운영 관리 요구사항을 포함하므로 전시/관리 기능과 경계가 겹치기 쉽다.
- 영향: 목록 정책서가 전시 운영 정책을 과도하게 포함하거나, 반대로 목록 고객 경험에 필요한 운영 품질 Gate를 누락할 수 있다.
- 개선 방향: Topic Knowledge의 `must_not`, `boundary_rule`, `trace_matrix`를 작성 전 Core Design 단계에서 명시적으로 분리한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-03 수동 작성물에는 반영 완료. Agent 프롬프트/Context Pack에도 동일한 경계 지침을 유지할 필요가 있다.

### 8. 상태 코드는 상태명만으로 통과시키지 말고 정의·후속처리를 필수화

- 발견 위치: PM-03 상품 목록 상태 코드 표.
- 증상: 상태 ID와 상태명은 작성되었지만 `description`, `next_action`이 비어 있어 HTML의 정의/대표 후속 처리 칸이 공란으로 노출되었다.
- 영향: 상태가 고객 노출, 기능 허용, 정책 판단, 후속 처리 중 무엇을 바꾸는지 알 수 없어 상태 전이 장의 설계 가치가 약해진다.
- 개선 방향: State Gate에서 모든 상태 코드의 정의와 대표 후속 처리를 필수로 검증한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-03 산출물 보완 완료. `src/validator.py`와 회귀 테스트에 가드 반영 완료.

### 9. 렌더러 스키마 필드는 의미가 있어도 이름이 틀리면 공란이 된다

- 발견 위치: PM-04 외부 BP 서비스 관리 체계 수동 작성 초안.
- 증상: 주요 용어가 `term`/`definition`, 문서 히스토리가 `description`처럼 의미상 맞는 필드로 작성될 수 있지만 Renderer는 `id`/`name`/`description`, `change`를 기대한다.
- 영향: JSON에는 내용이 있어도 HTML 표가 공란으로 렌더링되어 사용자는 누락처럼 보게 된다.
- 개선 방향: 장별 Gate에서 핵심 렌더링 필드명을 검증하고, 수동 작성 프로토콜과 Agent 출력 스키마를 동일하게 고정한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-04 산출물은 보정 완료. `src/renderer.py`가 히스토리 `description`, 용어 `term/definition`, 문자열형 범위·원칙을 안전하게 렌더링하도록 보강했다.

### 10. 시스템 액터도 유즈케이스 커버리지가 필요하다

- 발견 위치: PM-04 외부 BP 서비스 관리 체계 액터-유즈케이스 연결.
- 증상: BSS를 액터로 정의했지만 별도 유즈케이스에 연결하지 않으면 전체 Gate에서 미연결 액터로 차단된다.
- 영향: BSS 판단·저장·상태 변경 책임은 정책서에 반드시 필요하지만, 유즈케이스 연결 없이 액터만 두면 후속 프로세스·기능·정책 추적성이 약해진다.
- 개선 방향: BSS는 독립 프로세스 대상은 아니더라도 `process_target=N` 시스템 유즈케이스로 판단 정보 제공, 상태 반영, 이력 저장 책임을 연결한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-04 산출물에 `고객 상태·제한 조건 제공` 유즈케이스를 추가해 보완 완료.

### 11. 유즈케이스-프로세스가 전부 1:1이면 유즈케이스 입자도를 먼저 의심한다

- 발견 위치: PM-04 외부 BP 서비스 관리 체계 프로세스 목록.
- 증상: `진입 전 정보 확인`, `호환 환경·권한 점검`, `미니앱 이용과 복귀`처럼 절차 단계가 각각 Y 유즈케이스로 올라가면서 모든 Y 유즈케이스가 프로세스 1개씩만 갖는 구조가 되었다.
- 영향: 유즈케이스가 고객/운영자의 완결 업무 목표가 아니라 프로세스 단계처럼 보이고, BPMN·기능·정책도 상위 업무 흐름보다 항목 나열형으로 굳을 수 있다.
- 개선 방향: 유즈케이스는 상위 업무 목표로 묶고, 기존 절차형 항목은 프로세스로 내려보낸다. 개수를 맞추는 것이 아니라 시작·판단·처리·결과·예외의 의미 있는 전환점을 기준으로 분해한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-04 산출물 보완 완료. `src/validator.py`에 전체 1:1 고착을 차단하는 Gate와 회귀 테스트를 추가했다.

### 12. 일부 운영 유즈케이스만 1개 프로세스로 끝나는 경우도 수동 검토 대상이다

- 발견 위치: PM-03 상품 목록 운영자·상품 기획자 유즈케이스.
- 증상: 고객 유즈케이스는 2개 프로세스로 적절히 분해되었지만, 필터·정렬·목록 카드 기준 관리, 목록 품질 검수·배포, 상품 속성 기준 제공은 각각 1개 프로세스로 끝났다.
- 영향: 고객 흐름은 충분해도 운영 통제 흐름이 등록·검증·배포·회수·매핑 검증 단계로 분리되지 않아 상세 설계와 QA 기준이 약해질 수 있다.
- 개선 방향: 운영 유즈케이스는 단순 개수 강제가 아니라 등록, 검증, 승인/배포, 회수/복구, 매핑 검증처럼 책임 주체와 상태가 달라지는 지점에서 분해한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-03 산출물 보완 완료. PM-01/PM-02는 유즈케이스 목표와 프로세스 경계가 의미 있게 분리되어 유지했다.

### 13. 추천처럼 인접 주제가 많은 문서는 `범위 축소`보다 `연계 조건화`가 중요하다

- 발견 위치: PM-06 추천 수동 작성.
- 증상: 추천 요구사항은 전시, 상품 목록, AI 검색, 데이터 트래킹, 설정, 회원정보와 자연스럽게 겹친다. 이때 인접 주제를 무조건 제외하면 추천 근거, 실행 연결, 개인화 설정, 성과 측정 기준이 빈약해진다.
- 영향: 범위를 너무 줄이면 추천이 단순 슬롯 노출 문서가 되고, 너무 넓히면 전시/검색/데이터 정책서를 침범한다.
- 개선 방향: 인접 주제의 전체 절차는 쓰지 않되, 추천 판단에 필요한 연계 조건, 전후 상태, 이력, 대체 경로만 현재 문서에 남긴다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-06 산출물에 반영 완료. Agent Context Pack에도 `must_not`을 단순 제외가 아니라 `연계 조건으로만 다룬다`는 방식으로 유지하는 것이 좋다.

### 14. 고객 제어 요구사항은 별도 유즈케이스로 묶어야 정책 추적성이 좋아진다

- 발견 위치: PM-06 추천 수동 작성.
- 증상: 관심 없음, 숨김, 개인화 설정 진입, 피드백 이력은 추천 결과 확인의 부가 옵션처럼 보일 수 있지만 실제로는 고객 권한과 개인화 정확도에 영향을 주는 독립 업무다.
- 영향: 고객 제어를 정책 항목에만 숨기면 상태, 프로세스, 기능, 이력, 운영 개선 루프로 연결되지 않는다.
- 개선 방향: 추천 제어 및 피드백 관리를 고객 유즈케이스로 두고, 숨김 처리, 설정·이력 확인, 실패 대체 경로를 프로세스로 분해한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: PM-06 산출물에 반영 완료. 유사하게 알림 설정, 약관 동의, 개인정보 제어가 있는 주제에도 적용 가능하다.

- 증상: 고객센터 요구사항에는 조회, 처리, 관리, 상품, 가입, 콜센터, 상담, 문의 같은 단어가 섞여 있어 일반 semantic axis를 그대로 적용하면 `탐색 조건 적용`, `가입·신청`, `결제·혜택` 같은 인접 주제 프로세스가 유입될 수 있었다.
- 영향: 고객센터 통합허브가 고객 문제 해결 허브가 아니라 검색/가입/결제 보조 화면처럼 보이고, Self 해결, 상담 전환, 문의 접수, 상담 문맥 유지, 콜센터 재유입 같은 핵심 정책 판단이 약해질 수 있다.
- 개선 방향: 고객센터 주제에서는 `셀프 해결 경로`, `문의 접수·상태 관리`, `상담 문맥·이력`, `상담 채널·전환`, `운영·상담 Single-view` 축을 우선 분류한다. `관리` 같은 넓은 단어는 단독으로 운영 축에 보내지 않고 문맥 단어와 함께 판단한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: `src/schema.py`와 `src/chapter_agents.py`에 고객센터 통합허브 전용 semantic axis와 프로세스 템플릿을 반영했고, PM-32 산출물에서 인접 축 누수 0건과 Mock Inspector 100점/finding 0건을 확인했다.

### 27. 고객센터 FAQ·공지·이용안내는 통합허브 템플릿을 재사용하지 않는다

- 발견 위치: PM-33 고객센터_FAQ/공지/이용안내 초안 생성.
- 증상: 주제명에 `고객센터`가 포함되어 고객센터 통합허브 프로세스 템플릿이 잘못 적용되면서 `셀프 해결 가능 범위 판정`이 반복되고, `탐색 조건 적용`, `가입·신청` 같은 일반 축이 일부 유입됐다.
- 영향: FAQ·공지·이용안내가 콘텐츠 탐색과 운영 기준이 아니라 상담 허브 또는 일반 탐색 업무처럼 보일 수 있다.
- 개선 방향: PM-33은 `FAQ 탐색·추천`, `이용안내·가이드`, `공지·장애·점검`, `해결 실패·CS 연결`, `콘텐츠 운영·버전 관리` 축으로 별도 분류한다. 프로세스 템플릿도 도움 콘텐츠 목적 선택, FAQ·가이드 검색, 최신성 확인, 해결 가이드 실행, 공지 영향 확인, 운영 기준 관리로 분리한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: `src/schema.py`, `src/chapter_agents.py`, `tests/test_schema_topic_specificity.py`에 반영했고, PM-33 산출물에서 `가입·신청`, `탐색 조건 적용`, `셀프 해결 가능 범위 판정` 0건과 Mock Inspector 100점/finding 0건을 확인했다.

### 28. 고객센터 매장안내는 위치 기반 방문 가능성 판단으로 분리한다

- 발견 위치: PM-34 고객센터_매장안내 초안 생성.
- 증상: 고객센터 하위 주제라는 이유만으로 FAQ·공지 또는 통합허브 템플릿을 재사용하면 매장 검색, 위치 동의, 단골 매장, 대리점 URL, 처리 가능 업무, 예약 가능 여부 같은 핵심 판단이 상담·콘텐츠 흐름에 묻힐 수 있었다.
- 영향: 매장 안내가 고객지원 콘텐츠처럼 작성되면 고객이 실제 방문 가능한 매장인지, 어떤 업무를 처리할 수 있는지, 예약·대체 경로를 어떻게 판단해야 하는지 개발/QA 기준으로 전환하기 어렵다.
- 개선 방향: PM-34는 `매장 검색·위치 기준`, `방문 가능성·예약 기준`, `대리점 사이트·URL 운영`, `단골 매장·개인화`, `매장 안내 운영·통합 기준` 축으로 분리한다. 프로세스는 방문 목적·위치 기준 선택, 매장 검색·속성 필터, 운영시간·처리 가능 업무 확인, 방문 업무·예약 가능 여부 확인, 대체 매장·온라인 경로 안내로 구성한다.
- 적용 범위: LLM 모드, Mock 모드, 수동 작성.
- 상태: `src/schema.py`, `src/chapter_agents.py`, `tests/test_schema_topic_specificity.py`에 반영했고, PM-34 산출물에서 고객센터 허브/FAQ 인접 축 누수 0건과 Mock Inspector 100점/finding 0건을 확인했다.
