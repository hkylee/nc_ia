# PM-09 외부 쿠폰 Topic Learning & Core Design

## 1. 요구사항 해석
- 외부 쿠폰은 고객이 발급 전 실제 사용 가능성을 판단하고, 발급·다운로드·즉시 사용·쿠폰함 확인까지 이어지는 혜택 실행 업무다.
- 최신 상세 요구사항 32건은 고객 탐색·발급·사용·쿠폰함 흐름, 운영 기준정보·코드·연동·정산 흐름, 위치 기반 매장 안내·지오펜싱 알림·결제 자동 적용·쿠폰 양도/회수·2종 중 택1·사용기간/할인 적용 기준으로 확장되었다.
- 요구사항 원문은 그대로 문서에 넣지 않고, 발급 조건·상태·기능·정책 판단 기준으로 재구성했다.

## 2. Core Design Map
- 고객 업무: 외부 쿠폰 혜택 탐색과 발급 준비 → 외부 쿠폰 발급 여정 완성 → 외부 쿠폰 사용 경험 완결 → 보유 외부 쿠폰 재사용 관리.
- 운영 업무: 쿠폰 기준 등록·검수·배포 → 쿠폰 코드·재고·연동 운영 → 쿠폰 중지·회수·재발급 및 고객 고지 관리 → 제휴처 정산·오류 확인.
- 시스템 보조: 채널 업무 시스템의 조건·재고 판정, 외부 제휴처 상태 동기화, 통합 쿠폰함 저장, 위치·매장 데이터 동기화, 결제 시 자동 적용 판정, 알림센터 발송, BSS/연계 조건 제공은 process_target=N으로 두고 고객·운영 프로세스의 기능·정책에 흡수했다.

## 3. 주요 설계 판단
- 외부 쿠폰과 멤버십 혜택/포인트/결제 할인은 경계를 분리했다.
- 쿠폰 상태는 고객 노출, 기능 허용, 운영 조치가 달라지는 경우에만 상태 코드로 승격했다.
- 위치 기반 매장 안내와 지오펜싱 알림은 별도 유즈케이스로 쪼개지 않고 외부 쿠폰 탐색·사용 경험을 강화하는 기능과 정책으로 내렸다.
- 결제 자동 적용은 결제 정책서로 넘기지 않고 외부 쿠폰 사용 가능성, 할인 적용 대상, 오적용 복구 기준까지 외부 쿠폰 사용 정책에 포함했다.
- 선택형 쿠폰, 기보유 쿠폰, 재다운로드, 오픈 예정, 제휴처 오류, 정산 보류, 쿠폰 양도·안심 회수는 별도 정책 판단축으로 분리했다.
- 프로세스는 유즈케이스당 2개 이상으로 분해해 이전 재발 이슈인 유즈케이스 과세분화와 프로세스 1개 고착을 피했다.

## 4. Stage Inspection 요약
- Actor coverage: 고객, 쿠폰 운영자, 제휴처 운영자, 시스템 액터가 모두 유즈케이스에 연결되었다.
- Process linkage: 모든 process_target=Y 유즈케이스가 복수 프로세스를 가진다.
- Function linkage: 모든 프로세스는 관련 기능과 정책을 ID 기준으로 연결한다.
- Policy specificity: 정책 항목은 상태, 조건, 시간, 횟수, 위치·매장 데이터, 자동 적용, 양도·회수, 이력, 외부 연동, 고지 기준을 포함한다.

## 5. 유즈케이스 보정
- 유즈케이스명에서 절차 단계처럼 보이는 가능 여부 판단, 결과 확인 표현을 제거하고 고객의 완결 과업 단위로 재명명했다.
- 세부 판단과 확인은 프로세스로 내렸고, 프로세스는 기능으로 다시 분해 가능한 업무 경계만 유지했다.
- 추가 요구사항으로 들어온 위치 안내, 지오펜싱, 자동 적용, 쿠폰 양도/회수는 고객 완결 과업을 새로 쪼개지 않고 기존 탐색·사용·쿠폰함 관리 유즈케이스의 프로세스와 정책 항목으로 반영했다.
