# PM-21 회선 변경/관리 Topic Learning & Core Design Map

## 1. 작성 지향점
- 회선 변경·관리는 계약자, 사용자, 대표 회선, 구성원의 권한과 상태를 정리하는 고객 자산 운영 업무로 정의한다.
- 고객이 어떤 회선을 누가 관리할 수 있는지, 변경하면 요금·약정·결합·혜택·수신·후속 서비스에 어떤 영향이 생기는지 먼저 이해하도록 작성했다.
- 정지, 해지, 분실, 계약자 변경, 해외·대리·사망자 등 예외 상황을 상담 의존이 아닌 셀프 처리와 운영 검토의 명확한 분기 기준으로 설계했다.

## 2. 요구사항 분류
- 총 요구사항: 73건
- 주요 축: 회선 권한, 처리 가능 여부, 일시정지/정지해제, 분실·도난, 해지·예약, 계약자 변경·양도, 대표 회선·구성원, 해외·사망·대리, BSS 반영, 이력·고지.

## 3. Core Design
- 액터는 고객, 대리 권한자, 상담사, 회선 운영자, 채널 업무 시스템, BSS/연계 시스템으로 제한했다.
- 2026-05-10 보완: `변경 대상 회선과 처리 가능 여부 확인`, `변경 신청 내역과 후속 조치 확인`은 절차 단계처럼 보일 수 있어 각각 `회선 변경 업무 진입과 처리 경로 결정`, `회선 변경 신청 이력과 복구 행동 관리`로 상위 고객 목적을 드러내도록 정리했다.
- 2026-05-10 보완: 정책 상세는 최신 Validator와 Health Check가 직접 판정할 수 있도록 정책 항목 단위 flat 구조로 정규화했다.
- 로그인/정상/제한/미성년/해외 고객은 액터로 분리하지 않고 권한·상태·정책 조건으로 처리했다.
- 유즈케이스는 고객이 인식하는 완결 업무 단위로 묶고, 프로세스는 각 유즈케이스의 판단·입력·영향 확인·신청·결과 확인 흐름으로 분해했다.
- 정책은 기능 설명이 아니라 권한, 조건, 기간, 제한, 이력, BSS 반영, 개인정보 보호 기준으로 분리했다.

## 4. 주요 설계 판단
- 회선 상태 변경은 고객에게 보이는 상태와 BSS 원장 반영 상태가 다를 수 있으므로 `처리 중`, `운영 검토 필요`, `반려·보완 필요` 상태를 별도로 둔다.
- 분실·도난은 고객 피해 방지를 위해 긴급 차단을 우선하되, PUK·유심·위치조회·보험·임대폰 후속 경로를 같은 업무 흐름에서 안내한다.
- 해지와 정지는 비용·결합·가족 할인·수신 가능성에 영향을 주므로 처리 전 영향도 확인을 필수 정책으로 둔다.
