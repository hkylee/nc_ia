## 3. Core Design Map

- 고객 유즈케이스는 화면 단계가 아니라 `신청 준비`, `계약 조건 비교`, `계약 제출`, `진행 관리`, `실패 복구`라는 완결 업무 목표로 묶었다.
- 대리 권한자, 상담사, 운영자는 독립 책임 주체로 유지하되 로그인/비로그인, 정상/제한, 미성년 등 고객 상태는 액터가 아니라 정책 조건과 상태로 내렸다.
- Y 유즈케이스 9개는 각각 3~6개 프로세스로 분해해 유즈케이스 과분화와 유즈케이스당 프로세스 1개 문제를 모두 피했다.
- 프로세스는 고객 의사결정과 처리 목적 기준으로 분해하고, 시스템 조회·저장·판정은 기능과 정책으로 연결했다.
- 상태는 주문 전, 주문서 작성 중, 조건 검증 중, 인증·동의 필요, 최종 확인 대기, 주문 접수, BSS 반영 중, 계약 생성 완료, 배송·개통 대기, 가입 완료, 보류·상담 필요, 고객 보완 필요, 주문 실패, 주문 취소, 재진입 가능으로 분리했다.
- 정책은 주문 단위, 진입/유형, 자동입력, 가입 가능성, 예약, 혜택·약정, 비용, 그룹, 인증, 대리, 약관, 입력, 제출, BSS, 상태, 문서, 배송·개통, 실패 복구, 상담, 운영, 문구, 리스크, 이력, 알림으로 구성했다.

## 4. 작성 경계

- 결제 승인 상세, 배송 상세 운영, 교환·반품·환불 정산, 해지·취소 사후관리 상세는 인접 정책서로 넘긴다.
- 단, 주문 제출 가능 여부, 고객 후속 행동, 상태 표시, BSS 반영 결과, 실패 복구에 영향을 주는 결제 필요 여부, 배송·개통 방식, 취소 가능 시점은 본 문서에서 연결한다.
- UI 화면명과 운영자 화면 상세는 직접 명세하지 않고, 통합 운영 가능 여부, 노출 조건, 검증 기준, 이력 저장 기준으로 재구성한다.
