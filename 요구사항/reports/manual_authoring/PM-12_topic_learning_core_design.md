# PM-12 카트/장바구니 Topic Learning & Core Design

## 입력 근거
- 상세 요구사항 수: 43건
- Depth4: 카트/장바구니
- 기준: 상세 요구사항명과 상세 요구사항 설명을 업무 흐름과 정책 판단 기준으로 재구성했다.

## 문서 지향점
1. 카트/장바구니는 임시 저장함이 아니라 선택한 상품·옵션·혜택 조합을 주문 가능한 상태로 정리하는 계약 전 검증 공간이다.
2. 고객은 한 화면에서 전체 구매안을 보되, 실제 주문·결제·배송·발급 단위가 다른 항목의 분리 이유와 다음 행동을 이해해야 한다.
3. 비로그인 복원, 로그인 이관, 세트 구성, 조합 검증, 변동 고지, 자동삭제, 상담 전환까지 주문 직전 실패를 줄이는 정책 기준으로 작성한다.
4. 최신 요구사항에서 추가된 담기 후 동선, 액세서리 수량·금액 산정, 함께쓰기 요금제 모회선 인증은 단순 화면 기능이 아니라 주문 전환 가능성을 좌우하는 정책 기준으로 반영한다.

## Core Design Map
- 상위 유즈케이스: 장바구니 구매안 통합 관리, 상품 구성 변경과 정리, 조합 검증과 주문 전환 준비, 이어보기와 복구, 관심상품 저장과 공유 활용, 추천과 단독구매 전환, 문제 상담 전환.
- 운영 유즈케이스: 장바구니 제공 기준 운영, 조합·분리주문 정책 운영, 보존·이관·만료 정책 운영, 변동 모니터링과 복구 운영.
- 주요 상태: 진입 가능, 임시 보관, 계정 보관, 구매안 구성 중, 검증 필요, 인증 필요, 주문 전환 가능, 분리 주문 필요, 단독구매 필요, 변경 확인 필요, 수량 한도 초과, 구매 불가, 만료 예정, 자동 삭제 완료, 상담 전환 필요, 주문 완료 반영, 장바구니 비어 있음.
- 핵심 정책 질문: 무엇을 장바구니에 담을 수 있는가, 실제 처리 단위는 언제 분리되는가, 담기 후 고객은 어떤 다음 행동을 선택할 수 있는가, 비로그인 선택은 언제 계정으로 이관되는가, 액세서리 수량과 금액은 언제 재산정되는가, 함께쓰기 요금제는 언제 모회선 인증을 요구하는가, 몇 개·몇 일까지 보존하는가, 조합 불가·변동·만료 시 고객은 무엇을 할 수 있는가.

## 작성상 주의
- 상품 상세의 담기 버튼 기준과 주문서 작성/결제 처리를 본문 범위로 확장하지 않는다.
- 로그인 고객, 비로그인 고객, 개인고객, 그룹고객을 액터로 분리하지 않고 상태와 정책 조건으로 관리한다.
- 유즈케이스를 절차 단계로 쪼개지 않고 고객 또는 운영자의 완결 업무 목적 단위로 묶은 뒤 프로세스에서 시작·판단·처리·결과·예외를 분해한다.
- 상세 요구사항 43건은 요구사항 원문을 본문에 복사하지 않고 `PM-12_requirement_trace.md`에서 유즈케이스, 상태, 프로세스, 기능, 정책, 정책 항목 후보로 추적한다.

## 액터 정리 기준
- 액터는 고객, 운영자, 상담사, 채널 업무 시스템, BSS/연계 시스템의 5개 책임 단위로 통합했다.
- 상품 운영자, 장바구니 운영자, 가격 운영자, 재고 운영자는 별도 액터로 분리하지 않고 운영자 기능·정책으로 내려 작성했다.
- 장바구니 시스템, 상품 마스터, 재고 시스템, 주문 시스템은 별도 액터로 분리하지 않고 채널 업무 시스템 또는 BSS/연계 시스템의 기능 책임으로 표현했다.

## 직접 Inspector / Health Check 기록
- Codex 직접 Inspector: PASS. 액터 5개 책임 단위가 적정하고, 고객 유즈케이스는 완결 업무 목적 단위로 유지되며, 신규 보강 항목은 프로세스·기능·정책으로 연결되어 있다.
- Codex 직접 Health Check: PASS. 주문 전 검증, 보존·삭제, 변동 고지, 인증 필요 상태, BSS 판정, 요구사항 Trace가 개발/QA 테스트 조건으로 전환 가능하다.
- 자동 검증과 수동 판단 차이: 자동 Health Check는 96점/우수, 필수 Gate 통과. 직접 검토에서도 P1/P2 잔여 이슈는 없다.
