# PM-02 전시/관리 기능 수동 작성 기록

## 1. Topic Learning

- 전시/관리 기능은 메뉴와 배너를 단순 배치하는 문서가 아니라, 고객 여정의 확인, 이해, 실행, 완료를 하나의 안내 체계로 연결하는 정책서로 정의했다.
- 상세 요구사항명과 상세 요구사항 설명 기준으로 홈 빠른 메뉴, 개인화 홈, 목적 기반 IA, 전시 컴포넌트, 배너·콘텐츠 관리, 운영자 승인·배포·회수, A/B 테스트, AI 전시 최적화, 변경 이력 요구를 묶어 해석했다.
- TK의 지향점은 전시를 고객 목적 기반 탐색 허브로 재구성하고, 운영자가 노출 조건·우선순위·검수·배포·회수 기준을 정책으로 통제하는 방향으로 반영했다.

## 2. Blueprint / Core Design

- 액터는 고객, 전시 운영자, 마케팅 운영자, 검수 승인자, AI 추천·전시 시스템, BSS로 정리했다.
- 유즈케이스는 고객 탐색·실행, 운영자 전시 등록·검수·배포, 마케팅 캠페인 운영, AI 최적화, BSS 원천 정보 제공을 분리했다.
- 상태는 작성 전, 초안 작성 중, 검수 대기, 승인 완료, 예약 대기, 노출 중, 일시 중지, 회수 완료, 반려, 롤백 필요로 구성했다.
- 프로세스는 고객 홈 탐색부터 운영자 등록, 검수, 예약, 배포, 회수, 롤백, 실험, 성과 확인까지 이어지는 운영 흐름으로 설계했다.
- 정책은 노출 범위, 우선순위, 개인화, 콘텐츠 품질, 승인·배포, 캠페인 피로도, AI 최적화, 이력·감사, BSS 연계, 실험·지표, 랜딩·복귀 맥락으로 나눴다.

## 3. Stage Inspection / Repair

- 1차 검증에서 BSS 액터가 유즈케이스와 연결되지 않은 점과 랜딩 정책 참조가 정책 목록에 없는 점을 발견했다.
- `US-DSP-BSS-001`을 추가해 BSS의 고객 상태·보유 관계 원천 제공 책임을 연결했다.
- `PG-DSP-LND-001` 랜딩·복귀 맥락 정책을 추가해 관련 프로세스와 정책 상세의 참조를 일치시켰다.
- 정책 상세는 43개 항목으로 작성했으며, 요구사항 문구를 그대로 복사하지 않고 운영 판단 기준과 고객 경험 기준으로 재구성했다.

## 7. 추가 보정

- 세부 기능 구성이 정책 설명문처럼 작성된 항목을 기능 하위 처리명으로 보정했다.
- 보정 후 세부 기능 구성은 전시 후보 수집, 고객 상태 매칭, 승인 권한 확인, 롤백 이력 기록처럼 실제 구현 검토 가능한 하위 처리 단위로 정리했다.
