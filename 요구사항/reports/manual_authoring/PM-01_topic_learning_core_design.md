## 3. 이 정책서가 해결해야 하는 업무 문제

이 문서는 34개 개별 정책서가 서로 다른 방식으로 작성되는 문제를 막는 공통 설계 기준서다.
개별 기능을 설명하는 문서가 아니라, 통합채널 전반에서 고객 과업을 일관되게 완료시키기 위한 공통 UX, 문구, 품질, 상태, 운영, 적응형 기준을 정의해야 한다.

특히 다음 문제를 해결해야 한다.

- 같은 의미의 문구, 상태, 버튼, 오류 안내가 화면과 업무별로 다르게 쓰이는 문제.
- 고객이 로그인, 권한 허용, 회원가입, 입력, 비용 부담을 하기 전에 얻는 가치와 제한을 알기 어려운 문제.
- 검색, 비교, 주문, 상담 전환 과정에서 이전 맥락과 입력값이 끊기는 문제.
- 각 팀이 독립적으로 운영하더라도 고객 경험과 정책 기준이 흔들리는 문제.
- 품질 점검과 적응형 처리가 화면 단위 가이드에 머물고 정책 판단 기준으로 연결되지 않는 문제.

## 4. 작성 지향점

통합채널의 공통 정책 기준을 정의해 모든 정책서가 같은 고객 경험 원칙과 품질 Gate 위에서 작성되도록 한다.
고객이 과업을 시작하기 전에 가치, 조건, 제한, 다음 행동을 예측할 수 있게 하고, 채널과 기능이 달라도 상태·문구·CTA·이력 기준이 일관되게 유지되도록 한다.
적응형 처리는 개인화 장식이 아니라 고객 조건, 이전 맥락, 처리 가능성, 상담 전환 기준을 줄이는 정책 판단 체계로 정의한다.

## 5. Core Customer Jobs

- 고객은 과업을 시작하기 전에 얻는 가치, 비용, 제한, 예상 결과를 이해해야 한다.
- 고객은 3초 안에 답하기 어려운 입력을 반복하지 않고, 이전 맥락과 보유 정보를 활용해 과업을 이어갈 수 있어야 한다.
- 고객은 실패, 제한, 보류 상태에서 원인과 다음 행동을 확인할 수 있어야 한다.
- 고객은 중요 알림과 프로모션을 구분하고 수신·숨김·나중에 보기·관심 없음 기준을 조정할 수 있어야 한다.
- 운영자는 공통 문구, 상태, CTA, 오류 안내, 품질 Gate를 등록·검수·배포·회수할 수 있어야 한다.

## 6. Core Policy Questions

- 어떤 상황에서 고객에게 로그인, 권한 허용, 회원가입, 입력, 비용 부담을 요구할 수 있는가?
- 고객에게 먼저 보여야 하는 가치, 조건, 제한, 유의사항은 무엇인가?
- 공통 CTA, 토스트, 팝업, 모달, 오류 문구는 어떤 기준으로 선택하는가?
- 이전 검색어, 필터, 비교 대상, 입력값, 스크롤 위치, 다음 행동 정보는 언제 유지하거나 복원하는가?
- 고객 상태, 권한, 이용 이력, 채널 맥락에 따라 어떤 정보와 행동을 적응형으로 제공하는가?
- 고객 경험이 단절되거나 실패했을 때 상담 전환과 셀프 복구 기준은 무엇인가?
- 운영자는 공통 기준을 어떤 승인·배포·이력 절차로 관리하는가?
- 품질 Gate는 어떤 조건에서 통과, 보류, 회수, 재검수 상태로 전환되는가?

## 7. Must Include

- 공통 UX 원칙: Value First, Clear Action, Easy to Answer, 여정 연속성, 강요하지 않는 톤앤매너.
- 공통 문구/용어/상태/CTA 기준.
- 토스트, 팝업, 모달, 오류 안내 사용 기준.
- 고객 맥락 유지와 복원 기준.
- 적응형 처리 기준: 개인화, 조건별 노출, 기본값, 자동채움, 상담 전환.
- 품질 Gate: 등록, 검수, 승인, 배포, 모니터링, 회수, 재검수.
- 데이터/로그/이력 기준: 고객 입력, 변경 이력, 노출 이력, 품질 점검 이력.
- 권한/보안 기준: 로그인, 권한 허용, 동의, 개인정보 노출 제한.

## 8. Must Not

- 특정 화면 UI 상세, 와이어프레임, 버튼 위치, API 필드, DB 컬럼을 본문에 과도하게 쓰지 않는다.
- 34개 개별 정책서가 다룰 업무별 세부 정책값을 여기서 확정하지 않는다.
- “공통 기준에 따른다”처럼 빈 정책으로 끝내지 않는다.
- 개인화/적응형을 추천 노출 장식으로만 정의하지 않는다.
- 고객에게 과도한 판매 압박, 반복 팝업, 불안 조성 문구를 허용하는 기준을 만들지 않는다.

## 9. Core Design Map 초안

### 9.1 액터 후보

- 고객
- 운영자
- 품질 검수자
- 채널 업무 시스템
- BSS
- 연계 시스템

### 9.2 유즈케이스 후보

- 고객이 공통 기준에 따라 과업 가능성과 다음 행동을 확인한다.
- 고객이 이전 맥락을 유지한 채 과업을 이어간다.
- 고객이 실패·제한·보류 상태에서 복구 경로를 확인한다.
- 운영자가 공통 문구·상태·CTA 기준을 관리한다.
- 품질 검수자가 공통 기준과 적응형 기준을 검수한다.
- 채널 업무 시스템이 고객 맥락과 품질 기준을 적용한다.
- BSS가 고객 상태와 업무 가능 조건을 판정한다.

### 9.3 상태 후보

- 기준 등록 전
- 검수 대기
- 검수 반려
- 승인 완료
- 배포 대기
- 배포 완료
- 회수 필요
- 재검수 필요
- 적용 제한
- 복구 가능

### 9.4 프로세스 후보

- 공통 기준 진입 및 가치 안내
- 고객 맥락 확인 및 재사용 가능성 판단
- 공통 문구·CTA·상태 기준 선택
- 적응형 노출 및 기본값 적용
- 실패·제한·보류 안내 및 복구 경로 제공
- 공통 기준 등록·검수·승인·배포
- 품질 모니터링 및 회수·재검수

### 9.5 기능 후보

- 공통 문구/용어 기준 관리 기능
- CTA/안내 유형 선택 기능
- 고객 맥락 유지·복원 기능
- 적응형 조건 판정 기능
- 품질 Gate 검수 기능
- 운영 이력 및 배포 이력 저장 기능
- 실패/제한 안내 및 상담 전환 기능

### 9.6 정책 후보

- 가치 우선 노출 정책
- 명확한 행동 기준 정책
- 입력 최소화 및 맥락 유지 정책
- 공통 문구·상태·CTA 정책
- 토스트/팝업/모달 사용 정책
- 적응형 처리 정책
- 품질 Gate 정책
- 운영 변경 및 이력 정책
- 권한·보안·개인정보 노출 제한 정책

## 10. 다음 장 작성 전 Gate

- 유즈케이스가 화면 구성 기준이 아니라 고객/운영자의 완결 과업인지 확인한다.
- 프로세스가 공통 기준의 등록·적용·검수·회수 흐름으로 이어지는지 확인한다.
- 기능은 공통 기준을 실행하는 처리 역량으로 작성한다.
- 정책은 개별 화면 설명이 아니라 34개 정책서가 공유할 판단 기준으로 작성한다.
- 내부 연결값은 순수 ID로 유지하고 HTML에서만 ID+명칭으로 표시한다.

## 11. 추가 보정

- 세부 기능 구성이 설명 문장으로 작성된 항목을 짧은 기능 하위 처리명으로 보정했다.
- 보정 기준은 `조건 정보 조합`, `권한 상태 검증`, `결과 안내 구성`, `이력 저장`처럼 개발자가 하위 처리로 이해할 수 있는 명사형 처리 단위다.
