# PM-13 할인/시뮬레이션 Topic Learning & Core Design

## 1. 작성 범위 판단

- PM-13은 결제나 청구 확정 문서가 아니라, 고객이 현재 조건과 후보 조건을 바탕으로 예상 비용·혜택·가입 가능성을 비교하고 다음 행동을 결정하는 정책서다.
- 상품 상세, 장바구니, 주문/계약, 결제 처리와 경계가 맞닿아 있으므로 PM-13은 전환 전 판단 기준과 결과 문맥 전달까지만 다루고, 승인·결제·원장 반영은 후속 정책서로 분리했다.
- 요구사항은 상세 요구사항명과 상세 요구사항 설명을 기준으로 읽고, 원문을 복사하지 않고 계산 조건, 할인 조합, 가입 가능 판정, 오차 고지, 운영 품질 기준으로 재구성했다.

## 2. 핵심 고객 과업

- 고객은 현재 요금제, 회선, 결합, 구독, 약정, 단말, 로밍, 복지할인 조건을 바꿔 보며 예상 월 납부액과 일시 비용을 비교해야 한다.
- 고객은 최대 혜택 추천을 그대로 믿는 것이 아니라, 약정·승계·해지·재가입 제한과 같은 조건 영향을 함께 이해해야 한다.
- 고객은 시뮬레이션 결과를 장바구니, 주문서, 상품 상세, 상담으로 이어가기 전에 결과가 유효한지 확인해야 한다.

## 3. Core Design Map

### 액터

- 고객: 시나리오를 만들고 비교 결과를 이해한 뒤 다음 행동을 선택한다.
- 운영자: 계산식, 마케팅프로그램, 오차 문구, 운영 상태, 변경 이력을 관리한다.
- 상담사: 셀프 해결이 어려운 부적격·충돌·고관여 할인 이슈를 문맥과 함께 이어받는다.
- 채널 업무 시스템: 조건 수집, 결과 표시, 저장, 전환 문맥 전달을 수행한다.
- BSS/연계 시스템: 고객 조건, 상품 관계, 혜택 관계, 비용, 가입 가능 여부를 판정한다.

### 유즈케이스 설계 기준

- `가입 가능 확인`, `조건 입력`, `결과 저장` 같은 절차 단계는 유즈케이스가 아니라 프로세스로 내렸다.
- 고객 유즈케이스는 `개인화 할인 비교 기준 구성`, `비용·혜택 비교 의사결정`, `할인 적용 구매안 준비`, `시뮬레이션 문제 해결과 지원 전환`처럼 고객이 인식하는 완결 업무 목표로 묶었다.
- 운영 유즈케이스는 계산식/규칙 운영과 품질 모니터링으로 분리했다.
- 20260509 요구사항 업데이트 후 운영성 요구사항이 70건으로 확대되어, 공통·별도·추가지원금, 프론트 연동, 삭제 보호, 입력 유효성은 별도 고객 유즈케이스로 만들지 않고 운영자 유즈케이스의 프로세스·기능·정책으로 내렸다.

## 4. 정책 질문

- 어떤 조건을 자동 반영하고, 고객이 직접 수정한 값은 언제 우선하는가?
- 월 납부액, 일시 비용, 정상가, 할인 후 총 비용은 어떻게 구분해 표시하는가?
- 최대 혜택은 무엇을 기준으로 산정하며, 더 높은 요금제에서만 가능한 할인은 어떻게 분리하는가?
- 마케팅프로그램, 선택약정, T지원금, 공시지원금, 복지할인, 로밍, 결합 할인은 어떤 중복·배타 기준으로 계산하는가?
- 가입 불가, 승계 제한, 재가입 제한, 변경 횟수 초과, 계산 실패는 어떤 사유와 다음 행동으로 안내하는가?
- 저장된 시뮬레이션 결과는 언제까지 유효하고, 장바구니·주문서 전환 전에는 무엇을 재검증하는가?
- 공통지원금 A/B 트랙, 별도지원금 가입유형, 추가지원금 15% 합산 기준은 월 납부금액과 고객 노출 금액에 어떻게 반영하는가?
- 운영자가 등록한 상품 코드, 노출 순서, 추천 여부, Bar 문구, 유의사항 문구는 FO 반영 전 어떤 검증을 거치는가?
- 요금제, 프로모션, 요금약정할인, 추가할인, 추가지원금 삭제 시 참조 데이터와 사용여부 Y 항목은 어떻게 보호하는가?

## 5. 작성 중 보완한 점

- 최초 초안에서 `가입 가능·승계 가능 여부 확인`, `부적격·오차·실패 사유 확인`이 유즈케이스로는 절차적이라는 검증 결과가 나와, 고객 업무 목표 기준으로 상위 유즈케이스를 재구성했다. 이후 첫 고객 유즈케이스에 프로세스가 과도하게 집중되어 `개인화 할인 비교 기준 구성`과 `비용·혜택 비교 의사결정`으로 한 번 더 균형을 조정했다.
- 요구사항 추적에서 자동 매칭이 약했던 `가입일자·청구조건`, `고객 조회 시점 가입 가능여부`, `상품 구매 최대할인 구독가`는 정책 상세에 명시적으로 보강했다.
- 20260509 요구사항 DB 기준 70건을 재검토했고, 자동 Trace에서 수동 검토 필요로 남던 공통지원금 트랙, 가입유형별 별도지원금, 합산 노출, 요금제 비교 수 확장, 선택값 유지, 유의사항 문구 관리, 추가지원금 등록·삭제, 프론트 연동, 삭제 보호, Bar 문구 검증을 정책 항목으로 보강했다.
- 최종 70개 상세 요구사항은 모두 자동 매칭되었고, 수동 검토 필요 항목은 0건이다.

## 6. Codex 직접 Inspector / Health Check

- 직접 Inspector: 액터는 고객, 운영자, 상담사, 채널 업무 시스템, BSS/연계 시스템 5개로 유지했고, 로그인/비로그인·가입유형·고객 상태는 액터가 아니라 상태와 정책 조건으로 내렸다.
- 직접 Inspector: 고객 유즈케이스는 비교 기준 구성, 비용·혜택 의사결정, 구매안 준비, 문제 해결로 묶고, 지원금 운영·삭제·프론트 연동은 운영자 프로세스로 분리했다.
- 직접 Inspector: 모든 프로세스는 관련 기능과 관련 정책을 갖고, 모든 기능과 정책 그룹은 역참조와 정책 상세를 갖도록 확인했다.
- 직접 Health Check: 정책 정의 범위, CX, PI, 업무 구조, 상태/결과, 정책 구체성, 예외/실패, BSS/통합 리스크, 데이터 기준, 요구사항 추적성을 수동 점검했다. 상태/예외/BSS 영역은 자동 점수 9점이지만 실제 문서에는 지원금 검증·삭제 제한·프론트 반영 대기·BSS 응답 지연·상담 전환 기준이 포함되어 제출 가능으로 판단했다.
