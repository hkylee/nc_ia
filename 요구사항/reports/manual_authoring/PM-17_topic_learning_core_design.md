# PM-17 결제 Topic Learning & Core Design Map

## 1. 작성 목표
- 결제는 승인 성공 여부만이 아니라 결제수단 선택, 혜택 적용, 승인, BSS 수납 반영, 실패 복구, 증빙까지 이어지는 금전 거래 완료 업무다.
- 고객이 최종 결제 금액, 결제수단 제한, 할인 근거, 실패 시 다음 행동을 결제 전에 이해하도록 구성한다.
- PG 승인과 BSS 수납 반영의 불일치, 중복 결제, 외부 결제 복귀 실패, 정기결제 실패를 상태와 정책으로 명확히 둔다.

## 2. 요구사항 해석 기준
- 상세 요구사항명과 상세 요구사항 설명을 우선 근거로 사용했다.
- 요구사항 ID는 추적 보조값으로만 사용하고 문서 문구를 그대로 복사하지 않았다.
- 104건 요구사항은 결제 금액 확인, 결제수단 선택·등록, 할인·포인트·프로모션 적용, 승인·완료, 실패 복구, 정기결제, 증빙, 운영 관리로 재구성했다.
- 제휴카드, 제휴몰, 약관 버전, 노출상태, 삭제·상태관리, 노출순서·적용상품 매핑, 공통 항목 일괄 설정은 운영 화면 설명이 아니라 결제·혜택 운영 기준으로 정책화한다.

## 3. Core Design
- 액터는 고객, 운영자, 상담사, 채널 업무 시스템, BSS/연계 시스템 5개로 통합했다.
- 유즈케이스는 고객 완결 업무 7개, 운영자 1개, 상담사 1개, 시스템 보조 2개로 구성했다.
- 프로세스는 Y 유즈케이스당 2~5개로 분해해 유즈케이스당 1개 프로세스가 되는 문제를 방지했다.
- 기능은 각 프로세스 수행에 필요한 처리 역량으로 1:1 연결하고 세부 기능 구성은 명사형 처리 단위로 작성했다.
- 정책은 범위, 금액, 결제수단, 인증, 할인·포인트, 프로모션, 제휴카드, 승인, 실패, 정기결제, 증빙, BSS, 운영 기준 24개 그룹으로 정리했다.

## 4. 중점 검수 기준
- 결제수단 상세사, PG사, 카드사를 액터로 과분화하지 않는다.
- 할인 정책은 할인/시뮬레이션 문서로 넘기지 않고 결제 단계에서 필요한 적용·재검증·고지 기준을 완결적으로 작성한다.
- PG 승인 성공과 BSS 수납 반영 실패가 분리되는 보류 상태를 반드시 둔다.
- 비로그인 쿠폰 접근 대안, 제휴카드 탐색 위치, 약관 버전, 노출상태, 삭제 가능 여부처럼 상세 요구사항의 판단축은 요구사항 원문을 복사하지 않되 정책 항목명과 상세 기준에 추적 가능하게 남긴다.
