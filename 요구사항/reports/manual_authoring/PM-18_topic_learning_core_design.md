# PM-18 주문 상태/사후 관리 Topic Learning + Core Design Map

## 1. 핵심 지향점
- 주문 이후 관리는 단순 조회가 아니라 배송, 설치, 픽업, 변경, 클레임, 제휴 연동, 증빙의 다음 행동을 안내하는 사후 처리 허브로 정의했다.
- 고객 경험 관점에서는 “왜 이 상태인지”, “언제 완료되는지”, “지금 무엇을 할 수 있는지”, “실패하면 어떻게 복구하는지”를 상태 기반으로 제공하는 것을 목표로 잡았다.
- PI 관점에서는 배송사, 설치, 매장, 제휴, BSS의 분산 상태를 고객 표시 상태와 가능 행동으로 통합하고, 운영자는 상태 코드와 예외 보정 기준을 관리하게 했다.

## 2. 계위 설계
- 액터는 고객, 운영자, 상담사, 채널 업무 시스템, BSS/연계 시스템 5개 책임 단위로 제한했다.
- 유즈케이스는 고객/운영자/상담사의 완결 업무 목표로 묶고, 시스템 처리 유즈케이스는 process_target=N으로 분리했다.
- 프로세스는 주문 상태 조회, 변경, 배송·설치·픽업, 클레임, 연동·인증, 증빙, 운영 기준의 실제 절차 단위로 분해했다.
- 기능은 각 프로세스의 처리 역량으로 1:1 연결하고, 세부 기능 구성은 설명문이 아니라 짧은 하위 처리명으로 작성했다.
- 정책은 상태·변경·배송·클레임·연동·인증·BSS·이력 기준을 판단축별로 분리했다.

## 3. 검수 포인트
- 주문/계약/가입, 결제, 해지/환불/취소는 본문 범위로 확장하지 않고 상태·후속 행동 연계로만 다뤘다.
- 최신 요구사항 DB에서 PM-18로 재분류된 상세 요구사항 111건은 별도 Trace Report로 1차 확인한다.
- BSS/연계 시스템이 고객 표시 상태, 제휴 권한, 배송·클레임 이력, 운영 보정 결과를 바꾸는 경우에는 시스템 유즈케이스가 상태 전이에 함께 연결되어야 한다.
