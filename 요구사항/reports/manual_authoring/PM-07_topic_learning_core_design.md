# PM-07 데이터 트래킹 체계 Topic Learning & Core Design

## 1. 작성 목적
데이터 트래킹 체계 정책서는 로그 적재 방식이 아니라 통합채널에서 고객 행동, 업무 완료, 실패, 이탈, 재시도, 상담 전환, 외부 연계 결과를 같은 기준으로 수집하고 해석하는 정책 기준을 정의한다.
수집된 데이터는 고객 여정 개선, 추천·전시 품질 고도화, 장애·품질 이상 탐지, CS/마케팅 운영 성과 분석, 자가해결 확대에 활용되어야 한다.

## 2. 요구사항 해석 기준
요구사항은 상세 요구사항명과 상세 요구사항 설명을 기준으로 재구성했다.
전시/상세 로깅, 로깅·추적 표준화, 추천/이벤트 퍼널 로깅, 공통 스키마, 행동 흐름 이벤트, 검색·결제·주문·배송·서비스 관리·혜택·캠페인·유입 채널·세그먼트 분석 요구사항을 모두 이벤트 체계와 지표 활용 관점으로 묶었다.
최신 요구사항 DB 88건 기준으로 본인인증 통계, 게시판/문의 처리현황, 노이즈 통합 빈도캡, 채널 안정성 로그, 장바구니·구매 퍼널, 앱 이벤트, Open API 로그, 나의 T 혜택 리포트, 리텐션 분석, 판매·개통 일원화 통계, 메뉴별 통계, 담기 실시간 통계 요구사항을 추가 반영했다.
요구사항 문구는 그대로 복사하지 않고 프로세스, 기능, 정책 판단 기준으로 전환했다.

## 3. Core Design Map
- 액터: 고객, 데이터/분석 운영자, 채널 운영자, CS/마케팅 운영자, 채널 업무 시스템, 데이터 분석 플랫폼, BSS/연계 시스템.
- 핵심 유즈케이스: 고객의 데이터 활용 고지·동의 확인, 운영자의 이벤트 표준 관리, KPI 원인 분석, 데이터 품질 이상 대응, 채널 성과 운영, 캠페인·CS 리포트 활용.
- 상태 흐름: 수집 대기 -> 수집 정상 -> 집계 대기 -> 집계 완료를 정상 흐름으로 두고, 검증 실패, 품질 이상, 접근 제한, 수집 중지를 예외·제한 상태로 관리한다.
- 프로세스: 이벤트 설계와 스키마 검증, KPI 분석, 품질 이상 조치, 채널 성과 개선, 캠페인·CS 성과 리포트 흐름으로 구성한다. 개별 통계 화면은 별도 유즈케이스로 분리하지 않고 KPI 조회, 리포트 공유, 품질 대응 프로세스 안에서 다룬다.
- 기능: 이벤트 수집, 스키마 검증, 메타 태깅, 동의 필터, 퍼널 조립, KPI 산정, 대시보드, 통계 조회 조건, 결과 컬럼, 엑셀 생성, 품질 모니터링, 권한·감사, 외부 연계 등 처리 역량으로 정의한다.
- 정책: 수집 범위, 공통 스키마, 개인정보/동의, 식별자, 퍼널, KPI, 품질, 실패 사유, 접근 권한, 리포트, 알람, 보관, 외부 연계, 캠페인, 추천, 자가해결, 운영 통계, 변경 관리 기준을 둔다.

## 4. 작성 금지 및 경계
- AI 검색, 추천, 전시, 상품, 결제, 주문, CS 등 개별 업무 자체를 대신 정의하지 않는다.
- API 필드, DB 컬럼, 로그 테이블 설계, ETL 상세, 데이터 마트 설계는 본문에서 제외한다.
- 단, 정책 판단에 필요한 수집 범위, 필수 이벤트, 보관, 마스킹, 접근 권한, 이력 기준은 정책 항목으로 남긴다.
- 통계 조회 기능은 화면 목록 설명으로 쓰지 않고 조회 기준, 결과 없음, 결과 컬럼, 다운로드, 권한, 감사 기준으로 정책화한다.

## 5. Inspector 메모
- 이벤트 수집 기준이 너무 기술 로그로 흐르지 않도록 고객 여정과 업무 완료 기준으로 검수한다.
- 개인정보·동의·가명화·마스킹 기준이 빠지면 P1로 본다.
- KPI와 리포트가 단순 대시보드 설명으로 끝나면 보완한다.
- 운영 통계 요구사항은 누락되기 쉬우므로 본인인증, 게시판/문의, 판매·개통, 메뉴별, 구매 퍼널, 담기, 알림, 혜택, 리텐션, 채널 안정성, Open API 로그가 정책 상세에 반영됐는지 확인한다.
