# PM-05 AI 검색 Topic Learning & Core Design

## 1. 입력 근거
- 요구사항: PM-05 상세 요구사항 96건.
- 핵심 해석 기준: 상세 요구사항명과 상세 요구사항 설명을 기준으로 검색 진입, 입력, 결과 구성, 비교·실행, 실패 복구, 운영 품질 관리로 재구성했다.
- TK 지향점: AI 검색은 키워드 조회가 아니라 고객 의도를 해석해 정보 탐색, 의사결정, 업무 실행, 문제 해결까지 연결하는 원스톱 경험으로 정의한다.

## 2. 작성 범위
- 전역/전시 내 검색 진입과 검색 입력 UI 표준.
- 검색어 정규화, 자동완성, 최근·인기·연관 검색어.
- 통합 결과, 유형별 탭, 필터, 정렬, 결과 내 재검색, 상세 복귀 상태 유지.
- 자연어/질문형 의도 해석, 조건 추출, Answer Card, 비교·추천 요약, 실행 연결.
- 무결과, 근거 부족, 오류, 노출 제한의 복구 경로.
- 검색 운영 사전, 랭킹, 금칙어, AI 후보 검수, 품질 지표, 실패 케이스 개선.

## 3. Core Design Map

### 액터
- 고객: 검색, 비교, 실행, 상담 전환을 선택하는 주체.
- 검색/AI 운영자: 사전, 랭킹, AI 후보, 품질 지표를 관리하는 주체.
- 콘텐츠/상품 운영자: 검색 대상 자산과 대표 엔티티를 관리하는 주체.
- 상담사: 검색 맥락을 이어 받아 고객 지원을 수행하는 주체.
- AI 검색 엔진, 채널 업무 시스템, BSS/연계 시스템: 의도 해석, 실행 연결, 고객 상태 판정을 수행하는 시스템 액터.

### 유즈케이스 설계
- 사람 액터 Y 유즈케이스 6개, 시스템 N 유즈케이스 3개로 재정리했다.
- 고객 유즈케이스는 검색 입력·결과 확인 같은 단계가 아니라 정보 탐색 완료, 의사결정·실행 연결, 실패 복구·상담 전환이라는 완결 업무 목표로 묶었다.
- 상담사와 콘텐츠/상품 운영자는 각각 상담 응대, 검색 대상 자산 기준 관리 유즈케이스로 직접 연결했다.
- 검색 입력, 결과 구성, 조건 확인, 상담 전환 같은 세부 단계는 프로세스에 배치해 유즈케이스 과분화를 줄였다.

### 상태 설계
- 고객 검색 흐름 상태 14개와 운영 검수 상태 3개를 정의했다.
- 순간 오류는 상태로 남기되, 고객의 다음 행동 또는 운영 후속 조치가 달라질 때만 상태로 승격했다.

### 프로세스 설계
- 고객 검색 흐름 10개 프로세스, 운영 품질 루프 6개 프로세스로 구성했다.
- 각 프로세스는 관련 기능과 관련 정책 ID를 함께 보유한다.

### 정책 설계
- 통합 검색 범위, 입력, 자동완성, 랭킹, 카드·출처, 고객 상태, 비교·추천, 실행 연결, 실패 복구, 이력, 상담, 운영, AI 검수, 품질 지표, 데이터 보호 15개 정책 그룹으로 구성했다.
- 정책 항목은 기간, 단계 수, 우선순위, 표시 기준, 제한 기준, 이력 기준을 포함하도록 작성했다.

## 5. 작성 리스크와 보완
- AI 검색 범위가 추천, 상담, 상품 상세, 결제 등 인접 정책서로 확장될 위험이 있어 실행 연결 기준까지만 작성하고 세부 업무 정책은 인접 정책서로 남겼다.
- AI가 고객 대신 결정을 내리는 표현을 피하고 고객 최종 확인, 원본 확인, 추천 근거 고지를 정책 항목에 포함했다.
- 운영 품질 루프를 별도 운영자 유즈케이스로 구성해 검색 품질 개선이 단순 지표 수집에 머물지 않도록 했다.
