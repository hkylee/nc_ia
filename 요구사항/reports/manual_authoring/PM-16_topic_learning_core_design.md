# PM-16 상품변경 Topic Learning & Core Design Map

## 1. 작성 목표
- 상품변경은 현재 계약 조건을 다른 조건으로 전환하는 업무다.
- 고객이 변경 전에 비용, 혜택, 약정, 구독, 적용 시점, 되돌림 가능 범위를 이해하고 직접 처리할 수 있게 한다.
- BSS와 연계 시스템의 원장 반영, 보류, 실패, 결과 회신 기준을 정책으로 남긴다.

## 2. 요구사항 해석 기준
- 상세 요구사항명과 상세 요구사항 설명을 우선 근거로 사용했다.
- 요구사항 ID는 추적 보조값으로만 사용하고 문서 문구를 그대로 복사하지 않았다.
- 39건 요구사항은 변경 가능 조건, 영향 비교, 예약 변경, 구독·옵션 변경, USIM/eSIM, 일시정지, 상담 fallback, 운영 모니터링으로 재구성했다.
- 대체 요금제 제안, 통합 변경 예상 절감액, 로밍 무료충전 신청 경로, 패키지 상품 교체 금액 비교처럼 세부 요구사항명이 명확한 항목은 정책 상세 판단 기준으로 별도 반영한다.

## 3. Core Design
- 액터는 고객, 법정대리인, 운영자, 상담사, 채널 업무 시스템, BSS/연계 시스템 6개로 통합했다.
- 유즈케이스는 고객 완결 업무 단위 5개, 법정대리인 1개, 운영자 1개, 상담사 1개, 시스템 보조 2개로 구성했다.
- 프로세스는 Y 유즈케이스당 2~5개로 분해해 유즈케이스당 1개 프로세스가 되는 문제를 방지했다.
- 기능은 각 프로세스 수행에 필요한 처리 역량으로 1:1 연결하고 세부 기능 구성은 명사형 처리 단위로 작성했다.
- 정책은 변경 대상·조건·영향·예약·인증·BSS·실패·상담·이력·운영 기준 18개 그룹으로 정리했다.

## 4. 중점 검수 기준
- 상태 전이는 상태 코드 목록에 존재하는 상태만 사용한다.
- 정책 상세는 조건, 시간, 횟수, 상태, 고지, 이력, BSS 반영 기준을 포함한다.
- 상품 목록, 주문, 배송, 결제 승인, 멤버십 혜택 사용 영역으로 범위를 확장하지 않는다.
