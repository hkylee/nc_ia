# PM-32 고객센터_통합허브 Topic Learning & Core Design

## 1. 요구사항 확인

- 기준 데이터: `reports/evidence/requirements.db`
- 정책 모듈: `PM-32`
- Depth 4: `고객센터_통합허브`
- 상세 요구사항 수: 111건
- 검토 기준: 상세 요구사항 ID가 아니라 `상세 요구사항명`, `상세 요구사항 설명`을 중심으로 업무 의미를 재구성했다.

## 2. 작성 목표

고객센터_통합허브는 단순 FAQ 또는 상담 연결 화면이 아니라, 고객이 문제를 스스로 해결할 수 있는지 먼저 판단하고, 해결이 어려운 경우 상담·문의·콜센터 재유입까지 문맥을 잃지 않고 연결하는 고객 지원 허브로 정의한다.

고객이 전화하기 전에 앱/웹에서 해결 가능 여부, 필요 정보, 제한 사유, 다음 행동을 이해할 수 있어야 한다.

상담으로 전환되는 경우에는 직전 과업, 고객 상태, 실패 사유, 상담 문맥, 문의 이력, 처리 상태가 이어져야 한다.

## 3. 핵심 범위

- 셀프 해결 가능 범위와 불가 사유 판정
- 문제 유형별 Self 해결 카드, 준비 정보, 대체 경로 안내
- 상담 채널 선택, 운영 시간, 대기, 문의 유형 기준
- 1:1 문의 접수, 첨부, 개인정보 동의, 상태 관리
- 상담 문맥 전달, Single-view, 콜센터 상담 후 온라인 재유입
- 상품 복잡도 기반 셀프처리·상담 이관 기준
- 고객센터 운영 기준, 품질 기준, 기준정보 변경 이력

## 4. Core Design Map

### 액터

- 고객: 고객센터 통합허브에서 문제 유형을 확인하고 셀프 해결, 문의 접수, 상담 전환, 후속 조치를 수행하는 주체
- 상담사: 고객의 문의와 상담 문맥을 확인하고 응대 결과와 후속 조치를 기록하는 주체
- 운영자: 고객지원 기준, 문의 유형, 상담 채널, 품질 지표, 운영 예외를 관리하는 주체
- AI 상담 시스템: 고객 문의 의도를 분류하고 셀프 해결 답변, 상담 전환 요약, 위험 신호를 생성하는 시스템
- BSS: 고객 상태, 가입 상태, 주문·결제·혜택·회선 상태, 처리 가능 여부를 판정하는 시스템
- 연계 시스템: 콜센터, 채팅, 이메일, 매장 예약, 알림, 외부 상담 채널의 접수·처리 결과를 제공하는 시스템
- 채널 업무 시스템: 고객센터 허브 상태, 문의 이력, 상담 문맥, 알림, 결과 안내를 저장하고 화면에 회신하는 시스템

## 6. Agent 개선 인사이트

고객센터_통합허브는 요구사항에 상담, 문의, 운영, 재유입, Single-view, Self 해결이 함께 나타난다. 일반 검색·가입·결제 축으로 해석하지 않고 셀프 해결, 문의 접수, 상담 문맥, 상담 전환, 운영 품질 축을 우선 적용해야 한다.

또한 `관리`라는 단어만으로 운영 축으로 분류하면 실제 고객 문의·상담 문맥 요구사항이 운영 프로세스로 빠질 수 있으므로, 문맥 단어와 인접 동사를 함께 보고 분류해야 한다.
