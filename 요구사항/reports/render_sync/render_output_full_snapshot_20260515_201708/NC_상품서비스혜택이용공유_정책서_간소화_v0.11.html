1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객이 상품·서비스 혜택을 확인하고 수령, 연결, 사용, 공유, 만료 관리, 오류 복구까지 직접 완료하는 업무 기준을 정의한다.

• 대상 업무는 보유 혜택 요약, 미수령·연결 필요 혜택 관리, 쿠폰형 혜택 전송·양도, 제휴 서비스 이동, 사용처·사용 방식 안내, 만료·소멸 알림, 저활용 구독 관리, 누락·오적용 정정 요청을 포함한다.

• 대상 채널은 앱·웹에서 고객과 공유 수신자가 직접 처리하는 통합채널이며, 혜택 연계 시스템과 제휴사 시스템의 판정·결과 회신을 정책 판단 기준으로 포함한다.

• 상품 가입·변경 자체, 통합 쿠폰/이용권함의 보관 구조, 멤버십 포인트 별도 적립 정책, 외부 BP 계약 운영 상세는 제외하되 혜택 사용 가능성에 영향을 주는 조건은 포함한다.

• 핵심 정책 질문은 고객이 어떤 혜택을 사용할 수 있는가, 언제 수령·공유·양도할 수 있는가, 몇 번까지 재발급·재전송할 수 있는가, 실패하면 어떤 경로로 복구하는가, 이력과 BSS·제휴사 회신은 어디에 남기는가로 둔다.

• 상세 요구사항 30건은 혜택 확인, 수령·연결, 사용·공유·양도, 만료·손실 관리, 누락·오적용 복구, 운영 정책 중 하나 이상에 매핑한다.

나. 설계 원칙

• 혜택 실행 중심: 혜택은 보유 목록이 아니라 고객이 지금 사용할 수 있는 가치와 다음 행동으로 제공한다.

• 조건 사전 판정: 사용, 공유, 수령, 제휴 이동 전 자격, 만료, 제휴 제한, 공유 가능 범위와 오남용 차단 기준을 먼저 판정한다.

• 손실 예방 우선: 만료 예정, 예상 손실, 저활용 혜택은 고객이 놓치기 전에 알림과 우선 사용 경로로 연결한다.

• 제휴 실패 복귀 보장: 외부 서비스 이동 실패는 고객을 막다른 화면에 두지 않고 원래 혜택 문맥으로 복귀시킨다.

• 상태·책임 경계 명확화: 고객 표시 상태는 사용 가능, 미수령, 연결 필요, 만료 예정, 복구 필요처럼 고객 행동 기준으로 구분하고, 시스템 내부 상태는 혜택 연계 시스템 판정, 제휴사 회신, 채널 업무 시스템 화면·알림 구성 상태로 분리한다.

• 개발·QA 전환 가능성: 각 프로세스와 정책 항목은 개발자가 기능 설계 항목으로 전환하고 QA가 조건, 경계값, 실패 사유, 기대 결과를 테스트 케이스로 만들 수 있게 작성한다.

1) 혜택 이용·공유 범위 정책 (PG-BEN-SCOPE-001)

• 대상 업무 범위 (PI-BEN-SCOPE-001-001)
- 본 문서는 혜택 확인, 수령, 연결, 사용, 공유, 만료 관리, 누락 복구, 정정 요청, 상담 전환과 운영자 상태·노출·알림 관리를 포함한다.
• 인접 문서 경계 (PI-BEN-SCOPE-001-002)
- 상품 가입·변경, 통합 쿠폰/이용권함, 멤버십 포인트, 외부 BP 계약 관리는 별도 정책서에서 다루되 혜택 사용 가능성에 영향을 주는 조건은 본 문서에 남긴다.
• 채널 통합 기준 (PI-BEN-SCOPE-001-003)
- 통합채널은 여러 서비스에 흩어진 혜택을 고객 과업 기준으로 묶고, 사용 가능·미수령·연결 필요·만료 예정 상태를 한 화면에서 구분한다.
• 빈 상태 퀵메뉴 기준 (PI-BEN-SCOPE-001-004)
- 구독 중인 상품이 없거나 로그아웃 상태인 고객이 이용 페이지에 진입하면 빈 화면 대신 주요 혜택 카테고리 퀵메뉴를 제공하고, 로그인·가입·혜택 탐색 중 다음 행동을 선택할 수 있게 한다.

2) 혜택 권한·자격 판정 정책 (PG-BEN-ELIG-001)

• 고객 자격 판정 (PI-BEN-ELIG-001-001)
- 혜택 사용 자격은 고객 식별, 가입상품 유지, 회선 상태, 등급, 제휴 계정 연결, 약관·마케팅 동의 중 해당 혜택이 요구하는 조건을 모두 충족할 때 허용한다.
• 상품 유지 조건 (PI-BEN-ELIG-001-002)
- 상품 변경, 결합 해지, 등급 하락, 가입조건 미충족이 발생하면 해당 혜택은 즉시 사용 가능 목록에서 제외하고 자동 정지 또는 해지 후보로 표시한다.
• 공유 수신자 조건 (PI-BEN-ELIG-001-003)
- 공유 수신자는 휴대폰 번호, 고객 식별, 수신 가능 상태, 만료 전 사용 가능 여부를 통과해야 하며 계정 귀속 혜택은 수신자 사용을 제한한다.

3) 혜택 사용 가능성 정책 (PG-BEN-USE-001)

• 즉시 사용 가능 표시 (PI-BEN-USE-001-001)
- 혜택은 사용 가능, 미수령, 발급중, 발급실패, 연결 필요, 만료 예정, 종료, 정지 상태 중 하나로 표시하고 고객 다음 행동을 함께 제공한다.
• 사용처·방식 구분 (PI-BEN-USE-001-002)
- 온라인 사용, 오프라인 제시, 쿠폰 번호, 바코드, 자동 적용, 제휴 앱 이동 방식은 별도 라벨로 구분하고 사용 전 필수 조건을 노출한다.
• 사용 제한 기준 (PI-BEN-USE-001-003)
- 만료, 자격 상실, 제휴 계정 미연결, 동일 혜택 중복 사용, 대상 외 상품, 제휴사 점검 중에는 사용 요청을 제한하고 대체 행동을 안내한다.

4) 공유·양도 정책 (PG-BEN-SHARE-001)

• 공유 가능 혜택 (PI-BEN-SHARE-001-001)
- MMS 쿠폰, 일반 쿠폰, 기프티콘형 혜택은 제휴 정책이 허용하는 경우 수신자에게 공유할 수 있으며 만료일과 사용처 제한을 함께 전달한다.
• 양도 제한 혜택 (PI-BEN-SHARE-001-002)
- 제휴사 계정에 직접 발급되거나 고객 등급·상품 상태에 귀속된 혜택은 타인 사용을 제한하고 공유 버튼을 노출하지 않는다.
• 공유 이력 기준 (PI-BEN-SHARE-001-003)
- 공유 요청, 수신, 사용, 거절, 만료, 재전송 이력은 발신 고객과 수신자 기준으로 각각 저장하고 고객별 조회 가능 범위를 분리한다.

5) 수령·발급·재전송 정책 (PG-BEN-RECV-001)

• 수령 상태 기준 (PI-BEN-RECV-001-001)
- 수령 상태는 수령완료, 미수령, 발급중, 발급실패로 구분하며 발급중이 10분을 초과하면 고객에게 재조회 또는 재발급을 안내한다.
• 재발급 허용 (PI-BEN-RECV-001-002)
- 쿠폰 미수신, 바코드 미노출, 기기변경 후 연결 끊김은 24시간 셀프 재발급 또는 재연결 경로를 제공한다.
• 반복 요청 제한 (PI-BEN-RECV-001-003)
- 동일 혜택 재전송은 기본 1일 3회까지 허용하고 4회 이상 요청 시 추가 인증 또는 상담 전환을 요구한다.
- QA는 1회, 3회, 4회 요청과 실패 재시도 기대 결과를 분리해 검증한다.

6) 외부 제휴 이동·복귀 정책 (PG-BEN-EXT-001)

• 이동 전 고지 (PI-BEN-EXT-001-001)
- 외부 제휴 서비스로 이동하기 전 외부 로그인, 제휴 약관, 앱 설치 필요 여부, 통합채널 복귀 기준을 안내한다.
• 실패 복귀 처리 (PI-BEN-EXT-001-002)
- 인증 실패, 앱 미설치, 세션 만료, 제휴 정책 오류가 발생하면 원래 혜택 상세 문맥으로 복귀하고 재시도, 가이드, 상담 전환을 제공한다.
• 외부 결과 반영 (PI-BEN-EXT-001-003)
- 제휴사 사용 성공 또는 실패 결과는 혜택 연계 시스템에 반영한다.
- FO 채널은 고객 안내와 재시도 경로를 제공하고, BSS·혜택 연계 시스템은 권한·원장 상태를 저장하며, 외부 시스템은 사용·실패 회신 책임을 가진다.

7) 만료·소멸·손실 정책 (PG-BEN-EXP-001)

• 만료 예정 기준 (PI-BEN-EXP-001-001)
- 만료 예정 혜택은 기본 D-7부터 노출하고 고객이 알림 시점을 변경한 경우 고객 설정값을 우선 적용한다.
• 소멸 가치 산정 (PI-BEN-EXP-001-002)
- 예상 손실 가치는 혜택 금액, 포인트, 쿠폰 금액, 사용 가능 회수, 최근 사용 빈도를 기준으로 산정하고 큰 손실부터 우선 노출한다.
• 만료 ID 분리 (PI-BEN-EXP-001-003)
- 만료된 혜택 ID는 사용 가능한 혜택 목록과 분리하고 만료일, 만료 사유, 재발급 가능 여부를 함께 표시한다.

8) 리마인드·마케팅 알림 정책 (PG-BEN-REM-001)

• 알림 동의 기준 (PI-BEN-REM-001-001)
- 소멸 예정, 추천 혜택, 상품 유지 혜택 알림은 고객의 알림 설정과 마케팅 수신 동의가 필요한 경우 해당 동의를 통과해야 발송한다.
• 리마인드 시점 (PI-BEN-REM-001-002)
- 소멸 알림은 D-7과 D-1을 기본 시점으로 제공하고 고객이 끈 혜택 또는 종료된 혜택은 발송 대상에서 제외한다.
• 수신 거부 반영 (PI-BEN-REM-001-003)
- 고객이 알림을 OFF로 변경하면 다음 발송 배치부터 제외하고 변경 이력은 최소 1년 보관한다.

9) 저활용·유지 혜택 관리 정책 (PG-BEN-UNDER-001)

• 저활용 감지 (PI-BEN-UNDER-001-001)
- 혜택이 30일 이상 사용되지 않았거나 월 사용 가능 횟수의 20% 미만으로 사용되면 저활용 후보로 표시한다.
• 대체 행동 연결 (PI-BEN-UNDER-001-002)
- 저활용 안내는 유지, 일시정지, 해지, 플랜 변경, 혜택 사용 가이드 중 하나 이상의 행동으로 연결해야 한다.
• 반복 안내 제한 (PI-BEN-UNDER-001-003)
- 같은 저활용 혜택 안내는 고객이 확인한 뒤 7일 동안 반복 노출하지 않고 고객이 숨김 처리한 경우 30일 동안 제외한다.

10) 정정·확인 요청 정책 (PG-BEN-CORR-001)

• 정정 요청 상태 (PI-BEN-CORR-001-001)
- 정정·확인 요청은 접수, 검토중, 추가 확인 필요, 완료, 반려 상태로 관리하고 각 상태별 고객 안내와 예상 처리일을 제공한다.
• SLA 기준 (PI-BEN-CORR-001-002)
- 일반 정정 요청은 3영업일, 제휴사 확인 필요 건은 5영업일을 기본 SLA로 두며 초과 시 운영 검토 중 상태로 승격한다.
• 반려 사유 고지 (PI-BEN-CORR-001-003)
- 정정 반려 시 자격 불충족, 만료, 제휴 정책 제한, 중복 사용, 증빙 부족 중 하나 이상의 사유를 표시하고 재요청 가능 여부를 안내한다.

11) 가입정보 변경 자동 정지·해지 정책 (PG-BEN-AUTO-001)

• 자동 정지 조건 (PI-BEN-AUTO-001-001)
- 상품 변경, 결합 해지, 등급 하락, 가입조건 미충족으로 혜택 자격을 잃으면 혜택 상태를 자동 정지 또는 해지로 전환한다.
• 사전 영향 프리뷰 (PI-BEN-AUTO-001-002)
- 조건 변경 전 사라지거나 줄어드는 혜택, 만료 전 사용 가능 기간, 회복 조건을 고객에게 미리 보여준다.
• 조건 회복 처리 (PI-BEN-AUTO-001-003)
- 자격 조건이 회복되면 혜택 연계 시스템에서 재판정하고 재연결 또는 재수령이 필요한 혜택은 연결 필요 상태로 안내한다.

12) 운영 노출·상태 정책 (PG-BEN-OPS-001)

• 상태 문구 관리 (PI-BEN-OPS-001-001)
- 운영자는 사용 가능, 미수령, 예정, 만료 예정, 종료, 미적용, 정지 상태별 문구, 유의사항, 다음 행동을 관리할 수 있어야 한다.
• 우선 노출 통제 (PI-BEN-OPS-001-002)
- 개인 혜택 배너와 메시지는 고객군, 등급, 회선 상태, 보유 상품, 만료 임박, 손실 가치 기준으로 우선순위를 정한다.
• 정책 변경 영향 (PI-BEN-OPS-001-003)
- 상태, 노출, 알림 정책 변경 시 영향 고객 수, 노출 화면, 적용 시작일, 롤백 기준을 운영자가 확인해야 한다.

13) 오남용·재노출 차단 정책 (PG-BEN-ABUSE-001)

• 오남용 탐지 조건 (PI-BEN-ABUSE-001-001)
- 동일 혜택 중복 사용, 1일 5회 이상 재조회, 3회 이상 재전송 실패, 비정상 선물 전송, 캡처·복제 위험은 오남용 후보로 분류한다.
• 차단 조치 (PI-BEN-ABUSE-001-002)
- 오남용 후보는 재노출, 재발급, 바코드 표시, 공유 전송을 제한하고 고객에게 추가 인증 또는 상담 확인 기준을 안내한다.
• 오탐 예외 (PI-BEN-ABUSE-001-003)
- 제휴사 장애, 발급 지연, 네트워크 실패가 확인된 경우 오남용 차단 대신 실패 복구 또는 정정 요청으로 전환한다.

14) 이력·보관 정책 (PG-BEN-HIST-001)

• 사용·공유 이력 (PI-BEN-HIST-001-001)
- 혜택 사용, 공유, 수신, 거절, 만료, 재전송, 재발급 이력은 고객, 수신자, 혜택 ID, 처리 시각, 결과 상태로 저장한다.
• 정책 변경 이력 (PI-BEN-HIST-001-002)
- 운영자 정책 변경은 변경 전후 값, 영향 대상, 적용 시작일, 승인자, 롤백 가능 여부를 최소 3년 보관한다.
• 고객 조회 범위 (PI-BEN-HIST-001-003)
- 고객은 최근 1년의 혜택 사용 이력과 최근 3개월의 공유·재전송 이력을 조회할 수 있으며 타인의 민감 정보는 마스킹한다.

15) 상담 fallback 정책 (PG-BEN-CS-001)

• 상담 전환 조건 (PI-BEN-CS-001-001)
- 제휴 정책상 셀프 처리가 불가능하거나 24시간 복구 실패, SLA 초과, 고액 손실 혜택 오적용이면 상담 전환을 제공한다.
• 문맥 전달 범위 (PI-BEN-CS-001-002)
- 상담사에게는 가입상품, 혜택 상태, 오류 단계, 고객 재시도 이력, 제휴사 응답, 정정 요청 번호만 전달하고 불필요한 민감 정보는 제외한다.
• 상담 결과 회신 (PI-BEN-CS-001-003)
- 상담 처리 결과는 복구 완료, 운영 검토, 제휴사 확인, 반려, 고객 추가 행동 필요 중 하나로 회신하고 고객 화면에 반영한다.

최종 점검 기준

상품·서비스 혜택 이용/공유 정책서 제출 전 점검
1. 혜택 이용·공유 범위가 수령, 연결, 사용, 공유, 만료, 복구까지 완결 흐름으로 정의되어 있는가.
2. 액터가 책임 주체 단위로만 정의되고 고객 상태가 액터로 분리되지 않았는가.
3. 각 Y 유즈케이스가 2개 이상의 프로세스로 분해되어 있는가.
4. 상태 전이 기준이 고객 안내, 기능 허용, 정책 판단, 후속 처리 기준으로 연결되어 있는가.
5. 프로세스가 관련 기능과 관련 정책 ID·명칭으로 연결되어 있는가.
6. 기능의 세부 기능 구성이 설명문이 아니라 하위 처리명으로 작성되어 있는가.
7. 정책 항목에 공유 제한, 만료, 알림, 재발급, 상담 전환, BSS/제휴 반영 기준이 구체적으로 들어 있는가.
8. 요구사항 30건이 요구사항 원문 복사가 아니라 프로세스, 기능, 정책 항목으로 재구성되어 있는가.
9. 외부 제휴 이동 실패, 누락·오적용, 오남용 차단, 정정 요청, SLA 초과 예외 흐름이 고객 후속 행동과 연결되어 있는가.
10. 고객 이력, 운영 변경 이력, 감사 로그와 개인정보 최소 전달 기준이 명확한가.
11. 고객 표시 상태와 시스템 내부 상태가 분리되어 있고 FO, BSS, 외부 시스템 간 책임 경계가 명확한가.
12. 개발자는 기능 설계 항목으로, QA는 조건·경계값·실패 사유·기대 결과 테스트 케이스로 전환할 수 있는가.
13. 요구사항 추적성
상세 요구사항은 유즈케이스, 상태, 프로세스, 기능, 정책 또는 정책 항목 중 하나 이상으로 추적 가능해야 한다.
14. 개발·QA 전환성
각 정책 항목은 테스트 조건, 기대 결과, 실패·복구 경로로 전환 가능해야 한다.
15. 책임 경계 정합성
FO, BSS, 연계 시스템, 외부 주체의 판단·저장·회신 책임이 혼재되지 않아야 한다.
