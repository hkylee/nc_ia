1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객이 해지, 주문 취소, 결제 취소, 환불, 예약 해지 철회를 직접 처리하기 위한 기준을 정의한다.

• 포함 범위는 가능 여부 판정, 영향도 고지, 환불·정산 산정, 요청 접수, 예약·철회, BSS 원장 반영, 알림, 운영 보정, 이력·증빙 보관이다.

• 대상 채널은 앱·웹 FO이며 고객이 처리 대상 선택부터 결과 확인까지 셀프 처리할 수 있는 흐름을 포함한다.

• 상품 상세, 주문 변경, 멤버십 혜택 자체 기준, 외부 BP 운영 상세는 제외하되 해지·취소·환불 결과에 영향을 주는 조건은 본 문서에 포함한다.

• 인터페이스 항목명, 저장소 물리 구조, 운영자 화면 상세, 정산 배치 설계는 후속 상세 설계 산출물에서 정의한다.

• 해지·취소·환불 진입은 알림, 주문완료, 서비스 상세, 비회원 인증, 외부 제휴 서비스 등 다양한 경로에서 들어올 수 있으나 동일한 가능 여부·영향도·최종 확인 기준으로 수렴한다.

• 포함 범위는 해지·취소·환불 가능 여부 확인, 손실 영향 고지, 요청 접수, 예약·철회, 환불 산정, 혜택 회수, BSS 원장 반영, 결제 금융기관 회신, 알림·증빙 이력이다.

• 제외 범위는 상품 탐색, 신규 주문 생성, 배송·교환·반품 물류 상세, 외부 BP 운영 화면, 정산 배치 설계이며 해지·취소·환불 결과에 직접 영향을 주는 조건만 정책 항목으로 연결한다.

• 핵심 정책 질문은 누가 해지·취소·환불을 요청할 수 있는가, 언제 즉시취소와 일반해지가 갈리는가, 몇 번까지 재처리할 수 있는가, 어떤 조건이면 위약금·혜택 회수·재가입 제한이 발생하는가, 실패하면 어떤 보류와 복구 경로를 남기는가이다.

• 통합 리스크는 BSS 원장 반영과 결제 금융기관 환불 회신 불일치, 외부 제휴 서비스 해지 지연, 알림 딥링크 해지 경로 오진입, 채널별 처리 상태 차이이며 완료 표시는 모든 필수 반영 결과 확인 후 확정한다.

• 상세 요구사항 48건은 취소·해지 가능 여부, 손실 영향 확인, 즉시취소·일반해지, 환불·위약금·혜택 회수, 예약·철회, 외부 제휴 해지 기준으로 재구성해 유즈케이스·프로세스·기능·정책 중 하나 이상에 매핑했다.

나. 설계 원칙

• 손실 사전 인지: 고객이 환불액, 차감액, 위약금, 소멸 혜택, 재가입 제한을 요청 전에 이해하도록 한다.

• 가능 여부 선판정: 처리 진입 초기에 대상 상태와 정책 조건을 판정해 불필요한 입력과 상담 전환을 줄인다.

• 예약·철회 분리: 즉시 종료와 예약 해지를 분리하고 효력 발생 전 철회 가능 여부를 상태로 관리한다.

• 정산 투명성: 환불 산정 항목과 차감 사유를 고객 표시 기준과 원장 반영 기준에서 일치시킨다.

• 부분 실패 보정: 일부 반영 실패나 정산 불일치는 완료로 숨기지 않고 보류·운영 보정 상태로 관리한다.

• 증빙 가능성: 고객 고지, 최종 확인, BSS 반영, 알림, 보정 이력을 요청 단위로 추적 가능하게 남긴다.

현재 상태전이 이벤트다음 상태처리 기준 및 후속 처리
진입 전처리 대상 선택대상 확인 중고객이 주문, 계약, 구독, 부가서비스 중 처리 대상을 선택하면 대상 상태를 조회한다.
대상 확인 중대상 상태 조회 완료영향 확인 필요출고, 개통, 사용 개시, 약정, 결합, 미납, 환불 가능 기간을 기준으로 영향 산정을 시작한다.
대상 확인 중정책 제한 확인처리 제한당일 취소 불가, 정산 확정, 외부 처리 완료, 권한 부족이면 자동 처리를 제한한다.
영향 확인 필요고객 영향도 확인 완료인증·동의 필요고객이 예상 환불액, 차감액, 위약금, 소멸 혜택, 적용 시점을 확인하면 인증·동의 단계로 이동한다.
인증·동의 필요인증·최종 동의 완료신청 가능본인확인과 최종 확인 이력을 저장하고 요청 접수를 허용한다.
신청 가능예약 해지 선택예약 유지 중정기결제 또는 구독 상품에서 효력 발생 전까지 예약 상태를 유지한다.
신청 가능즉시 처리 요청처리 중BSS와 결제 금융기관에 처리 요청을 전달하고 중복 요청을 제한한다.
예약 유지 중고객 철회 요청처리 완료효력 발생 전 철회가 허용되면 기존 이용 상태로 복구하고 철회 이력을 저장한다.
처리 중처리 성공처리 완료원장, 결제, 혜택 상태가 모두 반영되면 완료 상태로 전환한다.
처리 중외부 응답 지연 또는 정산 불일치정산 보류환불 예정일 또는 정산 금액이 확정되지 않으면 보류 상태로 표시한다.
정산 보류운영 보정 필요 판정운영 보정 필요반복 실패, 오지급, 일부 반영 실패는 운영 보정 큐에 등록한다.
처리 완료원장 반영 확정복구 불가해지 완료 또는 환불 완료 후 원장 반영이 확정되면 원복을 제한한다.

3) 상태 전이 다이어그램

상태 전이 다이어그램은 대표 상태와 전이 이벤트를 개념적으로 보여주며, 상세 판정 기준은 상태 전이표를 따른다.

1) 해지·취소·환불 범위 정책 (PG-TRM-SCP-001)

• 처리 유형 구분 (PI-TRM-SCP-001-01)
- 고객 요청은 해지, 주문 취소, 결제 취소, 환불, 예약 해지 철회 중 하나로 분류한다.
- 동일 요청 안에서 복수 유형이 필요한 경우 고객에게 주 처리 유형과 후속 처리 유형을 분리해 고지한다.
• 상품 유형별 후속 처리 (PI-TRM-SCP-001-02)
- 주문형 상품은 출고·개통 전에는 취소를 우선 적용하고, 사용 개시 후에는 해지 또는 환불 심사로 전환한다.
- 구독형 상품은 해지 예약과 즉시 종료를 분리해 고객이 선택하게 한다.
• 인접 업무 경계 (PI-TRM-SCP-001-03)
- 상품 상세, 주문 변경, 멤버십 혜택 자체의 상세 기준은 본 문서에서 직접 정의하지 않는다.
- 단, 해지·취소·환불 결과에 영향을 주는 결합·혜택·주문 상태는 영향도와 정책 항목으로 반영한다.

2) 가능 여부 판정 정책 (PG-TRM-ELG-001)

• 가능 상태 판정 (PI-TRM-ELG-001-01)
- 처리 가능 여부는 주문 상태, 계약 Activation, 개통·출고·사용 개시 여부, 결합·약정 상태, 미납·분쟁 이력, 외부 결제 응답을 모두 확인한 뒤 확정한다.
• 당일 취소 기준 (PI-TRM-ELG-001-02)
- 신규가입 또는 번호이동 당일에 해지를 요청하면 즉시 취소 가능 여부를 먼저 판정한다.
- 가입 익일부터는 일반 해지 기준으로 전환하고 환불·위약금·혜택 회수 기준을 함께 산정한다.
• 처리 제한 기준 (PI-TRM-ELG-001-03)
- 이미 출고, 개통, 혜택 사용, 정산 확정, 외부 제휴 처리 완료 상태이면 자동 처리를 제한한다.
- 제한 시에는 불가 사유, 상담 전환 또는 운영 검토 경로를 함께 제공한다.
• 고객 유형별 제한 기준 (PI-TRM-ELG-001-04)
- 명의자 불일치, 법정대리인 동의 필요, 법인·대리 처리, 분쟁 접수 고객은 자동 해지·환불을 제한한다.
- 제한 고객은 필요한 증빙, 승인 주체, 상담 전환 기준을 함께 안내한다.
• 해지 가능 정책 (PI-TRM-ELG-001-05)
- 신규가입 또는 번호이동 상품은 가입 당일에만 즉시 취소를 우선 판정한다.
- 가입 익일부터는 일반 해지로 전환하고 위약금, 할인 반환금, 혜택 회수, 재가입 제한을 함께 산정한다.
• 고객·약정 프로그램 단독해지 (PI-TRM-ELG-001-06)
- 고객 프로그램 또는 약정 프로그램이 본 계약과 독립 관리되는 경우 단독 해지를 허용한다.
- 단독 해지 시 남는 상품, 할인, 약정 영향은 고객 영향 요약에 포함한다.
• 비회원 간편해지 (PI-TRM-ELG-001-07)
- 가입 완료 문자, 주문번호, 본인확인 키로 대상이 식별되는 부가서비스는 로그인 없이 최소 인증으로 간편 해지 경로를 제공한다.
- 민감 정보 조회나 환불 계좌 변경이 필요한 경우에는 추가 인증으로 전환한다.
• 가입 익일부터 해지를 요청하는 경우 일반해지 처리 (PI-TRM-ELG-001-08)
- 가입 익일부터 해지를 요청하는 경우 일반해지로 처리한다.
- 일반해지 전환 시 고객에게 위약금, 할인 반환금, 잔여 청구, 혜택 회수, 재가입 제한 기준을 함께 고지한다.

3) 영향도 사전 고지 정책 (PG-TRM-IMP-001)

• 핵심 영향 요약 (PI-TRM-IMP-001-01)
- 최종 요청 전 예상 환불액, 차감액, 위약금, 소멸 혜택, 결합 영향, 재가입 제한, 적용 시점을 요약 카드로 표시한다.
- 되돌릴 수 없는 항목은 별도 확인을 받아야 한다.
• 동시해지 안내 (PI-TRM-IMP-001-02)
- 이동전화 계약과 단말 구매 계약이 지원금, 할부, 결합 조건으로 연결된 경우 동시 해지를 기본 안내로 제공한다.
- 단독 해지가 가능한 고객·약정 프로그램은 단독 처리 가능 조건을 분리해 고지한다.
• 해지 후 후속 경로 (PI-TRM-IMP-001-03)
- 완료 화면에는 나의 요금, 결합 관리, 구독 관리, 환불 내역, 재가입 가능일 확인 경로를 제공한다.
- 고객이 추가로 해야 할 행동이 없으면 완료 상태와 이력 저장 결과를 명확히 표시한다.
• 해지 요금 시뮬레이션 (PI-TRM-IMP-001-04)
- 고객이 해지 기준일을 선택하면 다음 달 예상 청구 금액, 잔여 이용료, 할인 반환금, 위약금, 결합 할인 종료 영향을 사전에 계산해 표시한다.
• 위약금 시뮬레이션 (PI-TRM-IMP-001-05)
- 약정 중 해지 또는 변경 시 잔여 기간, 지원금, 할인 누적액, 반환 제외 조건을 반영해 예상 위약금을 산정한다.
- 산정 기준일과 실제 청구 금액 차이가 발생할 수 있는 사유를 함께 고지한다.
• 결합 해지 요금 변화 시뮬레이션 (PI-TRM-IMP-001-06)
- 결합 해지 시 결합 할인 종료, 대표 회선 변경, 가족 구성원 영향, 다음 청구월 예상 금액 변화를 사전 계산해 고객에게 제공한다.

4) 환불·정산 산정 정책 (PG-TRM-SET-001)

• 환불 산정 기준 (PI-TRM-SET-001-01)
- 환불액은 결제 금액, 사용 개시 여부, 이용 일수, 혜택 사용량, 차감 대상 금액, 위약금, 수수료를 분리해 산정한다.
- 산정 결과는 고객 표시 금액과 BSS 원장 반영 금액이 일치해야 한다.
• 부분 환불 기준 (PI-TRM-SET-001-02)
- 부분 취소, 부분 환불, 부분 해지는 대상 상품 단위와 사용 이력 단위가 모두 식별될 때만 허용한다.
- 일부 항목이 정산 확정 상태이면 해당 항목만 보류하고 나머지 항목은 독립 처리한다.
• 환불 계좌 기준 (PI-TRM-SET-001-03)
- 원결제수단 환불이 불가하거나 자동납부 계좌가 없는 고객은 환불 계좌 입력과 계좌 인증을 완료해야 한다.
- 계좌 인증 실패 시 환불 요청은 접수 보류 상태로 저장하고 재입력 경로를 제공한다.
• 환불 일정 기준 (PI-TRM-SET-001-04)
- 고객에게 환불 예정일 또는 처리 단계별 예정 범위를 사전에 안내한다.
- 외부 결제기관 응답 지연으로 일정 확정이 불가하면 지연 사유와 다음 확인 가능 시점을 알림으로 제공한다.
• 환불 실패 재처리 기준 (PI-TRM-SET-001-05)
- 원결제 취소 실패, 계좌 오류, 외부기관 응답 실패는 1차 자동 재시도 후 보류로 전환한다.
- 동일 실패가 2회 반복되면 운영 보정 대상으로 등록하고 고객에게 재입력 또는 상담 경로를 제공한다.

5) 요청 접수·철회 정책 (PG-TRM-REQ-001)

• 요청 접수 기준 (PI-TRM-REQ-001-01)
- 고객 최종 확인과 필수 인증·동의가 완료된 경우에만 해지·취소·환불 요청을 접수한다.
- 동일 고객의 동일 대상 요청은 처리 중 상태에서 중복 접수를 제한한다.
• 예약 해지 기준 (PI-TRM-REQ-001-02)
- 정기결제 또는 구독 상품은 다음 결제일 전까지 해지 예약을 접수할 수 있다.
- 예약 상태에서는 효력 발생 시점, 이용 가능 기간, 철회 가능 여부를 함께 저장한다.
• 철회·복구 기준 (PI-TRM-REQ-001-03)
- 예약 해지 상태는 효력 발생 전까지 철회를 허용한다.
- 즉시 해지 또는 환불 완료 후에는 원장 반영 전 상태일 때만 복구 가능으로 표시하고, 원장 반영 완료 후에는 재가입 또는 재신청 경로로 안내한다.
• 당일가입 즉시취소 (PI-TRM-REQ-001-04)
- 가입 당일 부가서비스 또는 주문형 상품이 사용 개시 전이면 상담 없이 즉시 취소를 허용한다.
- 취소 완료 후 고객에게 처리 결과와 재가입 가능 여부를 즉시 표시한다.
• 알림연계 원터치해지 (PI-TRM-REQ-001-05)
- 유지기간 만료 또는 종료 예정 알림에서 진입한 경우 로그인 후 동일 서비스 해지 플로우로 바로 연결한다.
- 알림 진입 컨텍스트가 확인되면 대상 선택 단계를 생략하고 영향 확인부터 진행한다.
• 외부 제휴서비스 원터치 해지 (PI-TRM-REQ-001-06)
- 외부 제휴 부가서비스는 서비스 상세 화면에서 해지 버튼을 제공하고, 제휴 응답 성공 전에는 완료로 표시하지 않는다.
- 제휴 응답 실패 시 재시도 또는 상담 전환 경로를 제공한다.
• 즉시 해지 지원 (PI-TRM-REQ-001-07)
- 우주패스 등 구독형 상품은 즉시 해지 가능 조건을 먼저 판정하고, 가능하면 앱에서 셀프 해지를 완료한다.
- 즉시 해지 불가 상태이면 예약 해지, 다음 결제일 종료, 상담 전환 중 가능한 경로를 안내한다.

6) 상태 전환·복구 정책 (PG-TRM-STA-001)

• 상태 단계 기준 (PI-TRM-STA-001-01)
- 요청 상태는 접수, 검토 중, 처리 중, 완료, 제한, 보류, 운영 보정 필요로 구분한다.
- 각 상태에는 현재 처리 주체와 고객이 수행할 다음 행동을 함께 제공한다.
• 상태 복구 기준 (PI-TRM-STA-001-02)
- 실패 또는 보류 상태는 원인 제거 후 동일 요청을 재처리할 수 있으면 복구 가능으로 표시한다.
- 이미 해지 완료, 환불 완료, 정산 확정이면 복구 불가로 표시하고 재가입 가능일 또는 상담 경로를 안내한다.
• 정지해제 상태 전환 (PI-TRM-STA-001-03)
- 정지해제는 행위유형, 적용시점, 요청주체 조합으로 판정한다.
- 계약 단위 상태가 정지해제 가능이면 처리 중으로 전환하고, 원장 반영 성공 후 이용 가능 상태로 복구한다.
• 상품 재가입 가능일 (PI-TRM-STA-001-04)
- 재가입 불가 상태에서는 이전 해지일과 제한 기간을 기준으로 재가입 가능일을 계산해 표시한다.
- 재가입 가능일이 산정되지 않으면 제한 사유와 상담 경로를 함께 제공한다.

7) BSS·원장 반영 정책 (PG-TRM-BSS-001)

• 원장 반영 기준 (PI-TRM-BSS-001-01)
- BSS는 계약 Activation, 주문 상태, 청구·수납 상태, 혜택 상태를 한 요청 단위로 반영한다.
- 일부 반영 실패 시 고객 표시 상태는 보류로 두고 성공·실패 항목을 분리 저장한다.
• 외부 결과 동기화 (PI-TRM-BSS-001-02)
- PG, 제휴사, 배송·개통 시스템 결과는 처리 완료 전 최신 상태로 재조회한다.
- 외부 응답과 BSS 상태가 불일치하면 자동 완료를 금지하고 운영 보정 대상으로 등록한다.

8) 결과 고지·알림 정책 (PG-TRM-NTC-001)

• 상태 변경 알림 (PI-TRM-NTC-001-01)
- 해지 예약 접수, 처리 일정 확정, 완료, 실패, 보류, 환불 예정일 변경 시 고객에게 푸시 또는 문자 알림을 발송한다.
- 알림에는 처리 대상, 적용 시점, 고객 후속 행동을 포함한다.
• 불가 사유 안내 (PI-TRM-NTC-001-02)
- 처리 불가 또는 제한 시에는 한 문장 사유와 고객이 선택할 수 있는 다음 행동을 함께 제공한다.
- 단순 오류 문구만 제공하는 처리는 금지한다.
• 고지 재확인 기준 (PI-TRM-NTC-001-03)
- 비용 손실, 혜택 소멸, 복구 불가, 환불 지연처럼 고객 영향이 큰 항목은 최종 요청 전 별도 확인을 받는다.
- 확인 이력이 없으면 요청 접수 상태로 전환하지 않는다.

9) 운영 예외·보정 정책 (PG-TRM-OPR-001)

• 예외 승인 기준 (PI-TRM-OPR-001-01)
- 운영자는 상품 유형, 결제 상태, 사용 이력, 민원·분쟁 여부, 환불 금액 영향을 확인한 뒤 예외 승인 또는 반려를 결정한다.
- 예외 승인에는 승인자, 사유, 적용 범위, 고객 고지 이력을 남긴다.
• 지연·불일치 보정 기준 (PI-TRM-OPR-001-02)
- 환불 지연, 오지급, 정산 불일치, 부분 실패 건은 운영 보정 큐에 등록한다.
- 보정 완료 전에는 고객 표시 상태를 보류로 유지하고 임의 완료 처리하지 않는다.

10) 이력·증빙 보관 정책 (PG-TRM-HIS-001)

• 처리 이력 기준 (PI-TRM-HIS-001-01)
- 요청 접수, 가능 여부 판정, 영향도 고지, 고객 최종 확인, BSS 반영, 외부 결과, 알림 발송, 운영 보정 이력을 요청 ID 기준으로 저장한다.
• 증빙 조회 기준 (PI-TRM-HIS-001-02)
- 분쟁 또는 감사 대응을 위해 요청 당시 고객에게 고지한 영향도, 환불 산정 결과, 알림 발송 결과, 운영 승인 이력을 동일 화면에서 조회할 수 있어야 한다.
• 개인정보·로그 보관 기준 (PI-TRM-HIS-001-03)
- 환불 계좌, 인증 결과, 알림 수신 정보처럼 개인정보가 포함된 이력은 마스킹 조회를 기본으로 한다.
- 로그와 증빙 이력은 감사 목적 보관 기간이 지난 뒤 파기 대상과 보존 대상을 분리한다.

최종 점검 기준

해지·환불·취소 정책서 제출 전 점검
