0-2. v0.17 기준선 확정 [v0.17 확정]

v0.17은 2026-05-14까지 협의한 내용을 기준선으로 확정한 버전이다. 상품유형은 BSS 분류를 변경하지 않고 채널 상품상세 코드로 매핑한다. 상품상태는 BSS PLM 상태와 채널 Front 표시 상태를 분리하며, 고객 노출 대상·CTA·담기/가입 가능 여부를 채널 상태 기준으로 판단한다. 내일 작업은 v0.18로 이어간다.

0-1. v0.16.8 변경 요약 [v0.16.8 수정]

본 문서는 v0.16 HTML 템플릿 구조와 서식을 유지한 변경표시본이다. v0.16.8에서는 2.1.2를 “상품 유형 협의가 필요한 영역”으로 재정리하고, 협의 대상 항목에 붉은 별표와 “1팀 협의 필요” 표시를 추가했다. 정책서 반영 방향은 채널 판단을 기준으로 수정하되, BSS LV1/LV2/LV3 원문 분류는 변경하지 않는 원칙을 유지한다.

1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객이 상품 상세를 이해하고, 비교·조건 확인·구성 선택 후 담기까지 완료하는 기준을 정의한다.

• 대상 업무는 상품 상세 요약, 미디어·스펙, 가격·혜택, 가입 가능성, 비교·추천, 옵션 선택, 담기 실행, 담기 실패 복구를 포함한다.

• 대상 채널은 앱·웹 FO이며 Product Catalog, BSS, AI 추천, 장바구니, 채널 업무 시스템과의 정책 판단 연계를 포함한다.

• 장바구니 안의 수량 변경, 장바구니 목록 유지, 주문서 작성, 결제, 배송 상세는 후속 정책서에서 다룬다.

• API 필드, DB 컬럼, 운영자 화면 상세, 추천 모델 알고리즘 상세는 제외하되 고객 판단과 정책 판정에 영향을 주는 기준은 정책 항목으로 남긴다.

• 최신 요구사항 기준으로 Product Catalog 연동, 상품 라이프사이클, 전시 상태 코드, 운영 템플릿, 후기·Q&A, 키워드·카테고리, 디바이스·액세서리·연동기기, 외부채널 상품 설정 운영 기준을 포함한다.

• 운영자 화면의 세부 UI와 입력 필드 자체는 제외하되, 등록·수정·삭제 허용 조건, 검증 실패, 일괄 변경 부분 실패, 영향도 확인, 변경 이력처럼 정책 판단에 필요한 기준은 포함한다.

• 채널 업무 시스템은 상품 상세 표시, 조건 안내, 담기 실행, 담기 실패 복구를 책임지고 상품 연계 시스템은 상품 기준 정보, 고객 조건, 재고·판매 가능성, BSS 기준 정보 검증을 책임진다.

• 고객 표시 상태는 상세 확인 가능, 조건 확인 필요, 담기 가능, 담기 제한, 담기 실패로 안내하고 내부 상태는 상품 기준 검증, 고객 조건 판정, 재고·판매 상태, 담기 이력으로 분리한다.

나. 설계 원칙

• 고객 의사결정 중심: 상품 정보는 상품군 구조가 아니라 고객이 이해, 비교, 선택, 담기하는 순서로 구성한다.

• 조건 사전 고지: 가입·구매·배송·재고·결제수단·혜택 제한은 주문 진입 전 또는 담기 직전에 먼저 알려야 한다.

• 선택 유지: 옵션 선택, 로그인, 인증, 딥링크, 상담 전환 후에도 고객이 선택한 상품·옵션·비교 맥락을 유지한다.

• 정책 기반 담기: 담기 가능 여부는 상품 상태, 고객 상태, 조합 정책, 재고, 판매 기간, 채널 판매 가능성을 함께 판정한다.

• 운영 통제 가능성: 템플릿, 정책 문구, 비교 기준, 상품 관계, 재고·판매 상태, 알림 기준은 변경 이력과 영향도 기준으로 관리한다.

• AI 보조 설명 책임: AI 요약과 추천은 원장 정보와 고객 상태를 근거로 표시하고, 생성 기준과 원문 이동 경로를 제공한다.

• 원장 우선·채널 보조: 상품명, 가격, 가입 조건, 혜택, 라이프사이클, 전시 상태는 Product Catalog와 BSS 기준을 우선하며 채널 문구는 정책값을 덮어쓸 수 없다.

• 운영 안전성: 삭제, 일괄 변경, 카테고리 이동, 키워드 변경, 외부채널 설정은 참조 데이터와 고객 노출 영향 확인 후 허용한다.

2. 정의 [v0.16.3 수정]

본 장은 기존 “주요 용어”를 “정의”로 변경하고, 정책 판단에 필요한 기준을 2.1 상품유형과 2.2 상품용어 정의로 분리한다.
상품유형은 BSS 상품정책의 Product Offering 유형을 기준으로 삼되, 채널 화면·CTA·담기·혜택 표시가 달라지는 T4S 실사용 유형을 보조 축으로 반영한다.

v0.16.3 반영 기준: BSS 유형은 원장·계약·오더 기준의 기준 유형으로 유지한다. 채널 정책서에서는 고객 화면, CTA, 담기, 가격·혜택, 불가 사유 안내를 위해 채널 상품유형 보조 축을 함께 둔다. 삭제 대상은 없으며, 로밍·구독·USIM/eSIM·혜택/멤버십·커머스/사은품·복합서비스는 채널 기준에서 보강한다.

2.1 상품유형 [v0.16.7 수정]

상품유형은 BSS의 상품 분류 체계를 기준으로 정의한다.
채널 정책서에서는 BSS의 LV1·LV2·LV3를 변경하지 않고, 원장·계약·오더 판정의 기준으로 그대로 사용한다.
다만 고객 화면에서 동일한 상세 템플릿, CTA, 담기, 판정 로직으로 처리할 수 있는 경우에는 우측의 상품상세 코드로 묶어 관리한다.
즉 BSS 분류는 기준 분류이고, 상품상세 코드는 채널 표시·처리 단위다.

v0.16.7 반영 기준: 이동전화 base 제휴서비스에서 T우주 제휴상품을 제외했다. 보험형과 클럽형은 BSS LV3를 분리 유지하되 채널 상품상세 코드는 제휴 보장·혜택형으로 묶었다. BSS LV3의 플랫폼 명칭은 그대로 두고, 채널 판단에는 플랫폼형(커머스)라는 표현을 사용했다. 기기서비스와 유선서비스는 상품상세의 CTA와 검증 로직 차이가 커서 채널 코드도 분리했다.

1) 상품상세/담기 범위·경계 정책 (PG-PDD-SCOPE-001)

• 범위 경계 (PI-PDD-SCOPE-001-01)
- 상품 상세는 고객의 상품 이해와 선택 판단을 다루고, 담기는 선택 구성을 주문·장바구니로 넘기기 전 임시 확정까지 다룬다.
- 장바구니 내 수량 변경, 결제, 배송 상세는 후속 정책서에서 제외한다.
• 담기 정의 (PI-PDD-SCOPE-001-02)
- 담기는 상품, 옵션, 요금제, 부가서비스, 혜택, 프로그램 조합과 판정 결과를 함께 저장한 상태로 본다.
- 단순 버튼 클릭이나 장바구니 목록 관리는 담기 정책의 본문 범위가 아니다.
• 인접 정책서 경계 (PI-PDD-SCOPE-001-03)
- 카트/장바구니는 담긴 상품의 유지·변경·삭제를 담당하고, 주문/결제는 신청서와 결제 처리를 담당한다.
- 상품상세/담기는 담기 직전 조건과 담기 완료 후 분기만 정의한다.

2) 상품 상세 표준 템플릿 정책 (PG-PDD-TPL-001)

• 표준 섹션 (PI-PDD-TPL-001-01)
- 상품 상세 표준 템플릿은 핵심 요약, 혜택 구성, 이용 조건, 가격·혜택, 비교, 후기·Q&A, 유의 문구, 담기 행동 영역을 기본 섹션으로 둔다.
- 상품군별로 숨김은 허용하되 섹션 순서 기준은 유지한다.
• 상품군 특화 (PI-PDD-TPL-001-02)
- 휴대폰은 단말 가격과 배송 유형, 요금제는 기본 혜택과 약정 비교, 로밍·부가서비스는 이용 조건·과금 방식·적용 시점·해지 조건을 필수 항목으로 표시한다.
• 앵커 구조 (PI-PDD-TPL-001-03)
- 탭 또는 앵커 전환 시 스크롤과 선택 상태는 유지한다.
- 정보량이 많은 상품은 요약+상세 펼침으로 나누고, 고객이 담기 전에 확인해야 할 제한 조건은 접힌 영역 안에만 두지 않는다.
• 공통 모듈 (PI-PDD-TPL-001-04)
- 가격, 상품 구성, 체감 혜택, 사용 방법, 고시 정보는 공통 모듈로 관리한다.
- 상품군별 데이터가 달라도 위치, 명칭, 단위, 문구 구조는 동일하게 유지한다.
• 상품군별 필수 템플릿 (PI-PDD-TPL-001-05)
- 상품군별 표준 템플릿은 공통 모듈과 특화 모듈을 구분한다.
- 휴대폰은 단말·배송·공시지원금, 요금제는 데이터·약정·혜택, 부가서비스·로밍은 이용 조건·과금 방식·적용 시점·해지 조건을 필수 검토 항목으로 둔다.
• 플러스·구독 상품 안내 (PI-PDD-TPL-001-06)
- 플러스 상품과 우주패스처럼 결합 구매 구조가 낯선 상품은 상품 상세 상단에서 개념, 동시 구매 조건, 할인 적용 방식, 해지 영향, 구독 전환 단계를 별도 안내한다.
• 상품군 CTA 기준 (PI-PDD-TPL-001-07)
- 인터넷, B tv, 구독, 단품, 복합 상품은 담기, 바로가입, 바로결제, 구독하기 CTA를 상품군과 주문 방식에 맞게 구분한다.
- 고객에게 다음 단계가 장바구니인지 결제창인지 신청서인지 명확히 알려야 한다.

3) 상품 요약·미디어·스펙 표시 정책 (PG-PDD-SUMMARY-001)

• 핵심 요약 (PI-PDD-SUMMARY-001-01)
- 상품 상세 요약은 상품명, 대표 이미지, 상품 유형, 가격·할인 유형, 핵심 혜택, 가입 가능성 중 최소 5개 항목을 상단에 제공한다.
- 고객 상태 기반 문구는 로그인 상태에서만 개인화한다.
- 상품정보 및 가격 확인은 고객이 담기 전 동일 요약 영역에서 수행할 수 있어야 한다.
• 미디어 뷰어 (PI-PDD-SUMMARY-001-02)
- 상품 이미지, 배너, 동영상은 확대와 스와이프를 허용하고, 대체텍스트를 필수로 둔다.
- 미디어가 상품 조건을 설명하는 경우 동일 정보를 텍스트로도 제공한다.
• 스펙 요약 (PI-PDD-SUMMARY-001-03)
- 상품 설명과 스펙은 요약과 상세 펼침으로 구분한다.
- 요약 영역에는 고객이 담기 전 판단해야 할 조건, 제한, 기간, 대상 여부를 우선 표시한다.
• 상품 리뷰·평점·Q&A (PI-PDD-SUMMARY-001-04)
- 상품 리뷰·평점·Q&A 제공 영역은 최신순, 도움순, 유형별 필터를 제공한다.
- 신고·가림 대상은 운영 정책에 따라 제한하고, 답변 상태와 원문 이동 경로를 함께 제공한다.
• 브랜드·판매주체·이행방식 (PI-PDD-SUMMARY-001-05)
- 상품 상세에는 브랜드, 판매주체, 즉시 사용·배송·설치·외부 제휴·전문 채널 연계 같은 이행방식을 명확히 표시한다.
- 고객이 주문 후 누가 이행하고 어디에서 처리되는지 알 수 있어야 한다.

4) 가격·혜택·가치 표시 정책 (PG-PDD-PRICE-001)

• 가격 기준 (PI-PDD-PRICE-001-01)
- 가격은 정가, 할인가, 실구매가, 예상 절감, 월 기준, 1회성 비용을 구분한다.
- 고객 상태가 반영된 실구매가는 담기 전까지 최신 조건으로 재산정한다.
• 혜택 분리 (PI-PDD-PRICE-001-02)
- 쿠폰, 포인트, 제휴 혜택, 사은품, 마케팅 프로그램 혜택은 혜택별 적용 조건, 기간, 사용처, 제외 사유를 분리해 표시한다.
- 중복 적용 불가 혜택은 불가 사유를 함께 표시한다.
• 공시지원금 (PI-PDD-PRICE-001-03)
- 단말 상품은 출고가, 공시지원금, 선택약정 할인 적용가, 약정 기간 차이를 비교 가능한 단위로 제공한다.
- 공시지원금 고지 기준은 고객이 담기 전 확인 가능한 위치에 둔다.
• 마케팅 정보 (PI-PDD-PRICE-001-04)
- 제휴카드 할인, 사은품, 구매 유도 혜택은 적용 조건, 유지 조건, 제외 조건을 함께 표시한다.
- 마케팅 문구가 가격 또는 가입 가능성을 오인하게 하면 노출을 제한한다.
• 실사용 빈도 가치 (PI-PDD-PRICE-001-05)
- 구독·혜택성 상품은 월 N회 사용 시 구독료 이상의 가치처럼 고객이 이해할 수 있는 실사용 빈도 예시를 제공한다.
- 예시는 과장 없이 적용 조건과 제외 조건을 함께 표시한다.
• 선물가 문구 (PI-PDD-PRICE-001-06)
- 일반 가격과 동일한 선물가는 별도 선물가로 강조하지 않는다.
- 고객에게 가격 차이가 있는 것처럼 오인될 수 있는 툴팁이나 문구는 노출하지 않는다.
• 구독가 비교 (PI-PDD-PRICE-001-07)
- 기존 일반 구독가와 T우주 또는 채널 내 구독가 혜택을 비교할 때는 월 기준 금액, 할인 기간, 할인 종료 후 금액, 적용 제외 조건을 함께 표시한다.

5) 가입·구매 가능성 사전 안내 정책 (PG-PDD-ELIG-001)

• 사전 판정 (PI-PDD-ELIG-001-01)
- 가입·구매 가능 여부는 연령, 회선, 고객 등급, 지역, 재고, 보유 상품, 판매 기간, 채널 판매 가능성을 기준으로 주문 진입 전 또는 담기 직전에 판정한다.
• 불가 사유 (PI-PDD-ELIG-001-02)
- 상품 선택 불가 시 재고 부족, 옵션 상태, 가입 조건 불충족, 중복 가입, 선행 조건 미충족, 판매 중지 중 하나 이상의 사유와 해결 방법을 함께 표시한다.
• 비회원 전환 (PI-PDD-ELIG-001-03)
- 비회원은 구매·가입·개통을 완료할 수 없다.
- 비로그인 고객이 담기 또는 구매를 시도하면 로그인과 본인확인이 필요한 이유와 전환 이점을 먼저 안내한다.
• 제한 고지 (PI-PDD-ELIG-001-04)
- 결제수단, 포인트, 쿠폰, 할인, 가입 가능 시점, 예약 가능 시점의 제한은 상세와 담기 단계에서 사전 고지한다.
- 제한 조건이 바뀌면 기존 선택 상태에 즉시 반영한다.
• 연령·기간 유효성 (PI-PDD-ELIG-001-05)
- 가입 제한 기준연령은 시작 나이와 종료 나이의 범위가 유효한 경우에만 저장한다.
- 판매 기간, 예약 가능 시점, 가입 가능 시점이 있는 상품은 주문 전에 제한 조건을 표시한다.
• 묶음 해지 제약 (PI-PDD-ELIG-001-06)
- 패키지 상품 또는 묶음 혜택은 구성 혜택별 개별 해지 가능 여부를 가입 전에 안내한다.
- 묶음 단위로만 해지 가능한 경우 고객에게 해지 영향과 환불 기준 참조 경로를 함께 제공한다.

6) 비교·추천·AI 요약 정책 (PG-PDD-COMPARE-001)

• 비교 기준 (PI-PDD-COMPARE-001-01)
- 비교표는 상품군별 표준 속성, 단위, 용어, 노출 우선순위를 따른다.
- 요금제, 로밍, 약정, 보험, 웨이브, 단말 비교는 상품군별 필수 비교 항목을 다르게 둔다.
• AI 요약 (PI-PDD-COMPARE-001-02)
- 상품 핵심 특징 AI 요약은 원장 정보 상위에 제공하되 생성 기준, 반영 시점, 요약 대상 범위, 원문 이동 경로를 함께 표시한다.
• 추천 근거 (PI-PDD-COMPARE-001-03)
- AI 추천은 고객의 가입 정보, 데이터 사용량, 결합 여부, 보유 혜택, 월 예상 부담 중 사용한 기준을 표시한다.
- 고객이 조건을 수정하면 추천 결과를 재탐색할 수 있어야 한다.
• 후기 요약 (PI-PDD-COMPARE-001-04)
- 상품 구매 후기 AI 요약은 장점과 단점을 균형 있게 제공한다.
- 요약만으로 판단하지 않도록 원문 후기, 평점 세부, Q&A로 이동할 수 있게 한다.
• atomic view (PI-PDD-COMPARE-001-05)
- 상품 상세 AI atomic view는 고객 세그먼트, 가입·해지 맥락, 현재 상태에 맞춰 원장 정보를 재조립하되 원장 값과 다르게 표현할 수 없다.
• 상품군별 비교 템플릿 (PI-PDD-COMPARE-001-06)
- 요금제, 로밍, 약정, 보험, 웨이브, 단말 비교는 각 상품군의 필수 비교 속성을 다르게 둔다.
- BSS 또는 Product Catalog에서 제공한 기준값을 사용하고 고객이 조건을 수정하면 비교 결과를 다시 산정한다.
• 대체 옵션 추천 (PI-PDD-COMPARE-001-07)
- 선택한 색상·용량이 품절 또는 일시품절이면 재고가 있는 대체 색상·용량, 입고 알림, 다른 상품 비교 중 하나 이상을 제공한다.
- 대체 추천은 재고 기준 시각을 함께 표시한다.

7) 옵션·구성 선택 정책 (PG-PDD-OPTION-001)

• 옵션 선택 (PI-PDD-OPTION-001-01)
- 색상, 용량, 약정, 요금제, 부가서비스, 배송 유형은 선택 즉시 가격, 혜택, 재고, 가입 조건 변화를 재산정한다.
• 선택형 혜택 (PI-PDD-OPTION-001-02)
- 여러 혜택 중 1개 또는 N개를 선택해야 하는 경우 선택 가능 기한, 선택 전 비교 정보, 선택 완료 상태, 재선택 가능 여부를 표시한다.
• 그룹 상품 (PI-PDD-OPTION-001-03)
- 그룹 상품은 필수 구성과 선택 구성을 구분한다.
- 구성원 상태값과 그룹 상품 조건은 실시간 또는 수동 검증 결과를 기준으로 표시한다.
- 그룹상품 선택 시 고객은 그룹에 속한 상품과 구성원 조건을 확인한 뒤 담을 수 있어야 한다.
• 가격 반영 (PI-PDD-OPTION-001-04)
- 옵션과 프로그램 선택에 따라 가격 또는 혜택이 바뀌면 고객 액션 시점에 즉시 반영한다.
- 재계산 중에는 이전 금액을 확정 금액처럼 표시하지 않는다.
• 로밍 시작 옵션 (PI-PDD-OPTION-001-05)
- 로밍 상품은 데이터 용량, 자동개시·수동개시, 데이터 이용 안 함, 해지·차단·QoS 가능 여부를 고객이 같은 선택 흐름에서 확인할 수 있게 한다.
• 인기 옵션 적용 (PI-PDD-OPTION-001-06)
- 통계 팝업이나 인기 옵션 안내에서 고객이 색상·용량을 선택하면 상품 상세의 현재 옵션에 즉시 반영한다.
- 반영 전 재고와 가격 변동 여부를 다시 확인한다.

8) 상품 조합·담기 가능성 정책 (PG-PDD-COMBO-001)

• 동시 주문 (PI-PDD-COMBO-001-01)
- 상품 유형별 동시 주문 가능 조합을 정책으로 정의한다.
- 단말+요금제+부가+구독은 허용 조합을 둘 수 있고, 단독 구매 상품은 함께 담기를 제한한다.
• 중복가입 가능여부 확인 (PI-PDD-COMBO-001-02)
- 중복가입 가능여부 확인은 고객 회선과 계정 기준으로 수행한다.
- 중복 가입이 불가한 상품은 담기, 장바구니, 가입 단계에서 동일한 제한 사유를 표시한다.
• 필수 구성 (PI-PDD-COMBO-001-03)
- 필수 옵션, 필수 요금제, 필수 프로그램, 그룹 구성 조건이 누락되면 담기를 제한한다.
- 고객에게 누락 항목과 대체 가능한 구성을 함께 제시한다.
• 담기 판정 (PI-PDD-COMBO-001-04)
- 담기 가능 여부는 가입 가능 여부, 중복 보유, 선행 조건, 판매 상태, 재고·수량, 판매 기간, 채널 판매 가능성을 실행 시점에 재검증한다.

9) 담기 실행·다음 행동 정책 (PG-PDD-SAVE-001)

• 담기 저장 (PI-PDD-SAVE-001-01)
- 담기 성공 시 상품, 옵션, 프로그램, 혜택, 예상 비용, 판정 결과, 기준 시각을 저장한다.
- 동일 요청은 멱등 키 또는 동일 고객·상품·옵션·기준 시각으로 중복 요청 여부를 확인하고, 중복 요청이면 새 건을 만들지 않고 기존 담기 상태를 갱신한다.
• 다음 행동 (PI-PDD-SAVE-001-02)
- 담기 완료 후 계속 탐색, 장바구니 이동, 바로 신청, 비교 계속하기 중 최소 3개 행동을 제공한다.
- 행동별로 현재 선택 기준의 핵심 혜택 또는 주의사항을 짧게 표시한다.
• 주문 전환 (PI-PDD-SAVE-001-03)
- 바로 신청 또는 주문 전환 시 상품 상태, 가격, 재고, 혜택, 가입 가능성은 다시 확인한다.
- 변경이 있으면 변경 전후와 고객 선택지를 안내한다.
• 재검증 (PI-PDD-SAVE-001-04)
- 담기 이후 장바구니 또는 주문으로 넘어갈 때 10분 이상 경과했거나 상품 상태가 바뀐 경우 재검증을 수행한다.
- 재검증 실패 시 담기 완료 상태는 유지하되 주문 전환은 제한한다.
• CTA 의미 구분 (PI-PDD-SAVE-001-05)
- 담기와 구독하기는 장바구니 또는 신청 준비 단계로, 바로 결제하기는 결제 진입으로 구분한다.
- 상품 유형별 CTA 명칭과 다음 단계는 고객에게 혼동 없이 안내해야 한다.
• 고객 표시 상태와 내부 상태 구분 (PI-PDD-SAVE-001-06)
- 고객 표시 상태는 탐색 가능, 담기 완료, 주문 전환 가능, 선택 불가처럼 고객 행동을 결정하는 문구로 관리한다.
- 시스템 내부 상태는 조건 확인 필요, 조합 충돌, 재고 부족, 인증 필요, 운영 반영 대기처럼 판정·저장·연동 처리를 위한 상태로 구분하고, 내부 상태 변경이 고객 행동을 제한할 때만 고객 표시 상태를 변경한다.

10) 불가·충돌·인증 복구 정책 (PG-PDD-FAIL-001)

• 불가 안내 (PI-PDD-FAIL-001-01)
- 선택 불가 또는 가입 불가가 발생하면 고객에게 상품, 옵션, 조건, 정책 중 어느 축에서 실패했는지 구분해 안내한다.
- 단순 오류 문구만 표시하는 것은 허용하지 않는다.
• 충돌 복구 (PI-PDD-FAIL-001-02)
- 조합 충돌은 충돌 상품, 충돌 이유, 해제해야 할 항목, 대체 가능한 구성을 함께 제공한다.
- 고객이 수정하면 기존 비교·선택 상태로 복귀한다.
• 인증 복귀 (PI-PDD-FAIL-001-03)
- 로그인, 회선 확인, 추가 인증이 필요하면 필요 사유를 먼저 설명한다.
- 인증 완료 후에는 고객이 선택한 상품, 옵션, 비교 조건, 이전 스크롤 위치 중 핵심 상태를 복원한다.
• 대체 경로 (PI-PDD-FAIL-001-04)
- 담기 실패 시 대체 상품 보기, 조건 충족 방법 보기, 상담 연결, 나중에 다시 보기 중 최소 2개 이상의 후속 행동을 제공한다.

11) 상담·문의 맥락 전달 정책 (PG-PDD-CS-001)

• 상담 문맥 (PI-PDD-CS-001-01)
- 상품 상세 또는 담기 단계에서 상담으로 전환하면 상품 ID, 선택 옵션, 회선, 비교 조건, 실패 사유, 시도 횟수, 최근 판정 결과를 상담 문맥으로 전달한다.
• 실패 이력 (PI-PDD-CS-001-02)
- 상담 전환 이력에는 실패 유형, 발생 시각, 고객 선택, 상담 전환 여부, 최종 안내 결과를 저장한다.
- 고객이 같은 설명을 반복하지 않도록 상담 화면에서 참조 가능해야 한다.
• 대체 안내 (PI-PDD-CS-001-03)
- 상담사는 상품 정책을 변경하지 않고 고객 조건에 맞는 대체 상품, 옵션, 신청 경로, 재시도 가능 시점을 안내한다.

12) Product Catalog 연동 정책 (PG-PDD-CATALOG-001)

• 상품 I/F (PI-PDD-CATALOG-001-01)
- Product Catalog에서 상품, 마케팅, 서비스, 혜택, 정책, 가입조건, 가격 정보를 통합 수신한다.
- 채널은 원장 기준과 다른 임의 값을 표시하지 않는다.
• Spec/Item (PI-PDD-CATALOG-001-02)
- Product Spec은 제공 단위, Item은 사용 단위로 구분한다.
- Item Type 관계가 있는 상품은 상세와 담기 모두에서 동일한 구성 관계를 사용한다.
• 상품 관계 (PI-PDD-CATALOG-001-03)
- Product Offering과 마케팅 프로그램 관계는 동시 가입, 자동 가입, 자동 해지, 선가입·선해지 기준으로 판정한다.
• 정책 수신 (PI-PDD-CATALOG-001-04)
- 가입 조건, 혜택 조건, 조합 정책, 제한 문구는 Product Catalog 또는 BSS 기준을 우선한다.
- 채널 보조 문구는 정책값을 덮어쓸 수 없다.
• 외부채널 연동 (PI-PDD-CATALOG-001-05)
- 외부채널 상품 설정은 Product Catalog와 채널 운영값의 매핑을 기준으로 수신한다.
- 외부채널 또는 Product Catalog 연동 장애가 발생하면 고객 표시를 제한하거나 보조 안내로 전환하고, 채널 간 영향 확대 여부와 복구 결과를 이력으로 남긴다.

13) 상품 라이프사이클·코드 체계 정책 (PG-PDD-LIFE-001)

• 라이프사이클 (PI-PDD-LIFE-001-01)
- 상품기획, 검토, 개발, 검증, 출시, 운영, 가입중단, 퇴출 상태는 채널 노출과 담기 가능성을 결정한다.
- 가입중단·퇴출 상품은 신규 담기를 제한한다.
• 코드 체계 (PI-PDD-LIFE-001-02)
- 요금제, 부가서비스, 혜택, 단말, 서비스 코드는 구분 가능한 코드 체계로 수신한다.
- 신규 상품군이 추가되어도 채널 판매 불가가 발생하지 않도록 코드 유형을 확장 가능하게 둔다.
• 메타데이터 (PI-PDD-LIFE-001-03)
- 전시 단위 기본 식별 정보 관리 대상에는 이미지, 마케팅 문구, Product 설명, 전사 상품·서비스 메타데이터가 포함된다.
- 채널은 NOVA 연동 데이터를 우선 사용하고 별도 운영 값은 제한한다.
• 판매 상태 (PI-PDD-LIFE-001-04)
- 판매 가능, 판매 중지, 품절, 예약 판매, 채널 판매 제외 상태는 상품 상세와 담기 단계에서 동일하게 적용한다.
• 전시 상태 코드 (PI-PDD-LIFE-001-05)
- 전시, 비전시, 사전예약, 판매종료, 사용, 미사용 상태는 표준 코드로 관리한다.
- 채널별 다른 표현을 쓰더라도 담기 가능성 판정은 같은 상태 코드를 따른다.

14) 운영자 템플릿·정책문구 관리 정책 (PG-PDD-OPS-001)

• 버전 관리 (PI-PDD-OPS-001-01)
- 템플릿, 정책 문구, 비교 기준, 상품 조합 정책 변경은 버전, 적용일, 변경자, 변경 전후, 영향 상품을 저장한다.
• 영향도 확인 (PI-PDD-OPS-001-02)
- 운영 정책 변경 전에는 고객 판정 결과, 담기 가능성, 노출 문구, 기존 담기 상태에 미치는 영향을 미리 확인한다.
• 미리보기 (PI-PDD-OPS-001-03)
- 운영자는 상품군, 고객 상태, 판매 상태, 재고 상태, 혜택 조건을 바꿔가며 상품 상세와 담기 문구를 미리보기로 확인할 수 있어야 한다.
• 변경 이력 (PI-PDD-OPS-001-04)
- 정책 문구와 템플릿 변경 이력은 최소 1년 보관한다.
- 고객 안내에 영향을 준 변경은 고객 문의와 재현 검증이 가능해야 한다.

15) 재고·배송·예약판매 운영 정책 (PG-PDD-STOCK-001)

• 재고 기준 (PI-PDD-STOCK-001-01)
- 옵션별 재고 현황 표시와 수량 제한은 상품 선택 전과 담기 직전에 모두 확인한다.
- 재고 부족 시 수량 조정, 다른 옵션, 입고 알림을 제공한다.
• 배송 가능성 (PI-PDD-STOCK-001-02)
- 배송, 픽업, 설치가 필요한 상품은 배송 유형과 지역·일정 조건을 기준으로 주문 가능 여부를 표시한다.
• 예약판매 (PI-PDD-STOCK-001-03)
- 예약판매 상품은 사전 체크리스트, 입력값 세팅, 오픈 시각, 종료 시각, 수량 제한을 정책으로 관리한다.
- 시작 전에는 예약 가능 시점과 제한 조건을 안내한다.
• 시장 대응 (PI-PDD-STOCK-001-04)
- 전략 단말과 시장 정책 변동은 추가지원금, 판매 상태, 마케팅 혜택에 영향을 주는 경우 운영자가 변경 기준과 반영 시점을 관리한다.

16) 담기 모니터링·알림 정책 (PG-PDD-MON-001)

• 활동 현황 (PI-PDD-MON-001-01)
- 운영자는 상품, 옵션·조합, 고객 여정 단계, 담기 성공·실패, 실패 사유, 인증·연계 상태, 상담 전환 여부를 실시간으로 조회한다.
• 불편 이벤트 (PI-PDD-MON-001-02)
- 불편 이벤트는 시스템 오류, 상품·정책 충돌, 판매 가능 상태 오류, 인증·연계 실패, 반복 실패, 이탈 급증, 상담 전환 급증으로 분류한다.
• 실시간 알림 (PI-PDD-MON-001-03)
- 불편 이벤트가 기준을 초과하면 발생 시각, 이벤트 유형, 영향 상품·조합, 발생 규모, 주요 실패 사유, 권장 조치를 포함해 알림을 발송한다.
• 에스컬레이션 (PI-PDD-MON-001-04)
- 미확인 또는 미조치 상태가 설정 시간 이상 지속되면 심각도와 담당 상품 기준으로 상위 담당자 또는 연관 운영 조직에 자동 에스컬레이션한다.

17) 이력·성과·개선 정책 (PG-PDD-AUDIT-001)

• 판정 이력 (PI-PDD-AUDIT-001-01)
- 가입 가능성, 담기 가능성, 조합 충돌, 재고 부족, 인증 필요 판정은 기준 시각과 판정 근거를 이력으로 저장한다.
• 변경 이력 (PI-PDD-AUDIT-001-02)
- 상품 원장, 템플릿, 비교 기준, 마케팅 정보, 정책 문구 변경은 변경 전후와 승인 결과를 함께 저장한다.
• 성과 리포트 (PI-PDD-AUDIT-001-03)
- 상품 상세과 담기 성과는 노출 수, 진입 수, 비교 이용 수, 담기 성공률, 실패율, 상담 전환율, 주문 전환율 기준으로 집계한다.
• 개선 추적 (PI-PDD-AUDIT-001-04)
- 반복 실패 유형과 미조치 알림은 개선 대상 목록으로 관리한다.
- 조치 완료 후 동일 유형 재발 여부를 확인해 정책 또는 원장 개선으로 연결한다.

18) 운영 입력·검증·삭제 보호 정책 (PG-PDD-ADMIN-001)

• 입력 검증 (PI-PDD-ADMIN-001-01)
- 상품 등록·수정 시 필수값, 가격 형식, 이미지 규격, 노출순서, 유효기간, 카테고리 선택, 공통코드 변경 반영 여부를 저장 전에 검증한다.
• 삭제 보호 (PI-PDD-ADMIN-001-02)
- 모상품, 판매정책, 카테고리, 부가서비스, 자급제 상품, 액세서리, 사은품 정책 삭제는 참조 데이터와 프론트 노출 영향이 없거나 승인된 예외인 경우에만 허용한다.
• 일괄 변경 부분 실패 (PI-PDD-ADMIN-001-03)
- 전시·비전시, 공개·비공개, 사용여부 일괄 변경에서 일부 항목이 실패하면 성공 항목과 실패 항목을 분리해 표시하고 실패 항목은 기존 상태를 유지한다.
• 이미지 규격 (PI-PDD-ADMIN-001-04)
- 이미지와 영상 등록은 권장 해상도, 파일 형식, 용량, 대체텍스트 기준을 사전 안내하고 기준 미달 시 저장을 제한하거나 보완 안내를 제공한다.
• 변경 보호 (PI-PDD-ADMIN-001-05)
- 등록·수정 중 취소 또는 화면 전환이 발생하면 변경사항 유실 여부를 안내한다.
- 저장 완료 후에는 수정일, 수정자, 변경 항목을 이력으로 남긴다.
• 관리자 승인 (PI-PDD-ADMIN-001-06)
- 잔여재고현황 다운로드처럼 영업 민감 정보가 포함된 자료는 관리자 승인 후 제공한다.
- 승인 권한, 승인 시각, 다운로드 이력은 감사 이력으로 저장한다.

19) 후기·Q&A 운영 정책 (PG-PDD-REVIEW-001)

• 공통 컬럼 (PI-PDD-REVIEW-001-01)
- 구매후기와 Q&A는 상품군별로 다른 컬럼을 쓰더라도 고객 표시 기준은 작성자, 작성일, 평점, 유형, 답변 상태, 신고·가림 여부 중심으로 표준화한다.
• 마스킹 (PI-PDD-REVIEW-001-02)
- 후기 작성자 정보는 개인정보가 드러나지 않도록 마스킹한다.
- 마스킹 전 원문은 권한 있는 운영자만 확인하고 조회 이력을 저장한다.
• 답변 상태 (PI-PDD-REVIEW-001-03)
- Q&A 답변은 답변 등록 여부에 따라 미답변, 답변완료, 수정완료 상태를 자동 전환한다.
- 답변 수정 시 변경 전후와 수정자를 이력으로 저장한다.
• 결과 없음 (PI-PDD-REVIEW-001-04)
- 후기 필터와 카테고리 탭을 동시에 적용해 결과가 없으면 조건 완화, 전체 후기 보기, 후기 작성 가능 여부 중 하나 이상을 안내한다.
• 후기 작성 유도 (PI-PDD-REVIEW-001-05)
- 구매후기가 없는 상품에서 고객에게 구매 이력이 확인되면 후기 작성 유도 안내를 제공한다.
- 구매 이력이 없거나 작성 제한 상태인 고객에게는 작성 요청을 노출하지 않는다.

20) 키워드·카테고리 운영 정책 (PG-PDD-KEYWORD-001)

• 키워드 저장 단위 (PI-PDD-KEYWORD-001-01)
- 키워드 세트는 카테고리, 상품군, 노출 위치, 적용 기간을 기준으로 저장한다.
- 최소 1개 이상의 키워드가 있어야 저장할 수 있다.
• 중복 제한 (PI-PDD-KEYWORD-001-02)
- 동일 조건의 키워드 세트와 동일 카테고리 내 중복 카테고리명은 등록을 제한한다.
- 중복 사유와 기존 항목으로 이동하는 경로를 제공한다.
• 역할 구분 (PI-PDD-KEYWORD-001-03)
