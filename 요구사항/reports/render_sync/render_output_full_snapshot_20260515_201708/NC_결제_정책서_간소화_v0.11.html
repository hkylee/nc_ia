1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객이 결제 금액을 확인하고 결제수단과 혜택을 선택해 결제를 완료하는 업무 기준을 정의한다.

• 포함 범위는 결제 단계 진입, 결제수단 선택·등록·변경, 할인·포인트·쿠폰·프로모션 적용, 결제 승인, 외부 간편결제 복귀, 실패 복구, 정기결제, 결제 증빙, 운영 정책 관리를 포함한다.

• 고객은 결제 전에 최종 금액, 할인 근거, 결제수단 제한, 인증 필요 여부, 취소·환불 핵심 조건, 실패 시 다음 행동을 이해할 수 있어야 한다.

• 주문 생성, 배송·교환·반품, 계약 조항, 상품 탐색은 인접 정책서에서 다루며 본 문서는 결제와 직접 연결되는 금액·승인·수납·증빙 기준만 정의한다.

• API 필드, PG 전문, DB 컬럼, 운영자 화면 상세, 정산 배치 설계는 후속 상세 설계에서 다루고 본 문서는 정책 판단에 필요한 기준만 포함한다.

• 채널 업무 시스템은 결제 금액 표시, 결제수단 선택, 고객 안내, 재시도·복구 경로를 책임지고 결제 연계 시스템은 PG 승인, BSS 수납 반영, 환불·취소, 증빙 결과 회신을 책임진다.

• 고객 표시 상태는 결제 대기, 승인 완료, 수납 반영 대기, 결제 실패, 환불 처리 중, 환불 완료로 안내하고 내부 상태는 승인 거래, 멱등키, BSS 수납 반영, 혜택 복원, 증빙 발급 이력으로 분리한다.

• 통합 리스크는 PG 승인과 BSS 수납 반영 불일치, 간편결제 복귀 실패, 콜백 중복, 채널별 결제 상태 캐시 차이이며 완료 표시는 수납 반영 확인 후 확정한다.

• 상세 요구사항 104건은 결제수단 선택, 할인·포인트·쿠폰 적용, 승인·수납 분리, 실패 복구, 정기결제, 증빙, 운영 정책 기준으로 재구성해 유즈케이스·프로세스·기능·정책 중 하나 이상에 매핑했다.

• FO는 결제 금액 표시, 결제수단 선택, 고객 안내와 재시도 경로를 책임지고, BSS는 수납 원장과 주문·계약 상태 반영을 책임지며, 외부 시스템은 PG 승인·간편결제 복귀·카드사 회신 결과를 책임진다.

나. 설계 원칙

• 금액 투명성 우선: 고객이 상품금액, 할인, 포인트, 결제수단 프로모션, 최종 결제 금액을 한 흐름에서 이해하도록 구성한다.

• 실패 불안 최소화: 결제 실패, 중단, 외부 복귀 실패, 승인·수납 불일치 상태에서 고객이 다음 행동을 즉시 선택하게 한다.

• 중복 결제 차단: 동일 주문서의 중복 요청, 연타, 콜백 중복, 재진입 결제는 거래ID와 멱등키 기준으로 차단한다.

• 혜택 판단 일관성: 쿠폰, 포인트, 할인, 결제수단 프로모션의 우선순위와 제한 사유는 결제 전후 동일한 기준으로 표시한다.

• BSS 수납 정합성: PG 승인과 BSS 수납, 주문·계약 상태 반영이 일치해야 결제 완료로 확정한다.

• 운영 통제 가능성: 결제수단 노출, 제한, 프로모션, 할인 예외, 실패 모니터링은 운영자가 이력과 승인 절차로 관리한다.

나. 정책 상세

1) 결제 범위·경계 정책 (PG-PAY-SCOPE-001)

• 대상 범위 (PI-PAY-SCOPE-001)
- 본 문서는 결제 처리, 결제수단, 할인·포인트·쿠폰, 결제수단 프로모션, 정기결제, 증빙 발급, 결제 실패 복구를 포함한다.
• 제외 범위 (PI-PAY-SCOPE-002)
- 상품 탐색, 주문 생성, 배송·교환·반품, 계약 상세 조항은 인접 정책서에서 정의하고 본 문서는 결제 영향과 연계 결과만 다룬다.
• 결제 단계 표준 (PI-PAY-SCOPE-003)
- 주문 최종 검토 이후 결제 단계는 상품금액, 수단 선택, 혜택 적용, 최종 금액 확인, 결제 진행 순서로 구성한다.

2) 결제 금액 확정 정책 (PG-PAY-AMOUNT-001)

• 금액 구성 (PI-PAY-AMOUNT-001)
- 결제 화면은 상품금액, 할인 합계, 포인트·쿠폰 차감, 즉시 결제 금액, 월 반복 비용, 다음 청구 반영 금액을 구분해 표시한다.
• 금액 변동 (PI-PAY-AMOUNT-002)
- 결제 직전 금액이 바뀌면 변동 사유, 변경 전 금액, 변경 후 금액을 표시하고 고객 재확인을 받아야 한다.
• 요약 표시 (PI-PAY-AMOUNT-003)
- 결제 요약은 주문단위별 최종 결제 금액을 기본값으로 표시하고, 상품금액·할인·포인트·쿠폰·청구 반영 금액은 항목별 상세 펼침에서 확인할 수 있게 한다.
• 가격 정책 기준 (PI-PAY-AMOUNT-004)
- Product Offering의 Price Plan, 매출 구분, 정액·정률·종량·일시불 구조는 결제 금액 구성의 기준 데이터로 사용한다.

3) 결제수단 선택 정책 (PG-PAY-METHOD-001)

• 수단 노출 (PI-PAY-METHOD-001)
- 사용 가능한 결제수단은 최근 사용, 기본 수단, 채널 공통 결제 프로필 순으로 우선 표시한다.
• 수단 등록 (PI-PAY-METHOD-002)
- 카드, 계좌, 간편결제 등록은 본인확인과 유효성 검증을 통과한 경우에만 저장한다.
• 입력 최소화 (PI-PAY-METHOD-003)
- 저장된 수단, 자동완성, 카드 OCR, 스캔 입력은 고객 동의가 있을 때만 사용하고 인식 실패 시 수동 입력으로 전환한다.
• 프로필 연속성 (PI-PAY-METHOD-004)
- 기본 수단, 최근 사용 수단, 할부 개월 수, 현금영수증 유형, 환불 계좌는 고객 단위 공통 결제 프로필로 저장하고 고객이 삭제 또는 기본값 변경을 요청하면 즉시 반영한다.

4) 결제수단 제한 정책 (PG-PAY-LIMIT-001)

• 허용 수단 (PI-PAY-LIMIT-001)
- 상품 또는 주문단위별 Allowlist에 포함된 결제수단만 선택 가능으로 표시한다.
• 제한 수단 (PI-PAY-LIMIT-002)
- Blocklist 또는 한도·인증·상품 조건에 걸린 수단은 비활성화하고 제한 사유와 대체 수단을 함께 표시한다.
• 제한 사유 (PI-PAY-LIMIT-003)
- 제한 사유는 상품 제한, 결제수단 한도, 고객 인증 부족, 프로모션 조건 불충족, 외부 연계 불가 중 하나로 표준화한다.

5) 결제 인증 정책 (PG-PAY-AUTH-001)

• 추가 인증 (PI-PAY-AUTH-001)
- 고액, 기기 변경, 이상 거래, 외부 결제, 저장 수단 변경은 Step-up 인증 대상이다.
• 인증 유효 시간 (PI-PAY-AUTH-002)
- 결제 추가 인증 유효 시간은 10분이며 만료되면 결제수단과 혜택 선택값은 유지하고 인증만 다시 요구한다.
• 원탭 결제 (PI-PAY-AUTH-003)
- 등록 인증 기반 원탭 결제는 동일 기기, 정상 위험 등급, 저장 수단 유효 상태를 모두 만족할 때만 허용한다.
• 거래 바인딩 (PI-PAY-AUTH-004)
- 최종 강한 인증은 주문번호, 결제금액, 결제수단, 거래ID와 묶어서 수행한다.

6) 할인 적용 정책 (PG-PAY-DISCOUNT-001)

• 적용 상태 (PI-PAY-DISCOUNT-001)
- 혜택 영역은 지금 적용 가능, 조건 충족 시 가능, 적용 불가 세 상태로 구분해 표시한다.
• 최적 조합 (PI-PAY-DISCOUNT-002)
- 최적 조합 제안은 고객이 그대로 적용하거나 일부 변경할 수 있어야 하며 변경 시 최종 금액을 즉시 갱신한다.
• 적용 불가 (PI-PAY-DISCOUNT-003)
- 만료, 잔액 부족, 최소 금액 미충족, 중복 불가, 특정 수단 제한은 적용 불가 사유와 대체 적용안을 함께 표시한다.

7) 포인트·쿠폰 적용 정책 (PG-PAY-POINT-001)

• 포인트 사용 (PI-PAY-POINT-001)
- 포인트는 전액 사용, 일부 사용, 최대 사용, 적용 해제 방식으로 선택할 수 있고 변경 즉시 남은 결제 금액을 갱신한다.
• 쿠폰 선택 (PI-PAY-POINT-002)
- 사용 가능한 쿠폰은 대상, 기간, 최소 금액, 만료 임박 기준으로 필터링하고 불가 쿠폰은 사유를 표시한다.
• 중복 우선순위 (PI-PAY-POINT-003)
- 포인트, 쿠폰, 할인은 운영자가 등록한 우선순위와 중복 가능 여부를 적용하되 고객에게 적용 순서를 문장으로 안내한다.
• 제휴 포인트 (PI-PAY-POINT-004)
- 제휴 포인트는 연동 완료, 잔액 조회 성공, 사용 조건 충족 시에만 결제수단 또는 혜택으로 선택할 수 있다.
• 비로그인 쿠폰 대안 (PI-PAY-POINT-005)
- 비로그인 상태에서 쿠폰 목록을 조회할 수 없으면 로그인 유도만 표시하지 않는다.
- 일반 적용 조건 보기, 로그인 후 적용 가능 안내, 대체 혜택 탐색 경로를 함께 제공하고 쿠폰 선택값은 로그인 완료 전까지 확정하지 않는다.

8) 결제수단 프로모션 정책 (PG-PAY-PROMO-001)

• 즉시할인 (PI-PAY-PROMO-001)
- 즉시할인은 결제 금액 구성의 별도 라인으로 표시하고 조건 미충족 시 사유를 표시한다.
• 사후 혜택 (PI-PAY-PROMO-002)
- 청구할인과 캐시백은 사후 혜택 섹션으로 분리하고 지급 주체, 지급 시점, 확인 경로, 한도를 표시한다.
• 적용 범위 (PI-PAY-PROMO-003)
- 프로모션 적용 범위는 전체, 카테고리, 상품, 주문단위 중 하나로 관리하고 제외 대상은 결제 전에 표시한다.
• 충돌 규칙 (PI-PAY-PROMO-004)
- 결제수단 프로모션과 쿠폰·포인트가 중복 불가이면 우선 적용 항목과 제외 항목을 함께 안내한다.
• 혼합 주문 프로모션 (PI-PAY-PROMO-005)
- 혼합 주문에서 프로모션이 일부 상품 또는 카테고리에만 적용되면 적용 대상과 제외 대상을 라벨로 분리 표시하고, 합산 결제 조건을 충족하지 못하면 분리 결제로 전환한다.

9) 제휴카드 운영 정책 (PG-PAY-PARTNER-001)

• 제휴카드 탐색 위치 (PI-PAY-PARTNER-001)
- 제휴카드 상세 또는 비교 팝업에서 카드가 2개 이상이면 전체 카드 수와 현재 위치를 함께 표시한다.
- 고객이 좌우 탐색 중 동일 카드에 머무르거나 적용 가능 혜택을 놓치지 않도록 현재 위치, 이전·다음 이동 가능 여부, 선택 카드명을 함께 갱신한다.
• 제휴카드 유의사항 관리 (PI-PAY-PARTNER-002)
- 요금할인카드와 휴대폰할인카드의 유의사항은 카드 유형, 적용 기간, 상품군, 고객 유형별로 등록·수정·중지할 수 있다.
- 변경 내용은 검수 승인 후 노출하고 작성자, 승인자, 적용 시작 시각, 이전 문구를 이력으로 저장한다.
• 카드코드·할인적용·노출상태 산정 (PI-PAY-PARTNER-003)
- 제휴카드 코드는 카드 유형 Prefix와 자동 증가 일련번호 조합으로 생성하며 수동 중복 등록을 허용하지 않는다.
- 할인적용 상태와 프론트 노출상태는 사용여부, 사용기간, 승인상태, 적용상품 매핑 결과가 모두 유효한 경우에만 가능으로 산정한다.
• 카드 이미지 검증 (PI-PAY-PARTNER-004)
- 카드 이미지는 승인된 확장자, 용량, 비율, 최소 해상도 기준을 통과한 경우에만 저장한다.
- 기준 미달이면 저장을 차단하고 운영자에게 보완 사유와 재업로드 기준을 안내한다.
• 할인대상 매핑 (PI-PAY-PARTNER-005)
- 제휴카드 할인 대상은 단말, 인터넷, Btv, 요금제, 상품군 중 적용 단위를 선택해 매핑한다.
- 동일 카드에 중복 또는 충돌 매핑이 있으면 우선순위와 제외 대상을 확인한 뒤 승인 대상으로 전환한다.
• 입력 유효성 검증 (PI-PAY-PARTNER-006)
- 제휴카드 등록·수정 시 카드명, 카드코드, 사용기간, 노출순서, 약관 버전, 유의사항, 할인대상 매핑을 필수 검증한다.
- 필수값 누락이나 기간 역전, 중복 코드, 승인 전 노출 시도는 저장 전 차단한다.
• 저장 후 결과 확인 (PI-PAY-PARTNER-007)
- 제휴카드 저장이 완료되면 운영자는 저장된 상세 화면으로 이동해 카드코드, 노출상태, 약관 버전, 적용상품 매핑, 유의사항 반영 결과를 확인한다.
- 저장 실패 시 입력 화면을 유지하고 실패 항목별 보완 위치를 표시한다.
• 약관 버전 관리 (PI-PAY-PARTNER-008)
- 제휴카드 약관은 버전 단위로 등록하고 결제 또는 혜택 적용 시점의 약관 버전을 거래 이력에 저장한다.
- 사용 중인 약관 버전은 삭제하지 않고 종료 처리하며, 신규 버전은 승인 후 적용 시작 시각부터 사용한다.
• 제휴카드 삭제 및 상태관리 (PI-PAY-PARTNER-009)
- 제휴카드 삭제는 진행 중 프로모션, 적용상품 매핑, 고객 거래 참조가 없는 경우에만 허용한다.
- 참조가 있으면 삭제 대신 사용중지 또는 노출중지로 상태관리하고 변경 사유와 영향 범위를 이력으로 저장한다.
• 제휴카드 노출순서 및 적용상품 매핑·공통 항목 (PI-PAY-PARTNER-010)
- 제휴카드 노출순서와 적용 가능한 인터넷/Btv 상품 매핑은 상품군과 카드 유형별로 관리하며 동일 순번 충돌 시 운영자가 우선순위를 확정해야 배포할 수 있다.
- 라이트할부 카드와 요금할인카드의 공통 노출 조건은 일괄 설정할 수 있으나 개별 override 여부와 변경 이력은 카드별로 남긴다.

10) 분리·분할 결제 정책 (PG-PAY-SPLIT-001)

• 분리 결제 (PI-PAY-SPLIT-001)
- 혼합 주문에 서로 다른 결제 제한이 섞이면 합산 가능 여부를 표시하고 불가 시 주문단위별 분리 결제로 유도한다.
• 분할 납부 (PI-PAY-SPLIT-002)
- 분할 납부는 상품별 허용 조건을 충족한 경우에만 제공하고 비율 또는 순서를 선택할 수 있게 한다.
• 할부 방식 (PI-PAY-SPLIT-003)
- 단말할부금은 일할계산, 월할계산, 당월 시작, 익월 시작 중 허용된 방식만 표시한다.

11) 0원 결제 정책 (PG-PAY-ZERO-001)

• 무결제 완료 (PI-PAY-ZERO-001)
- 최종 결제금액이 0원이면 결제수단 선택 없이 완료할 수 있으나 주문·계약 상태와 증빙은 생성한다.
• 0원 증빙 (PI-PAY-ZERO-002)
- 0원 결제 완료 화면에는 할인·포인트 적용 근거, 결제금액 0원 표시, 주문·계약 반영 상태, 증빙 조회 경로를 모두 노출한다.
• 0원 실패 (PI-PAY-ZERO-003)
- 0원 결제라도 계약 반영이나 혜택 차감이 실패하면 결제 실패가 아니라 수납 반영 보류로 표시한다.

12) 결제 승인 정책 (PG-PAY-APPROVAL-001)

• 최종 확인 (PI-PAY-APPROVAL-001)
- 결제 전 최종 확인은 결제수단, 최종 금액, 할인 내역, 취소·환불 핵심 조건, 필수 동의 항목이 모두 표시된 경우에만 완료로 인정한다.
• 처리 중 (PI-PAY-APPROVAL-002)
- 결제 요청 후 30초 안에 승인 또는 실패 결과가 없으면 처리 중 상태로 전환하고, 고객에게 상태 조회와 상담 전환 가능 시점을 함께 표시한다.
• 성공 결과 (PI-PAY-APPROVAL-003)
- 결제 성공 안내에는 결제수단, 승인번호, 주문번호, 다음 행동, 증빙 조회 경로를 포함한다.

13) 외부 결제 연동 정책 (PG-PAY-EXTERNAL-001)

• 외부 전환 (PI-PAY-EXTERNAL-001)
- 간편결제 앱·웹뷰·리다이렉트 전환 전 고객에게 외부 이동과 복귀 방식, 예상 소요 시간을 안내한다.
• 복귀 복원 (PI-PAY-EXTERNAL-002)
- 외부 결제 복귀 시 결제 단계, 선택 수단, 혜택 조합, 주문서 상태를 복원한다.
• 콜백 실패 (PI-PAY-EXTERNAL-003)
- 콜백이 누락되면 동일 거래ID로 상태 조회를 수행하고 승인 여부가 확인되기 전까지 재결제를 차단한다.

14) 중복 결제 방지 정책 (PG-PAY-DUP-001)

• 연타 방지 (PI-PAY-DUP-001)
- 결제하기 버튼은 최초 요청 후 즉시 비활성화하고 로딩 상태를 표시한다.
• 멱등키 (PI-PAY-DUP-002)
- 동일 주문서의 결제 요청은 거래ID와 멱등키로 구분하며 5초 안의 동일 요청은 중복으로 차단한다.
• 재진입 차단 (PI-PAY-DUP-003)
- 이탈 후 재진입 시 승인 완료 또는 처리 중 거래가 있으면 결제 화면 대신 상태 확인 화면으로 이동한다.

15) 결제 실패 정책 (PG-PAY-FAIL-001)

• 실패 분류 (PI-PAY-FAIL-001)
- 실패 사유는 승인 거절, 한도 부족, 인증 실패, 수단 제한, 외부 복귀 실패, BSS 반영 실패로 분류한다.
• 실패 CTA (PI-PAY-FAIL-002)
- 실패 화면은 다시 결제하기, 수단 변경, 할인·포인트 재적용, 주문서로 돌아가기, 상담 연결 중 가능한 CTA만 표시한다.
• 재시도 횟수 (PI-PAY-FAIL-003)
- 동일 주문서 재시도는 동일 세션에서 최대 3회까지 허용하고 이후에는 수단 변경 또는 상담 전환을 안내한다.

16) 결제 복구 정책 (PG-PAY-RECOVERY-001)

• 주문서 복귀 (PI-PAY-RECOVERY-001)
- 결제 실패 후 주문서 복귀 시 결제수단, 혜택 조합, 인증 상태, 입력값을 가능한 범위에서 유지한다.
• 혜택 재검증 (PI-PAY-RECOVERY-002)
- 결제수단 변경 또는 재시도 시 쿠폰, 포인트, 프로모션 적용 여부를 다시 검증한다.
• 복구 불가 (PI-PAY-RECOVERY-003)
- 승인 완료 후 BSS 반영 보류, 환불 대기, 주문 만료 상태는 고객 재시도 대상에서 제외하고 상담 또는 상태 조회로 안내한다.

17) 주문서 유지 정책 (PG-PAY-ORDER-001)

• 유지 기간 (PI-PAY-ORDER-001)
- 결제 실패 후 주문서는 24시간 유지하고 만료 시 장바구니 복귀 또는 주문 재작성 필요 여부를 안내한다.
• 중단 안내 (PI-PAY-ORDER-002)
- 고객이 뒤로가기 또는 취소를 선택하면 결제 중단 상태, 주문 유지 여부, 재진입 가능 시간을 안내한다.
• 만료 영향 (PI-PAY-ORDER-003)
- 주문서가 만료되면 예약 재고, 할인 적용, 쿠폰 선택, 인증 상태는 모두 재검증 대상으로 전환한다.

18) 정기결제 정책 (PG-PAY-RECUR-001)

• 대표·예비 수단 (PI-PAY-RECUR-001)
- 정기결제 상품은 대표 결제수단 1개와 예비 결제수단 1개 이상을 등록할 수 있다.
• 실패 유예 (PI-PAY-RECUR-002)
- 정기결제 실패 후 유예 기간에는 혜택 제한 전 수단 변경, 예비 수단 결제, 선결제 경로 중 하나 이상을 고객에게 선택지로 제공한다.
• 갱신 알림 (PI-PAY-RECUR-003)
- 카드 만료, 잔액 부족, 갱신 실패 위험은 다음 결제일 전 사전 알림 대상으로 등록하고, 알림에는 결제수단 변경 경로와 실패 시 영향 범위를 포함한다.
• 선불형 (PI-PAY-RECUR-004)
- 선불형 정기결제는 결제 주기, 유지 조건, 다음 결제 일정, 실패 시 혜택 제한 시점을 결제 단계에서 고지한다.

19) 청구·납부 방식 정책 (PG-PAY-BILLING-001)

• 청구정보 (PI-PAY-BILLING-001)
- 후불 청구형 상품은 주문 단계에서 청구 방식과 납부 시점을 안내하고 청구 관리 화면으로 연결한다.
• 납부정보 (PI-PAY-BILLING-002)
- 구매자 통합회원의 청구정보와 납부정보를 선택할 수 있는 경우 현재 등록 상태와 변경 가능 여부를 표시한다.
• 변상금 (PI-PAY-BILLING-003)
- 위약금 또는 변상금 수납은 즉시 수납과 후불 청구 중 허용된 방식을 선택할 수 있게 한다.
• 일시불·선불 수납 (PI-PAY-BILLING-004)
- 일시불 또는 선불 결제는 회원 청구정보의 등록 카드, 고객 보유 카드, 현금성 수단 중 상품별 허용 수단만 선택할 수 있다.
• SKT 자산 단말 수납 (PI-PAY-BILLING-005)
- SKT 자산 단말 결제는 단말 소유 기준, 수납 방식, 청구 반영 방식이 확인된 경우에만 일시불 또는 할부 수납으로 접수한다.
• 단말할부금 청구방식 선택 (PI-PAY-BILLING-006)
- 단말할부금 청구방식은 일할계산 또는 월할계산 중 상품별 허용 방식만 선택할 수 있으며 월할계산은 당월 시작 또는 익월 시작 기준을 함께 표시한다.

20) 결제 증빙 정책 (PG-PAY-PROOF-001)

• 영수증 (PI-PAY-PROOF-001)
- 결제 완료 후 영수증, 거래내역, 확인서 조회는 결제 완료 상태에서만 허용하고 저장·공유 진입은 고객 식별 정보 마스킹 후 제공한다.
• 현금영수증 (PI-PAY-PROOF-002)
- 현금영수증 발급 정보는 휴대폰번호 또는 사업자번호를 마스킹하여 저장하고 결제 단계에서 기본값으로 적용한다.
• 세금계산서 (PI-PAY-PROOF-003)
- 법인 또는 B2B 과금 유형은 사업자 정보, 발급 대상, 발급 가능 상태가 모두 확인된 경우에 세금계산서 신청을 허용한다.

21) BSS 수납 반영 정책 (PG-PAY-BSS-001)

• 수납 반영 (PI-PAY-BSS-001)
- 결제 완료는 PG 승인, BSS 수납, 주문·계약 상태 반영이 모두 성공한 경우에만 확정한다.
• 불일치 (PI-PAY-BSS-002)
- PG 승인 성공 후 BSS 반영이 5분 안에 완료되지 않으면 수납 반영 보류로 전환하고 재처리 이력을 저장한다.
• 취소·환불 (PI-PAY-BSS-003)
- 결제 취소 또는 환불은 승인 취소, 부분 취소, 수납 정정, 주문 상태 복구 결과가 모두 확정된 경우에만 고객에게 완료로 표시한다.
• 부분 성공 (PI-PAY-BSS-004)
- 프로모션 차감 또는 제휴 포인트 차감만 성공한 경우 고객 결제 완료로 표시하지 않고 복구 또는 상담 전환으로 안내한다.

22) 결제 이력 정책 (PG-PAY-HIST-001)

• 거래 이력 (PI-PAY-HIST-001)
- 결제 요청, 승인, 실패, 중단, 복귀, 재시도, 취소, 환불, 증빙 발급 이력은 거래ID 기준으로 저장한다.
• 고객 조회 (PI-PAY-HIST-002)
- 고객 결제 이력의 기본 조회 범위는 최근 1년이며, 그 이전 이력은 증빙 조회 또는 고객센터 요청 경로로 분리한다.
• 운영 이력 (PI-PAY-HIST-003)
- 결제수단 정책, 프로모션, 할인 우선순위 변경은 작성자, 검수자, 승인자, 배포 시각, 롤백 여부를 저장한다.

23) 운영 정책 관리 기준 (PG-PAY-OPR-001)

• 수단 정책 (PI-PAY-OPR-001)
- 운영자는 채널, 상품군, 고객조건별 결제수단 노출 순서, 추천 수단, 배지, 제한 문구를 등록하고 변경 시 미리보기와 영향 확인을 거친다.
• 프로모션 관리 (PI-PAY-OPR-002)
- 프로모션은 유형, 대상 수단, 기간, 조건, 한도, 적용 범위, 제외 규칙, 요약·상세 문구를 필수 항목으로 등록한다.
• 검수 배포 (PI-PAY-OPR-003)
- 결제 정책 변경은 작성, 검수, 승인, 예약 배포, 종료, 롤백 단계를 거쳐 반영한다.
• 모니터링 (PI-PAY-OPR-004)
- 운영 모니터링은 결제 실패율, 정기결제 갱신 실패, 수납 반영 보류, 상담 전환 사유를 일 단위로 집계하고 기준치를 초과하면 운영 확인 대상으로 분류한다.
• 제휴몰 코드 관리 (PI-PAY-OPR-005)
- 토스, 카카오페이, 네이버 등 제휴몰과 요금제 코드는 운영자가 등록·수정·중지할 수 있으며 변경 이력과 적용 시작 시각을 저장한다.

24) 결제 상담 전환 정책 (PG-PAY-CS-001)

• 상담 문맥 (PI-PAY-CS-001)
- 상담 전환 시 주문번호, 거래ID, 결제수단, 최종 금액, 혜택 조합, 실패 사유, 수납 상태를 함께 전달한다.
• 대체 안내 (PI-PAY-CS-002)
- 상담사는 실패 원인에 따라 대체 수단, 재결제, 주문 재작성, 수납 보정, 환불·취소 경로를 안내한다.
• 후속 이력 (PI-PAY-CS-003)
- 상담 처리 결과는 결제 거래 이력과 상담 이력 양쪽에 저장하고 고객에게 후속 상태를 안내한다.
