1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객이 멤버십 카드의 사용 가능 상태를 확인하고 디지털 카드 또는 실물 카드를 사용할 수 있도록 하는 기준을 정의한다.

• 대상 업무는 멤버십 자격·카드 상태 조회, 디지털 카드 제시, 카드번호 복사, 실물 카드 발급·재발급, 대표 회선 변경, 해지 유예·확정, 운영자 예외 관리다.

• 멤버십 연계 시스템이 수행하는 자격 판정, 등급·카드 식별자 회신, 대표 회선 반영, 해지 유예·확정, 삭제 이력 저장은 정책 판단 기준에 포함한다.

• 제휴 혜택 상세, 포인트 적립·사용, 쿠폰·이용권, 제휴처별 혜택 정책은 인접 정책서에서 다루며 본 문서는 카드 접근 수단과 카드 상태 관리 기준만 포함한다.

• 화면 UI 상세, API 필드, DB 컬럼, 제휴처 내부 승인 절차는 제외하되 카드 사용 가능 여부와 고객 후속 행동에 영향을 주는 기준은 정책 항목으로 남긴다.

• 포함 범위는 카드 사용 가능성 확인, 디지털 카드 제시, 카드번호 복사, 실물 카드 발급·재발급, 대표 회선 변경, 해지 유예·확정, 카드 운영 예외 관리다.

• 제외 범위는 멤버십 등급 산정, 포인트 적립·사용, 제휴 혜택 상세 정산이며, 본 문서는 카드가 혜택 접근 수단으로 동작하기 위한 상태와 사용 기준만 다룬다.

• 핵심 정책 질문은 현재 카드를 사용할 수 있는지, 어떤 제시 수단을 허용할지, 사용 실패 시 어떤 대체 경로를 제공할지, 카드 상태 변경 이력을 어디에 남길지다.

• FO는 카드 상태와 제시 수단을 노출하고, 멤버십 연계 시스템은 자격·등급·카드 상태·BSS 원장 반영 결과를 회신하며, 외부 사용처는 제시 성공·실패 결과를 회신한다.

• 고객 표시 상태와 시스템 내부 상태를 구분해 카드 사용 가능, 제한, 유예, 비활성, 재발급 진행, 해지 확정 상태별 안내와 후속 행동을 다르게 제공한다.

• 상세 요구사항 14건은 멤버십 카드 사용, 발급, 상태 변경, 운영 예외, 보안 정책 중 하나 이상에 매핑한다.

• 동일 고객·동일 카드·동일 발급 또는 재발급 요청이 반복되면 기존 접수번호와 카드 상태를 우선 안내하고, 멤버십 원장에 중복 발급이나 중복 비활성 이력이 남지 않도록 멱등 기준을 적용한다.

• 멤버십 카드 상태는 통합채널, 멤버십 원장, 외부 사용처가 함께 참조하므로 동기화 지연이나 장애가 발생하면 현장 사용 실패, 중복 발급, 잘못된 제한 안내로 확대될 수 있다.

나. 설계 원칙

• 상태 기반 카드 관리: 카드는 이미지가 아니라 자격, 등급, 사용 가능 상태, 제한 사유를 증명하는 수단이므로 모든 표시와 조치는 상태 기준으로 관리한다.

• 현장 사용 우선: 고객이 제휴처에서 즉시 사용할 수 있도록 카드 상태, 제시 수단, 제한 사유, 대체 경로를 한 번에 확인하게 한다.

• BSS 반영 전 완료 금지: BSS/멤버십 원장 반영이 성공하기 전에는 발급 완료, 대표 회선 변경 완료, 해지 취소 완료로 표시하지 않는다.

• 보안 노출 최소화: 카드번호와 바코드는 필요한 시점에만 제한 시간 동안 노출하고 기본 표시는 마스킹과 자동 가림을 적용한다.

• 유예·삭제 영향 선고지: 해지 유예와 복구 불가 삭제는 고객 혜택과 이력에 직접 영향을 주므로 실행 전 소멸·유지·복구 가능성을 명확히 고지한다.

• 예외의 운영 추적성: 발급 보류, 재발급 예외, BSS 불일치, 반복 실패는 운영 확인 상태와 이력을 남겨 고객 안내와 품질 개선에 재사용한다.

1) 카드 관리 범위·경계 정책 (PG-MBS-SCOPE-001)

• 대상 업무 범위 (PI-MBS-SCOPE-001-01)
- 멤버십 카드 상태 확인, 디지털 카드 제시, 카드번호 복사, 실물 카드 발급·재발급, 대표 회선 변경, 해지 유예·확정, 운영자 예외 관리를 포함한다.
• 인접 혜택 업무 경계 (PI-MBS-SCOPE-001-02)
- 혜택 상세, 제휴처 혜택 사용 조건, 포인트 적립·사용 상세는 인접 정책서에서 다루고 본 문서는 카드 접근 수단과 카드 상태 판정 기준만 정의한다.
• 카드와 혜택 사용 분리 (PI-MBS-SCOPE-001-03)
- 카드 사용 가능 상태가 혜택 사용 가능성을 보장하지 않는다.
- 제휴처별 혜택 조건은 카드 상태와 별도로 판정하고 안내한다.
• 채널 일관성 (PI-MBS-SCOPE-001-04)
- 앱, 웹, 상담에서 카드 상태와 제한 사유는 동일 기준으로 표시한다.
- 문구 표현이 다르더라도 처리 가능 여부와 후속 행동은 동일해야 한다.

2) 멤버십 자격·사용 가능 판정 정책 (PG-MBS-ELIG-001)

• 회원 자격 판정 (PI-MBS-ELIG-001-01)
- 멤버십 가입 상태, 이동상품 가입 조건, 등급 유효성, 정지·해지 상태, 대표 회선 관계가 모두 유효한 경우에만 카드 사용 가능으로 판정한다.
• 제한 사유 분류 (PI-MBS-ELIG-001-02)
- 사용 제한 사유는 미가입, 자격 미충족, 탈퇴 처리중, 유예중, 제휴처 제외, 일시 제한, 연계 지연으로 구분해 고객에게 표시한다.
• 상태 변경 반영 시점 (PI-MBS-ELIG-001-03)
- 가입, 탈퇴, 재가입, 등급 변경은 BSS 반영 성공 전까지 완료로 표시하지 않는다.
- 차월 반영 또는 익일 반영이면 적용 예정일을 함께 표시한다.
• 번호이동 후 재가입 안내 (PI-MBS-ELIG-001-04)
- 번호이동 완료 후 멤버십 재연동 또는 재가입이 필요한 경우 고객에게 자동 안내하고, 재가입 전까지 카드 사용 제한 가능성을 표시한다.

3) 카드 상태 표시 정책 (PG-MBS-STATUS-001)

• 상태 표시 기준 (PI-MBS-STATUS-001-01)
- 카드 상태는 사용 가능, 사용 제한, 발급 대기, 발급 처리중, 발급 완료, 비활성, 해지 유예, 해지 확정으로 구분한다.
• 중간 상태 고지 (PI-MBS-STATUS-001-02)
- 처리중, 유예, 차월 반영, 운영 확인 필요 상태는 완료 상태와 구분해 표시하고 예상 반영 시점과 고객이 할 수 있는 행동을 함께 안내한다.
• 사용 가능 수단 표시 (PI-MBS-STATUS-001-03)
- 디지털 카드, 카드번호, 실물 카드 중 현재 사용할 수 있는 식별 수단을 구분해 표시하고, 사용 불가 수단은 사유를 함께 표시한다.
• 후속 행동 연결 (PI-MBS-STATUS-001-04)
- 사용 제한이나 발급 보류 상태에는 재인증, 재발급, 대표 회선 변경, 해지 취소, 상담 연결 중 가능한 다음 행동을 1개 이상 제공한다.

4) 디지털 카드 제시 정책 (PG-MBS-DIGI-001)

• 제시 수단 제공 (PI-MBS-DIGI-001-01)
- 사용 가능 상태의 고객에게 바코드, QR, 카드번호 중 정책상 허용된 제시 수단을 제공한다.
- 사용처별 허용 수단이 다르면 사용처 조건을 함께 안내한다.
• 바코드 유효 시간 (PI-MBS-DIGI-001-02)
- 바코드 또는 QR 표시 유효 시간은 10분을 기본으로 한다.
- 시간이 만료되면 화면을 갱신하고 자격과 카드 상태를 다시 확인한다.
• 대체 제시 수단 (PI-MBS-DIGI-001-03)
- 바코드 호출 실패 또는 스캔 불가 상황에서는 카드번호 복사, 카드번호 수기 제시, 상담 또는 제휴처 문의 경로를 안내한다.
• 제휴처 사용 조건 (PI-MBS-DIGI-001-04)
- 제휴처 제외, 특정 등급 제한, 오프라인 사용 불가 조건이 있으면 카드 화면에서 사전에 안내한다.

5) 카드번호 복사·마스킹 정책 (PG-MBS-COPY-001)

• 복사 허용 범위 (PI-MBS-COPY-001-01)
- 카드번호와 바코드 번호는 사용 가능 상태에서만 복사할 수 있다.
- 해지 유예, 사용 제한, 복구 불가 삭제 상태에서는 복사를 허용하지 않는다.
• 복사 완료 안내 (PI-MBS-COPY-001-02)
- 복사가 완료되면 완료 토스트와 사용 가능 채널 또는 제휴처 조건을 즉시 안내한다.
• 마스킹 기준 (PI-MBS-COPY-001-03)
- 화면 기본 표시는 뒤 4자리를 제외하고 마스킹한다.
- 고객이 복사를 요청할 때만 전체 번호를 일시적으로 사용할 수 있게 한다.
• 복사 후 사용 제한 (PI-MBS-COPY-001-04)
- 복사한 번호가 제휴처에서 항상 사용 가능한 것으로 보장하지 않는다.
- 제휴처별 사용 가능 여부는 제시 시점의 카드 상태와 제휴처 정책으로 다시 판정한다.

6) 실물 카드 발급 정책 (PG-MBS-PHYS-001)

• 발급 신청 자격 (PI-MBS-PHYS-001-01)
- 실물 카드 발급은 멤버십 자격이 유효하고 동일 고객의 진행 중 신청이 없는 경우 허용한다.
• 중복 신청 제한 (PI-MBS-PHYS-001-02)
- 동일 고객·동일 카드 유형의 실물 카드 신청은 발급 완료 또는 취소 전까지 1건만 허용한다.
• 접수·처리 상태 (PI-MBS-PHYS-001-03)
- 신청 완료 후 접수, 처리중, 보류, 발급 완료, 반려 상태를 제공하고 접수번호와 예상 처리 시점을 저장한다.
• 예상 발급 시점 (PI-MBS-PHYS-001-04)
- 발급 예상 시점은 영업일 기준 5일을 기본으로 안내하되, 배송 또는 수령 방식에 따라 달라질 수 있음을 함께 고지한다.

7) 재발급·분실·비활성 대응 정책 (PG-MBS-REISSUE-001)

• 재발급 대상 (PI-MBS-REISSUE-001-01)
- 분실, 훼손, 비활성, 보안 의심, 기존 카드 사용 제한 상태에서 고객 본인확인과 카드 소유 관계가 확인되면 재발급을 허용한다.
• 기존 카드 제한 (PI-MBS-REISSUE-001-02)
- 재발급 접수 후 기존 카드가 부정 사용 위험이 있으면 즉시 비활성 처리하고, 고객에게 사용 제한 시점과 대체 수단을 안내한다.
• 복구 가능 조건 (PI-MBS-REISSUE-001-03)
- 단순 비활성 또는 상태 동기화 오류는 재발급보다 복구를 우선 제공한다.
- 복구가 불가한 경우에만 재발급 경로로 전환한다.
• 상담·매장 연계 (PI-MBS-REISSUE-001-04)
- 셀프 재발급이 제한되는 고객은 상담 또는 매장 경로를 안내하고, 상담 전환 시 고객 상태와 접수 이력을 함께 전달한다.

8) 멤버십 상태 변경 영향 정책 (PG-MBS-CHANGE-001)

• 가입·탈퇴 영향 고지 (PI-MBS-CHANGE-001-01)
- 가입, 탈퇴, 재가입 진입 전 카드 사용 가능성, 등급 영향, 혜택 소멸·유지, 포인트 영향, 반영 예정일을 먼저 안내한다.
• 혜택 소멸·유지 안내 (PI-MBS-CHANGE-001-02)
- 탈퇴 또는 해지 확정으로 소멸되는 혜택과 유지되는 이력은 구분해 표시한다.
- 소멸 대상은 고객 확인 후 다음 단계로 진행한다.
• 재가입 가능 조건 (PI-MBS-CHANGE-001-03)
- 재가입 가능 여부는 이동상품 가입 상태, 해지 확정 여부, 유예 종료 여부, 복구 불가 삭제 여부를 기준으로 판정한다.
• 변경 처리 이력 (PI-MBS-CHANGE-001-04)
- 상태 변경 요청, 취소, 반려, 완료, 자동 처리 이력은 고객과 운영자가 조회할 수 있도록 저장한다.

9) 해지 유예·확정·삭제 정책 (PG-MBS-GRACE-001)

• 유예 기간 기준 (PI-MBS-GRACE-001-01)
- 해지 신청 후 유예 기간은 7일로 관리한다.
- 유예 기간 동안 카드 사용은 제한하고 해지 취소 또는 가입조건 재충족 경로를 제공한다.
• 유예 취소 기준 (PI-MBS-GRACE-001-02)
- 유예 기간 내 고객이 해지 취소를 완료하거나 가입조건을 재충족하면 다음날 00시 이후 멤버십 유지 상태로 복구한다.
• 해지 확정 기준 (PI-MBS-GRACE-001-03)
- 유예 종료 시점까지 취소 또는 자격 재충족이 없으면 해지 확정 상태로 전환하고 고객에게 확정 결과를 안내한다.
• 복구 불가 삭제 기준 (PI-MBS-GRACE-001-04)
- 해지 확정 후 T멤버십ID, 카드 식별자, 잔여 포인트, 사용 이력은 복구 불가 삭제 대상으로 처리한다.
- 법적 보존이 필요한 감사 이력은 별도 보존한다.

10) 대표 회선 알림 정책 (PG-MBS-LINE-001)

• 대표 회선 조회 (PI-MBS-LINE-001-01)
- 고객은 현재 멤버십 혜택 알림 대표 회선을 조회할 수 있어야 하며, 회선명과 수신 가능 상태를 함께 확인한다.
• 대표 회선 변경 권한 (PI-MBS-LINE-001-02)
- 대표 회선 변경은 고객 본인 소유 또는 위임 권한이 확인된 회선으로만 허용한다.
• 변경 적용 시점 (PI-MBS-LINE-001-03)
- 대표 회선 변경은 BSS 반영 성공 후 즉시 적용을 기본으로 하며, 지연 시 적용 예정 시점과 현재 수신 회선을 함께 표시한다.
• 알림 수신 이력 (PI-MBS-LINE-001-04)
- 대표 회선 변경 요청, 성공, 실패, 원복 이력은 알림 발송 기준과 함께 저장한다.

11) 운영자 예외 관리 정책 (PG-MBS-OPS-001)

• 수동 승인 대상 (PI-MBS-OPS-001-01)
- 자격 불일치, 중복 발급, 고객 정보 불일치, 보안 의심, BSS 응답 불일치가 있으면 운영자 수동 승인 대상으로 전환한다.
• 발급 보류 기준 (PI-MBS-OPS-001-02)
- 신청 정보가 부족하거나 발급 조건이 일시적으로 확인되지 않으면 발급 보류 상태로 관리하고 보완 요청 내용을 고객에게 안내한다.
• 재처리 이력 (PI-MBS-OPS-001-03)
- 운영자가 수동 승인, 반려, 보류 해제, 재처리를 수행하면 처리자, 시각, 사유, 변경 전후 상태를 저장한다.
• 운영 변경 감사 (PI-MBS-OPS-001-04)
- 카드 상태와 발급 정책의 운영 변경은 즉시 고객 상태에 영향을 줄 수 있으므로 변경 이력과 영향 범위를 감사 대상으로 보존한다.

12) BSS·연계 반영 정책 (PG-MBS-BSS-001)

• BSS 원장 우선 (PI-MBS-BSS-001-01)
- 고객에게 완료로 표시하기 전에 BSS/멤버십 원장의 자격, 카드 식별자, 대표 회선, 해지 상태 반영 성공을 확인한다.
• 반영 실패 상태 (PI-MBS-BSS-001-02)
- BSS 반영이 실패하거나 지연되면 처리중 또는 운영 확인 필요 상태로 표시하고 완료 안내를 하지 않는다.
• 외부 사용처 결과 (PI-MBS-BSS-001-03)
- 제휴처 또는 외부 사용처에서 실패 결과가 발생하면 카드 상태, 제휴처 제외, 일시 제한, 사용처 정책 중 어느 사유인지 분리해 안내한다.
• 동기화 주기 (PI-MBS-BSS-001-04)
- 카드 상태와 대표 회선 상태는 고객 요청 시 실시간 조회를 우선하고, 배치 동기화 결과와 불일치하면 실시간 결과를 기준으로 표시한다.

13) 실패·대체수단·상담 전환 정책 (PG-MBS-FAIL-001)

• 바코드 호출 실패 (PI-MBS-FAIL-001-01)
- 바코드 또는 QR 생성 실패 시 1회 자동 재시도 후 카드번호 표시 또는 상담 연결 경로를 제공한다.
• 제휴처 실패 대응 (PI-MBS-FAIL-001-02)
- 제휴처 스캔 실패가 발생하면 카드 상태 정상 여부, 제휴처 제외 여부, 사용처 네트워크 오류 여부를 구분해 고객에게 안내한다.
• 재시도 제한 (PI-MBS-FAIL-001-03)
- 동일 카드 표시 또는 복사 실패가 3회 이상 반복되면 보안 보호를 위해 추가 시도를 제한하고 상담 전환을 안내한다.
• 상담 전환 기준 (PI-MBS-FAIL-001-04)
- 해지 유예, 복구 불가 삭제, 운영 보류, BSS 불일치처럼 셀프 처리로 해결할 수 없는 상태는 상담 전환 문맥을 자동 구성한다.

14) 보안·개인정보 보호 정책 (PG-MBS-SEC-001)

• 노출 최소화 (PI-MBS-SEC-001-01)
- 카드번호, 바코드, QR은 고객이 명시적으로 요청한 화면에서만 노출하고 기본 목록이나 알림에는 전체 식별자를 표시하지 않는다.
• 화면 자동 가림 (PI-MBS-SEC-001-02)
- 디지털 카드 화면은 앱 전환, 화면 캡처 감지, 일정 시간 미사용 시 자동으로 가림 처리한다.
• 개인정보 저장 제한 (PI-MBS-SEC-001-03)
- 복사 여부, 제시 이력, 발급 상태 이력은 필요한 범위만 저장하고 카드 식별자 전체값은 정책상 필요한 경우를 제외하고 마스킹한다.
• 감사 이력 보존 (PI-MBS-SEC-001-04)
- 해지 확정 삭제 후에도 법적·감사상 필요한 최소 이력은 별도 보존하며 고객 재사용 목적의 복구에는 사용하지 않는다.

2. 디지털 카드, 카드번호, 실물 카드의 사용 가능 수단과 보안 기준이 분리되어 있는가?
3. 실물 카드 발급, 재발급, 비활성 복구, 운영 보류 상태가 추적 가능한가?
4. 해지 유예, 해지 확정, 복구 불가 삭제 기준과 고객 고지가 정책 항목으로 존재하는가?
5. 대표 회선 변경과 알림 수신 기준이 BSS 반영 및 이력 저장과 연결되는가?
6. 상태 전이표의 상태명과 상태 코드가 일치하고 전이별 후속 처리가 비어 있지 않은가?
7. 모든 프로세스가 관련 기능과 관련 정책을 가지고, 정책 상세에서 실제 판단 기준이 정의되는가?
8. 요구사항 14건이 유즈케이스, 프로세스, 기능, 정책 중 하나 이상에 매핑되는가?
9. 요구사항 추적성
상세 요구사항은 유즈케이스, 상태, 프로세스, 기능, 정책 또는 정책 항목 중 하나 이상으로 추적 가능해야 한다.
10. 개발·QA 전환성
각 정책 항목은 테스트 조건, 기대 결과, 실패·복구 경로로 전환 가능해야 한다.
11. 책임 경계 정합성
FO, BSS, 연계 시스템, 외부 주체의 판단·저장·회신 책임이 혼재되지 않아야 한다.
