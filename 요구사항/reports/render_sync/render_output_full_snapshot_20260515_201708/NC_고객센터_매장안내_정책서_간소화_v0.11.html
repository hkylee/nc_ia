1. 개요

가. 범위

• 본 정책서는 통합채널에서 제공하는 고객센터 매장안내 업무의 처리 기준을 정의한다.

• 요구사항에서 확인된 핵심 판단축은 매장 검색·위치 정렬, 단골 매장 관리, 대리점 마이크로사이트와 전용 URL, 매장별 처리 가능 업무·예약 가능 여부, 방문 준비 정보, 대체 경로와 매장 정보 갱신 기준을 포함한다.

• 대상 채널은 앱·웹 FO이며 고객이 직접 조회, 조건 확인, 처리 요청, 결과 확인을 수행하는 셀프 처리 흐름을 포함한다.

• BSS와 연계 시스템이 수행하는 가능 여부 검증, 상태 판정, 처리 결과 회신은 정책 판단 기준에 포함한다.

• 화면 UI 상세, API 필드, DB 컬럼, 운영자 화면 상세는 제외하되 정책 판단에 필요한 기준은 포함한다.

• 상세 화면 설계, API 필드, DB 컬럼, 테스트 케이스는 후속 화면 설계서와 기능 상세 명세서에서 상세화한다.

• 포함 범위는 매장 검색, 위치 기반 정렬, 단골 매장, 처리 가능 업무, 예약 가능 여부, 대리점 마이크로사이트·전용 URL, 가입·개통 매장 조회이며 실제 매장 운영 화면과 재고 상세는 후속 산출물에서 다룬다.

• 핵심 정책 질문은 어떤 매장을 방문 가능 후보로 보여줄지, 처리 가능 업무와 예약 가능 여부를 어떻게 판단할지, 위치·단골·대리점 URL 정보를 언제 갱신·회수할지, 방문 불가 시 어떤 대체 경로를 안내할지를 기준으로 둔다.

• 고객 표시 상태는 방문 가능, 예약 가능, 준비물 필요, 방문 제한, 대체 경로 안내처럼 고객 행동 기준으로 관리하고, 시스템 내부 상태는 매장 정보 검증, 예약 슬롯 회신, URL 활성·만료, 갱신 대기 상태로 구분한다.

• 동일 고객, 동일 매장, 동일 방문 목적, 동일 시간대 반복 예약 요청은 기존 예약번호와 처리 상태를 우선 안내하고 중복 예약을 제한한다.

나. 설계 원칙

• 고객 과업 중심: 고객센터 매장안내은 내부 시스템 단위가 아니라 고객이 완료하려는 목적을 기준으로 구성한다.

• 셀프 처리 우선: 고객이 앱·웹에서 직접 완료할 수 있는 업무는 상담 전환보다 셀프 처리 경로를 우선 제공한다.

• BSS 판단 통합: 고객 상태, 가입 정보, 요금, 혜택, 제한 조건처럼 BSS 판단이 필요한 정보는 프로세스와 정책에 포함한다.

• 영향도 사전 고지: 비용, 혜택, 약정, 포인트, 쿠폰, 회선, 주문 상태에 영향이 있으면 처리 전에 안내한다.

• 권한·보안 우선: 민감정보 노출 또는 변경 처리는 본인확인, 동의, 권한 기준을 충족한 경우에만 허용한다.

• 근거 기반 운영: 요구사항과 참고자료에서 확인한 기준은 프로세스, 기능, 정책 중 하나 이상으로 추적 가능하게 관리한다.

- 전체 완료로 보지 않고 보류 상태와 남은 처리 대상을 저장한다.
• 추적 관리 (PI-YMX-REQMAP-001-04)
- 추적 관리 기준에서 실패·보류·상담 전환이 같은 조건에서 3회 이상 반복되면 품질 모니터링 대상으로 등록한다.
- 반복 원인은 개선 과제로 관리한다.
• 매장 검색·위치 기준 탐색·정렬 기준 (PI-YMX-RQCOV-001)
- 매장 검색·위치 기준 탐색·정렬 기준 위치 동의가 없으면 지역·키워드 검색을 기본 경로로 제공하고 위치 기반 추천을 사용하지 않는다.
• 대리점 사이트·URL 운영 노출·공유 기준 (PI-YMX-RQCOV-002)
- 대리점 사이트·URL 운영 노출·공유 기준 콘텐츠 변경 시 노출 기간, 예약·진행·종료 상태, 승인자, 변경 이력, 롤백 가능 여부를 저장한다.
• 대리점 사이트·URL 운영 노출·공유·개인화 대리점 사이트·URL 운영 노출·공유 기준 (PI-YMX-RQCOV-003)
- 대리점 사이트·URL 운영 노출·공유 기준은 대리점 식별, 운영 상태, URL 규칙, 접근 권한, 만료·리다이렉트 기준이 충족된 경우에만 노출한다.
• 대리점 사이트·URL 운영 노출·공유 대리점 사이트·URL 운영 노출·공유 기준 (PI-YMX-RQCOV-004)

• 단골 매장·개인화 기준 (PI-YMX-RQCOV-005)
- 단골 매장·개인화 기준 등록·해제 결과는 고객별 단골 목록, 동의 상태, 알림 수신 여부, 변경 이력으로 저장한다.
• 방문 가능성·예약 기준 (PI-YMX-RQCOV-006)
- 방문 가능성·예약 기준은 매장별 처리 가능 업무, 운영 여부, 예약 가능 여부, 준비물 필요 여부가 확인된 경우에만 방문 가능으로 안내한다.
• 단골 매장·개인화 탐색·정렬 기준 (PI-YMX-RQCOV-007)
- 단골 매장·개인화 탐색·정렬 기준 혜택·알림 동의는 채널 공통 수신 동의와 분리해 안내하고 미동의 시 단골 등록 자체를 제한하지 않는다.

21) 연계 오류·장애 처리 정책 (PG-YMX-ERR-002)

• 장애 구분 (PI-YMX-ERR-002-01)
- BSS 장애, 연계 장애, 인증기관 장애, 채널 장애, 일시 지연을 구분해 처리 기준을 적용한다.
• 처리 제한 (PI-YMX-ERR-002-02)
- 정확한 판정이 불가능한 장애 상황에서는 상태 변경과 비용 발생 처리를 제한한다.
• 재시도 기준 (PI-YMX-ERR-002-03)
- 일시 장애는 제한 횟수 내 재시도를 허용하고 반복 장애는 상담 또는 운영 확인으로 전환한다.
• 장애 이력 (PI-YMX-ERR-002-04)
- 장애 유형, 발생 시점, 고객 영향, 처리 결과는 운영 모니터링 이력으로 저장한다.

22) 데이터 보관·파기 정책 (PG-YMX-DATA-001)

• 보관 대상 (PI-YMX-DATA-001-01)
- 처리 이력, 인증 이력, 고지 확인, 연계 결과, 운영 변경 증적은 목적별 보관 대상으로 분류한다.
• 파기 대상 (PI-YMX-DATA-001-02)
- 업무 목적이 종료되고 법정 보관 사유가 없는 개인정보와 임시 데이터는 파기 대상으로 분류한다.
• 보관 기간 (PI-YMX-DATA-001-03)
- 보관 기간은 법정 보관 사유, 고객 분쟁 대응, 처리 이력 추적 필요성을 기준으로 정한다.
- 기간 만료 시 파기 대상으로 전환하고 파기 이력을 저장한다.
• 파기 이력 (PI-YMX-DATA-001-04)
- 파기 대상, 파기 시점, 파기 방식, 처리자는 감사 가능한 이력으로 저장한다.

최종 점검 기준

고객센터 매장안내 정책서 제출 전 점검
1. 고객센터 매장안내 범위, 제외 범위, 후속 상세화 영역이 고객 과업 기준으로 명확한지 확인한다.
2. 고객이 업무를 시작해 완료하거나 제한 사유와 대체 경로를 확인할 수 있는지 확인한다.
3. 유즈케이스, 상태, 프로세스, 기능, 정책의 ID와 연결 관계가 누락 없이 이어지는지 확인한다.
4. BSS와 연계 시스템의 조회, 판정, 상태 변경, 결과 회신 기준이 프로세스와 정책에 반영됐는지 확인한다.
5. 모든 프로세스가 관련 기능과 실제 정책 그룹명에 연결되어 있는지 확인한다.
6. 모든 기능이 화면 단위가 아니라 처리 단위이며 세부 기능 구성이 있는지 확인한다.
7. 정책 상세가 기능 설명이 아니라 실제 기능 동작값, 허용 목록, 횟수, 시간, 제한 조건, 채널, 고지, 저장 기준으로 작성됐는지 확인한다.
8. 개인정보, 민감정보, 로그, 이력, 보관·파기 기준이 포함되어 있는지 확인한다.
9. 운영 기준 관리, 변경 이력, 품질 모니터링, 개선 과제 관리 기준이 포함되어 있는지 확인한다.
10. 요구사항 통합 list의 관련 요구사항이 유즈케이스, 상태, 프로세스, 기능, 정책 중 하나 이상에 반영됐는지 확인한다.
11. 참고자료에서 확인한 고객 불편, 채널 전략, IA, VoC, 운영 기준이 정책 판단 기준으로 재구성됐는지 확인한다.
12. 핵심 정책 질문이 범위, 상태 전이, 프로세스, 기능, 정책 항목에 일관되게 반영됐는지 확인한다.
13. 고객 표시 상태와 시스템 내부 상태가 분리되어 고객 안내, BSS 판정, 이력 저장 기준으로 추적되는지 확인한다.
14. 동일 고객·동일 대상·동일 조건의 반복 요청은 기존 처리 상태를 우선 안내하고 중복 처리와 중복 저장을 제한하는지 확인한다.
15. 개발자는 프로세스, 기능, 정책 ID를 기준으로 설계 항목을 전환할 수 있고 QA는 정상, 제한, 실패, 복구 기대 결과를 도출할 수 있는지 확인한다.
