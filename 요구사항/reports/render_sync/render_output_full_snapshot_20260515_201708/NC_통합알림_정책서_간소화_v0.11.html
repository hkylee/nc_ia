1. 개요

가. 범위

• 본 정책서는 통합채널에서 제공하는 통합 알림 업무의 처리 기준을 정의한다.

• 요구사항에서 확인된 핵심 판단축은 알림함 조회, 행동 필요 알림 우선순위, 수신 유형·채널 설정, OS 푸시 권한, 필수·선택 알림 구분, 중복·빈도 제어, 발송 실패 복구와 발송 이력을 포함한다.

• 대상 채널은 앱·웹 FO이며 고객이 직접 조회, 조건 확인, 처리 요청, 결과 확인을 수행하는 셀프 처리 흐름을 포함한다.

• BSS와 연계 시스템이 수행하는 가능 여부 검증, 상태 판정, 처리 결과 회신은 정책 판단 기준에 포함한다.

• 화면 UI 상세, API 필드, DB 컬럼, 운영자 화면 상세는 제외하되 정책 판단에 필요한 기준은 포함한다.

• 상세 화면 설계, API 필드, DB 컬럼, 테스트 케이스는 후속 화면 설계서와 기능 상세 명세서에서 상세화한다.

• 핵심 정책 질문은 누가 이 업무를 시작할 수 있는지, 어떤 조건에서 제한되는지, 중복 요청을 어떻게 처리하는지, 실패 시 어떤 상태와 후속 조치로 복구하는지를 기준으로 둔다.

• 고객 표시 상태는 고객이 이해해야 하는 진행·완료·제한 상태로, 시스템 내부 상태는 BSS·연계 시스템의 판정·저장·회신 상태로 구분한다.

• 개발·QA는 유즈케이스 ID, 프로세스 ID, 기능 ID, 정책/정책항목 ID를 기준으로 정상·제한·실패·복구 테스트 조건과 기대 결과를 도출한다.

나. 설계 원칙

• 고객 과업 중심: 통합 알림은 내부 시스템 단위가 아니라 고객이 완료하려는 목적을 기준으로 구성한다.

• 셀프 처리 우선: 고객이 앱·웹에서 직접 완료할 수 있는 업무는 상담 전환보다 셀프 처리 경로를 우선 제공한다.

• BSS 판단 통합: 고객 상태, 가입 정보, 요금, 혜택, 제한 조건처럼 BSS 판단이 필요한 정보는 프로세스와 정책에 포함한다.

• 영향도 사전 고지: 비용, 혜택, 약정, 포인트, 쿠폰, 회선, 주문 상태에 영향이 있으면 처리 전에 안내한다.

• 권한·보안 우선: 민감정보 노출 또는 변경 처리는 본인확인, 동의, 권한 기준을 충족한 경우에만 허용한다.

• 근거 기반 운영: 요구사항과 참고자료에서 확인한 기준은 프로세스, 기능, 정책 중 하나 이상으로 추적 가능하게 관리한다.

2) 상태 전이 기준

현재 상태전이 이벤트다음 상태처리 기준 및 후속 처리
알림 생성 전고객 알림함 진입 또는 대상 이벤트 발생알림 생성됨주문, 결제, 혜택, 상담, 보안, 공지 등 알림 대상 사건이 식별되면 알림 유형과 연결 업무를 생성한다.
알림 생성됨알림 유형·수신 채널 판정 완료발송 대기필수·선택 알림 여부, 고객 동의, 연락처, 채널 우선순위가 확인되면 발송 대기로 전환한다.
발송 대기OS 권한·수신 연락처 유효발송 완료OS 푸시 권한, 서비스 수신 동의, 연락처 유효성, 빈도 제한을 모두 충족하면 발송하고 알림함에 표시한다.
발송 대기수신 거부·권한 거부·빈도 제한 확인수신 제한수신 거부, OS 권한 거부, 조용한 시간, 발송량 제한, 연락처 오류가 확인되면 발송을 제한하고 사유를 저장한다.
수신 제한중요 알림 미전달 또는 필수 확인 필요대체 확인 필요거래성·보안·처리 필요 알림이 전달되지 않으면 인앱 알림함, 상세 화면, 상담 전환 등 대체 확인 경로를 제공한다.
발송 완료고객 알림 확인 또는 후속 업무 완료확인·처리 완료고객이 알림을 읽거나 연결 업무를 완료하면 읽음, 처리 완료, 업무 결과 이력을 동기화한다.
발송 완료대상 업무 해결·조건 변경·기한 만료만료·무효혜택 만료, 주문 상태 변경, 상담 종료, 조건 변경으로 알림 목적이 사라지면 만료 또는 무효로 표시한다.
알림 생성 전운영 공지·점검 기준 배포알림 생성됨운영자가 공지, 점검, 긴급 안내 기준을 승인·배포하면 대상 고객군 기준으로 알림을 생성한다.

나. 정책 상세

1) 접근·권한 정책 (PG-ARZ-ACC-001)

• 접근 허용 범위 (PI-ARZ-ACC-001-01)
- 공개 정보는 비로그인 조회를 허용하고, 개인화 정보와 처리 요청은 로그인 또는 본인확인 후 허용한다.
• 권한 제한 대상 (PI-ARZ-ACC-001-02)
- 대리 처리, 법정대리인, 법인, 미성년, 제한 고객은 본인확인, 대리 권한, 법정대리인 동의, 고객 상태가 모두 확인된 경우에만 진행한다.
• 인증 전환 대상 (PI-ARZ-ACC-001-03)
- 민감정보 노출, 비용 발생, 상태 변경, 계약 영향 업무는 본인확인 또는 추가 인증 단계로 전환한다.
• 권한 이력 저장 항목 (PI-ARZ-ACC-001-04)
- 고객ID, 접근 결과, 제한 사유, 인증 전환 여부, 요청 채널, 처리 일시를 저장한다.

2) 대상 정보 노출 정책 (PG-ARZ-INF-001)

• 노출 대상 (PI-ARZ-INF-001-01)
- 고객에게 제공하는 대상 정보는 업무 완료에 필요한 기준 정보, 상태 정보, 안내 정보로 제한한다.
• 노출 우선순위 (PI-ARZ-INF-001-02)
- 고객 조건에 직접 맞는 정보, 진행 중 업무, 최근 이용 정보, 공통 안내 순으로 노출 우선순위를 둔다.
• 노출 제한 (PI-ARZ-INF-001-03)
- 고객 권한, 약관 동의, 보안 기준, 기준일이 충족되지 않은 정보는 개인화 결과에서 제외한다.
• 기준일 표시 (PI-ARZ-INF-001-04)
- 요금, 혜택, 약정, 상품 조건처럼 변동 가능한 정보는 기준일 또는 적용 시점을 함께 안내한다.

3) 고객 상태 조회 정책 (PG-ARZ-CST-001)

• 상태 조회 (PI-ARZ-CST-001-01)
- 고객 상태는 정상, 제한, 보류, 처리 중, 운영 확인 필요로 구분하고 업무 진입 시 최신 상태를 조회한다.
• 상태 사용 기준 (PI-ARZ-CST-001-02)
- 상태값은 업무 가능 여부, 인증 필요 여부, 고지 필요 여부, 상담 전환 여부를 판단하는 기준으로 사용한다.
• 상태 불일치 처리 (PI-ARZ-CST-001-03)
- 채널 상태와 BSS 상태가 다르면 BSS 최신 결과를 우선하고 불일치 이력을 운영 확인 대상으로 저장한다.
• 상태 이력 (PI-ARZ-CST-001-04)
- 상태 조회 결과, 조회 시점, 상태 전환 사유, 고객 안내 여부는 감사 가능한 이력으로 저장한다.

4) 가능 여부 검증 정책 (PG-ARZ-VAL-001)

• 검증 항목 (PI-ARZ-VAL-001-01)
- 고객 상태, 가입 정보, 보유 상품, 미납·제한, 약정, 혜택, 처리 가능 시간을 검증 항목으로 관리한다.
• 가능 판정 (PI-ARZ-VAL-001-02)
- 필수 검증 항목이 모두 충족되고 연계 시스템 성공 응답이 확인된 경우에만 처리 가능으로 판정한다.
• 제한 판정 (PI-ARZ-VAL-001-03)
- 미납, 권한 부족, 약정 제한, 진행 중 업무, 연계 실패가 있으면 처리 제한으로 판정하고 사유를 안내한다.
• 재검증 기준 (PI-ARZ-VAL-001-04)
- 재검증의 제한 조건은 제한·보류·만료 상태이거나 필수 조건이 누락된 경우이다.
- 제한 시 다음 가능 시점과 상담 전환 여부를 안내한다.

5) 인증·동의 정책 (PG-ARZ-AUT-001)

• 인증 수단 (PI-ARZ-AUT-001-01)
- 휴대폰 본인확인, PASS 인증, 공동인증서, 간편인증 중 업무 보안 수준에 맞는 수단을 제공한다.
• 인증 가능 횟수 (PI-ARZ-AUT-001-02)
- 동일 업무 세션 기준 최대 5회까지 허용한다.
• 인증번호 유효시간 (PI-ARZ-AUT-001-03)
- 인증 유효 시간은 10분이며 만료되면 재인증을 요구하고 실패 이력을 저장한다.
• 인증 실패 처리 (PI-ARZ-AUT-001-04)
- 가능 횟수를 초과하거나 유효시간이 만료되면 재요청 또는 상담 경로를 안내하고 상태 변경은 제한한다.
• 동의 이력 저장 항목 (PI-ARZ-AUT-001-05)
- 동의 항목, 약관 버전, 동의 일시, 철회 여부, 처리 채널을 저장한다.

6) 입력값 검증 정책 (PG-ARZ-INP-001)

• 필수 입력 항목 (PI-ARZ-INP-001-01)
- 업무 처리에 필요한 최소 항목만 필수로 지정하고 선택 항목과 구분한다.
• 입력 허용 범위 (PI-ARZ-INP-001-02)
- 이름, 연락처, 이메일, 주소, 식별값, 요청값은 업무별 형식과 허용 범위를 검증한다.
• 입력 오류 안내 항목 (PI-ARZ-INP-001-03)
- 오류 항목, 수정 방법, 재시도 가능 여부를 안내한다.
• 입력값 민감정보 제한 (PI-ARZ-INP-001-04)
- 비밀번호, 인증번호, 결제정보 등 민감정보는 저장과 화면 노출 범위를 제한하고 로그에서 마스킹한다.

7) 영향도 고지 정책 (PG-ARZ-IMP-001)

• 고지 대상 (PI-ARZ-IMP-001-01)
- 비용, 혜택, 약정, 회선, 쿠폰, 포인트, 주문, 환불, 데이터 보관에 영향이 있으면 처리 전 고지한다.
• 고지 시점 (PI-ARZ-IMP-001-02)
- 고객이 최종 처리 요청을 확정하기 전에 영향도, 제한 사유, 적용 시점, 철회 가능 여부를 안내한다.
• 확인 기준 (PI-ARZ-IMP-001-03)
- 고객이 필수 고지 항목을 확인한 경우에만 다음 단계 또는 최종 처리 요청을 허용한다.
• 고지 이력 (PI-ARZ-IMP-001-04)
- 고지 항목, 확인 여부, 확인 일시, 적용 기준은 처리 이력과 함께 저장한다.

8) 처리 요청 접수 정책 (PG-ARZ-REQ-001)

• 접수 조건 (PI-ARZ-REQ-001-01)
- 필수 입력, 인증·동의, 영향도 확인, 가능 여부 검증이 완료된 경우에만 처리 요청을 접수한다.
• 요청 상태값 (PI-ARZ-REQ-001-02)
- 접수, 진행 중, 완료, 실패, 보류, 제한으로 구분한다.
• 접수 실패 (PI-ARZ-REQ-001-03)
- 요청 접수 실패 시 고객에게 실패 사유와 재시도 가능 여부를 안내하고 완료 상태로 확정하지 않는다.
• 접수 이력 저장 항목 (PI-ARZ-REQ-001-04)
- 요청자, 접수 일시, 요청 내용, 처리 상태, 접수 실패 사유를 저장한다.

9) 중복 요청 제한 정책 (PG-ARZ-DUP-001)

• 중복 판단 기준 (PI-ARZ-DUP-001-01)
- 중복 판단 기준에서 이전 요청이 실패·만료·고객 취소로 종료된 경우에만 재요청을 허용한다.
- 진행 중 요청이 있으면 중복 요청으로 제한한다.
• 중복 제한 (PI-ARZ-DUP-001-02)
- 중복 요청은 추가 접수를 제한하고 기존 요청 상태와 예상 처리 기준을 안내한다.
• 재요청 허용 (PI-ARZ-DUP-001-03)
- 이전 요청이 실패, 취소, 만료 상태이면 업무별 재요청 가능 기준에 따라 다시 접수할 수 있다.
• 중복 이력 저장 항목 (PI-ARZ-DUP-001-04)
- 중복 제한 여부, 기존 요청 ID, 고객 안내 여부를 저장한다.
• 만료/무효/해결 완료 알림 구분 제공 (PI-ARZ-RQCOV-028)
- 만료, 무효, 해결 완료 알림은 일반 미확인 알림과 구분한다.
- 고객이 더 이상 처리할 수 없는 경우에는 사유를 표시하고, 복구 또는 재요청 가능성이 있으면 해당 경로만 제공한다.

10) BSS 연계 판정 정책 (PG-ARZ-BSS-001)

• 조회 기준 (PI-ARZ-BSS-001-01)
- BSS 조회는 고객 상태, 계약, 청구, 혜택, 제한 조건처럼 처리 가능 여부에 영향을 주는 항목으로 제한한다.
• 판정 우선순위 (PI-ARZ-BSS-001-02)
- 채널 보유 정보와 BSS 결과가 다르면 BSS 최신 판정 결과를 우선 적용한다.
• 결과 반영 (PI-ARZ-BSS-001-03)
- BSS 성공 회신과 상태 변경 완료가 확인된 경우에만 처리 완료로 확정한다.
• 실패 처리 방식 (PI-ARZ-BSS-001-04)
- BSS 조회 실패 또는 처리 실패 시 개인화 처리와 상태 변경을 제한하고 공통 안내 또는 상담 경로를 제공한다.

11) 상태 전환 정책 (PG-ARZ-STAT-001)

• 전환 조건 (PI-ARZ-STAT-001-01)
- 상태 전환은 필수 검증과 연계 처리 결과가 충족된 경우에만 수행한다.
• 전환 실패 (PI-ARZ-STAT-001-02)
- 상태 전환 실패 시 완료로 확정하지 않고 보류, 실패, 운영 확인 필요 중 하나로 분류한다.
• 전환 고지 (PI-ARZ-STAT-001-03)
- 고객 이용 가능 여부나 혜택에 영향을 주는 상태 전환은 고객에게 결과와 적용 시점을 안내한다.
• 전환 이력 (PI-ARZ-STAT-001-04)
- 전환 전 상태, 전환 후 상태, 전환 사유, 처리자, 처리 시점은 감사 가능한 이력으로 저장한다.

12) 처리 결과·이력 정책 (PG-ARZ-RSLT-001)

• 결과 구분 (PI-ARZ-RSLT-001-01)
- 처리 결과는 성공, 실패, 보류, 제한, 운영 확인 필요로 구분하고 고객에게 다음 행동을 안내한다.
• 처리 결과·이력 반영 기준 (PI-ARZ-RSLT-001-02)
- 연계 시스템 성공 회신과 상태 변경 완료가 모두 확인된 경우에만 처리 완료로 확정한다.
• 처리 결과·이력 실패 안내 (PI-ARZ-RSLT-001-03)
- 실패 또는 제한 결과는 사유, 재시도 가능 여부, 상담 전환 가능 여부, 적용되지 않은 항목을 안내한다.
• 이력 저장 (PI-ARZ-RSLT-001-04)
- 요청자, 처리 일시, 결과, 실패 사유, 연계 응답, 고객 고지 여부는 감사 가능한 이력으로 저장한다.

13) 후속 업무 연결 정책 (PG-ARZ-NEXT-001)

• 연결 대상 (PI-ARZ-NEXT-001-01)
- 처리 완료, 실패, 제한, 보류 결과에 따라 다음 업무, 재시도, 상담, 관련 메뉴 중 하나 이상을 제공한다.
• 연결 제한 (PI-ARZ-NEXT-001-02)
- 권한 부족, 인증 필요, 고위험 업무, 연계 오류가 있으면 직접 연결 대신 안내 또는 상담 경로를 제공한다.
• 컨텍스트 전달 (PI-ARZ-NEXT-001-03)
- 후속 업무로 이동할 때 고객 요청, 대상 정보, 처리 결과, 제한 사유를 필요한 범위에서 전달한다.
• 연결 이력 (PI-ARZ-NEXT-001-04)
- 후속 업무 선택, 진입 성공 여부, 실패 사유는 품질 관리와 운영 개선 이력으로 저장한다.
• 알림 상세 화면 이동 및 컨텍스트 전달 (PI-ARZ-RQCOV-012)
- 알림 상세 화면 이동 시 주문번호, 혜택ID, 문의ID, 구독ID처럼 대상 업무를 식별할 최소 컨텍스트를 전달한다.
- 컨텍스트가 만료되었거나 권한이 없으면 알림 상세에서 사유와 대체 경로를 안내한다.
• 알림 클릭 후 업무 화면 연결 (PI-ARZ-RQCOV-021)
- 알림 클릭 후 이동 대상은 관련 상세 화면 또는 후속 처리 화면으로 정한다.
- 이동 대상이 없는 알림은 알림 상세에서 발생 이유, 현재 상태, 고객이 할 수 있는 다음 행동을 안내한다.
• 알림 우선순위/긴급도 체계 제공 (PI-ARZ-RQCOV-023)
- 알림 긴급도는 고객 손실 가능성, 보안 위험, 계약·결제 영향, 후속 처리 필요 여부를 기준으로 산정한다.
- 긴급 알림은 목록 상단에 표시하고 참고성 알림과 시각적으로 구분한다.
• 행동 필요 알림과 참고성 알림 분리 제공 (PI-ARZ-RQCOV-024)

- 참고성 알림은 상태 안내 중심으로 표시하고 행동 필요 알림과 별도 필터로 제공한다.

- 병합 시 최신 상태, 이전 알림 이력, 처리 완료 여부를 함께 표시한다.
• 알림 상태 동기화 제공 (PI-ARZ-RQCOV-027)
- 알림 상태는 읽음, 삭제, 중요 표시, 처리 완료, 무효 상태를 기기 간 동기화한다.
- 대상 업무가 이미 완료되면 알림은 처리 완료로 표시하고 고객이 같은 행동을 반복하지 않도록 제한한다.

- 변경자, 변경일, 변경 사유, 변경 전후 값, 승인자는 운영 이력으로 저장한다.

- 알림 미수신이 확인되면 푸시 차단, 연락처 오류, 발송 실패, 수신 거부, 빈도 제한 중 원인을 구분한다.
- 고객에게는 알림센터, 공지 영역, 상세 화면, 상담 전환 중 가능한 대체 확인 경로를 제공한다.

• 대리점: 단골 대상 마케팅 메시지 발송 (PI-ARZ-RQCOV-018)
- 대리점 단골 대상 마케팅 메시지는 고객 마케팅 수신 동의, 대리점 관계 유효성, 발송 빈도 제한을 모두 충족한 경우에만 허용한다.
- 신고·차단 이력이 있거나 금칙어 기준을 위반하면 발송을 제한한다.
• 고객 단위 알림 발송량 제한 (PI-ARZ-RQCOV-019)
- 고객 단위 알림 발송량은 알림 유형과 중요도에 따라 제한한다.

• 카테고리별 알림 필터 및 보기 제공 (PI-ARZ-RQCOV-025)
- 알림 카테고리는 주문, 결제, 혜택, 멤버십, 구독, 상담, 보안, 공지 등 업무 기준으로 제공한다.
- 고객이 선택한 필터와 정렬 조건은 세션 또는 계정 기준으로 유지할 수 있다.

20) 요구사항 반영 관리 정책 (PG-ARZ-REQMAP-001)

• 근거 부족 (PI-ARZ-REQMAP-001-03)

- 추적 관리 기준에서는 필수 약관과 개인정보 처리 동의가 완료되어야 업무를 허용한다.

• 이벤트 알림 발송 및 상세 화면 연결 (PI-ARZ-RQCOV-001)
- 이벤트 알림은 당첨 발표, 지급 완료, 만료 임박, 주문·결제·상담 상태 변경처럼 고객 후속 행동이 필요한 상태값이 변경된 경우 생성한다.
- 알림은 인앱 알림센터에 저장하고, 고객이 수신 동의한 채널에는 푸시 또는 외부 메시지로 발송하며 클릭 시 동일 업무 상세 화면으로 연결한다.

- OS 접근권한 요청은 알림의 고객 효용이 분명한 시점에만 노출한다.
- 거부 시 기능 제한 범위, 대체 확인 경로, 설정 앱 이동 방법을 안내하고 동일 맥락의 반복 요청은 제한한다.
• OS 접근권한 상태 안내/설정 이동 (PI-ARZ-RQCOV-004)
- OS 푸시 권한이 꺼져 있으면 앱푸시 발송은 제한하고 권한 상태, 제한되는 알림 유형, 설정 앱 이동 경로를 안내한다.
- 필수 알림은 인앱 알림센터 또는 허용된 대체 채널로 확인 가능하게 한다.
• 알림 채널별 수신 연락처 설정 (PI-ARZ-RQCOV-005)
- 채널별 수신 연락처는 대표 연락처와 고객이 개별 지정한 연락처를 구분해 관리한다.
- 문자, 이메일, 카카오 등 채널별 연락처가 유효하지 않으면 발송을 제한하고 변경 안내를 제공한다.
• 대표 연락처 변경에 따른 알림 우선순위 갱신 및 예외 채널 개별 지정 (PI-ARZ-RQCOV-006)
- 대표 연락처가 변경되면 기본 수신 연락처와 채널 우선순위를 자동 갱신한다.
- 단, 고객이 특정 채널에 개별 연락처를 지정한 경우 해당 예외값을 우선 적용하고 변경 전후 값을 이력으로 저장한다.
• 푸시 권한 점진적 요청 (PI-ARZ-RQCOV-007)
- 푸시 권한은 앱 최초 실행 시 일괄 요청하지 않고 주문 상태 변경, 혜택 만료, 상담 답변처럼 알림 필요성이 발생한 시점에 요청한다.
- 거부 이력이 있으면 즉시 재요청하지 않고 알림함 또는 설정 화면에서 다시 선택할 수 있게 한다.
• 알림 유형별 수신 설정 및 알림센터 연계 (PI-ARZ-RQCOV-008)
- 주문·배송, 혜택 만료, 데이터 임계, 상담 답변, 보안, 공지 등 알림 유형별 수신 설정을 제공한다.
- 설정값은 알림센터 표시 기준과 연동해 고객이 어떤 알림을 받거나 받지 않는지 확인할 수 있게 한다.
• 알림 채널 정책 및 알림 유형별 수신 기준 안내 (PI-ARZ-RQCOV-009)
- 알림 채널 정책은 필수 알림과 선택 알림을 구분해 안내한다.
- 필수 알림은 거래, 보안, 처리 결과처럼 고객 보호와 계약 이행에 필요한 경우로 제한하고 선택 알림은 고객 동의와 채널별 수신 설정을 따른다.
• 인앱 알림센터 (PI-ARZ-RQCOV-011)
- 인앱 알림센터는 여러 서비스에서 발생한 알림을 하나의 목록으로 제공한다.
- 행동 필요 알림은 우선 표시하고, 고객은 읽음·미읽음, 삭제, 중요 표시, 상세 이동, 후속 처리 상태를 확인할 수 있다.
• 알림/고지 채널/언어 설정 (PI-ARZ-RQCOV-016)
- 고객의 언어, 대표 연락처, 채널별 수신 가능 상태를 기준으로 알림·고지 채널을 선택한다.
- 외국어 안내가 필요한 경우 고객 언어 설정을 우선 적용하고, 지원하지 않는 언어는 기본 언어와 대체 안내를 함께 제공한다.
• 알림 우선순위/중복 방지/빈도 제어 (PI-ARZ-RQCOV-017)
- 알림 우선순위는 거래·보안·행동 필요·참고·마케팅 순으로 적용한다.
- 동일 사건 중복 알림은 병합하고 선택 알림은 고객 단위 일일 발송량, 조용한 시간, 다시알림 설정을 따른다.
• 통합 알림함 제공 (PI-ARZ-RQCOV-020)
- 통합 알림함은 서비스 유형, 카테고리, 중요도, 행동 필요 여부, 읽음 상태를 함께 제공한다.
- 고객은 여러 서비스의 알림을 하나의 목록에서 확인하되 필터로 필요한 업무만 좁혀 볼 수 있다.

21) 연계 오류·장애 처리 정책 (PG-ARZ-ERR-002)

• 장애 구분 (PI-ARZ-ERR-002-01)
- BSS 장애, 연계 장애, 인증기관 장애, 채널 장애, 일시 지연을 구분해 처리 기준을 적용한다.
• 처리 제한 (PI-ARZ-ERR-002-02)
- 정확한 판정이 불가능한 장애 상황에서는 상태 변경과 비용 발생 처리를 제한한다.
• 재시도 기준 (PI-ARZ-ERR-002-03)
- 일시 장애는 제한 횟수 내 재시도를 허용하고 반복 장애는 상담 또는 운영 확인으로 전환한다.
• 장애 이력 (PI-ARZ-ERR-002-04)
- 장애 유형, 발생 시점, 고객 영향, 처리 결과는 운영 모니터링 이력으로 저장한다.

22) 데이터 보관·파기 정책 (PG-ARZ-DATA-001)

• 보관 대상 (PI-ARZ-DATA-001-01)
- 처리 이력, 인증 이력, 고지 확인, 연계 결과, 운영 변경 증적은 목적별 보관 대상으로 분류한다.
• 파기 대상 (PI-ARZ-DATA-001-02)
- 업무 목적이 종료되고 법정 보관 사유가 없는 개인정보와 임시 데이터는 파기 대상으로 분류한다.
• 보관 기간 (PI-ARZ-DATA-001-03)
- 보관 기간은 법정 보관 사유, 고객 분쟁 대응, 처리 이력 추적 필요성을 기준으로 정한다.
- 기간 만료 시 파기 대상으로 전환하고 파기 이력을 저장한다.
• 파기 이력 (PI-ARZ-DATA-001-04)
- 파기 대상, 파기 시점, 파기 방식, 처리자는 감사 가능한 이력으로 저장한다.

최종 점검 기준

통합 알림 정책서 제출 전 점검
1. 통합 알림 범위, 제외 범위, 후속 상세화 영역이 고객 과업 기준으로 명확한지 확인한다.
2. 고객이 업무를 시작해 완료하거나 제한 사유와 대체 경로를 확인할 수 있는지 확인한다.
3. 유즈케이스, 상태, 프로세스, 기능, 정책의 ID와 연결 관계가 누락 없이 이어지는지 확인한다.
4. BSS와 연계 시스템의 조회, 판정, 상태 변경, 결과 회신 기준이 프로세스와 정책에 반영됐는지 확인한다.
5. 모든 프로세스가 관련 기능과 실제 정책 그룹명에 연결되어 있는지 확인한다.
6. 모든 기능이 화면 단위가 아니라 처리 단위이며 세부 기능 구성이 있는지 확인한다.
7. 정책 상세가 기능 설명이 아니라 실제 기능 동작값, 허용 목록, 횟수, 시간, 제한 조건, 채널, 고지, 저장 기준으로 작성됐는지 확인한다.
8. 개인정보, 민감정보, 로그, 이력, 보관·파기 기준이 포함되어 있는지 확인한다.
9. 운영 기준 관리, 변경 이력, 품질 모니터링, 개선 과제 관리 기준이 포함되어 있는지 확인한다.
10. 요구사항 통합 list의 관련 요구사항이 유즈케이스, 상태, 프로세스, 기능, 정책 중 하나 이상에 반영됐는지 확인한다.
11. 참고자료에서 확인한 고객 불편, 채널 전략, IA, VoC, 운영 기준이 정책 판단 기준으로 재구성됐는지 확인한다.
12. 핵심 정책 질문(누가, 언제, 어떤 조건, 중복·실패·복구 기준)이 범위와 정책 상세에 반영되었는지 확인한다.
13. 고객 표시 상태와 시스템 내부 상태가 구분되어 있고 상태별 후속 행동이 테스트 가능하게 정의되었는지 확인한다.
14. 개발자가 기능 설계 항목으로 전환할 수 있도록 프로세스-기능-정책 연결과 ID 추적성이 유지되는지 확인한다.
15. QA가 정상·제한·실패·복구 테스트 조건과 기대 결과를 요구사항 매핑 기준으로 도출할 수 있는지 확인한다.
