1. 개요

가. 범위

• 본 정책서는 통합채널에서 제공하는 고객센터 통합허브 업무의 처리 기준을 정의한다.

• 요구사항에서 확인된 핵심 판단축은 셀프 해결 가능 범위, 문제 유형별 해결 경로, 상담 채널 선택, 1:1 문의 접수·상태 관리, 상담 문맥 전달, 콜센터·온라인 재유입, 운영 품질 기준을 포함한다.

• 대상 채널은 앱·웹 FO이며 고객이 직접 조회, 조건 확인, 처리 요청, 결과 확인을 수행하는 셀프 처리 흐름을 포함한다.

• BSS와 연계 시스템이 수행하는 가능 여부 검증, 상태 판정, 처리 결과 회신은 정책 판단 기준에 포함한다.

• 화면 UI 상세, API 필드, DB 컬럼, 운영자 화면 상세는 제외하되 정책 판단에 필요한 기준은 포함한다.

• 상세 화면 설계, API 필드, DB 컬럼, 테스트 케이스는 후속 화면 설계서와 기능 상세 명세서에서 상세화한다.

• 포함 범위는 셀프 해결 허브, 문의 접수, 상담 전환, 상태 추적, 상담 문맥 전달, 고객지원 품질 환류이며 FAQ 콘텐츠 상세 작성과 매장 안내 상세는 별도 정책서 기준을 따른다.

• 핵심 정책 질문은 고객이 어떤 문제를 셀프로 해결할 수 있는지, 언제 상담으로 전환해야 하는지, 동일 문의 반복 시 기존 접수와 어떻게 연결하는지, 상담 문맥과 결과 이력을 어디까지 전달·저장하는지를 기준으로 둔다.

• 고객 표시 상태는 해결 가능, 상담 연결 필요, 접수 완료, 답변 완료처럼 고객 행동 기준으로 관리하고, 시스템 내부 상태는 배정, 검토, 연계 대기, BSS 회신, 품질 환류 상태로 구분한다.

• 동일 고객, 동일 문제 유형, 동일 대상 식별자의 5분 이내 반복 문의는 기존 접수번호와 처리 상태를 우선 안내하고 중복 접수를 제한한다.

나. 설계 원칙

• 고객 과업 중심: 고객센터 통합허브은 내부 시스템 단위가 아니라 고객이 완료하려는 목적을 기준으로 구성한다.

• 셀프 처리 우선: 고객이 앱·웹에서 직접 완료할 수 있는 업무는 상담 전환보다 셀프 처리 경로를 우선 제공한다.

• BSS 판단 통합: 고객 상태, 가입 정보, 요금, 혜택, 제한 조건처럼 BSS 판단이 필요한 정보는 프로세스와 정책에 포함한다.

• 영향도 사전 고지: 비용, 혜택, 약정, 포인트, 쿠폰, 회선, 주문 상태에 영향이 있으면 처리 전에 안내한다.

• 권한·보안 우선: 민감정보 노출 또는 변경 처리는 본인확인, 동의, 권한 기준을 충족한 경우에만 허용한다.

• 근거 기반 운영: 요구사항과 참고자료에서 확인한 기준은 프로세스, 기능, 정책 중 하나 이상으로 추적 가능하게 관리한다.

1) 접근·권한 정책 (PG-PIN-ACC-001)

• 접근 허용 범위 (PI-PIN-ACC-001-01)
- 공개 정보는 비로그인 조회를 허용하고, 개인화 정보와 처리 요청은 로그인 또는 본인확인 후 허용한다.
• 권한 제한 대상 (PI-PIN-ACC-001-02)
- 대리 처리, 법정대리인, 법인, 미성년, 제한 고객은 본인확인, 대리 권한, 법정대리인 동의, 고객 상태가 모두 확인된 경우에만 진행한다.
• 인증 전환 대상 (PI-PIN-ACC-001-03)
- 민감정보 노출, 비용 발생, 상태 변경, 계약 영향 업무는 본인확인 또는 추가 인증 단계로 전환한다.
• 권한 이력 저장 항목 (PI-PIN-ACC-001-04)
- 고객ID, 접근 결과, 제한 사유, 인증 전환 여부, 요청 채널, 처리 일시를 저장한다.

2) 대상 정보 노출 정책 (PG-PIN-INF-001)

• 노출 대상 (PI-PIN-INF-001-01)
- 고객에게 제공하는 대상 정보는 업무 완료에 필요한 기준 정보, 상태 정보, 안내 정보로 제한한다.
• 노출 우선순위 (PI-PIN-INF-001-02)
- 고객 조건에 직접 맞는 정보, 진행 중 업무, 최근 이용 정보, 공통 안내 순으로 노출 우선순위를 둔다.
• 노출 제한 (PI-PIN-INF-001-03)
- 고객 권한, 약관 동의, 보안 기준, 기준일이 충족되지 않은 정보는 개인화 결과에서 제외한다.
• 기준일 표시 (PI-PIN-INF-001-04)
- 요금, 혜택, 약정, 상품 조건처럼 변동 가능한 정보는 기준일 또는 적용 시점을 함께 안내한다.

3) 고객 상태 조회 정책 (PG-PIN-CST-001)

• 상태 조회 (PI-PIN-CST-001-01)
- 고객 상태는 정상, 제한, 보류, 처리 중, 운영 확인 필요로 구분하고 업무 진입 시 최신 상태를 조회한다.
• 상태 사용 기준 (PI-PIN-CST-001-02)
- 상태값은 업무 가능 여부, 인증 필요 여부, 고지 필요 여부, 상담 전환 여부를 판단하는 기준으로 사용한다.
• 상태 불일치 처리 (PI-PIN-CST-001-03)
- 채널 상태와 BSS 상태가 다르면 BSS 최신 결과를 우선하고 불일치 이력을 운영 확인 대상으로 저장한다.
• 상태 이력 (PI-PIN-CST-001-04)
- 상태 조회 결과, 조회 시점, 상태 전환 사유, 고객 안내 여부는 감사 가능한 이력으로 저장한다.

4) 가능 여부 검증 정책 (PG-PIN-VAL-001)

• 검증 항목 (PI-PIN-VAL-001-01)
- 고객 상태, 가입 정보, 보유 상품, 미납·제한, 약정, 혜택, 처리 가능 시간을 검증 항목으로 관리한다.
• 가능 판정 (PI-PIN-VAL-001-02)
- 필수 검증 항목이 모두 충족되고 연계 시스템 성공 응답이 확인된 경우에만 처리 가능으로 판정한다.
• 제한 판정 (PI-PIN-VAL-001-03)
- 미납, 권한 부족, 약정 제한, 진행 중 업무, 연계 실패가 있으면 처리 제한으로 판정하고 사유를 안내한다.
• 재검증 기준 (PI-PIN-VAL-001-04)
- 재검증의 우선순위는 고객 피해 가능성, 금전 영향, 상태 변경 여부 순으로 판단한다.
- 충돌 시 제한 기준을 먼저 적용하고 예외 사유를 기록한다.

5) 인증·동의 정책 (PG-PIN-AUT-001)

• 인증 수단 (PI-PIN-AUT-001-01)
- 휴대폰 본인확인, PASS 인증, 공동인증서, 간편인증 중 업무 보안 수준에 맞는 수단을 제공한다.
• 인증 가능 횟수 (PI-PIN-AUT-001-02)
- 동일 업무 세션 기준 최대 5회까지 허용한다.
• 인증번호 유효시간 (PI-PIN-AUT-001-03)
- 인증 유효 시간은 10분이며 만료되면 재인증을 요구하고 실패 이력을 저장한다.
• 인증 실패 처리 (PI-PIN-AUT-001-04)
- 가능 횟수를 초과하거나 유효시간이 만료되면 재요청 또는 상담 경로를 안내하고 상태 변경은 제한한다.
• 동의 이력 저장 항목 (PI-PIN-AUT-001-05)
- 동의 항목, 약관 버전, 동의 일시, 철회 여부, 처리 채널을 저장한다.

• 입력 오류 안내 항목 (PI-PIN-INP-001-03)
- 오류 항목, 수정 방법, 재시도 가능 여부를 안내한다.
• 입력값 민감정보 제한 (PI-PIN-INP-001-04)
- 비밀번호, 인증번호, 결제정보 등 민감정보는 저장과 화면 노출 범위를 제한하고 로그에서 마스킹한다.

• 고지 시점 (PI-PIN-IMP-001-02)
- 고객이 최종 처리 요청을 확정하기 전에 영향도, 제한 사유, 적용 시점, 철회 가능 여부를 안내한다.
• 확인 기준 (PI-PIN-IMP-001-03)
- 고객이 필수 고지 항목을 확인한 경우에만 다음 단계 또는 최종 처리 요청을 허용한다.
• 고지 이력 (PI-PIN-IMP-001-04)
- 고지 항목, 확인 여부, 확인 일시, 적용 기준은 처리 이력과 함께 저장한다.

- 필수 입력, 인증·동의, 영향도 확인, 가능 여부 검증이 완료된 경우에만 처리 요청을 접수한다.

• 접수 이력 저장 항목 (PI-PIN-REQ-001-04)
- 요청자, 접수 일시, 요청 내용, 처리 상태, 접수 실패 사유를 저장한다.
• 고객센터 알림 (PI-PIN-RQCOV-003)

• 변경 이력 (PI-PIN-OPR-001-04)
- 변경자, 변경일, 변경 사유, 변경 전후 값, 승인자는 운영 이력으로 저장한다.
• 이용안내·FAQ·공지 공통화 기준 (PI-PIN-OPR-001-05)
- 이용안내, FAQ, 공지는 업무 유형, 고객 상태, 검색 키워드, 노출 채널 기준으로 공통 구조를 관리한다.
- 콘텐츠 등록·수정·종료는 승인자, 적용 시작일, 종료일, 대체 콘텐츠 여부를 함께 저장하고 검색과 탐색 경로가 단절되지 않도록 한다.

18) 운영 변경 이력 관리 정책 (PG-PIN-OPR-002)

• 운영 관리 결과 구분 (PI-PIN-OPR-002-01)
- 처리 결과는 성공, 실패, 보류, 제한, 운영 확인 필요로 구분하고 고객에게 다음 행동을 안내한다.
• 운영 관리 반영 기준 (PI-PIN-OPR-002-02)
- 연계 시스템 성공 회신과 상태 변경 완료가 모두 확인된 경우에만 처리 완료로 확정한다.
• 운영 관리 실패 안내 (PI-PIN-OPR-002-03)
- 실패 또는 제한 결과는 사유, 재시도 가능 여부, 상담 전환 가능 여부, 적용되지 않은 항목을 안내한다.
• 운영 관리 이력 저장 (PI-PIN-OPR-002-04)
- 요청자, 처리 일시, 결과, 실패 사유, 연계 응답, 고객 고지 여부는 감사 가능한 이력으로 저장한다.

19) 품질 관리 정책 (PG-PIN-QUL-001)

• 품질 지표 (PI-PIN-QUL-001-01)
- 완료율, 실패율, 재시도율, 상담 전환율, 처리 지연, 고객 피드백을 품질 지표로 관리한다.
• 품질 관리 모니터링 주기 (PI-PIN-QUL-001-02)
- 품질 관리 모니터링 주기 기준에서 실패·보류·상담 전환이 같은 조건에서 3회 이상 반복되면 품질 모니터링 대상으로 등록한다.
- 반복 원인은 개선 과제로 관리한다.
• 개선 과제 (PI-PIN-QUL-001-03)
- 반복 실패, 상담 전환 과다, 고객 불만 반복, 기준 정보 오류는 개선 과제로 등록한다.
• 효과 추적 (PI-PIN-QUL-001-04)
- 운영 보정 후 완료율, 실패율, 상담 전환율, 고객 피드백 변화를 비교해 개선 효과를 확인한다.

20) 요구사항 반영 관리 정책 (PG-PIN-REQMAP-001)

• 반영 기준 (PI-PIN-REQMAP-001-01)
- 요구사항은 유즈케이스, 상태, 프로세스, 기능, 정책 중 하나 이상에 반영해야 한다.
• 미반영 기준 (PI-PIN-REQMAP-001-02)
- BSS 반영은 결과 확정 후 원장 또는 업무 상태 변경이 필요한 경우에만 요청한다.
- 반영 실패 시 실패 상태와 재시도 가능 여부를 저장한다.
• 근거 부족 (PI-PIN-REQMAP-001-03)
- 근거 부족 기준에서 이전 요청이 실패·만료·고객 취소로 종료된 경우에만 재요청을 허용한다.
- 진행 중 요청이 있으면 중복 요청으로 제한한다.
• 추적 관리 (PI-PIN-REQMAP-001-04)
- 인증 유효 시간은 10분이며 만료되면 재인증을 요구한다.

21) 연계 오류·장애 처리 정책 (PG-PIN-ERR-002)

• 장애 구분 (PI-PIN-ERR-002-01)
- BSS 장애, 연계 장애, 인증기관 장애, 채널 장애, 일시 지연을 구분해 처리 기준을 적용한다.
• 처리 제한 (PI-PIN-ERR-002-02)
- 정확한 판정이 불가능한 장애 상황에서는 상태 변경과 비용 발생 처리를 제한한다.
• 재시도 기준 (PI-PIN-ERR-002-03)
- 일시 장애는 제한 횟수 내 재시도를 허용하고 반복 장애는 상담 또는 운영 확인으로 전환한다.
• 장애 이력 (PI-PIN-ERR-002-04)
- 장애 유형, 발생 시점, 고객 영향, 처리 결과는 운영 모니터링 이력으로 저장한다.

22) 데이터 보관·파기 정책 (PG-PIN-DATA-001)

• 보관 대상 (PI-PIN-DATA-001-01)
- 처리 이력, 인증 이력, 고지 확인, 연계 결과, 운영 변경 증적은 목적별 보관 대상으로 분류한다.
• 파기 대상 (PI-PIN-DATA-001-02)
- 업무 목적이 종료되고 법정 보관 사유가 없는 개인정보와 임시 데이터는 파기 대상으로 분류한다.
• 보관 기간 (PI-PIN-DATA-001-03)
- 보관 기간은 법정 보관 사유, 고객 분쟁 대응, 처리 이력 추적 필요성을 기준으로 정한다.
- 기간 만료 시 파기 대상으로 전환하고 파기 이력을 저장한다.
• 파기 이력 (PI-PIN-DATA-001-04)
- 파기 대상, 파기 시점, 파기 방식, 처리자는 감사 가능한 이력으로 저장한다.

최종 점검 기준

고객센터 통합허브 정책서 제출 전 점검
1. 고객센터 통합허브 범위, 제외 범위, 후속 상세화 영역이 고객 과업 기준으로 명확한지 확인한다.
2. 고객이 업무를 시작해 완료하거나 제한 사유와 대체 경로를 확인할 수 있는지 확인한다.
3. 유즈케이스, 상태, 프로세스, 기능, 정책의 ID와 연결 관계가 누락 없이 이어지는지 확인한다.
4. BSS와 연계 시스템의 조회, 판정, 상태 변경, 결과 회신 기준이 프로세스와 정책에 반영됐는지 확인한다.
5. 모든 프로세스가 관련 기능과 실제 정책 그룹명에 연결되어 있는지 확인한다.
6. 모든 기능이 화면 단위가 아니라 처리 단위이며 세부 기능 구성이 있는지 확인한다.
7. 정책 상세가 기능 설명이 아니라 실제 기능 동작값, 허용 목록, 횟수, 시간, 제한 조건, 채널, 고지, 저장 기준으로 작성됐는지 확인한다.
8. 개인정보, 민감정보, 로그, 이력, 보관·파기 기준이 포함되어 있는지 확인한다.
9. 운영 기준 관리, 변경 이력, 품질 모니터링, 개선 과제 관리 기준이 포함되어 있는지 확인한다.
10. 요구사항 통합 list의 관련 요구사항이 유즈케이스, 상태, 프로세스, 기능, 정책 중 하나 이상에 반영됐는지 확인한다.
11. 참고자료에서 확인한 고객 불편, 채널 전략, IA, VoC, 운영 기준이 정책 판단 기준으로 재구성됐는지 확인한다.
12. 핵심 정책 질문이 범위, 상태 전이, 프로세스, 기능, 정책 항목에 일관되게 반영됐는지 확인한다.
13. 고객 표시 상태와 시스템 내부 상태가 분리되어 고객 안내, BSS 판정, 이력 저장 기준으로 추적되는지 확인한다.
14. 동일 고객·동일 대상·동일 조건의 반복 요청은 기존 처리 상태를 우선 안내하고 중복 처리와 중복 저장을 제한하는지 확인한다.
15. 개발자는 프로세스, 기능, 정책 ID를 기준으로 설계 항목을 전환할 수 있고 QA는 정상, 제한, 실패, 복구 기대 결과를 도출할 수 있는지 확인한다.
