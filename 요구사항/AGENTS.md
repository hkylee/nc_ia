# AGENTS.md

## 1. 역할

이 저장소는 **통합채널 정책서 작성 자동화 Agent**를 개발·운영하기 위한 프로젝트다.

Agent는 업로드된 HTML 템플릿, 요구사항, 채널 전략 자료, 고객 조사, VoC, IA 분석 자료를 참고하여 NC 정책서를 장별로 작성하고, 유즈케이스 → 상태 → 프로세스 → 기능 → 정책의 연결성을 검수한다.

정책서는 단순 설명 문서가 아니라, 업체가 상세 설계할 수 있도록 업무 구조, 처리 흐름, 기능 범위, 정책 기준을 정의하는 문서다.

---

## 2. 기본 작성 원칙

정책서는 아래 원칙에 따라 작성한다.

1. **채널 전략 및 과제 방향과 정렬**
   - 기존 채널 기능을 그대로 옮기는 것이 아니라, 통합채널에서 고객 과업을 더 쉽고 명확하게 끝낼 수 있도록 작성한다.
   - 고객 셀프 처리 확대, 채널 경량화, 중복 제거, 목적 기반 UX, AI 적용 가능성, 보안·인증 안정성, BSS 연계 구조를 고려한다.

2. **중복되더라도 완결적으로 작성**
   - 다른 정책서, 다른 모듈, BSS 문서, 상세 설계서에서 작성될 것이라고 가정하지 않는다.
   - 담당 주제 안에서 고객이 업무를 시작해 완료하기까지 필요한 액터, 유즈케이스, 상태, 프로세스, 기능, 정책을 모두 작성한다.
   - 중복은 최종 통합 단계에서 정리할 수 있지만, 누락은 구축 범위에서 빠질 수 있다.

3. **BSS 업무도 충분히 통합 작성**
   - 정책서가 채널 산출물이더라도 고객 업무 완결에는 BSS의 판단과 처리가 포함된다.
   - BSS가 수행하는 검증, 판정, 상태 변경, 원장 반영, 이력 저장, 데이터 조회, 연계 결과 회신을 문서에 포함한다.
   - API 필드, DB 컬럼, 내부 구현 방식은 제외하되, 정책 판단에 영향을 주는 기준은 정책 항목으로 남긴다.

4. **요구사항을 모두 포함**
   - 기준 요구사항은 정책서의 최소 작성 범위다.
   - 요구사항은 프로세스, 기능, 정책 중 하나 이상에 반드시 매핑한다.
   - 요구사항을 그대로 복사하지 말고, 업무 흐름과 정책 판단 기준으로 재구성한다.

5. **새로 발견한 사항은 요구사항에 추가**
   - 작성 중 새로 도출된 고객 과업, 처리 흐름, 기능, 정책, 예외 조건, BSS 연동, 알림, 이력 관리 필요성은 요구사항 추가 대상으로 기록한다.

## 3. 산출물 작성 방식

정책서는 업로드된 HTML 템플릿을 기준으로 작성한다.

### 3.1 HTML 작성 기준

- 템플릿 HTML의 구조와 CSS를 유지한다.
- 기존 문서 폭, 폰트, 여백, 표 스타일을 유지한다.
- 각 단계마다 업데이트된 HTML 파일을 `output/` 폴더에 저장한다.
- 문장은 마침표 뒤에 `<br/>`로 줄바꿈한다.
- 표 안에서도 긴 문장은 `<br/>`로 줄바꿈한다.
- 문서 히스토리에는 매 단계 변경 내용을 추가한다.
- 화면 UI 상세, API 필드, DB 컬럼, 오류 코드 전체 목록, 운영자 화면 상세, 배치 설계, 비기능 상세는 본 정책서에 작성하지 않는다.
- 단, 해당 항목이 정책 판단에 직접 영향을 주면 정책 항목으로 남긴다.

### 3.2 파일명 기준

파일명은 아래 규칙을 따른다.

```text
NC_{주제}_정책서_간소화_v0.10.html
NC_{주제}_정책서_간소화_v0.11.html
NC_{주제}_정책서_간소화_v1.00.html
NC_{주제}_정책서_간소화_v1.01_보완본.html
```

### 3.3 버전 기준

- 최초 생성: `v0.10`
- 장별 추가 또는 보완 저장: `v0.11`, `v0.12`, ...
- 버전은 소수점 숫자가 아니라 시퀀스 라벨로 관리한다. `v0.19` 다음은 `v0.20`이다.
- 전체 장 작성 완료 후에도 동일한 시퀀스를 유지하고, 별도 메이저 전환이 필요할 때만 `v1.00`처럼 두 자리 minor로 시작한다.
- 점검 후 보완: 다음 시퀀스 버전 또는 명시 요청 시 `v1.01_보완본`

## 4. 장별 작성 순서

정책서는 한 번에 전체를 작성하지 않는다.

1. 표지 / 메타 정보
2. 0. 문서 히스토리
3. 1. 개요
   - 가. 범위
   - 나. 설계 원칙
4. 2. 주요 용어
5. 3. 유즈케이스 정의
   - 가. 액터
   - 나. 유즈케이스
   - 다. 유즈케이스 다이어그램
   - 라. 상태 전이표
6. 4. 프로세스 정의
   - 가. 프로세스 목록
   - 나. 전체 업무 흐름도
7. 5. 기능 정의
   - 가. 기능 목록
8. 6. 정책 정의
   - 가. 정책 목록
   - 나. 정책 상세
9. 최종 점검 기준
10. 점검 후 보완

각 장을 작성할 때는 이전 장과의 연결성을 반드시 확인한다.

## 5. 개요 작성 기준

### 5.1 범위

범위에는 아래 내용을 포함한다.

- 대상 업무
- 대상 채널
- 대상 고객
- 포함 범위
- 제외 범위
- 후속 산출물에서 상세화할 영역

작성 기준:

- 고객이 인식하는 업무 기준으로 작성한다.
- 내부 조직 기준, 시스템 기준, 기존 채널 기준으로만 쓰지 않는다.
- “T월드 업무”, “T멤버십 업무”보다 “요금제 변경”, “상품 상세”, “멤버십 혜택 사용”처럼 고객 과업 중심으로 작성한다.

### 5.2 설계 원칙

설계 원칙은 4~6개로 제한한다.

좋은 원칙의 조건:

- 판단 기준이 있다.
- 예외를 만들 수 있다.
- 기능과 정책에 연결된다.
- 채널과 BSS의 역할을 구분한다.
- 고객에게 보이는 경험 기준이 있다.

## 6. 주요 용어 작성 기준

주요 용어는 정책, 프로세스, 기능, 시스템 간 해석 차이를 줄이기 위한 기준이다.

### 6.1 작성 대상

- 상태 용어
- 고객 유형
- 인증·권한 용어
- 약관·동의 용어
- 상품·요금·혜택 용어
- 데이터 보관 용어
- 외부기관·제휴사 용어
- 정책 판단값에 쓰이는 용어

### 6.2 제외 대상

- 일반 명사
- 누구나 아는 단어
- 화면에서만 쓰이는 마케팅 표현
- 정책 판단에 쓰이지 않는 설명성 용어

### 6.3 설명 기준

- “무엇이다”로만 끝내지 않는다.
- 업무상 판단 기준을 포함한다.
- 유사 용어와의 차이가 드러나야 한다.

## 7. 액터 작성 기준

액터는 독립적인 책임 주체만 작성한다.

### 7.1 작성 대상

- 고객
- 법정대리인
- 대리인
- 운영자
- 핵심 처리 시스템
- 외부 승인·인증 기관
- 연계 시스템

복합 액터명은 쓰지 않는다.

- `법정대리인/대리인`, `발신 고객/수신 고객`처럼 여러 사람 책임 주체를 `/`, `·`, `및`으로 묶지 않는다.
- 시스템 액터도 `BSS/멤버십 연계 시스템`, `제휴처/외부 사용처`처럼 여러 책임 주체를 `/`, `·`, `및`으로 묶지 않는다.
- 책임이 같으면 하나의 책임명으로 통합하고, 책임 차이는 유즈케이스 설명·상태·정책 조건으로 내린다.
- 책임이 다르면 각각 별도 액터로 분리하고, 각 액터가 최소 1개 이상의 유즈케이스에 연결되도록 한다.

### 7.2 작성 제외

아래는 액터로 분리하지 않는다.

- 로그인 고객
- 비로그인 고객
- 정상 고객
- 제한 고객
- 휴면 고객
- 미성년 고객
- VIP 고객
- 일반 고객

이들은 액터가 아니라 상태, 권한 조건, 정책 상세에서 다룬다.

### 7.3 시스템 액터 통합 기준

시스템은 과도하게 쪼개지 않는다.

아래 시스템은 업무 특성에 따라 `연계 시스템`으로 통합할 수 있다.

- BSS
- 지식 베이스
- 상품 마스터
- 채널 업무 시스템
- 외부 연계 시스템
- 상담 시스템
- 결제 시스템
- 재고 시스템
- 주문 시스템

단, 통합 액터명은 하나의 책임명으로 쓴다. 예를 들어 멤버십 자격과 카드 결과를 회신하는 주체는 `멤버십 연계 시스템`, 제휴 사용처 결과를 회신하는 주체는 `외부 사용처`처럼 쓰고, 세부 시스템 차이는 설명·기능·정책 항목에서 다룬다.

### 7.4 액터 분리 기준

아래 질문 중 2개 이상에 해당하면 별도 액터로 분리한다.

1. 해당 주체가 유즈케이스를 직접 시작하는가?
2. 해당 주체가 검증·승인·처리 결과를 직접 생성하는가?
3. 해당 주체의 책임이 다른 액터와 명확히 다른가?
4. 해당 주체가 빠지면 프로세스가 성립하지 않는가?
5. 해당 주체별로 정책 기준이나 권한 기준이 달라지는가?

## 8. 유즈케이스 작성 기준

유즈케이스는 고객 또는 시스템이 완결해야 하는 업무 단위다.

### 8.1 작성 기준

- 고객이 인식하는 업무는 상위 유즈케이스로 작성한다.
- 시스템 내부 검증·저장·결과 제공은 보조 유즈케이스로 작성한다.
- 설명에는 시작 목적과 완료 상태가 드러나야 한다.

### 8.2 프로세스 정의 대상 판단

프로세스 정의 대상은 액터가 사람인지 시스템인지에 따라 1차 판단한다.

- 고객, 운영자, 법정대리인, 대리인처럼 사람이 수행하는 유즈케이스는 원칙적으로 `Y`로 둔다.
- AI 시스템, BSS, 연계 시스템, 외부기관처럼 시스템이 수행하는 검증·생성·조회·저장·결과 제공은 원칙적으로 `N`으로 둔다.
- 단, 시스템 처리라도 고객에게 노출되는 상태 변화, 업무 제한, 인증, 상담 전환, 처리 결과에 직접 영향을 주면 관련 고객 또는 운영자 프로세스 안에서 기능·정책으로 반영한다.

판단 예시:

```text
고객이 검색한다, 신청한다, 변경한다, 해지한다, 확인한다, 피드백한다 → Y
운영자가 등록한다, 검수한다, 승인한다, 보정한다, 관리한다 → Y
시스템이 조회한다, 분류한다, 저장한다, 결과를 생성한다, 연계 결과를 회신한다 → N
로그인 고객, 비로그인 고객, 정상 고객, 제한 고객처럼 고객 상태가 다른 경우 → 액터를 나누지 않고 정책·상태 조건으로 관리
```

## 9. 유즈케이스 다이어그램 작성 기준

유즈케이스 다이어그램은 액터와 유즈케이스 간 관계를 한눈에 보여준다.

### 9.1 표현 기준

- 고객 중심 유즈케이스를 중앙에 배치한다.
- 시스템 액터는 보조 처리로 연결한다.
- include 관계는 반복되는 공통 처리에만 사용한다.
- 화면 이동, 버튼 클릭, 팝업 노출 등 UI 단계는 넣지 않는다.

### 9.2 작성 방식

HTML에서는 `diagram-wrap` 영역에 텍스트 도식으로 작성할 수 있다.

예시:

```text
[고객] → (상품 상세 조회) → include: 상품 상세 구성, 상품 기준 정보 제공
[고객] → (고객별 적용 정보 확인) → include: 고객별 적용 정보 산정, 고객 상태 및 조건 정보 제공
[상품 운영자] → (상품 상세 정보 관리) → include: 등록, 검수, 승인, 배포
```

## 10. 상태 전이 작성 기준

상태는 고객 노출, 기능 허용, 정책 판단, 후속 처리의 기준이다.

### 10.1 작성 대상

아래 항목은 상태 후보 예시다.
단, 고객 노출, 기능 허용, 정책 판단, 후속 처리 기준이 실제로 달라질 때만 상태 코드로 관리한다.
인증 실패, 검증 실패, 처리 보류처럼 순간적인 처리 결과는 후속 가능 여부나 복구 경로가 달라지는 경우에만 상태로 승격하고, 그 외에는 전이 기준 또는 정책 항목으로 작성한다.

- 신청 전
- 신청 중
- 인증 완료
- 검증 실패
- 처리 완료
- 처리 보류
- 취소
- 제한
- 만료
- 복구 가능
- 복구 불가
- 운영 검토 필요
- 운영 반영 완료

### 10.2 작성 제외

- 화면 로딩
- 버튼 활성화
- 단순 안내 노출
- 임시 UI 상태

단, UI 상태가 업무 가능 여부를 결정하면 상태 코드로 관리한다.

### 10.3 상태 전이 기준

상태 전이는 아래 구조로 작성한다.

```text
현재 상태 → 전이 이벤트 → 다음 상태 → 처리 기준 및 후속 처리
```

### 10.4 정합성 기준

- 상태 전이표에서 사용하는 현재 상태와 다음 상태는 상태 코드 목록에 존재해야 한다.
- 상태 코드에 없는 표현을 상태 전이표에서 사용하지 않는다.
- 전이 이벤트에는 유즈케이스 또는 업무 사건을 작성한다.
- 상태 전이표에 별도 `관련 유즈케이스` 컬럼을 추가하지 않는다.
- 정상 흐름, 예외 흐름, 제한 흐름, 운영 검토 흐름을 함께 작성한다.
- `process_target=N`인 시스템/BSS/연계 유즈케이스라도 고객 노출 상태, 가입·사용 가능 여부, 제한 사유, 오류·복구 경로, BSS 반영 결과를 바꾸면 상태 전이의 내부 `usecase_ids`에 함께 연결한다.
- 시스템 보조 유즈케이스를 프로세스로 승격하지 않는 대신, 그 판단 결과가 고객/운영자 프로세스와 정책 항목에서 추적되도록 연결한다.

## 11. 프로세스 작성 기준

프로세스는 고객 또는 운영자가 경험하는 순서대로 작성한다.

### 11.1 작성 기준

- 고객 시작점부터 완료까지 순서대로 작성한다.
- 시작, 상태 확인, 조건 판단, 입력·선택, 영향도 확인, 인증·동의, 처리 요청, 결과 반영, 완료 안내 순서가 일반적이다.
- 시스템 내부 조회·판정·저장은 별도 프로세스로 과도하게 분리하지 않고 관련 기능으로 연결한다.
- 하나의 프로세스는 하나의 업무 판단 또는 처리 목적을 가져야 한다.

### 11.2 필수 연결

각 프로세스에는 반드시 아래 항목이 있어야 한다.

- 프로세스 ID
- 프로세스명
- 설명
- 관련 기능
- 관련 정책

### 11.3 관련 정책 기준

관련 정책명은 6. 정책 정의의 실제 정책 그룹명과 일치해야 한다.

### 11.4 내부 연결값 기준

- 내부 JSON의 `related_functions`, `related_policies`에는 순수 ID만 저장한다.
- HTML 표에는 렌더링 단계에서 ID와 명칭을 함께 표시한다.
- 표시 편의를 위해 `FN-... 기능명`, `PG-... 정책명` 같은 문자열을 내부 연결 필드에 저장하지 않는다.
- 내부 연결값과 화면 표시값을 섞으면 자동 검증, Health Check, Full 버전 확장, 보완 루프에서 연결이 끊긴다.
- 기능의 `process_ids`와 프로세스의 `related_functions`는 양방향으로 일치해야 한다.
- 프로세스에서 실행되는 정책은 프로세스 `related_policies`와 해당 기능의 `related_policies`에 모두 추적 가능해야 한다.
- 정책 목록에 정의된 정책과 기능 목록에 정의된 기능은 최소 1개 이상의 프로세스에서 참조되어야 한다.

## 15. 체크리스트 / 제출 Gate 기준

정책서 제출 가능 조건은 아래 기준을 따른다.

```text
총점 80점 이상
Gate 항목 전체 충족
요구사항 커버리지 100%
Critical / Major Open 이슈 0건
자체판정과 검토자판정 모두 충족
```

Agent는 최종 제출 전 아래 항목을 점검해야 한다.

- 액터가 과도하게 분리되지 않았는가?
- 로그인/비로그인 고객이 액터로 들어가지 않았는가?
- 사람 액터 유즈케이스는 Y인가?
- 시스템 액터 유즈케이스는 N인가?
- 상태 전이표의 상태명이 상태 코드 목록과 일치하는가?
- 고객 상태·노출·결과·복구를 바꾸는 시스템/BSS 유즈케이스가 상태 전이 `usecase_ids`에 연결되었는가?
- 모든 프로세스에 관련 기능과 관련 정책이 있는가?
- 기능 `process_ids`와 프로세스 `related_functions`가 양방향으로 일치하는가?
- 프로세스에서 참조되지 않는 기능과 정책이 0건인가?
- 프로세스의 관련 정책명이 정책 목록과 일치하는가?
- 모든 기능에 세부 기능 구성이 있는가?
- 모든 정책 그룹에 정책 상세가 있는가?
- 정책 상세에 실행 기준 없는 표현이 없는가?
- 개인정보·민감정보 보호 기준이 있는가?
- 운영자 관리, 변경 이력, 품질 모니터링 기준이 있는가?
- 요구사항 ID별 매핑이 가능한가?

## 16. 요구사항 매핑 기준

요구사항 목록이 있는 경우 Agent는 아래 기준으로 매핑한다.

### 16.1 매핑 대상

요구사항 1건은 아래 중 하나 이상에 연결되어야 한다.

- 유즈케이스 ID
- 상태 코드
- 프로세스 ID
- 기능 ID
- 정책 ID
- 정책 항목 ID

### 16.2 매핑 기준

- 고객 흐름 요구사항 → 유즈케이스 또는 프로세스
- 시스템 처리 요구사항 → 기능
- 판단 기준 요구사항 → 정책
- 예외 조건 요구사항 → 상태 전이 또는 정책
- 운영 관리 요구사항 → 운영자 프로세스, 기능, 정책
- 데이터 보관·이력 요구사항 → 기능 또는 정책
- 알림·고지 요구사항 → 기능 또는 정책
- 상세 요구사항명이 대체안, 예상 금액, 신청 경로, 약관 버전, 노출 상태, 삭제 가능 여부처럼 구체적 판단축을 포함하면 해당 판단축이 정책 항목명 또는 기능명에 드러나야 한다.
- 여러 상세 요구사항이 같은 상위 정책으로 묶이더라도 고객 경로, 처리 시점, 계산 기준, 외부 회신, 복구 가능성이 달라지면 하나의 일반 항목으로 합치지 않는다.
- 예를 들어 즉시 처리와 예약 처리, 로그인 경로와 비회원 경로, 일반 처리와 예외 처리, 예상 금액과 위약금과 결합 영향처럼 개발/QA 케이스가 달라지는 축은 별도 정책 항목 또는 기능 세부 구성으로 분리한다.
- 요구사항 원문을 복사하지 않되, Trace가 가능하도록 핵심 판단축은 짧은 정책 항목명으로 보존한다.

### 16.3 미반영 처리

요구사항이 정책서에 반영되지 않으면 `미반영`으로 표시한다.

미반영 항목은 사유를 함께 작성한다.

### 16.4 요구사항 커버리지 추적 기준

요구사항 커버리지는 본문 문장만 보고 판단하지 않는다.

- 최신 `requirements.db`의 상세 요구사항 수와 queue의 `detail_requirement_count`가 일치해야 한다.
- 각 정책서 spec에는 요구사항 반영 근거가 `trace_matrix` 또는 산출물 요소에 남아 있어야 한다.
- 각 정책서별 `PM-XX_requirement_trace.md` 리포트를 생성하고 queue에서 해당 경로를 참조해야 한다.
- 상세 요구사항을 부모 요구축으로 묶어 반영하더라도 추적 데이터는 상세 요구사항 ID 단위로 남긴다.
- 자동 매칭에서 검토 필요가 나오면 실제 누락, `trace_matrix` 부재, 표현상 매칭 부족 중 무엇인지 먼저 판정한다.
- trace 부재를 본문 장황화로 해결하지 않는다. 문서에는 정책 판단 기준을 쓰고, 추적성은 trace 데이터로 보강한다.

### 16.5 요구사항 업데이트 동기화

요구사항 파일 또는 DB가 바뀌면 아래 산출물을 함께 갱신한다.

- Topic Learning의 요구사항 수와 주요 묶음.
- 문서 히스토리와 작성 기준.
- `requirements_mapping_summary`.
- Requirement Trace Report.
- 최종 점검 기준의 요구사항 반영 항목.
- HTML과 BPMN 등 렌더링 산출물.

JSON 본문만 수정하고 HTML, Trace, Topic Learning을 재생성하지 않으면 서비스 화면과 검증 결과가 서로 달라진다.

작성 전에는 기존 Trace 파일만 믿지 않고 최신 `reports/evidence/requirements.db`의 `Depth 4` 기준 요구사항 수와 상세 요구사항명을 다시 확인한다.
새로 확인된 상세 요구사항명은 원문 복사 대상이 아니라 누락 방지 신호이며, 프로세스명, 기능명, 정책 항목명 중 하나에 정책 판단 언어로 재구성해 드러낸다.

미구독, 로그아웃, 빈 화면, 퀵메뉴, 미사용 상태처럼 고객 상태나 화면 조건에 가까운 요구는 액터나 별도 유즈케이스로 늘리지 않는다.
진입 프로세스, 상태, 기능 세부 구성, 대체 행동 정책으로 내린다.

정책 목록의 `items`는 문자열이 아니라 항상 `{id, name}` 객체로 작성하고, 정책 상세의 `id`, `name`, `policy_id`와 연결한다.
문자열 항목은 HTML에는 보일 수 있어도 Full 확장, Health Check, 보완 루프에서 정책 상세와 연결이 끊길 수 있다.

## 17. Codex 작업 방식

Codex는 저장소 파일을 수정할 때 아래 방식을 따른다.

### 17.1 기본 명령어 예시

### 17.4 직접 작성 시 Agent 동작 준수

Codex가 Agent 대신 직접 정책서를 작성할 때도 실제 Agent 흐름을 따른다.

#### 17.4.0 직접 검수 누락 방지 강제 규칙

아래 규칙은 자동 검증 결과와 무관하게 반드시 수행한다.

- Codex는 정책서를 직접 작성하거나 보완할 때마다 자동 Inspector, 자동 Health Check, 테스트 결과만 근거로 완료 판단을 내려서는 안 된다.
- Codex는 매 정책서마다 직접 Inspector 역할과 직접 Health Check 역할을 별도로 수행해야 한다.
- 직접 Inspector는 범위, 액터, 유즈케이스, 상태, 프로세스, 기능, 정책, 정책 항목, 요구사항 재구성, 샘플 밀도, 렌더링 품질을 사람이 읽는 관점으로 확인한다.
- 직접 Health Check는 10개 영역과 필수 Gate를 자동 점수와 별개로 PASS/FAIL 판정한다.
- 자동 점수가 100점이어도 직접 검수에서 P1/P2 품질 문제가 보이면 완료로 처리하지 않는다.
- 다음 정책서로 넘어가기 전 `직접 Inspector 수행 여부`, `직접 Health Check 수행 여부`, `자동 결과와 다른 판단 여부`, `남은 P1/P2 이슈 여부`를 작업 기록 또는 응답에 남긴다.
- 이 직접 검수 기록이 없으면 해당 정책서는 아직 완료된 것으로 보지 않는다.

- 한 번에 전체 문서를 작성하지 않고 `Topic Learning → Blueprint/Core Design → 장별 작성 → 장별 검수 → 보완 → 다음 장` 순서로 진행한다.
- 간소화 버전 장 순서는 `overview → terms → actors → usecases → usecase_diagram → state → process → functions → policies → terms_refinement → final_check`를 따른다.
- Full 버전은 간소화 버전의 ID와 결정을 유지하고 `process_detail`, `function_detail`만 이어서 보강한다.
- Full 버전의 기능 상세 `처리 (상태-액션-결과)`는 각 줄을 반드시 `(상태) ... → (액션) ... → (결과) ...` 형식으로 작성한다.
- `(정상)`, `(분기)`, `(예외)`만 붙인 설명문은 Full 기능 상세 처리 로직으로 인정하지 않는다.
- 각 장 작성 후 JSON Critical Gate 수준의 구조 검증을 먼저 수행한다.
- 장별 검수 모드에서는 각 장마다 Inspector 관점의 구조·연결·품질 검수를 수행하고, 미통과 시 좁은 범위 보완 루프를 1회 이상 수행한다.
- 최종 검수만 모드에서도 JSON Critical Gate와 Final Inspector 관점 검수는 유지한다.
- 현재 장에서 해결할 수 없는 이전 장 계약 오류만 upstream 보완 대상으로 본다.
- 보완 루프가 반복되어 품질이 개선되지 않으면 최고 후보를 유지하고 남은 이슈를 Final Inspector 또는 다음 작업으로 넘긴다.
- 최종 완료 전에는 Schema/JSON Gate, Inspector, Requirement Trace, Health Check, Asset/Rendering Check를 모두 수행한다.
- Inspector가 통과해도 Health Check 필수 Gate가 실패하면 완료로 보지 않는다.
- 단, Health Check 실패가 trace 미연결 또는 검증기 오탐이면 문서 내용을 억지로 늘리지 않고 오탐 사유와 보완 계획을 기록한다.
- 자동 Inspector와 Health Check는 누락 탐지 도구이며, Codex 직접 품질 검수를 대체하지 않는다.
- Codex 직접 작성 큐에서는 자동 검증 통과 후에도 Topic/Core Design, 계위, 요구사항 재구성, 샘플 밀도, 최종 통합 편집 관점으로 수동 검수한다.
- Codex는 직접 작성 큐에서 별도 Inspector 역할을 수행해 장별 산출물의 범위, 계위, 연결, 정책 구체성, 요구사항 재구성 여부를 PASS/FAIL로 판정한다.
- Codex는 직접 작성 큐에서 별도 Health Check 역할을 수행해 10개 영역과 필수 Gate를 직접 확인하고, 자동 Health Check 점수와 다른 판단이 있으면 사유를 기록한다.
- 자동 Inspector 또는 Health Check가 100점이어도 Codex 직접 Inspector/Health Check에서 P1/P2 문제가 발견되면 완료로 보지 않는다.
- 직접 검수에서 상위 계약 문제가 발견되면 해당 장만 고치지 않고 액터, 유즈케이스, 상태, 프로세스, 기능, 정책, 정책 항목의 연쇄 영향 범위를 함께 점검한다.
- 자동 검증 결과와 Codex 직접 판단이 충돌하면 점수에 맞춰 문서를 늘리기 전에 실제 품질 문제, Trace 미연결, 검증기 오탐 중 무엇인지 먼저 판정한다.
- 직접 검수에서 발견한 재발 방지 규칙은 `reports/manual_authoring/manual_authoring_agent_insights.md`에 기록하고 다음 정책서 작성 전에 반영한다.

#### 17.4.1 인사이트 적용 등급

직접 작성 중 발견한 인사이트는 모두 같은 강도로 적용하지 않는다.

- `COMMON`: 모든 정책서에 적용 가능한 계위, 연결, 품질, 렌더링, 추적성 규칙이다. AGENTS.md, Validator, Inspector, Mock 규칙으로 승격할 수 있다.
- `TOPIC`: 특정 PM/주제에서 발견한 업무 경계 또는 판단축이다. 현재 주제, 요구사항, TK, 첨부자료가 같은 맥락일 때만 적용한다.
- `PATTERN`: 다른 주제에도 유사하게 참고할 수 있는 작성 패턴이다. 그대로 복사하지 말고 현재 주제의 요구사항 언어로 재구성한다.
- `DO_NOT_GENERALIZE`: 특정 주제 경계, 수치, 업무 이관 판단처럼 다른 정책서에 강제하면 오염되는 내용이다. 공통 prompt, mock 템플릿, validator로 승격하지 않는다.

적용 기준:

- `reports/manual_authoring/manual_authoring_agent_insights.md`의 PM-XX 섹션은 기본적으로 `TOPIC`으로 본다.
- 상태 전이 `criteria` 필수, 복합 액터명 금지, 순수 ID 연결, 요구사항 원문 복사 금지처럼 주제와 무관한 구조 문제만 `COMMON`으로 승격한다.
- 특정 주제에서 나온 업무 경계, 예를 들어 환불 산정 범위, 선물 발신/수신 구조, 청구·수납 상태 분리 등은 현재 주제 근거가 맞을 때만 적용한다.
- 인사이트를 적용하기 전 현재 주제의 상세 요구사항명/설명, TK, 첨부자료, 승인된 Blueprint와 충돌하지 않는지 확인한다.
- 주제별 인사이트를 근거 없이 일반화해 액터, 유즈케이스, 상태, 프로세스, 기능, 정책을 추가하지 않는다.

## 18. 문체 기준

정책서는 절제된 컨설팅 리포트 스타일로 작성한다.

- 쉬운 표현
- 바른 표현
- 짧은 문장
- 뚜렷한 판단 기준
- 불필요한 수식어 제거
- 구호성 표현 지양
- 정책서 본문과 표지의 채널 명칭은 `통합채널`로 표기하고, 앞에 `NC` 접두어를 붙이지 않는다.
- “고객이 무엇을 할 수 있는지”, “시스템이 무엇을 판단하는지”, “정책값이 무엇인지”가 드러나야 한다.

## 20. 금지 사항

아래는 하지 않는다.

- 템플릿 CSS를 임의로 크게 변경하지 않는다.
- 장별 작성 중 전체 문서를 한 번에 새로 쓰지 않는다.

- 상태 코드에 없는 상태명을 상태 전이표에서 사용하지 않는다.

## 21. 디자인팀/개발팀 export 변환기

`src/exporters/dev_format.py`는 작성된 정책서 HTML을 디자인팀(AI 목업 input)과 개발팀(ID 추적·N:N navigator) 양쪽에 친화적인 산출물 5종으로 변환한다. **LLM 호출 0**, stdlib만 사용하는 결정적 변환기.

본 정책서 작성 흐름(§1~§20, §17 Codex 작업 방식)과는 별도 트랙. 변환기는 정책서가 완성된 후 후처리 단계에서 실행한다.

### 21.1 기본 명령

```bash
python -m src.exporters.dev_format --input <html_path> [--output <dir>]
```

옵션:

- `--input` (필수): 정책서 HTML 경로 (예: `output/NC_회원가입탈퇴_정책서_Full_v1.0.html`)
- `--output` (선택): 출력 디렉토리. 생략 시 `output/exports/<slug>/`로 자동 결정.

### 21.2 산출물 (출력 디렉토리에 6종)

| 파일 | 용도 |
|---|---|
| `README.md` | 두 팀 담당자용 활용 가이드 (자동 생성) |
| `00_INDEX.md` | Claude Code/AI 진입점, UC 라우팅 표 |
| `usecase_*.md` × N | UC별 4단 nesting 분리 (Process > Function > Policy Group > Policy Item) |
| `mapping.csv` | 평탄 N:N 매핑 매트릭스 (Excel pivot 가능) |
| `entities.yaml` | 영문 키 + 양방향 + cross_refs + hierarchy 통합 데이터 모델 |
| `warnings.md` | 자동 검증 결과 (broken_refs, orphans, N:N 불일치, ID 형식 위반) |
| `diagrams/*.svg` | UC/State/BPMN 원본 SVG (Mermaid 본문 inline + SVG fallback) |

### 21.3 두 팀 활용처

- **디자인팀**: `usecase_*.md`를 AI 목업 도구(예: Claude Code)에 input으로 전달. 4단 nesting + Mermaid 다이어그램이 화면 단위 mockup 생성에 사용.
- **개발팀**: `00_INDEX.md` + `mapping.csv`로 ID·N:N 추적, `entities.yaml`로 데이터 모델 파악, `warnings.md`로 정합성 점검.

### 21.4 상세 가이드

- 사용법·산출물 구조·결정사항 10개·미해결 cosmetic·다음 작업 candidates는 **`docs/EXPORTERS.md`** 에 정리.
- 변환기 작업 시 (cosmetic 보완·기능 추가·테스트 확장 등) 반드시 `docs/EXPORTERS.md`를 먼저 읽고 결정사항과 충돌하지 않는지 확인한다.
- 회귀 방지 테스트는 `tests/test_dev_format.py`. 변환기 수정 시 `pytest tests/test_dev_format.py -v`로 회귀 확인.

### 21.5 변환기 작업 시 금지사항

- **LLM 호출 추가 금지**: 변환기는 결정적이어야 한다. `openai`, `anthropic` 등 외부 API import 금지.
- **stdlib 외 의존성 추가 금지**: `requirements.txt`에 새 패키지를 추가해 변환기에서 사용하지 않는다.
- **본체(`src/policy_agent.py` 외) 수정 금지**: 변환기 작업에서 정책서 작성 본체를 수정하지 않는다. 본체 변경은 별도 PR.
- **산출물 폴더 직접 수정 금지**: `output/exports/<slug>/` 안의 파일을 수동으로 편집하지 않는다. 변환기를 다시 실행해 재생성한다.
