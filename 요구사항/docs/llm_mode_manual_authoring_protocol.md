# LLM 모드 기반 수동 정책서 작성 지침

이 문서는 Codex가 에이전트 대신 직접 정책서를 작성할 때 따르는 작업 지침이다.
목표는 실제 LLM 모드의 작성 흐름을 사람이 검토 가능한 절차로 옮겨, 빠른 초안이 아니라 에이전트 파이프라인에 가까운 품질의 정책서를 만드는 것이다.

## 1. 적용 범위

- 대상: NC 정책서 34개 주제의 간소화/Full 정책서 작성, 보완, 재작성.
- 방식: Codex가 레포의 요구사항, 참고자료, TK 문서, 템플릿, 샘플, 기존 산출물을 읽고 직접 정책서 JSON/HTML 산출물을 작성한다.
- 원칙: 요구사항 문구를 그대로 붙여넣지 않고, 상세 요구사항명과 상세 요구사항 설명을 업무 흐름과 정책 판단 기준으로 재구성한다.
- 기준: 작성 결과는 업체가 상세 설계와 개발/QA 테스트 케이스로 전환할 수 있어야 한다.

## 3. 근거 우선순위

작성 중 근거가 충돌하면 아래 순서를 따른다.

1. 첨부 요구사항, 템플릿, 샘플, 사내자료, AGENTS.md.
2. TK 문서, 채널 방향성, 고객조사, VoC, IA 분석, 추가 PDF 추출 지식.
3. SKT 공식 서비스 안내, 약관, 고객지원 페이지.
4. 법령, 규제기관, 개인정보보호위원회, 방통위 등 컴플라이언스 자료.
5. 경쟁사, 벤치마킹, 일반 공개웹 자료.

공개웹 지식은 누락 후보와 검토 질문을 보강하는 용도로만 사용한다.
공개웹 지식이 첨부 요구사항이나 사내자료와 충돌하면 첨부자료를 우선하고, 충돌 내용은 문서 본문에 사실처럼 쓰지 않는다.

## 4. 작성 전 준비

정책서 하나를 시작할 때 아래 자료를 먼저 확인한다.

- 34개 기준 요구사항 파일: `input/requirements/20260509_요구사항 최종.xlsx`
- 요구사항 DB: `reports/evidence/requirements.db`
- 정책서 모듈 매핑표: 과거 전환 이력 확인용으로만 참고하고, 실제 작성 범위는 요구사항 DB의 `Depth 4`, `상세 요구사항명`, `상세 요구사항 설명`을 기준으로 한다.
- 템플릿: `input/templates/NC_정책서_간소화_템플릿_최종본.html`, `input/templates/NC_정책서_Full_템플릿_최종본.html`
- 샘플: `input/samples/`
- 작성 지향점: `input/references/34개_정책서_작성_지향점.md`
- TK 및 채널 방향성 PDF/MD: `input/references/`
- 정책 상세화 스타일 앵커: `input/references/스타일앵커_예전프로젝트_정책상세화패턴_20260510.md`
- 기존 산출물과 Health Check/Inspector 결과: `output/`, `reports/inspections/`, `output/quality/`

확인 후 아래 6개 질문에 먼저 답한다.

- 이 정책서가 해결해야 하는 고객/운영 업무 문제는 무엇인가?
- 이 정책서가 포함해야 하는 업무 범위와 제외해야 하는 범위는 무엇인가?
- 고객이 직접 완료해야 하는 일과 시스템/BSS가 판단해야 하는 일은 무엇인가?
- 상태, 프로세스, 기능, 정책으로 반드시 이어져야 하는 요구사항은 무엇인가?
- 샘플 문서 수준에 맞추기 위해 표 밀도, 정책 항목 수, 문장 길이를 어떻게 잡을 것인가?
- 정책 상세화 스타일 앵커 기준으로 유형, 가능/불가, 채널/주체, 시점, 권한, 사용/인증 여부, 자동 판정/수동 조정, 이력 중 어떤 판단축을 적용해야 하는가?

정책 상세화 스타일 앵커는 작성, 검증, 보완 단계 모두에서 참고한다.
단, 예전 프로젝트 문서는 정책 사실이 아니라 상세화 방식의 예시이므로 현재 요구사항, 첨부자료, TK, 샘플/템플릿과 연결되지 않는 값·상품명·채널명은 쓰지 않는다.

## 5. 에이전트 방식 수동 작성 흐름

실제 LLM 모드는 아래 순서로 움직인다.
Codex 수동 작성도 같은 순서로 진행한다.

수동 작성이라고 해서 한 번에 완성본을 쓰지 않는다.
반드시 에이전트 실행과 같은 단위로 `학습 → Blueprint/Core Design → 장별 작성 → 장별 검수 → 보완 → 다음 장` 순서를 따른다.
빠른 초안 생성은 허용하지 않는다.

### 5.1 Topic Learning

목적은 문장 작성이 아니라 작성 기준선을 만드는 것이다.

산출:

- 주제 정의.
- 고객 과업.
- 핵심 정책 질문.
- 주요 요구사항 묶음.
- TK/VoC/채널 방향성에서 도출한 작성 관점.
- 근거 부족 또는 판단 보류 항목.

작성 기준:

- 요구사항 ID 자체보다 상세 요구사항명과 상세 요구사항 설명을 먼저 읽는다.
- 상세 요구사항을 그대로 복사하지 않고, 업무 목적, 조건, 예외, 처리 결과로 재구성한다.
- 우선순위 필드는 정제되지 않은 값일 수 있으므로 단독 판단 기준으로 삼지 않는다.
- 상세 요구사항명이 대체안, 예상 금액, 신청 경로, 약관 버전, 노출 상태, 삭제 가능 여부처럼 구체적 판단축을 포함하면 해당 축이 정책 항목명 또는 기능명에 드러나야 한다.
- 같은 상위 정책 아래에 있어도 고객 경로, 처리 시점, 계산 방식, 외부 회신, 복구 가능성이 다르면 하나의 일반 정책으로 합치지 않는다.
- 즉시 처리/예약 처리, 로그인/비회원 경로, 원클릭/상담 전환, 예상 금액/위약금/결합 영향처럼 QA 케이스가 달라지는 축은 정책 항목 또는 기능 세부 구성으로 분리한다.
- 요구사항 건수가 바뀌면 Topic Learning, 문서 히스토리, `requirements_mapping_summary`, 최종 점검 기준, Requirement Trace, HTML/BPMN을 모두 함께 갱신한다.
- 작성 전에는 기존 Trace 파일만 믿지 않고 최신 `reports/evidence/requirements.db`에서 해당 `Depth 4` 요구사항 수를 다시 조회한다.
- 최신 DB에서 새로 확인된 상세 요구사항명은 원문 복사 대상이 아니라 누락 방지 신호로 보고, 프로세스명, 기능명, 정책 항목명 중 하나에 정책 판단 언어로 재구성해 드러낸다.

### 5.2 Blueprint / Core Design

목적은 장별 작성 전에 전체 업무 구조를 먼저 잡는 것이다.

산출:

- document_strategy: 이 정책서가 해결해야 하는 업무 문제.
- core_customer_jobs: 고객이 직접 처리·확인·복구해야 하는 일.
- core_policy_questions: 누가, 언제, 몇 번, 어떤 조건, 실패 시 복구, 이력, BSS 상태 변경 질문.
- hierarchy_map: 액터 → 유즈케이스 → 상태/상태전이 → 프로세스 → 기능 → 정책 → 정책 항목.
- bss_touchpoints: BSS가 판단·저장·상태 변경·이력 저장·결과 회신해야 하는 지점.
- must_include: 이 주제에서 반드시 포함할 구조.
- must_not: 화면 설명, API/DB 상세, 근거 없는 일반론, 빈 정책.

Core Design 원칙:

- 유즈케이스는 고객이나 운영자의 완결 업무 목적이다.
- 프로세스는 유즈케이스를 완료하는 절차다.
- 단순 유즈케이스가 프로세스 1개로만 끝난다면 유즈케이스가 너무 작거나 프로세스 분해가 부족한지 재검토한다.
- 상태 전이 이벤트에는 유즈케이스나 업무 사건을 쓴다.
- 상태 전이의 처리 기준 및 후속 처리는 내부 spec의 `criteria` 필드에 반드시 작성한다. `rule`, `description` 등 다른 필드명만 사용하면 HTML 상태 전이표가 빈 칸으로 렌더링된다.
- 상태 전이표에 별도 `관련 유즈케이스` 컬럼을 만들지 않는다.
- `process_target=N`인 시스템/BSS/외부 연계/분석 유즈케이스라도 고객 노출, 처리 가능 여부, 게시 가능 여부, 상태 변경, BSS 반영, 외부 회신, 실패·복구 경로에 영향을 주면 상태 전이의 내부 `usecase_ids`에 함께 연결한다.
- 시스템 보조 유즈케이스는 별도 프로세스로 승격하지 않되, 고객/운영자 프로세스의 기능·정책·상태 전이 안에서 판단 기준과 결과 회신 기준을 남긴다.
- 미구독, 로그아웃, 빈 화면, 퀵메뉴, 미사용 상태처럼 고객 상태나 화면 조건에 가까운 요구는 액터나 별도 유즈케이스로 늘리지 않는다. 진입 프로세스, 상태, 기능 세부 구성, 대체 행동 정책으로 내린다.
- 내부 데이터의 연결 필드는 순수 ID만 저장한다.
- 화면/HTML 표시에는 렌더링 단계에서 `ID 명칭` 형태로 보여준다.
- 상태 전이는 화면에는 상태명을 표시하되, 검증용 매트릭스에는 상태 ID와 상태명을 함께 유지한다.
- 프로세스, 기능, 정책 후보는 각자 따로 만들지 않고 하나의 Core Design Map에서 먼저 연결한다.

### 5.3 Context Pack 구성

각 장을 쓰기 전, 해당 장에 필요한 근거만 다시 추린다.

Context Pack에는 아래를 포함한다.

- writing_goal: 이번 장이 문서 전체에서 해결해야 할 일.
- evidence_summary: 요구사항/TK/샘플에서 확인한 핵심 근거.
- policy_questions: 이번 장에서 답해야 할 판단 질문.
- style_anchor: 정책 상세화 스타일 앵커에서 이번 장에 필요한 판단축. 단, 예전 문서의 실제 정책값은 포함하지 않는다.
- required_outputs: 이번 장에서 반드시 만들어야 하는 산출물.
- previous_decisions: 이전 장에서 확정한 ID, 명칭, 상태, 연결.
- evidence_gap: 근거가 부족해 TBD 또는 검토 필요로 남길 항목.

Context Pack에 넣지 않는다.

- 주제와 무관한 공개웹 일반론.
- API 필드, DB 컬럼, 화면 구성 상세.
- 다른 정책서가 담당할 범위의 세부 기능.
- 예전 프로젝트 문서의 수치, 상품명, 채널별 O/X, 조직명, 화면명 같은 과거 정책값.

### 5.4 장별 Agent 실행 루프

각 장은 아래 순서로 처리한다.

1. Stage start: 이전 장까지 확정된 ID, 명칭, 상태, 연결을 확인한다.
2. Chapter write: 현재 장만 작성하되 Blueprint/Core Design과 이전 장 결정을 따른다.
3. JSON Critical Gate: 필수 키, 필수 배열, 표 컬럼, ID 형식, 연결 필드 타입을 먼저 확인한다.
4. Stage Inspector: 장별 전문 검수 기준으로 구조, 연결, 품질, 범위를 확인한다.
5. Repair loop: 미통과 시 P1/P2/P3 finding을 좁은 범위로 반영한다.
6. Upstream backtrack: 현재 장에서 해결할 수 없고 이전 장 계약이 틀린 경우에만 이전 장을 1회 보완한다.
7. Handoff: 반복 보완에도 점수가 개선되지 않으면 최고 후보를 유지하고 남은 이슈를 Final Inspector로 넘긴다.
8. Checkpoint: 통과 또는 handoff된 장은 중간 HTML/JSON과 검사 결과를 남긴다.

장별 검수 모드에서는 모든 장이 위 루프를 거친다.
최종 검수만 모드에서는 장별 Inspector를 생략하되, JSON Critical Gate와 최종 Inspector는 유지한다.
검수 없음 모드에서도 JSON Critical Gate는 생략하지 않는다.

### 5.5 내부 데이터와 화면 표시 분리

작성 데이터는 기계 검증이 가능해야 한다.
따라서 연결 필드에는 표시용 문장을 넣지 않는다.

내부 데이터 기준:

- `usecase_id`, `process_id`, `policy_id`는 순수 ID만 저장한다.
- `related_functions`는 `FN-...` ID 배열만 저장한다.
- `related_policies`는 `PG-...` ID 배열만 저장한다.
- 상태 전이 검증용 데이터는 상태 ID 기준으로 매핑 가능해야 한다.
- 요구사항 추적 데이터는 상세 요구사항명과 상세 요구사항 설명을 기준으로 연결한다.

화면/HTML 표시 기준:

- 프로세스 목록의 관련 기능/정책은 `FN-... 기능명`, `PG-... 정책명`처럼 표시한다.
- 기능 상세의 관련 프로세스/정책도 ID와 명칭을 함께 보여준다.
- 상태 전이표에는 `관련 유즈케이스` 컬럼을 추가하지 않는다.
- 상태 전이의 `전이 이벤트` 컬럼에는 유즈케이스 또는 업무 사건을 쓴다.
- HTML 표시를 위해 ID와 명칭을 합친 문자열을 내부 연결 필드에 저장하지 않는다.

이 규칙을 어기면 화면상으로는 좋아 보여도 자동 검증, Health Check, Full 버전 확장, 후속 보완에서 연결이 끊긴다.

## 6. 장별 작성 지침

### 6.1 개요

작성 항목:

- 범위.
- 설계 원칙.

검사 기준:

- 범위 문장은 불릿으로 작성한다.
- 포함 범위와 제외 범위가 명확해야 한다.
- 설계 원칙은 4~6개로 제한한다.
- 설계 원칙은 기능과 정책으로 이어질 수 있는 판단 기준이어야 한다.

### 6.2 주요 용어

작성 항목:

- 상태, 권한, 인증, 약관, 상품, 요금, 혜택, 이력, 외부기관, 정책 판단값에 쓰이는 용어.

검사 기준:

- 용어 ID는 `TM-{업무코드}-001` 형식으로 반드시 부여한다.
- 일반 명사는 제외한다.
- 설명은 업무상 판단 기준을 포함해야 한다.
- 유사 용어와의 차이가 드러나야 한다.

### 6.3 액터

작성 항목:

- 고객, 법정대리인, 대리인, 운영자, 핵심 처리 시스템, 외부 승인/인증 기관, 연계 시스템.

검사 기준:

- 로그인 고객, 비로그인 고객, 정상 고객, 제한 고객, VIP 고객은 액터가 아니다.
- 고객 상태와 권한 조건은 상태/정책으로 내린다.
- 시스템 액터는 과도하게 쪼개지지 않게 한다.
- 시스템 액터도 `BSS/멤버십 연계 시스템`, `제휴처/외부 사용처`처럼 여러 책임 주체를 `/`, `·`, `및`으로 묶지 않는다.
- 통합 시스템 책임은 `연계 시스템`, `멤버십 연계 시스템`, `외부 사용처`처럼 하나의 책임명으로 쓰고, 세부 시스템 차이는 설명·기능·정책에서 다룬다.

### 6.4 유즈케이스

작성 항목:

- 고객/운영자가 완료하려는 상위 업무 목적.
- 시스템 보조 유즈케이스.
- 프로세스 정의 대상 Y/N.

검사 기준:

- 사람 액터의 직접 수행 업무는 원칙적으로 Y.
- 시스템 검증/저장/조회/결과 생성은 원칙적으로 N.
- 인증, 입력, 확인, 버튼 클릭 같은 절차 단계는 유즈케이스가 아니라 프로세스로 내린다.
- 시스템 유즈케이스가 N이더라도 고객 표시 상태, 업무 제한, 게시·노출 결과, BSS/외부 회신, 실패·복구 경로를 바꾸면 상태 전이와 정책 항목에 반드시 연결한다.

### 6.5 유즈케이스 다이어그램

작성 항목:

- 액터와 유즈케이스 관계.
- include 관계는 반복 공통 처리만 사용.

검사 기준:

- 화면 이동, 버튼 클릭, 팝업 노출을 넣지 않는다.
- 고객 중심 유즈케이스를 중앙에 둔다.
- 시스템 액터는 보조 처리로 표현한다.

### 6.6 상태

작성 항목:

- 상태 코드 목록.
- 상태 전이 기준.
- 상태 ID-상태명 검증 매트릭스.

검사 기준:

- 상태는 고객 노출, 기능 허용, 정책 판단, 후속 처리의 기준이어야 한다.
- 상태 전이표의 현재 상태와 다음 상태는 상태 코드 목록에 존재해야 한다.
- 전이 이벤트에는 유즈케이스 또는 업무 사건을 쓴다.
- 각 상태 전이는 `criteria`에 처리 기준과 후속 처리를 1문장 이상 작성한다.
- 완료, 실패, 대기, 제한, 만료, 운영 검토 필요 흐름을 함께 포함한다.
- 전이 이벤트를 보조하기 위해 별도 `관련 유즈케이스` 컬럼을 추가하지 않는다.
- 상태 전이가 유즈케이스와 연결되어야 할 때는 내부 검증 매트릭스에서 usecase ID를 관리한다.
- BSS 판정, 분석 결과, 외부 연계 회신, 채널별 반영 결과처럼 시스템이 만든 결과가 고객에게 보이는 상태나 후속 행동을 바꾸면 해당 시스템 유즈케이스 ID를 상태 전이 내부 연결에 포함한다.
- 완료 상태는 버튼 클릭이나 요청 접수 시점이 아니라 BSS/외부/채널 반영 결과가 성공·실패·지연으로 회신되고 이력이 저장된 시점으로 정의한다.

### 6.7 프로세스

작성 항목:

- 프로세스 목록.
- 전체 업무 흐름.

검사 기준:

- 고객 시작점부터 완료까지 순서대로 작성한다.
- 하나의 유즈케이스가 반드시 하나의 프로세스만 갖는 구조로 만들지 않는다.
- 내부 `related_functions`, `related_policies`에는 실제 기능/정책 ID만 저장한다.
- HTML 표에는 실제 기능/정책 ID와 명칭을 함께 표시한다.
- 프로세스의 관련 정책명은 정책 목록의 정책 그룹 ID와 반드시 연결되어야 한다.
- 기능의 `process_ids`와 프로세스의 `related_functions`는 양방향으로 일치해야 한다.
- 정책 목록에 정의된 정책과 기능 목록에 정의된 기능은 최소 1개 이상의 프로세스에서 참조되어야 한다.
- 프로세스에서 실행되는 정책은 프로세스 `related_policies`와 해당 기능의 `related_policies` 양쪽에서 추적 가능해야 한다.
- 시스템 내부 조회/판정/저장은 별도 프로세스로 과도하게 분리하지 않고 관련 기능으로 연결한다.

### 6.8 기능

작성 항목:

- 기능 목록.
- 세부 기능 구성.

검사 기준:

- 기능은 화면 단위가 아니라 처리 역량이다.
- 조회, 검증, 입력, 산정, 저장, 알림, 연동, 결과 확인 중 2~5개 하위 처리를 묶는다.
- 세부 기능 구성은 기능 아래의 짧은 하위 처리명으로만 작성한다.
- 세부 기능 구성에는 `...한다`, `...할 수 있다`, `...해야 한다` 형태의 설명문·정책문을 쓰지 않는다.
- 기능 설명과 세부 기능 구성이 여러 기능에서 같은 문장 구조로 반복되면 기능 단위가 약하다는 신호다. 각 기능은 체크리스트, 딥링크, SLA, 손실 가치, 오남용 차단, 이력 저장처럼 서로 다른 산출물을 드러내야 한다.
- 좋은 예: `조건 정보 조합`, `권한 상태 검증`, `결과 안내 구성`, `이력 저장`.
- 나쁜 예: `고객 상태와 BSS 판정 결과에 따라 진행 가능 여부를 구분한다.`
- 기능 상세에는 관련 프로세스와 관련 정책 ID를 연결한다.
- 내부 연결값은 ID만 저장하고, 표시 문구는 렌더링 단계에서 붙인다.
- 정책값을 기능 설명에 길게 쓰지 않는다.

### 6.9 정책

작성 항목:

- 정책 목록.
- 정책 상세.

검사 기준:

- 정책은 기능 설명이 아니라 판단 기준이다.
- 정책 항목 하나에는 하나의 판단 기준만 담는다.
- 정책 항목에는 값, 조건, 횟수, 시간, 상태, 예외, 고지, 이력, 처리 주체 중 하나 이상이 있어야 한다.
- 정책 상세 작성 전 유형 구분, 가능/불가, 채널/주체, 시점, 권한/명의/관계, 사용/인증 여부, 자동 판정/수동 조정, 외부 회신, 이력 저장 중 현재 주제에 필요한 판단축을 선택한다.
- 가능 기준만 쓰지 않고 제한 기준, 예외 불가 기준, 불가 시 고객 안내 또는 대체 경로를 함께 확인한다.
- 자동 판정이 기본인 항목은 자동 판정 입력 조건, 수동 조정 가능 주체, 수동 조정 이력 저장 기준을 분리한다.
- 줄바꿈된 정책 상세 문장은 각 줄을 `-` 불릿으로 통일한다.
- "검토 필요", "추후 협의", "시스템에서 처리", "정책에 따라 처리한다"는 단독 정책 내용으로 쓰지 않는다.
- 정책 그룹 ID는 프로세스와 기능에서 참조될 수 있도록 안정적으로 유지한다.
- 정책 상세 ID는 정책 그룹 ID를 접두로 삼아 추적 가능하게 작성한다.
- 정책 목록의 `items`는 문자열이 아니라 항상 `{id, name}` 객체로 작성하고, `policy_details`의 `id`, `name`, `policy_id`와 연결한다.
- Trace 보강을 위해 요구사항 원문을 정책 항목명으로 그대로 복사하지 않는다. 대신 요구사항의 판단축을 짧은 정책 항목명으로 재구성하고, `content`에 허용·제한·고지·이력·BSS/외부 회신 기준을 작성한다.

### 6.10 용어 재정렬

모든 장 작성 후 용어를 다시 본다.

검사 기준:

- 정책/상태/프로세스에서 실제로 쓰인 용어가 용어 장에 빠져 있지 않은지 확인한다.
- 모든 용어에 `TM-{업무코드}-NNN` ID가 있는지 확인한다.
- 더 이상 쓰지 않는 용어는 제거한다.
- 같은 개념이 다른 이름으로 반복되지 않게 한다.

### 6.11 최종 점검

작성 항목:

- 주제별 최종 점검 기준.

검사 기준:

- 점검 결과 장을 별도로 추가하지 않는다.
- 최종 점검 기준은 문서 본문을 보완하기 위한 체크리스트로만 사용한다.

## 7. Full 버전 확장 지침

간소화 버전을 Full로 확장할 때는 새로 쓰지 않는다.
간소화 버전의 ID, 명칭, 상태, 프로세스, 기능, 정책을 유지하고 필요한 장만 이어서 작성한다.

추가 작성:

- 프로세스 상세.
- 기능 상세.

Full 확장 기준:

- 프로세스 상세는 진입 조건, 종료 조건, 선행/후행 관계, 관련 기능, 관련 정책을 보강한다.
- 기능 상세는 입력 정보, 처리 기준, 출력 정보, 실패/예외, 이력/알림, 관련 정책을 보강한다.
- 프로세스 상세와 기능 상세는 모든 행에 같은 문장을 반복하지 않는다.
- 진입 조건, 종료 조건, 처리 기준에는 해당 프로세스명, 유즈케이스명, 관련 정책명을 반영해 업무별 차이가 드러나야 한다.
- 세부 기능 구성은 단어 조각이 아니라 실제 하위 처리명으로 작성한다.
- API 필드, DB 컬럼, 화면 상세는 쓰지 않는다.
- 이미 간소화 버전에서 정의한 정책값과 충돌하면 간소화 버전의 정책 기준을 우선한다.

## 8. Inspector 방식 수동 검수

각 장 작성 후 아래 순서로 검수한다.

1. 구조 검수: 필수 키, 표 컬럼, ID 체계, 템플릿 구조.
2. 연결 검수: 액터-유즈케이스, 유즈케이스-상태, 유즈케이스-프로세스, 프로세스-기능, 프로세스-정책, 정책 목록-정책 상세.
3. 품질 검수: 일반론, 빈 정책, 화면 설명 중심, 요구사항 누락.
4. 범위 검수: 현재 주제 밖 보안/운영/UX 일반론이 들어가지 않았는지 확인.
5. 근거 검수: 첨부자료와 TK/공개웹 지식이 충돌하지 않는지 확인.
6. 정책 상세화 스타일 검수: 유형, 가능/불가, 권한, 상태·시점, 사용 여부, 자동/수동 조정, BSS·외부 회신, 고지·이력 중 현재 주제에 필요한 판단축이 정책 항목에 실제 기준으로 반영됐는지 확인.

검수 결과는 아래 3가지 유형으로만 보완 항목을 만든다.

- 어디에 있는 어떤 내용을 어떻게 바꿀지.
- 어디에 어떤 내용을 추가할지.
- 어디에 있는 어떤 내용을 삭제할지.

우선순위:

- P1: 구조 연결이 끊기거나 개발/QA 착수가 어려운 문제.
- P2: 정책값, 예외, 이력, BSS 기준이 부족한 문제.
- P3: 용어, 문체, 표기, 가독성 문제.

## 9. 보완 루프 지침

보완은 전체 재작성보다 좁은 범위 수정이 기본이다.

1. Patch: 특정 행, 특정 정책 항목, 특정 상태 전이만 수정.
2. Scoped section revision: 같은 원인의 인접 항목을 함께 정렬.
3. Scoped full revision: 담당 장 전체의 계층과 입자도가 낮은 경우에만 사용.
4. Blueprint realign revision: 장이 Blueprint와 어긋났을 때 사용.

보완 금지:

- Inspector가 지적하지 않은 항목을 대량 수정.
- 이전 장에서 확정한 ID/명칭을 임의 변경.
- 내부 연결 필드에 표시용 `ID 명칭` 문자열을 넣는 수정.
- 정책서 범위 밖의 일반론 추가.
- 요구사항 원문을 그대로 붙여넣어 품질을 높인 것처럼 보이게 하기.

보완 완료 조건:

- 수정 대상 장의 Inspector finding이 실제로 줄어야 한다.
- 같은 finding이 반복되면 동일 보완을 반복하지 않고 원인을 재분류한다.
- 보완 후 점수가 떨어지면 이전 최고 후보를 유지하고 변경 범위를 축소한다.
- 장별 보완으로 해결할 수 없는 문제는 Final Inspector 또는 upstream backtrack으로 인계한다.
- 정책 상세 보완은 스타일 앵커의 판단축 중 현재 요구사항과 연결되는 축만 보강한다. 예전 프로젝트의 실제 수치, 과거 채널명, O/X 판정 방식은 보완 근거로 복사하지 않는다.

## 10. Health Check 기준

최종 문서는 Health Check 관점으로 다시 본다.

- 정책 정의 범위 및 채널 통합 방향성.
- CX 혁신성.
- 프로세스 효율성 및 PI 지향성.
- 업무 구조 정합성.
- 상태/결과 명확성.
- 정책 구체성.
- 예외/실패 대응.
- 시스템/BSS/통합 리스크.
- 데이터 입출력 및 처리 기준.
- 요구사항·개발·QA 추적성.

필수 Gate:

- 유즈케이스가 고객의 완결 업무 단위인가.
- 프로세스가 유즈케이스를 완성하는 절차인가.
- 정책이 프로세스/기능의 판단 기준으로 연결되는가.
- 정책 항목에 실제 값, 조건, 횟수, 시간, 상태가 있는가.
- BSS가 판단, 저장, 상태 변경해야 하는 항목이 정의되어 있는가.
- 기준 요구사항이 프로세스, 기능, 정책 중 하나 이상에 매핑되었는가.

### 10.1 최종 검증 실행 순서

정책서 1건을 완료 처리하기 전에는 아래 검증을 모두 수행한다.
어느 하나라도 실패하면 완료로 보지 않고, 실패 원인을 문서 품질 문제와 검증기 오탐으로 분리한다.

1. Schema / JSON Gate
   - 대상: `*_manual_spec.json`
   - 확인: 필수 키, 빈 값, ID 형식, 상태 전이 참조, 프로세스-기능-정책 연결, 정책 목록-정책 상세 연결.
   - 실패 시: HTML을 직접 고치지 않고 spec의 원천 데이터를 수정한 뒤 다시 렌더링한다.

2. Stage / Final Inspector
   - 대상: 최신 HTML과 spec.
   - 확인: 장별 구조, 템플릿 준수, 정책 구체성, 범위 이탈, 개발/QA 가능성.
   - 실패 시: finding을 `변경 / 추가 / 삭제` 중 하나로 분류하고, 해당 장 또는 해당 항목만 좁게 보완한다.

3. Requirement Trace
   - 대상: 요구사항 DB와 최신 spec.
   - 확인: 해당 주제의 상세 요구사항명과 상세 요구사항 설명이 유즈케이스, 상태, 프로세스, 기능, 정책, 정책 항목 중 하나 이상으로 재구성되었는가.
   - 통과 기준: 자동 매칭 또는 수동 검토 결과 기준 100% 커버리지.
   - 실패 시: 요구사항 원문을 본문에 붙여넣지 말고 누락된 업무 목적, 조건, 예외, 이력, BSS 기준을 해당 장에 반영한다.

4. Health Check
   - 대상: 최신 HTML, spec, requirement trace.
   - 확인: 10개 영역 점수, 필수 Gate, FAIL 항목.
   - 통과 기준: 총점 80점 이상, 필수 Gate 전체 통과, P1 FAIL 0건.
   - 실패 시: Health Check 결과를 그대로 수용하지 않고 아래 10.2 기준으로 분류한다.

5. Asset / Rendering Check
   - 대상: HTML, spec, BPMN, 다이어그램, 문서 작업실 미리보기.
   - 확인: HTML 파일 존재, spec 파일 존재, BPMN XML 생성, 다이어그램 가독성, 표 컬럼 폭, 빈 셀, 문서 히스토리.
   - 실패 시: 렌더러 또는 spec 원천 값을 수정하고 `scripts/ensure_manual_authoring_assets.py --check`로 큐 무결성을 확인한다.

6. Regression Test
   - 대상: 변경한 코드와 관련 테스트.
   - 최소 실행: 수동 작성 자산, 렌더러, Validator, Health Check, Inspector 관련 테스트.
   - 실패 시: 정책서 작성을 이어가지 않고 테스트 원인을 먼저 해소한다.

### 10.2 Codex 직접 품질 검수 Gate

자동 Inspector와 Health Check는 누락 탐지 도구이며, 최종 품질 판단을 대체하지 않는다.
Codex가 Agent 대신 직접 작성하는 큐 작업에서는 자동 검증 통과 후에도 반드시 직접 품질 검수 Gate를 수행한다.

자동 도구가 주로 맡는 역할:

- 필수 키, 빈 값, ID 불일치, 상태명 불일치, 표 컬럼 누락을 찾는다.
- 프로세스-기능-정책 연결, 정책 목록-정책 상세 연결, BPMN/XML 생성 여부를 확인한다.
- 요구사항 Trace 후보, Health Check FAIL 항목, Inspector finding을 1차로 제시한다.

Codex가 직접 판단해야 하는 역할:

- Topic/Core Design이 이 정책서의 고객 과업과 업무 경계를 제대로 잡았는지 확인한다.
- 액터 → 유즈케이스 → 상태 → 프로세스 → 기능 → 정책 → 정책 항목의 계위가 자연스럽게 이어지는지 검수한다.
- 상세 요구사항명과 상세 요구사항 설명이 원문 복사가 아니라 업무 흐름과 정책 판단 기준으로 재구성되었는지 확인한다.
- 샘플 정책서와 비교해 표 밀도, 정책 항목 구체성, 문장 길이, 다이어그램 가독성이 부족하지 않은지 본다.
- Codex가 직접 Inspector 역할을 맡아 각 장의 범위, 계위, 연결, 정책 구체성, 요구사항 재구성 여부를 PASS/FAIL로 판정한다.
- Codex가 직접 Health Check 역할을 맡아 10개 영역과 필수 Gate를 확인하고, 자동 Health Check 점수와 다른 판단이 있으면 이유를 남긴다.
- Inspector/Health Check 결과가 실제 품질 문제인지, Trace 미연결인지, 검증기 오탐인지 판별한다.
- 장별로는 맞지만 전체 문서가 따로 노는 경우 용어, 상태, 프로세스, 정책명을 통합 편집한다.
- 다음 문서와 Agent 개선에 반영할 재발 방지 인사이트를 `reports/manual_authoring/manual_authoring_agent_insights.md`에 기록한다.

완료 판정 기준:

- 자동 검증이 모두 통과해도 Codex 직접 품질 검수에서 구조, 범위, 계위, 요구사항 재구성, 샘플 밀도 중 하나라도 P1/P2 수준으로 미흡하면 완료하지 않는다.
- 자동 Inspector 또는 Health Check가 100점이어도 Codex 직접 Inspector/Health Check에서 P1/P2 문제가 발견되면 완료하지 않는다.
- 직접 검수에서 발견한 문제는 해당 장만 좁게 수정하되, 액터나 유즈케이스처럼 상위 계약이 바뀌면 프로세스, 기능, 정책, 상태 전이까지 연쇄 영향 범위를 함께 점검한다.
- 자동 도구와 Codex 판단이 충돌하면 즉시 자동 점수에 맞춰 문서를 늘리지 않고, 실제 문서 품질 문제인지 먼저 판정한다.

### 10.3 Health Check 실패 판정과 보완 기준

Health Check는 사후 품질 점검 도구이며, Inspector를 대체하지 않는다.
Health Check FAIL은 아래 3가지 중 하나로 분류한다.

1. 실제 문서 품질 문제
   - 예: 정책 항목에 값, 조건, 횟수, 시간, 상태가 없다.
   - 예: 상태 전이 기준의 처리 기준 및 후속 처리가 비어 있다.
   - 예: 프로세스 목록의 관련 기능 또는 관련 정책이 실제 ID와 연결되지 않는다.
   - 처리: spec을 보완하고 HTML을 재렌더링한 뒤 Inspector, Requirement Trace, Health Check를 다시 실행한다.

2. Trace / 보조 산출물 미연결 문제
   - 예: requirement trace는 100%인데 Health Check가 HTML 본문만 보고 `요구사항 매핑 없음`으로 판단한다.
   - 예: spec에는 `state_transitions.usecase_ids`가 있지만 HTML에는 추적성 ID가 노출되지 않아 연결성이 낮게 판단된다.
   - 처리: 문서 내용을 억지로 늘리지 않는다. Health Check 입력에 spec/trace 신호를 연결하거나 Evaluator 기준을 보완한다.

3. 검증기 오탐
   - 예: `시스템 기준에 따름 같은 빈 문장으로 끝나지 않아야 한다`처럼 금지 표현 설명문을 약한 정책으로 오인한다.
   - 예: 최종 점검 기준의 경고 문구를 본문 정책 문구로 오인한다.
   - 예: 전시/관리, 추천, AI 검색처럼 인증·결제가 핵심이 아닌 주제에서 BSS·분석 응답 지연, 딥링크 실패, 게시 반영 실패, 캐시 무효화 실패, 중복 요청 멱등성 같은 주제 고유 실패 기준을 반영했는데도 범용 인증/결제 실패만 없다고 FAIL 처리한다.
   - 처리: 문서 본문을 불필요하게 수정하지 않고 Health Check 신호 추출 범위 또는 bad phrase 판정 위치를 보완한다.

보완 우선순위는 아래와 같이 정한다.

- P1: 필수 Gate 미통과, 요구사항 커버리지 미달, 상태/프로세스/정책 연결 끊김, 빈 필수 셀.
- P2: 정책값, 예외, BSS 반영, 이력, 고객 안내, 재시도·복구 기준 부족.
- P3: Health Check 오탐, 표현 가독성, 다이어그램/표 렌더링, 문서 히스토리/메타 보완.

### 10.4 완료 판정 기록

정책서 완료 또는 다음 정책서 이동 전에는 `reports/manual_authoring/manual_authoring_queue.json`의 해당 항목 또는 작업 보고에 아래 결과를 남긴다.

- `schema`: pass/fail.
- `inspector`: 점수, 판정, finding 수.
- `requirement_trace`: 전체 요구사항 수, 매칭 수, 수동 검토 필요 수.
- `health_check`: 점수, 판정, 필수 Gate 통과 여부, FAIL 요약.
- `asset_check`: HTML/spec/BPMN 존재 여부와 XML 오류 여부.
- `codex_direct_review`: 계위, 요구사항 재구성, 샘플 밀도, 오탐 판별, 최종 통합 편집 결과.
- `known_exceptions`: 검증기 오탐 또는 의도적으로 보류한 이슈.

검증 결과가 서로 충돌할 때는 더 엄격한 결과를 기준으로 멈춘다.
예를 들어 Inspector가 100점이어도 Health Check 필수 Gate가 실패하면 완료하지 않는다.
단, Health Check 실패가 명확한 오탐이면 오탐 사유와 보완 계획을 기록한 뒤 다음 단계로 진행할 수 있다.

## 11. 수동 작성 작업 단위

정책서 하나를 작성할 때 Codex는 아래 단위로 사용자에게 보고한다.

1. 분석: 확인한 요구사항 묶음, TK/참고자료, 샘플 기준, 작성 지향점.
2. Blueprint: 액터, 유즈케이스, 상태, 프로세스, 기능, 정책 후보.
3. 작성: 장별 산출물 작성 및 HTML/JSON 반영.
4. 검수: 구조 연결, 정책 구체성, 요구사항 커버리지, Health Check 관점.
5. 보완: P1/P2/P3를 좁은 범위로 수정.
6. 산출물: 파일 경로, 버전, 다음 작성 대상.

## 12. 최종 산출물 기준

정책서 산출물은 아래 조건을 만족해야 한다.

- 템플릿 HTML 구조와 CSS를 유지한다.
- 문장은 짧고 명확하게 쓴다.
- 마침표 뒤에는 `<br/>` 또는 표 내 줄바꿈을 적용한다.
- 정책 상세의 줄바꿈은 `-` 불릿을 유지한다.
- 문서 히스토리에 작성/보완 내용을 남긴다.
- 버전은 서버 또는 로컬의 기존 최신 버전을 기준으로 증가시킨다.
- 작성 완료 여부는 사용자가 결정한다.

## 13. 직접 작성 시 최종 체크리스트

작성 완료 전 아래 질문에 모두 답한다.

- 이 주제의 상세 요구사항이 빠짐없이 검토되었는가?
- 요구사항이 유즈케이스, 프로세스, 기능, 정책 중 하나 이상으로 재구성되었는가?
- 요구사항 커버리지가 요구사항 ID가 아니라 상세 요구사항명/상세 요구사항 설명 기준으로 확인되었는가?
- 구체적 판단축이 넓은 정책명 아래에 숨지 않고 정책 항목명 또는 기능 세부 구성에서 확인되는가?
- 유즈케이스가 고객 완결 업무 단위인가?
- `process_target=N`인 시스템/BSS/외부 연계/분석 유즈케이스가 고객 상태·노출·결과·복구 경로를 바꾸는 경우 상태 전이에 연결되었는가?
- 상태 전이 이벤트가 유즈케이스 또는 업무 사건인가?
- 상태 전이 검증에서 상태 ID와 상태명이 모두 일치하는가?
- 프로세스가 한 유즈케이스당 1개로 기계적으로 축소되지 않았는가?
- 프로세스의 내부 관련 기능/정책 값이 순수 ID로 저장되었는가?
- 기능 `process_ids`와 프로세스 `related_functions`가 양방향으로 일치하는가?
- 정의된 기능과 정책 중 프로세스에서 참조되지 않는 항목이 0건인가?
- HTML 표에는 관련 기능/정책이 ID와 명칭으로 표시되었는가?
- 기능이 화면 단위가 아니라 처리 역량인가?
- 정책 항목이 실제 판단 기준인가?
- BSS 판단, 상태 변경, 이력 저장, 결과 회신 기준이 있는가?
- 예외, 실패, 재시도, 복구, 고지 기준이 있는가?
- 실패 기준이 주제 범위에 맞는가? 인증/결제 주제가 아니면 해당 주제의 연계 실패, 반영 지연, 중복 요청, 대체 경로, 캐시/딥링크/분석 실패 같은 고유 실패를 우선 확인했는가?
- 샘플 정책서의 표 밀도와 문체 수준에 맞는가?
- 다른 정책서 범위와 충돌하거나 중복을 이유로 누락한 항목은 없는가?
- Codex 직접 Inspector 판정과 직접 Health Check 판정을 수행하고, 자동 검증과 다른 판단이 있으면 사유를 기록했는가?

## 14. 실제 Agent 동작과의 정합성 점검

수동 작성은 아래 실제 구현 흐름과 어긋나면 안 된다.

확인한 구현 기준:

- `src/orchestrator.py`의 `orchestrate_policy_generation()`이 전체 실행 순서를 관리한다.
- `src/chapter_agents.py`의 `chapter_stages()`가 장별 작성 순서를 정의한다.
- 간소화 버전 순서는 `overview → terms → actors → usecases → usecase_diagram → state → process → functions → policies → terms_refinement → final_check`다.
- Full 버전은 간소화 흐름에 `process_detail`, `function_detail`을 추가한다.
- 각 장은 `run_chapter_stage()`에서 작성, JSON Critical Gate, Inspector, 보완 루프를 거친다.
- 장별 검수 모드에서는 장별 Inspector가 동작한다.
- 최종 검수만 모드에서는 장별 Inspector를 생략하되 JSON Critical Gate와 Final Inspector를 유지한다.
- 검수 없음 모드에서도 JSON Critical Gate는 유지해야 한다.
- Inspector가 이전 장 계약 오류를 명확히 지적하면 upstream backtrack을 1회 수행할 수 있다.
- 점수가 개선되지 않으면 같은 보완을 반복하지 않고 최고 후보를 유지한 뒤 후속 장 또는 Final Inspector로 이슈를 넘긴다.
- Final Inspector 이후에는 최종 보완 라우팅 또는 Final Revision Agent가 좁은 범위로 수정한다.

수동 작성 시 대응:

- 한 번에 전체 문서를 작성하지 않는다.
- 각 장을 작성할 때 이전 장의 ID와 결정사항을 고정 계약으로 본다.
- 장이 끝날 때마다 해당 장만 검수하고, 필요한 경우 같은 장을 보완한다.
- 이전 장의 오류 때문에 현재 장이 흔들리는 경우에만 이전 장을 되돌아가 1회 보완한다.
- 보완 루프가 길어지면 동일 수정을 반복하지 않고 남은 이슈를 명시적으로 기록한다.
- 최종 문서는 Health Check와 Final Inspector 관점으로 다시 점검한다.

이 절차를 지키지 않고 빠르게 작성한 산출물은 서비스 Agent 결과와 비교 가능한 초안으로 보지 않는다.
