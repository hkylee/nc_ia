## 2. 외부 학습에서 도출한 공통 정책 질문

### 2.1 고객 자격 / 권한 / 인증

- 누가 직접 처리할 수 있는가.
- 법정대리인, 대리인, 운영자, 연계 시스템의 역할이 어디에서 필요한가.
- 본인인증, 대리 권한 확인, 약관/동의 확인이 어느 단계에서 필요하고 언제 만료되는가.
- 인증 실패, 권한 부족, 미성년/법인/외국인/제한 고객 등 예외 상태는 어떻게 처리하는가.
- 인증 결과와 권한 판단 이력은 어떤 단위로 남기는가.

### 2.2 상품 / 주문 / 계약 / 가입

- 고객이 상품을 보기 전에 확인해야 하는 기본 정보, 고객별 적용 가능 정보, 혜택/제한 정보는 무엇인가.
- 고객이 주문/가입/계약을 요청하기 전에 가격, 약정, 할인, 유의사항, 철회/취소 가능 조건을 확인할 수 있는가.
- 주문, 계약, 개통, 배송, eSIM 발급, 회선 반영 상태는 서로 어떻게 구분되는가.
- 결제 성공과 BSS 반영 성공이 불일치할 때 어떤 보류 상태와 재처리 기준을 둘 것인가.
- 주문/계약/결제/배송/개통 이력은 각각 어떤 목적과 기간으로 관리해야 하는가.

### 2.3 결제 / 청구 / 수납 / 환불

- 청구 조회, 납부, 결제, 환불은 같은 금전 업무로 묶되 상태와 이력 기준은 구분되는가.
- 결제수단 등록, 결제 승인, BSS 수납 반영, 영수/납부 확인의 성공·실패 상태는 분리되는가.
- 부분 납부, 중복 납부, 환불 대상, 환불 불가, 환불 지연, 결제 실패 재시도 기준이 정책 항목으로 드러나는가.
- 계약·청약철회, 대금결제, 분쟁처리 기록은 보존 목적과 기간이 다를 수 있음을 반영하는가.

### 2.4 멤버십 / 혜택 / 쿠폰 / 이용권

- 혜택 보유, 발급, 사용 가능, 사용 완료, 만료, 회수, 복원, 재발급 상태가 구분되는가.
- 멤버십 등급, 쿠폰, 포인트, 이용권은 고객·회선·상품·제휴사 상태에 따라 사용 가능 여부가 달라지는가.
- 중복 사용, 양도/공유, 선물, 회수, 사용 취소, 환불/복원 기준이 정책 항목으로 분리되는가.
- 제휴사 처리 결과가 지연되거나 불일치할 때 고객 안내와 재처리 이력 기준이 있는가.

### 2.5 AI 검색 / 추천 / 개인화

- AI가 직접 계약, 결제, 해지, 환불 같은 확정 업무를 수행하지 않고 후보 제안 또는 안내에 머무르는 경계가 있는가.
- 추천/검색 결과에서 제외해야 할 고객 상태, 상품 상태, 법적/정책적 제한 조건이 있는가.
- AI 결과의 근거, 필터링, 오류 신고, 재처리, 고객 피드백 반영 기준이 있는가.
- 개인정보, 행태정보, 위치정보, 민감정보가 추천·검색에 사용될 때 목적, 보관, 거부, 대체 경로 기준이 있는가.

### 2.6 고객지원 / FAQ / 공지 / 매장안내

- 고객이 스스로 해결할 수 있는 업무와 상담/매장/운영자 전환이 필요한 업무를 구분하는가.
- 공지, FAQ, 이용안내, 매장안내는 최신성, 적용 대상, 노출 기간, 변경 이력 기준을 갖는가.
- 위치 기반 매장 안내는 위치 동의, 동의 거부 시 대체 검색, 위치정보 보관 여부를 구분하는가.
- 고객 문의/불만/분쟁 처리 이력은 처리 목적과 보존 기준을 갖는가.

## 3. 장별 작성에 반영할 체크포인트

### 3.1 개요

- 이 정책서가 해결하는 고객 업무 문제를 먼저 정의한다.
- 기존 채널명이나 내부 시스템명이 아니라 고객이 완료해야 하는 과업 기준으로 범위를 잡는다.
- 포함 범위와 제외 범위를 구분하되, 고객 업무 완료에 필요한 BSS 판정·상태 반영·이력 저장은 포함한다.

### 3.2 용어

- 정책 판단에 쓰이는 고객 상태, 계약 상태, 주문 상태, 결제 상태, 혜택 상태, 동의 상태를 우선 정의한다.
- 외부 공식 자료의 용어를 참고하되 첨부 템플릿과 샘플의 용어 체계를 우선한다.
- UI 문구나 마케팅 표현은 정책 판단에 직접 쓰이지 않으면 용어로 올리지 않는다.

### 3.3 액터 / 유즈케이스

- 로그인/비로그인, 정상/제한, VIP/일반 같은 고객 상태는 액터가 아니라 정책 조건으로 둔다.
- 고객과 운영자는 업무를 시작하는 액터이며, BSS·AI·연계 시스템은 검증·판정·결과 제공 액터로 둔다.
- 고객이 완료해야 하는 업무는 상위 유즈케이스로 두고, 시스템 내부 검증·저장·조회는 보조 유즈케이스나 기능으로 둔다.

### 3.4 상태 / 상태 전이

- 상태는 기능 허용, 고객 고지, BSS 반영, 후속 처리 가능 여부를 결정할 때만 둔다.
- 주문/계약/결제/배송/개통/환불/쿠폰/구독 상태를 섞지 않는다.
- 상태 전이는 현재 상태, 전이 이벤트, 다음 상태, 처리 기준, 이력 기준을 함께 작성한다.

### 3.5 프로세스 / 기능

- 프로세스는 고객 또는 운영자가 경험하는 순서대로 시작, 조건 확인, 입력/선택, 인증/동의, 영향도 안내, 요청, BSS/연계 처리, 결과 안내, 이력 저장을 포함한다.
- 시스템 내부 조회·검증·저장은 별도 프로세스로 과도하게 나누지 않고 관련 기능으로 묶는다.
- 기능은 조회, 검증, 산정, 저장, 알림, 연동, 결과 확인 중 2~5개 하위 처리를 포함하는 업무 완결 단위로 둔다.

### 3.6 정책 / 정책 항목

- 정책은 기능 설명이 아니라 판단 기준이다.
- 정책 항목 하나에는 하나의 기준만 둔다.
- 정책 항목은 허용 기준, 제한 기준, 조건, 횟수, 시간, 금액, 상태, 예외, 고지, 이력, 처리 주체, 연동 시스템, BSS 반영 기준, 결과 회신 기준 중 하나 이상을 포함한다.
- 외부 공개웹에서 확인한 수치가 있어도 첨부자료에 없으면 확정값으로 쓰지 말고 TBD 또는 정책 후보로 둔다.
- TBD를 쓸 때는 결정 주체, 결정 필요 사유, 결정 기한을 함께 작성한다.

- 주문/계약/가입: 주문 접수, 계약 확인, 본인인증, BSS 반영, 가입 완료.
- 선물주문: 발신자/수신자 권한, 수락/거절, 만료, 환불/회수.
- 상품변경: 변경 가능 조건, 변경 영향도, 예약/즉시 반영, 원복.
- 결제: 결제수단, 승인, 실패 재시도, BSS 수납 반영, 결제 이력.
- 주문 상태/사후 관리: 주문·배송·개통·교환·반품 상태, 사후 변경.
- 해지/환불/취소: 청약철회, 계약 해지, 환불 산정, 환불 불가, 이력.
- 나의 가입 정보: 가입 상품/회선/약정/청구 정보 조회, 민감정보 마스킹.
- 회선 변경/관리: 회선 선택, 대표 회선, 사용 제한, 권한 검증.
- 멤버십 카드 관리: 카드 발급/노출, 바코드, 도용 방지, 사용 이력.
- 청구 및 수납 관리: 청구 조회, 납부, 미납, 영수/수납 반영.
- 나의 데이터·통화: 사용량 조회, 잔여량, 리필/선물, 과금 안내.
- 상품·서비스 혜택 이용/공유: 혜택 사용, 공유/양도, 회수/복원.
- 통합 쿠폰/이용권함: 쿠폰/이용권 보유, 사용 가능, 만료, 제휴사 검증.
- 회원 가입/탈퇴: 회원 자격, 본인인증, 탈퇴 제한, 보관/분리/파기.
- 회원정보 조회/변경: 본인정보 변경, 인증, 변경 이력, 고지.
- 통합 알림: 수신 동의, 발송 대상, 재발송 제한, 이력.
- 통합 약관: 약관 게시, 동의/철회, 변경 공지, 버전 관리.
- 설정: 서비스 설정, 수신/개인화/위치/권한 설정, 기본값.
- 고객센터_통합허브: 자가 해결, 상담 전환, 문의 접수, 이력.
- 고객센터_FAQ/공지/이용안내: 콘텐츠 등록/검수/노출/종료, 변경 이력.
- 고객센터_매장안내: 위치 동의, 매장 검색, 예약/방문 가능성, 대체 경로.

## 5. Inspector 적용 제한

- 이 문서만 근거로 P1 오류를 만들지 않는다.
- 2~4순위 근거에서 발견한 항목은 현재 정책서 요구사항, 첨부자료, 승인된 블루프린트 계층과 연결될 때만 보완 대상으로 둔다.
- 범위 밖 일반론은 finding이 아니라 open issue 또는 evidence gap으로 남긴다.
- 개인정보, 위치정보, 전자상거래, AI 이용자보호 관련 항목도 현재 주제의 업무 흐름과 직접 연결될 때만 본문 보완 대상으로 삼는다.

## 6. Writer 적용 제한

- 외부 지식은 정책 문장을 풍부하게 만드는 재료가 아니라 정책 질문과 누락 체크리스트다.
- 경쟁사/공개웹 문구를 복사하지 않는다.
- 고객 과업, 상태, BSS 반영, 이력, 고지 기준으로 재구성한다.
- 첨부자료에 없는 수치·기간·비율·가격·혜택명은 확정값으로 쓰지 않는다.
- 불확실한 값은 TBD로 두되 결정 주체, 결정 필요 사유, 결정 기한을 함께 쓴다.
