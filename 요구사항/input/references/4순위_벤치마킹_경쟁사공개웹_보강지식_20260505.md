## 벤치마킹 관찰 축

### 통합 앱 구조

- 경쟁 통신사 앱은 MY, 혜택, 스토어, 맞춤 정보, AI 검색처럼 고객 목적 단위로 주요 영역을 재구성하는 방향이 관찰된다.
- 멤버십, 쿠폰, 혜택, 요금·이용 정보는 별도 앱보다 하나의 통합 앱 안에서 확인·사용하도록 모으는 패턴이 강하다.
- 정책서에서는 메뉴명을 복사하지 말고 “고객이 한곳에서 확인하고 다음 행동으로 이동할 수 있는 기준”으로 재구성한다.

### 요금·납부 셀프 처리

- 미납 요금 확인, 전체/부분 납부, 결제수단 선택, 본인인증, 납부 완료까지 앱 안에서 단계화하는 패턴이 있다.
- 자동이체, 납부수단 변경, 실시간 요금, 납부내역은 고객이 상담 없이 확인할 수 있어야 하는 핵심 셀프 처리 후보이다.
- 정책서에서는 요금 조회와 결제 실행을 구분하고, 결제 실패, 부분 납부, 인증 대체, 납부 이력 저장 기준을 정책 항목으로 둔다.

### 대표 회선·세션·개인화

- 복수 회선을 보유한 고객은 대표번호 또는 기준 회선 설정이 필요하다.
- 로그인 유지, 위젯, 개인화 추천은 편의 기능이지만 개인정보 보호를 이유로 유지기간, 재인증, 비노출 정보 범위를 둔다.
- 정책서에서는 대표 회선 기준, 회선 전환, 비로그인 또는 세션 만료 시 표시 제한, 재인증 조건을 분리한다.

### 멤버십·혜택·쿠폰

- 멤버십 혜택은 등급, 월/연 사용 가능 횟수, 쿠폰 유효기간, 중복 적용 제한, 오프라인/온라인 사용 가능 여부, 제시 방식 같은 사용 조건을 가진다.
- 쿠폰은 다운로드, 보유 쿠폰함 확인, 등록, 사용, 취소 후 재사용 가능 여부, 부분 취소 시 재사용 가능 여부 같은 세부 판단값이 필요하다.
- 정책서에서는 제휴사별 상세 혜택을 복사하지 말고 혜택 사용 가능 조건, 제한, 고지, 이력 기준을 공통 정책으로 만든다.

### 상품·가입·구매

- 온라인 가입 흐름은 가입 유형 선택, 요금제 선택, 혜택 선택, 지원금·약정·자급제 조건 확인, 제한 상품 안내를 포함한다.
- 일부 요금제나 상품은 고객 보호 또는 피해 예방 차원에서 온라인 가입을 제한하고 오프라인 채널로 유도할 수 있다.
- 정책서에서는 가입 가능 여부, 지원금·약정 영향 고지, 제한 사유, 대체 채널 안내를 정책 항목으로 둔다.

### eSIM·USIM·단말 조건

- eSIM/USIM 가입은 단말 지원 여부, 셀프 개통 가능 시간, 가입 유형, 본인인증, 유심 보유 여부, 기기 제한, 개통 실패 시 안내가 중요하다.
- 정책서에서는 개통 가능 조건, 개통 제한 사유, 실패 시 대체 경로, 고객 고지, 이력 저장 기준을 분리한다.

### 고객 보호와 제한 고지

- 고객 보호 목적의 온라인 가입 제한, 분실 단말 제한, 미납·회선 초과 제한, 약정/지원금 영향 안내 같은 패턴이 있다.
- 정책서에서는 제한 조건을 숨기지 않고 고객 행동 전 고지, 대체 채널, 상담 전환, 이의제기 가능 여부를 정책으로 둔다.

## 벤치마킹 직접 체크포인트

### 고객 과업 완결성 점검

- 고객이 업무를 시작한 뒤 완료까지 필요한 조회, 조건 확인, 선택, 인증, 처리 요청, 결과 확인, 이력 확인이 빠져 있지 않은지 점검한다.
- 경쟁사 화면이나 메뉴를 복사하지 않는다.
- 고객이 다음 행동을 알 수 없는 상태, 실패 사유가 불명확한 상태, 상담 전환 기준이 없는 상태를 누락 후보로 본다.

검토 질문:

- 고객이 현재 상태와 다음 행동을 알 수 있는가.
- 처리 실패 시 재시도, 취소, 대체 경로, 상담 전환 중 하나 이상이 정의되어 있는가.
- 처리 완료 후 고객 고지와 이력 저장 기준이 있는가.

### 요금·납부·결제 점검

- 전체 납부와 부분 납부, 결제 수단 변경, 자동납부, 미납 상태, 납부 이력, 환불을 분리해 검토한다.
- 경쟁사 결제 수단이나 수수료를 복사하지 않는다.
- 납부 가능 조건과 결제 실패 기준이 없으면 보완 후보로 본다.

검토 질문:

- 납부 대상 금액과 납부 가능 금액이 구분되어 있는가.
- 결제 실패, 승인 지연, 중복 결제, 취소, 환불 기준이 있는가.
- 고객의 회선 상태, 미납 상태, 법인·대리인 조건이 반영되어 있는가.

### 멤버십·혜택·쿠폰 점검

- 혜택 사용 가능 여부는 등급, 기간, 보유 쿠폰, 제휴처, 사용 횟수, 중복 적용 여부로 나눠 점검한다.
- 경쟁사 혜택명, 제휴사명, 포인트 값을 복사하지 않는다.
- 쿠폰 발급, 보유, 사용 완료, 만료, 재사용 가능 여부를 상태와 정책 항목으로 검토한다.

검토 질문:

- 혜택 대상 고객과 제외 고객이 구분되어 있는가.
- 쿠폰·포인트·바코드의 사용 실패 사유가 정의되어 있는가.
- 중복 적용 우선순위와 사용 이력 기준이 있는가.

### 상품·주문·개통 점검

- 상품 선택, 요금제 선택, 할인 조건 확인, 본인확인, 결제, 배송, 개통, 활성화, 교환·반품을 분리해 점검한다.
- 경쟁사 가입 가능 시간, 수수료, 단말 조건을 그대로 쓰지 않는다.
- 배송 완료와 개통 완료, 개통 완료와 활성화 완료를 혼동하지 않았는지 확인한다.

검토 질문:

- 주문·배송·개통·활성화 상태가 각각 정의되어 있는가.
- 개통 실패 시 재시도와 상담 전환 기준이 있는가.
- 교환·반품 가능 여부가 수령, 개통, 단말 상태, 구성품, 신청 시점으로 판정되는가.

### 구독·선물·공유 점검

- 구독 상품은 가입, 사용, 정기결제, 결제 실패, 일시중지, 해지, 환불, 선물, 공유를 함께 점검한다.
- 경쟁사 구독 상품명과 혜택 구성은 복사하지 않는다.
- 대표·멤버·수신자처럼 권한이 나뉘는 구조는 액터 또는 정책 조건으로 분리한다.

검토 질문:

- 정기결제 실패와 유예 기준이 있는가.
- 선물 수신, 거절, 사용, 만료, 환불 기준이 있는가.
- 공유 또는 패밀리 구조에서 대표와 멤버 권한이 분리되어 있는가.

### AI 검색·추천 점검

- AI 검색과 추천은 검색 대상, 제외 대상, 랭킹·추천 기준, 고객 제어, 오류 신고, 상담 전환, 로그 저장을 점검한다.
- 경쟁사 AI 명칭이나 기능명을 복사하지 않는다.
- AI 결과가 계약, 과금, 혜택 사용, 가입 제한을 직접 확정하지 않도록 BSS 또는 확정 정책 기준과 연결한다.

검토 질문:

- 추천 사유, 제외 사유, 부적절 결과 차단 기준이 있는가.
- 고객이 추천·개인화 사용을 제어할 수 있는 기준이 있는가.
- AI 결과 오류 신고와 후속 처리 이력이 정의되어 있는가.

## Inspector 제한 기준

- 4순위 벤치마킹 자료만으로 P1 finding을 만들지 않는다.
- 경쟁사에 있는 기능이 현재 정책서에 없다는 이유만으로 범위 누락으로 판단하지 않는다.
- 단, 현재 정책서 범위 안의 고객 과업을 완료하는 데 필요한 상태, 예외, 고지, 이력, 제한 기준이 빠졌다면 P2 또는 P3 보완 후보로 제시한다.
- 보완 권고는 "경쟁사도 한다"가 아니라 "현재 프로세스에서 고객이 다음 행동을 알 수 없다", "정책 항목에 제한 기준이 없다"처럼 문서 내부 기준으로 작성한다.

## 장별 사용 지침

- Overview: 벤치마킹은 고객 과업 완결성 후보로만 사용하고 직접 범위를 넓히지 않는다.
- Terms: 대표 회선, 로그인 유지, 보유 쿠폰함, 사용 가능 횟수, 유효기간, 중복 적용 제한, 대체 채널, 셀프 개통 같은 용어 후보를 검토한다.
- Usecases: 경쟁사 메뉴가 아니라 고객 행위 기준으로 “요금 납부”, “혜택 사용”, “대표 회선 설정”, “가입 가능 여부 확인”처럼 재구성한다.
- State: 보유, 사용 가능, 사용 완료, 만료, 재사용 가능, 재사용 불가, 인증 필요, 제한, 대체 채널 안내 상태를 검토한다.
- Process: 조회, 조건 확인, 선택, 인증, 처리, 결과 안내, 이력 저장 순서로 정리한다.
- Functions: 결제수단 확인, 대표 회선 설정, 쿠폰 상태 산정, 중복 적용 검증, 가입 가능 여부 산정, 고지 생성, 이력 저장을 기능 후보로 둔다.
- Policies: 경쟁사 수치를 그대로 쓰지 말고 적용 조건, 횟수, 유효기간, 중복 제한, 취소 후 재사용 기준을 SKT 요구사항과 사내 기준으로 확정한다.

## Evidence Gap

- 경쟁사 기능명, 혜택명, 수치, 시간, 횟수, 수수료는 NOVA 정책값으로 확정하지 않는다.
- 벤치마킹은 “고객이 기대할 수 있는 처리 단위와 예외 후보”를 찾는 용도이며, 실제 정책값은 첨부자료, 사내자료, 요구사항 또는 SKT 공식 근거로 확정해야 한다.
