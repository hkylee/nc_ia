# 1순위 첨부자료·사내자료·요구사항 정책 작성 체크포인트

## 근거 권위

- authority_tier: 1
- source_authority: attached_guideline
- 적용 범위: NOVA(Next Channel) 정책서 작성 시 최우선 작성 기준
- 사용 원칙: 첨부자료, 사내자료, 요구사항, 템플릿, 샘플은 정책서 작성의 확정 기준이다.
- 충돌 처리: 2순위 SKT 공식 안내, 3순위 컴플라이언스, 4순위 벤치마킹 자료와 충돌하면 본 문서를 포함한 1순위 근거를 따른다.
- 금지 사항: 공개웹 또는 경쟁사 자료를 이유로 첨부 요구사항의 범위를 임의로 줄이거나 늘리지 않는다.
- Inspector 사용 원칙: 1순위 근거와 산출물의 불일치는 finding 대상이다. 다만 1순위 근거끼리 상충하면 finding 대신 open_issue로 남기고 결정 주체를 요구한다.

## 1순위 자료별 해석 기준

### 요구사항 목록

- 요구사항은 최소 작성 범위다.
- 요구사항 1건은 유즈케이스, 상태, 프로세스, 기능, 정책, 정책 항목 중 하나 이상에 반드시 연결한다.
- 요구사항 문구를 그대로 복사하지 않고 고객 과업, BSS 판단, 기능 처리, 정책 기준으로 재구성한다.
- 요구사항이 기능 요구처럼 보여도 판단 기준, 제한 조건, 예외, 이력, 고지가 있으면 정책 항목까지 확장한다.
- 요구사항이 특정 화면이나 메뉴를 말하더라도 정책서에서는 업무 목적과 처리 단위로 바꿔 작성한다.
- 요구사항에 명시되지 않은 값은 확정값처럼 쓰지 않는다. 필요하면 TBD, 결정 주체, 결정 사유, 결정 기한을 함께 둔다.

정책 항목 후보:

- 요구사항 반영 대상 기준.
- 요구사항 미반영 사유 기록 기준.
- 신규 도출 요구사항 기록 기준.
- 요구사항-유즈케이스-프로세스-기능-정책 매핑 기준.

### 템플릿

- 템플릿은 문서 구조, 표 컬럼, 장 순서, HTML 스타일의 기준이다.
- 템플릿에 없는 장을 임의로 추가하지 않는다.
- 템플릿의 표 컬럼명과 장 제목을 유지한다.
- 문장은 마침표 뒤 줄바꿈 기준을 따른다.
- 화면 UI 상세, API 필드, DB 컬럼, 오류 코드 전체 목록, 운영자 화면 상세, 배치 설계, 비기능 상세는 본문에 과도하게 넣지 않는다.
- 단, 정책 판단에 직접 영향을 주면 정책 항목으로 남긴다.

정책 항목 후보:

- 문서 히스토리 반영 기준.
- 장별 필수 표 유지 기준.
- 정책 상세 작성 단위 기준.
- UI/API/DB 상세 제외 기준.

### 샘플 정책서

- 샘플은 작성 밀도, 문체, 표 구성, 정책 상세 수준의 품질 기준이다.
- 샘플의 주제나 정책값을 새 주제에 복사하지 않는다.
- 샘플에서 확인되는 계층 구조와 입자도만 따른다.
- 정책 항목은 한 항목에 하나의 판단 기준만 담는다.
- 정책 상세는 실제 기준값, 허용 목록, 제한 조건, 예외, 고객 고지, 이력 저장 기준을 포함해야 한다.
- 프로세스, 기능, 정책 ID와 명칭은 서로 추적 가능해야 한다.

정책 항목 후보:

- 정책 항목 입자도 기준.
- 정책 상세 필수 구성 기준.
- 프로세스-기능-정책 연결 기준.
- ID/명칭 정합성 기준.

### 사내 참고자료·IA·VoC·고객 조사

- IA와 채널 전략은 고객 과업의 진입, 탐색, 전환, 완료 흐름을 잡는 기준이다.
- VoC와 고객 조사는 불편을 그대로 쓰는 자료가 아니라 예외, 고지, 상태, 상담 전환 기준으로 환원한다.
- 고객이 상담을 찾는 이유는 셀프 처리 실패 조건, 정보 부족, 인증 실패, 상태 불명확, 정책 고지 부족으로 분해한다.
- BSS가 판단하거나 원장에 반영하는 업무는 채널 밖이라고 제외하지 않는다.
- BSS의 API 필드나 DB 컬럼은 쓰지 않되, 검증, 판정, 상태 변경, 이력 저장, 결과 회신 기준은 정책서에 남긴다.

정책 항목 후보:

- 고객 불편 기반 예외 처리 기준.
- 상담 전환 조건 기준.
- BSS 판정 및 이력 저장 기준.
- IA 기반 진입·종료 기준.

## 장별 직접 체크포인트

### 개요

- 대상 업무, 대상 고객, 대상 채널, 포함 범위, 제외 범위를 고객 과업 기준으로 작성한다.
- 요구사항에 포함된 업무는 범위에서 누락하지 않는다.
- 제외 범위는 후속 산출물로 넘길 수 있는 상세 설계 영역만 둔다.

### 주요 용어

- 요구사항, 정책 항목, 상태 전이, BSS 판단에 쓰이는 용어를 우선 정의한다.
- 일반 명사와 마케팅 표현은 제외한다.
- 용어 설명에는 업무상 판단 기준과 유사 용어 차이를 포함한다.

### 액터·유즈케이스

- 액터는 독립 책임 주체만 둔다.
- 로그인 고객, 비로그인 고객, 정상 고객, 제한 고객은 액터가 아니라 상태 또는 정책 조건이다.
- 고객 또는 운영자가 시작하는 업무는 프로세스 정의 대상 Y를 우선 검토한다.
- 시스템이 수행하는 조회, 저장, 결과 생성은 단독 프로세스보다 관련 고객·운영자 프로세스 안의 기능과 정책으로 연결한다.

### 상태·상태 전이

- 상태는 고객 노출, 기능 허용, 정책 판단, 후속 처리에 영향을 주는 값만 둔다.
- 상태 전이 이벤트는 유즈케이스와 연결한다.
- 현재 상태와 다음 상태는 상태 코드 목록에 존재해야 한다.
- 정상, 실패, 제한, 보류, 운영 검토 흐름을 함께 검토한다.

### 프로세스

- 프로세스는 고객 또는 운영자가 업무를 시작해 완료하는 순서대로 작성한다.
- 시스템 내부 조회·판정·저장은 과도하게 프로세스로 분리하지 않고 기능과 정책으로 연결한다.
- 각 프로세스는 관련 기능과 관련 정책을 반드시 가진다.
- 관련 정책명은 정책 목록의 정책 그룹명과 일치해야 한다.

### 기능·세부 기능 구성

- 기능은 화면 단위가 아니라 처리 단위다.
- 기능은 조회, 검증, 입력, 산정, 저장, 알림, 연동, 결과 확인 중 2~5개 하위 처리를 묶는 수준으로 작성한다.
- 세부 기능 구성은 기능 안에서 개발자가 처리 단위를 이해할 수 있을 만큼 구체적이어야 한다.
- 기능 설명에 정책값을 길게 쓰지 않고 판단 기준은 정책으로 분리한다.

### 정책·정책 항목

- 정책은 기능 설명이 아니라 판단 기준이다.
- 프로세스와 기능에 필요한 정책을 먼저 정의하고, 그 정책을 구성하는 세부 정책 항목과 값을 정의한다.
- 정책 항목에는 기준값, 허용 목록, 제한 조건, 적용 우선순위, 예외 허용·불가 조건, 고객 고지, 이력 저장 기준을 포함한다.
- "검토 필요", "추후 협의", "정책에 따라 처리" 같은 표현만 단독으로 남기지 않는다.

## Inspector 점검 기준

- 1순위 요구사항이 산출물에 연결되지 않으면 P1 finding으로 본다.
- 템플릿 장 구조나 샘플 수준의 정책 상세 밀도에서 벗어나면 P1 또는 P2 finding으로 본다.
- 정책 항목이 일반 설명이면 보완 대상이다.
- 2·3·4순위 근거와 달라도 1순위 근거에 맞으면 오류로 판단하지 않는다.
- 1순위 근거가 부족한 영역은 낮은 순위 근거로 후보를 만들 수 있지만, 확정 정책값은 첨부자료나 TBD로 처리한다.
