# 스타일 앵커: 예전 프로젝트 정책 상세화 패턴

## 1. 사용 목적

이 문서는 `input/old documents/`에 추가된 예전 프로젝트 PDF 5건을 검토해 도출한 정책 상세화 방식이다.

현재 NC 정책서의 요구사항, 템플릿, 샘플, TK, 채널 방향성을 대체하지 않는다.
예전 프로젝트의 실제 정책값, 상품명, 채널명, 수치, 시점, 조직명은 현재 정책서에 그대로 복사하지 않는다.

이 문서는 주로 현재 정책서의 `6. 정책 정의`를 풍부하게 만드는 스타일 앵커로 사용한다.
필요하면 `3. 유즈케이스 정의`, `4. 프로세스 정의`, `5. 기능 정의`, `상태 전이 기준`에서도 판단축을 보강하는 참고로만 사용한다.

## 2. 검토한 예전 문서

- 구독 결제 / 정기결제.
- 구독 상품 변경 / 관리.
- 구독 해지.
- 입점 / 퇴점 / 담당자.
- 패밀리.

PDF 텍스트는 전체 페이지 기준으로 충분히 추출되었다.
다만 Confluence 헤더, URL, 페이지 번호, 일부 표의 줄바꿈은 섞여 있으므로 문장 단위 근거가 아니라 패턴 학습용으로만 사용한다.

## 3. 가장 중요한 학습점

예전 문서는 정책을 추상 설명으로 끝내지 않고, 아래 질문에 답하는 방식으로 상세화한다.

- 어떤 유형으로 구분되는가?
- 어떤 조건이면 가능한가?
- 어떤 조건이면 불가능한가?
- 어느 채널 또는 주체가 처리할 수 있는가?
- 어느 시점 전후로 기준이 달라지는가?
- 고객 명의, 권한, 상품 유형, 사용 여부, 인증 여부에 따라 기준이 달라지는가?
- 자동 판정이 기본인지, 운영자 또는 권한자 변경이 허용되는지?
- 실패하거나 불가능할 때 고객에게 무엇을 안내하고 어떤 후속 행동을 제공하는가?
- 처리 결과와 이력은 어디에 남는가?

## 4. 정책 상세화 판단축

정책 상세를 작성할 때 아래 판단축 중 현재 주제에 맞는 것을 선택한다.
모든 축을 기계적으로 넣지 않는다.

### 4.1 유형 구분

정책 대상이 여러 형태로 나뉘면 먼저 유형을 정의한다.

예시 패턴:
- 결제 수단 유형.
- 변경 유형.
- 해지 유형.
- 환불 유형.
- 가입/입점 상태 유형.
- 패밀리 대표/멤버 유형.
- 혜택/SKU/쿠폰/이용권 유형.

작성 방식:
- 유형명.
- 유형의 업무 의미.
- 유형별 고객 영향.
- 유형별 처리 가능 조건.
- 유형별 후속 처리.

### 4.2 가능 / 불가 기준

정책은 “가능하다”로 끝내지 않고 불가능 조건까지 함께 둔다.

필수 확인:
- 기본 가능 기준.
- 제한 또는 불가 기준.
- 예외 허용 기준.
- 예외 불가 기준.
- 불가 시 고객 안내 기준.
- 대체 경로 또는 다음 행동.

### 4.3 채널 / 주체별 처리 가능 여부

동일 업무라도 셀프 채널, 상담, 운영자, 시스템, 제휴사 처리 가능 여부가 다를 수 있다.

작성 방식:
- 채널 또는 주체별 처리 가능 여부를 표로 정리한다.
- `O/X`만 쓰지 말고 제한 사유 또는 권한 조건을 정책 상세에 남긴다.
- 시스템 자동 처리와 사람이 요청하는 처리를 구분한다.
- 처리 주체가 달라지면 이력 저장 주체와 고객 고지 주체도 함께 본다.

### 4.4 시점 기준

정책값은 처리 시점에 따라 달라지는 경우가 많다.

확인할 시점:
- 신청 전.
- 신청 중.
- 처리 완료 전.
- 처리 완료 후.
- 다음 정기 처리일 전후.
- 만료 전후.
- 사용 전후.
- 인증 전후.
- 취소 가능 기한 전후.

작성 방식:
- 기준일 또는 기준 시점.
- 해당 시점 이전 가능 행동.
- 해당 시점 이후 제한 행동.
- 예약, 철회, 재처리 가능 여부.
- 고객 안내와 이력 저장 기준.

### 4.5 사용 여부 / 인증 여부 / 혜택 소진 여부

상품 변경, 혜택 공유, 해지, 환불, 쿠폰/이용권 정책은 사용 여부와 인증 여부에 민감하다.

확인할 조건:
- 미사용.
- 일부 사용.
- 전체 사용.
- 미인증.
- 인증 완료.
- 인증 후 혜택 미사용.
- 인증 후 혜택 사용.

작성 방식:
- 조건별 가능/불가를 분리한다.
- 자동 판정 가능한 조건과 운영자 확인이 필요한 조건을 구분한다.
- 조건 확인 실패 시 기본 처리와 보류 처리를 정의한다.

### 4.6 명의 / 권한 / 관계 기준

명의자, 가입자, 결제자, 대표자, 멤버, 담당자, 대리인 관계가 달라지면 정책이 달라진다.

작성 방식:
- 동일 명의와 타 명의 기준을 분리한다.
- 대표/멤버/대리/담당자 역할에 따라 허용 기능을 분리한다.
- 최대 인원, 위임 가능 범위, 권한 변경 가능 범위를 정책 항목으로 둔다.
- 관계가 바뀌어도 유지되는 정보와 초기화되는 정보를 구분한다.

### 4.7 자동 판정 / 수동 조정 기준

예전 문서는 “자동으로 결정되는 것이 기본이며 권한 또는 조건에 따라 변경 가능”한 구조를 자주 사용한다.

작성 방식:
- 자동 판정 기준.
- 자동 판정에 쓰이는 입력 조건.
- 수동 조정 허용 주체.
- 수동 조정 가능 범위.
- 수동 조정 불가 조건.
- 수동 조정 이력과 사유 저장 기준.

### 4.8 상태값과 고객 노출

상태는 내부 처리 상태가 아니라 고객 또는 운영자가 다음 행동을 판단하는 기준이어야 한다.

작성 방식:
- 고객에게 노출되는 상태명.
- 내부 처리 상태.
- 상태별 가능 행동.
- 상태별 제한 행동.
- 반려, 보류, 승인, 완료, 만료, 복구 가능, 복구 불가 같은 후속 차이를 명확히 둔다.

### 4.9 외부 제휴 / 연계 처리 기준

외부 제휴, 인증, 쿠폰, 이용권, 공유, 입점/퇴점처럼 외부 주체가 있는 업무는 책임 경계가 중요하다.

작성 방식:
- NC가 판단하는 기준.
- 외부 시스템 또는 제휴사가 회신해야 하는 기준.
- 회신 지연 또는 실패 시 보류/대체/복구 기준.
- 고객에게 노출하는 결과와 내부 이력 저장 기준.
- 제휴사 정책에 따르는 항목은 그대로 확정하지 않고, NC 문서에서는 영향과 처리 경계로 표현한다.

## 5. 현재 NC 정책서 장별 활용 방식

### 5.1 유즈케이스 정의

예전 문서의 버튼, 신청, 인증, 조회 단계는 그대로 유즈케이스로 올리지 않는다.
유즈케이스는 고객 또는 운영자가 완결하려는 업무 목적 단위로 묶는다.

좋은 적용:
- “결제 수단 등록”보다 “결제 가능 조건 확정”.
- “상품 변경 버튼 클릭”보다 “변경 가능 경로 결정”.
- “해지 신청”보다 “해지·환불 영향 확인과 실행 의사 확정”.
- “멤버 초대 입력”보다 “공유 관계 구성과 혜택 사용 권한 확정”.

### 5.2 상태 전이

예전 문서에서 상태값이나 처리 단계가 확인되면 현재 NC 문서에서는 아래 기준으로 상태 승격 여부를 판단한다.

상태로 둔다:
- 고객에게 노출되는 업무 상태.
- 기능 허용 여부가 달라지는 상태.
- 정책 판단 결과가 달라지는 상태.
- 복구 가능 여부가 달라지는 상태.

상태가 아니라 전이 기준 또는 정책 항목으로 둔다:
- 순간적인 입력 오류.
- 단순 인증 실패.
- 일회성 안내 노출.
- 내부 조회 중 상태.

### 5.3 프로세스 정의

프로세스는 “고객 시작 → 조건 확인 → 선택/입력 → 영향 확인 → 인증/동의 → 처리 요청 → BSS/외부 반영 → 결과 안내” 흐름으로 구성한다.

단, 모든 주제에 이 순서를 기계적으로 강제하지 않는다.
주제별로 빠질 수 있는 단계와 반복되는 단계를 조정한다.

### 5.4 기능 정의

기능은 화면이 아니라 처리 역량으로 작성한다.

예시 처리 역량:
- 대상 식별.
- 조건 판정.
- 가능 여부 산정.
- 영향도 계산.
- 인증/권한 확인.
- 요청 접수.
- 외부 회신 수신.
- 상태 반영.
- 이력 저장.
- 고객 안내 구성.

### 5.5 정책 정의

정책 상세는 아래 순서로 점검한다.

1. 정책 그룹이 프로세스/기능의 어떤 판단 지점을 통제하는가?
2. 정책 항목은 하나의 판단 기준만 담는가?
3. 각 정책 항목에 허용, 제한, 조건, 시점, 횟수, 상태, 예외, 고지, 이력 중 하나 이상이 있는가?
4. 가능 기준과 불가 기준이 모두 있는가?
5. 고객에게 안내할 메시지 또는 다음 행동이 있는가?
6. BSS, 외부 시스템, 운영자가 처리해야 하는 결과 반영 기준이 있는가?

## 6. 적용 금지

아래는 하지 않는다.

- 예전 문서의 수치, 기간, 인원수, 상품 유형, 채널별 O/X 값을 현재 정책값처럼 복사하지 않는다.
- 예전 프로젝트의 상품명, 제휴사명, 조직명, 화면명을 현재 NC 정책서의 근거로 쓰지 않는다.
- Confluence 목차 구조를 현재 NC 템플릿에 맞지 않게 복사하지 않는다.
- 예전 문서가 상세하다는 이유로 현재 간소화 버전의 장을 임의로 늘리지 않는다.
- 다른 정책서 범위의 상세 실행 로직을 중복으로 끌어오지 않는다.

## 7. 작성 전 체크 질문

정책 상세 작성 전 아래 질문을 확인한다.

- 이 정책은 어떤 유형 구분이 필요한가?
- 유형별 가능/불가 조건이 다른가?
- 고객 유형, 명의, 권한, 관계에 따라 기준이 달라지는가?
- 처리 시점 전후로 가능한 행동이 달라지는가?
- 사용 여부, 인증 여부, 소진 여부, 취소 가능 기한이 영향을 주는가?
- 셀프, 상담, 운영자, 시스템, 외부 주체별 처리 가능 여부가 다른가?
- 자동 판정과 수동 조정의 경계가 필요한가?
- 실패, 보류, 반려, 만료, 복구 가능, 복구 불가 상태가 필요한가?
- 고객 안내와 이력 저장 기준이 정책 항목에 남아 있는가?

## 8. Inspector / Health Check 참고 기준

검수 시 이 문서는 다음 기준으로만 사용한다.

- 정책 상세가 일반 설명으로 흐르는지 확인한다.
- 가능 기준만 있고 불가·예외·고지·이력 기준이 빠졌는지 확인한다.
- 채널/권한/시점/사용 여부별 차이가 필요한 주제인데 하나의 문장으로 뭉개졌는지 확인한다.
- 예전 문서의 실제 값과 다르다는 이유로 finding을 만들지 않는다.
- 현재 요구사항, 템플릿, 샘플과 충돌하면 현재 자료를 우선한다.
