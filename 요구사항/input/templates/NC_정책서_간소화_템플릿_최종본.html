작성 기준
1. 문서 목적
본 템플릿은 통합채널 정책서를 빠르게 작성하기 위한 간소화 양식이다.
업체가 상세 설계할 수 있도록 전체 구성과 흐름, 필요한 기능을 정의하고, SKT가 직접 결정해야 하는 정책 기준을 명확히 남기는 것을 목적으로 한다.
2. 작성 범위
업무 범위, 주요 용어, 액터, 유즈케이스, 상태 전이, 프로세스, 주요 기능, 정책 목록, 정책 상세까지 작성한다.
간소화 버전이므로 프로세스 상세 시나리오와 기능 상세 명세는 최소화하되, 프로세스-기능-정책의 연결은 반드시 유지한다.
3. 작성 수준
고객이 어떤 업무를 시작하고, 어떤 조건을 확인하며, 어떤 기준으로 처리 완료 또는 제한되는지 이해할 수 있는 수준으로 작성한다.
기능은 구현 검토가 가능한 단위까지 쓰고, 정책은 실제 판단값·조건·허용범위·제한 기준까지 작성한다.
4. 제외 범위
화면 UI 상세, 문구 상세, API 필드, DB 컬럼, 오류 코드 전체 목록, 운영자 화면 상세, 배치 설계, 비기능 상세는 본 문서에 쓰지 않는다.
다만 해당 항목이 정책 판단에 직접 영향을 주면 정책 항목으로 남긴다.
5. ID 작성 규칙
업무코드는 3자리 영문 대문자를 사용한다.
예: 회원은 MBR, 요금제는 PLN, 결제는 PAY. ID는 문서 전체에서 중복되지 않아야 하며, 삭제된 ID는 재사용하지 않는다.
정책 ID는 정책 그룹 단위, 정책 항목 ID는 상세 기준 단위로 관리한다.
6. 작성 순서
작성은 액터 → 액터별 유즈케이스 → 유즈케이스별 프로세스 → 프로세스별 기능·정책 → 기능별 세부 기능 → 정책별 상세 항목 → 세부 정책 정의 순서로 진행한다.
용어 정의는 별도 선행 작업이 아니라, 각 단계에서 새로 등장한 용어를 중간중간 보정하는 방식으로 관리한다.
7. 핵심 작성 원칙
정책서는 채널 전략 및 과제 방향과 정렬되게 작성한다.
다른 영역에서 작성될 것이라고 전제하지 말고, 중복되더라도 담당 업무 안에서 완결적으로 작성한다.
채널 문서라도 BSS가 수행하는 검증·판정·상태 변경·데이터 처리 업무를 충분히 포함한다.
기준 요구사항은 모두 반영하며, 작성 중 새로 도출된 사항은 요구사항 목록에 추가한다.

작성 원칙 가이드

정책서 작성 원칙
1. 채널 전략 및 과제 방향과 정렬
각 정책서는 단독 문서가 아니라 통합채널의 전략과 실행 과제를 구체화하는 하위 산출물이다.
따라서 기능과 정책은 “현재 채널에 있는 기능을 옮긴다”가 아니라, “통합채널에서 고객 과업을 더 쉽고 명확하게 끝낸다”는 방향에 맞춰 작성한다.
  • 작성 기준: 고객 셀프 처리 확대, 채널 경량화, 중복 제거, 목적 기반 UX, AI 적용 가능성, 보안·인증 안정성, BSS 연계 구조와 맞는지 확인한다.
  • 좋은 작성: “요금제 변경 전 예상 청구액과 연계 혜택 변동을 사전 안내한다.”
  • 나쁜 작성: “기존 T월드 요금제 변경 화면을 제공한다.”
2. 중복되더라도 완결적으로 작성
다른 정책서, 다른 모듈, BSS 문서, 상세 설계서에서 작성될 것이라고 가정하지 않는다.
담당 업무 안에서 고객이 업무를 시작해 완료하기까지 필요한 액터, 유즈케이스, 프로세스, 기능, 정책을 모두 작성한다.
중복은 최종 통합 단계에서 정리할 수 있지만, 누락은 설계와 구축 범위에서 빠질 위험이 크다.
  • 작성 기준: 동일 기능이나 정책이 다른 업무에도 쓰이더라도, 이 업무에서 필요한 이유와 연결 지점은 반드시 적는다.
  • 중복 허용 기준: 본인인증, 약관 동의, 상태 조회, 신청 가능 여부 확인, 결과 알림, 이력 저장처럼 여러 업무에 반복되는 공통 기능·정책은 각 프로세스에 연결한다.
  • 정리 기준: 기능 정의와 정책 정의에서는 공통 기능·공통 정책으로 묶되, 프로세스 목록에서는 해당 프로세스와의 관계를 빠뜨리지 않는다.
3. BSS 업무도 충분히 통합 작성
정책서가 채널 산출물이더라도 고객 업무 완결에는 BSS의 판단과 처리가 포함된다.
채널이 화면과 UX만 정의하고 BSS 처리를 비워두면, 프로세스가 실제로 완결되지 않는다.
BSS가 수행해야 하는 검증, 판정, 상태 변경, 원장 반영, 이력 저장, 데이터 분리·보관, 연동 결과 회신까지 문서에 포함한다.
  • 포함 대상: 가입 가능 여부 확인, 중복 여부 확인, 고객 상태 조회, 제한 조건 판정, 신청 결과 반영, 상태 전이, 청구·혜택·약정 영향 계산, 처리 이력 저장, 오류·실패 결과 회신.
  • 제외 대상: API 필드 상세, DB 컬럼 설계, 배치 상세, 시스템 내부 로직의 구현 방식.
    단, 정책 판단에 영향을 주면 정책 항목으로 남긴다.
  • 문장 예: “BSS는 대상 회선의 요금제 변경 가능 여부, 약정 영향, 연계 구독 영향, 예상 청구액을 산정해 채널에 회신한다.”
4. 요구사항을 모두 포함
기준 요구사항은 정책서의 최소 작성 범위다.
요구사항 목록에 있는 내용은 프로세스, 기능, 정책 중 하나 이상에 반드시 매핑한다.
요구사항을 그대로 복사하지 말고, 업무 흐름과 정책 판단 기준으로 재구성한다.
  • 작성 기준: 요구사항 1건이 고객 흐름이면 프로세스에, 시스템 행위이면 기능에, 판단 기준이면 정책에 반영한다.
  • 점검 기준: 요구사항 ID별로 “프로세스 ID / 기능 ID·기능명 / 정책 ID·정책명” 중 하나 이상이 연결되어야 한다.
  • 누락 위험: 요구사항이 문서에 반영되지 않으면 구축 범위, 검수 기준, 업체 산출물에서 빠질 수 있다.
5. 새로 발견한 사항은 요구사항에 추가
정책서를 작성하다 보면 기존 요구사항에 없던 기능, 정책, 예외 조건, 연동, 상태, 알림, 이력 관리 필요성이 나온다.
이 경우 정책서에만 적지 말고 요구사항 목록에도 신규 요구사항 또는 기존 요구사항 변경으로 반영한다.
  • 신규 요구사항 추가 대상: 새로운 고객 과업, 새로운 처리 흐름, 신규 기능, 신규 정책 판단, 신규 BSS 연동, 신규 예외 처리, 신규 이력·증적 관리, 신규 알림·고지.
  • 변경 요구사항 처리 대상: 기존 요구사항의 범위 확대, 처리 주체 변경, 채널/BSS 역할 변경, 정책값 변경, 우선순위 변경.
  • 문장 예: “정책서 작성 중 ‘탈퇴 유예 기간 내 철회’가 새로 도출되었으므로, 요구사항 목록에 탈퇴 철회 기능 및 정책 요구사항을 추가한다.”
최종 점검 질문
작성 완료 전 아래 질문에 모두 답할 수 있어야 한다.
  • 이 정책서는 통합채널의 전략과 과제 방향을 구체화하고 있는가?
  • 다른 문서에서 다룰 것이라고 보고 누락한 업무, 기능, 정책은 없는가?
  • BSS가 판단하거나 처리해야 하는 업무가 프로세스와 기능에 충분히 포함되어 있는가?
  • 기준 요구사항이 프로세스·기능·정책 중 하나 이상에 모두 반영되어 있는가?
  • 작성 중 새로 발견한 사항을 요구사항 목록에 추가했는가?

작성 흐름
작성 원칙
본 정책서는 목차 순서대로 채우는 문서가 아니라, 업무 구조를 먼저 확정한 뒤 세부 기준을 내려 쓰는 문서다.
먼저 누가 업무를 수행하는지 정하고, 그 액터가 어떤 일을 하는지 정의한다.
이후 유즈케이스별 프로세스를 잡고, 프로세스마다 필요한 기능과 정책을 연결한다.
마지막으로 기능은 세부 기능으로 풀고, 정책은 상세 항목과 실제 정책값으로 확정한다.

“정의 필요”, “추후 협의”처럼 판단을 미루는 문장은 최소화한다.		 6. 정책 정의 > 나.
정책 상세
10 용어 정의 보정 작성 중 새로 등장한 상태, 조건, 고객 유형, 인증 수단, 정책값을 용어로 추가한다.
마지막에는 중복 용어를 합치고, 정책 판단에 쓰이지 않는 설명성 용어는 삭제한다.		 2. 주요 용어
용어 정의 업데이트 기준
용어는 처음에 한 번에 확정하지 않는다.
작성 중간마다 업데이트한다.
액터·유즈케이스 작성 후에는 고객 유형과 업무 명칭을 보정한다.
프로세스 작성 후에는 상태와 처리 단계를 보정한다.
기능·정책 작성 후에는 조건값, 제한 기준, 인증 수단, 데이터 보관 기준을 보정한다.
최종 검토 전에는 같은 의미의 용어가 다른 이름으로 쓰이지 않았는지 확인한다.
되돌아가기 기준
프로세스를 쓰다가 새로운 주체가 나오면 액터로 돌아가 보완한다.
프로세스를 쓰다가 고객 목적이 둘 이상으로 갈라지면 유즈케이스를 분리한다.
기능을 쓰다가 판단 기준이 나오면 정책으로 분리한다.
정책을 쓰다가 화면 동작이 나오면 기능으로 옮긴다.
정책값을 쓰다가 새로운 상태가 필요하면 상태 전이표와 용어 정의를 함께 보정한다.
작성 완료 기준
모든 유즈케이스는 하나 이상의 프로세스 또는 시스템 지원 유즈케이스로 연결되어야 한다.
모든 프로세스는 관련 기능과 관련 정책을 가져야 한다.
모든 기능은 세부 기능 구성을 가져야 한다.
모든 정책은 정책 항목과 세부 정책값을 가져야 한다.
주요 용어는 프로세스, 기능, 정책에서 같은 의미로 반복 사용되어야 한다.

0. 문서 히스토리

작성 가이드
작성 목적
문서 변경의 의사결정 이력을 남긴다.
이후 검토자가 “왜 구조가 바뀌었는지”, “어떤 정책이 언제 확정되었는지”를 추적할 수 있어야 한다.
작성 대상
범위 변경, 유즈케이스 추가·삭제, 프로세스 순서 변경, 기능 병합·분리, 정책값 확정·변경, 제외 범위 변경을 기록한다.
단순 오탈자, 표현 정리, 표 서식 수정은 기록하지 않는다.

변경 내용은 “어느 장의 무엇을 어떻게 바꾸었는지”가 드러나게 쓴다.

1. 개요

가.
범위

작성 가이드
작성 목적
이 문서가 다루는 업무 경계를 정한다.
범위가 모호하면 기능과 정책이 과도하게 넓어지거나 누락된다.
반드시 포함할 내용
① 대상 업무, ② 대상 채널, ③ 대상 고객, ④ 포함 범위, ⑤ 제외 범위, ⑥ 후속 산출물에서 상세화할 영역을 쓴다.
작성 기준
고객이 인식하는 업무 기준으로 쓴다.
내부 조직 기준, 시스템 기준, 기존 채널 기준으로만 쓰지 않는다.
“T월드 업무”, “T멤버십 업무”보다 “요금제 변경”, “멤버십 혜택 사용”처럼 고객 과업을 기준으로 잡는다.
제외 범위 작성 기준
본 문서에서 다루지 않는 업무를 명확히 쓴다.
예: BO 상세 운영 화면, 상담원 처리 절차, 제휴사 내부 처리, 회계 정산, 법무 문안 검토 등.
문장 예: “본 문서는 앱·웹 FO의 고객 셀프 처리 기준을 다루며, 상담원 BO 처리 화면과 제휴사 내부 정산 기준은 제외한다.”

본 정책서는 통합채널에서 제공하는 {업무 영역} 중 {핵심 업무 1}, {핵심 업무 2}, {핵심 업무 3}에 대한 처리 기준을 정의한다.
대상 채널은 앱·웹 FO를 기준으로 하며, {대상 고객}을 기본 대상으로 한다.
본 문서는 상위 유즈케이스, 상태 전이, 프로세스, 주요 기능, 상세 정책을 정의하는 수준까지를 다룬다.
{제외 범위}는 본 문서에서 다루지 않으며, {후속 산출물명}에서 상세화한다.

나.
설계 원칙

작성 가이드
작성 목적
설계 원칙은 이후 프로세스, 기능, 정책을 판단하는 상위 기준이다.
방향성 문구가 아니라 의사결정 기준이어야 한다.
작성 수량
4~6개로 제한한다.
너무 많으면 원칙이 아니라 구호가 된다.
좋은 원칙의 조건
① 판단 기준이 있다.
② 예외를 만들 수 있다.
③ 기능·정책에 연결된다.
④ 채널과 BSS의 역할을 구분한다.
⑤ 고객에게 보이는 경험 기준이 있다.
권장 원칙 유형
셀프 처리 우선, 원스톱 처리 우선, BSS 판단 우선, 고객 상태 기반 분기, 고위험 업무 추가 인증, 법률·보안 우선, 변경 전 영향도 고지, 처리 결과 즉시 확인 등을 업무 특성에 맞게 선택한다.
좋은 예: “변경 전 영향도 고지: 요금·혜택·약정·구독에 영향이 있는 업무는 신청 전에 예상 결과를 표시한다.”
나쁜 예: “고객 편의성을 강화한다.”

• {원칙 1}: {해당 원칙의 적용 기준을 한 문장으로 작성한다.}

• {원칙 2}: {채널과 BSS의 역할 분담 기준을 작성한다.}

• {원칙 3}: {법률·보안·리스크 관련 상위 기준을 작성한다.}

• {원칙 4}: {셀프 처리, 원스톱 처리, 예외 처리 기준을 작성한다.}

• {원칙 5}: {업무 특성에 맞는 추가 원칙을 작성한다.}

2. 주요 용어

작성 가이드
작성 목적
정책, 프로세스, 기능, 시스템 간 해석 차이를 줄인다.
같은 단어를 다르게 해석하면 정책값과 기능 구현이 흔들린다.
작성 대상
상태 용어, 고객 유형, 인증·권한 용어, 약관·동의 용어, 데이터 보관 용어, 외부기관·제휴사 용어, 정책 판단값에 쓰이는 용어를 우선 정의한다.
용어는 초기에 완성하려 하지 말고, 액터·유즈케이스·프로세스·기능·정책 작성 과정에서 새로 등장할 때마다 추가·수정한다.
제외 대상
일반 명사, 화면에서만 쓰이는 마케팅 표현, 누구나 아는 단어, 정책 판단에 쓰이지 않는 설명성 용어는 제외한다.
설명 작성 기준
“무엇이다”로 끝내지 말고, 업무상 판단 기준을 포함한다.
유사 용어와 차이가 드러나야 한다.
좋은 예: “휴면 회원: 장기 미이용 기준에 따라 로그인과 주요 업무 처리가 제한되고 별도 해제 절차가 필요한 회원”
나쁜 예: “휴면 회원: 휴면 상태인 회원”
용어 ID 용어 설명
TM-{업무코드}-001 {용어명} {업무 또는 정책에서 사용하는 의미를 한 문장으로 정의한다.}
TM-{업무코드}-002 {상태/고객유형/처리유형} {다른 용어와 혼동되지 않도록 판단 기준을 포함한다.}
TM-{업무코드}-003 {시스템/데이터/정책 용어} {채널·BSS·외부기관 간 해석 차이가 없도록 정의한다.}

3. 유즈케이스 정의

가.
액터

작성 가이드
작성 목적
누가 업무를 시작하고, 누가 검증·승인·처리·결과 제공을 수행하는지 정의한다.
액터는 유즈케이스를 수행하거나, 처리 결과를 만들거나, 운영 책임을 갖는 독립적인 책임 주체로 한정한다.
액터를 과도하게 세분화하면 유즈케이스와 프로세스가 불필요하게 복잡해지므로, 업무 책임 단위로 압축한다.
작성 대상
고객, 대리인·법정대리인, 핵심 처리 시스템, 외부 승인·인증 기관, 운영자처럼 업무 수행 책임이 명확한 주체를 작성한다.
단, 시스템은 개별 시스템명을 모두 나열하지 말고, 역할이 유사하면 통합 액터로 묶는다.
예: BSS, 지식 베이스, 채널 업무 시스템, 외부 연계 시스템, 상담 시스템은 업무 특성에 따라 연계 시스템으로 통합할 수 있다.
작성 제외
로그인 고객, 비로그인 고객, 정상 고객, 휴면 고객, 미성년 고객처럼 동일 고객의 상태나 조건은 액터로 분리하지 않는다.
이러한 항목은 상태 전이, 권한 조건, 정책 상세에서 다룬다.
단순 참조 시스템, 화면 컴포넌트, 알림 수신만 하는 내부 조직, 보고용 이해관계자도 액터에서 제외한다.
액터 분리 기준
아래 질문 중 2개 이상에 해당하면 별도 액터로 분리한다.
① 해당 주체가 유즈케이스를 직접 시작하는가?
② 해당 주체가 검증·승인·처리 결과를 직접 생성하는가?
③ 해당 주체의 책임이 다른 액터와 명확히 다른가?
④ 해당 주체가 빠지면 프로세스가 성립하지 않는가?
⑤ 해당 주체별로 정책 기준이나 권한 기준이 달라지는가?
통합 기준
아래 항목은 별도 액터로 나누지 않고 통합한다.
① 로그인 여부, 고객 등급, 고객 상태처럼 같은 고객의 조건 차이
② 여러 시스템이 같은 역할로 정보를 제공하는 경우
③ 단순 데이터 저장소, 콘텐츠 저장소, 검색 인덱스처럼 독립 유즈케이스를 수행하지 않는 구성요소
④ 상담 연결, 알림 발송, 외부 링크 이동처럼 주 프로세스를 보조하는 수단
설명 작성 기준
액터 설명은 책임을 동사로 작성한다.
“업무를 수행한다”, “인증 결과를 제공한다”, “상태를 변경한다”, “검색 결과를 생성한다”처럼 역할이 분명해야 한다.
좋은 예: “고객: AI 검색에 질의를 입력하고 검색 결과를 확인하며, 필요한 경우 후속 업무로 진입하는 주체”
좋은 예: “연계 시스템: 검색 결과 생성과 후속 업무 연결에 필요한 고객 상태, 상품, 청구, 혜택, 인증, 상담 정보를 제공하거나 처리하는 시스템”
나쁜 예: “비로그인 고객: 로그인하지 않은 고객” — 로그인 여부는 액터가 아니라 상태·권한 조건이다.
나쁜 예: “FAQ DB, 공지 DB, 검색 인덱스, 상담 시스템, 인증 시스템” — 역할이 유사하거나 보조적이면 연계 시스템으로 통합한다.
액터 ID 액터명 설명
ACT-{업무코드}-001 고객 {해당 업무를 직접 신청·조회·변경·종료하는 주체}
ACT-{업무코드}-002 {대리인/법정대리인/가족} {대리 처리 또는 동의가 필요한 경우의 주체}
ACT-{업무코드}-003 {외부기관/BSS/제휴사} {검증, 승인, 상태 변경, 결과 제공을 수행하는 시스템 또는 기관}

나.
유즈케이스

작성 가이드
작성 목적
고객 또는 시스템이 완결해야 하는 업무 단위를 정의한다.
이후 프로세스와 기능 도출의 기준이 된다.
작성 기준
고객이 인식하는 업무는 상위 유즈케이스로 작성한다.
시스템 내부 검증·저장·결과 제공은 보조 유즈케이스로 작성하되, 프로세스 정의 대상은 일반적으로 N으로 둔다.
프로세스 정의 대상 판단
프로세스 정의 대상은 액터가 사람인지 시스템인지에 따라 1차 판단한다.
고객, 운영자, 법정대리인, 대리인처럼 사람이 수행하는 유즈케이스는 별도 흐름과 판단 기준이 필요하므로 원칙적으로 Y로 둔다.
AI 시스템, BSS, 연계 시스템, 외부기관처럼 시스템이 수행하는 검증·생성·조회·저장·결과 제공은 사람 유즈케이스를 지원하는 보조 유즈케이스로 보고 원칙적으로 N으로 둔다.
단, 시스템 처리라도 고객에게 노출되는 상태 변화, 업무 제한, 인증, 상담 전환, 처리 결과에 직접 영향을 주면 관련 고객 또는 운영자 프로세스 안에서 상세 기준을 정의한다.
판단 예시
고객이 검색한다, 신청한다, 변경한다, 해지한다, 확인한다, 피드백한다 → Y
운영자가 등록한다, 검수한다, 승인한다, 보정한다, 관리한다 → Y
시스템이 조회한다, 분류한다, 저장한다, 결과를 생성한다, 연계 결과를 회신한다 → N

작성 가이드
작성 목적
정책 상세는 실제 의사결정값을 확정하는 장이다.
이 장의 내용은 요구사항, 화면 설계, 개발, 테스트, 운영 기준의 상위 기준으로 사용된다.
작성 단위
정책 그룹 아래에 정책 항목 단위로 작성한다.
정책 항목 하나에는 하나의 판단 기준만 담는다.
정책 내용 작성 기준
실제 기준값, 허용 목록, 제한 조건, 적용 우선순위, 예외 허용 조건, 예외 불가 조건, 고객 고지 기준, 이력 저장 기준을 명확히 쓴다.
“검토 필요”, “추후 협의”, “시스템에서 처리”, “가능하도록 한다”는 정책 내용으로 쓰지 않는다.
TBD 작성 기준
미정 항목은 TBD로 표시한다.
단, 결정 주체, 결정 필요 사유, 결정 기한을 함께 쓴다.
TBD만 단독으로 남기지 않는다.

최종 점검 기준
1. 범위 정합성
이 문서가 다루는 업무와 제외하는 업무가 명확한지 확인한다.
다른 정책서와 중복되거나 빈틈이 생기지 않아야 한다.
2. 고객 완결성
고객이 전화 없이 앱·웹에서 처리 완료할 수 있는 구조인지 확인한다.
전화 전환이 필요한 경우, 그 사유와 예외 기준이 정책으로 정의되어야 한다.
3. 이해 가능성
고객이 처리 조건, 제한 사유, 예상 영향, 처리 결과를 이해할 수 있는지 확인한다.
특히 요금, 약정, 할인, 구독, 멤버십, 데이터, 인증, 법정대리인 관련 영향은 사전 고지가 필요하다.
4. 연결성
유즈케이스 → 상태 → 프로세스 → 기능 → 정책이 누락 없이 연결되어 있는지 확인한다.
프로세스에 관련 기능이 없거나, 정책 판단이 필요한데 정책이 없으면 보완한다.
5. 정책 구체성
정책 항목이 실제 결정값으로 작성되어 있는지 확인한다.
“협의”, “검토”, “처리 예정”처럼 실행 기준이 없는 문장은 보완한다.
6. 상세화 범위
화면 상세나 개발 상세로 넘어가야 할 내용이 과도하게 들어가지 않았는지 확인한다.
반대로 정책값이 빠져 업체가 임의 판단하게 되는 영역이 없는지도 확인한다.
