작성 기준
1. 문서 목적
본 템플릿은 통합채널 정책서를 상세 작성하기 위한 Full 버전 양식이다.
간소화 버전이 전체 구조와 정책 기준을 빠르게 정리하는 문서라면, Full 버전은 프로세스 상세와 기능 상세까지 포함하여 상세 설계의 기준을 명확히 남기는 문서다.
2. 작성 범위
업무 범위, 주요 용어, 액터, 유즈케이스, 상태 전이, 프로세스 목록, 프로세스 상세, 전체 업무 흐름도, 기능 목록, 기능 상세, 정책 목록, 정책 상세까지 작성한다.
업체가 화면 설계, 상세 기능 명세, 개발 범위 산정, 테스트 케이스 도출을 진행할 수 있도록 프로세스와 기능의 상세 기준을 포함한다.
3. 작성 수준
프로세스는 진입 조건, 종료 조건, 선행·후행 관계, 관련 기능, 관련 정책까지 작성한다.
기능은 입력 정보, 처리 로직, 세부 기능 구성, 출력 정보, 실패·예외 케이스까지 작성한다.
정책은 실제 판단값·허용범위·제한조건·예외 기준까지 작성한다.
4. 제외 범위
화면 UI 상세, 디자인 컴포넌트, API 필드 상세, DB 컬럼 상세, 배치 설계, 인프라 설계는 본 문서에 작성하지 않는다.
다만 해당 항목이 정책 판단, 기능 처리, 예외 처리에 직접 영향을 주면 정책 또는 기능 상세에 기준으로 남긴다.
5. ID 작성 규칙
업무코드는 3자리 영문 대문자를 사용한다.
예: 회원은 MBR, 요금제는 PLN, 결제는 PAY.
ID는 문서 전체에서 중복되지 않아야 하며, 삭제된 ID는 재사용하지 않는다.
6. 작성 순서
작성은 액터 → 액터별 유즈케이스 → 유즈케이스별 프로세스 → 프로세스 상세 → 프로세스별 기능·정책 → 기능 목록 → 기능 상세 → 정책 목록 → 정책 상세 순서로 진행한다.
용어 정의는 각 단계에서 새로 등장한 용어를 중간중간 보정하는 방식으로 관리한다.
7. 핵심 작성 원칙
정책서는 채널 전략 및 과제 방향과 정렬되게 작성한다.
다른 영역에서 작성될 것이라고 전제하지 말고, 중복되더라도 담당 업무 안에서 완결적으로 작성한다.
채널 문서라도 BSS가 수행하는 검증·판정·상태 변경·데이터 처리 업무를 충분히 포함한다.
기준 요구사항은 모두 반영하며, 작성 중 새로 도출된 사항은 요구사항 목록에 추가한다.

작성 원칙 가이드

정책서 작성 원칙
1. 채널 전략 및 과제 방향과 정렬
각 정책서는 통합채널의 전략과 실행 과제를 구체화하는 하위 산출물이다.
기능과 정책은 “기존 채널 기능을 옮긴다”가 아니라, “통합채널에서 고객 과업을 더 쉽고 명확하게 끝낸다”는 방향에 맞춰 작성한다.
  • 작성 기준: 고객 셀프 처리 확대, 채널 경량화, 중복 제거, 목적 기반 UX, AI 적용 가능성, 보안·인증 안정성, BSS 연계 구조와 맞는지 확인한다.
  • 좋은 작성: “요금제 변경 전 예상 청구액과 연계 혜택 변동을 사전 안내한다.”
  • 나쁜 작성: “기존 T월드 요금제 변경 화면을 제공한다.”
2. 중복되더라도 완결적으로 작성
다른 정책서, 다른 모듈, BSS 문서, 상세 설계서에서 작성될 것이라고 가정하지 않는다.
담당 업무 안에서 고객이 업무를 시작해 완료하기까지 필요한 액터, 유즈케이스, 프로세스, 기능, 정책을 모두 작성한다.
  • 중복 허용 기준: 본인인증, 약관 동의, 상태 조회, 신청 가능 여부 확인, 결과 알림, 이력 저장처럼 여러 업무에 반복되는 공통 기능·정책은 각 프로세스에 연결한다.
  • 정리 기준: 기능 정의와 정책 정의에서는 공통 기능·공통 정책으로 묶되, 프로세스 목록과 프로세스 상세에서는 해당 프로세스와의 관계를 빠뜨리지 않는다.
3. BSS 업무도 충분히 통합 작성
정책서가 채널 산출물이더라도 고객 업무 완결에는 BSS의 판단과 처리가 포함된다.
채널이 화면과 UX만 정의하고 BSS 처리를 비워두면, 프로세스가 실제로 완결되지 않는다.
  • 포함 대상: 가입 가능 여부 확인, 중복 여부 확인, 고객 상태 조회, 제한 조건 판정, 신청 결과 반영, 상태 전이, 청구·혜택·약정 영향 계산, 처리 이력 저장, 오류·실패 결과 회신.
  • 제외 대상: API 필드 상세, DB 컬럼 설계, 배치 상세, 시스템 내부 로직의 구현 방식.
    단, 정책 판단에 영향을 주면 정책 항목으로 남긴다.
4. 요구사항을 모두 포함
기준 요구사항은 정책서의 최소 작성 범위다.
요구사항 목록에 있는 내용은 프로세스, 기능, 정책 중 하나 이상에 반드시 매핑한다.
  • 작성 기준: 요구사항 1건이 고객 흐름이면 프로세스에, 시스템 행위이면 기능에, 판단 기준이면 정책에 반영한다.
  • 점검 기준: 요구사항 ID별로 “프로세스 ID / 기능 ID / 정책 ID” 중 하나 이상이 연결되어야 한다.
5. 새로 발견한 사항은 요구사항에 추가
정책서를 작성하다 보면 기존 요구사항에 없던 기능, 정책, 예외 조건, 연동, 상태, 알림, 이력 관리 필요성이 나온다.
이 경우 정책서에만 적지 말고 요구사항 목록에도 신규 요구사항 또는 기존 요구사항 변경으로 반영한다.
  • 신규 요구사항 추가 대상: 새로운 고객 과업, 새로운 처리 흐름, 신규 기능, 신규 정책 판단, 신규 BSS 연동, 신규 예외 처리, 신규 이력·증적 관리, 신규 알림·고지.
  • 변경 요구사항 처리 대상: 기존 요구사항의 범위 확대, 처리 주체 변경, 채널/BSS 역할 변경, 정책값 변경, 우선순위 변경.

최종 점검 기준
1. 범위 정합성
이 문서가 다루는 업무와 제외하는 업무가 명확한지 확인한다.
다른 정책서와 중복되거나 빈틈이 생기지 않아야 한다.
2. 고객 완결성
고객이 전화 없이 앱·웹에서 처리 완료할 수 있는 구조인지 확인한다.
전화 전환이 필요한 경우, 그 사유와 예외 기준이 정책으로 정의되어야 한다.
3. 프로세스 상세성
모든 프로세스에 진입 조건, 종료 조건, 선행·후행 관계, 관련 기능, 관련 정책이 작성되어 있는지 확인한다.
4. 기능 상세성
모든 기능에 입력 정보, 처리 로직, 세부 기능 구성, 출력 정보, 실패·예외 케이스, 관련 정책이 작성되어 있는지 확인한다.
5. 연결성
유즈케이스 → 상태 → 프로세스 → 프로세스 상세 → 기능 → 기능 상세 → 정책이 누락 없이 연결되어 있는지 확인한다.
6. 정책 구체성
정책 항목이 실제 결정값으로 작성되어 있는지 확인한다.
“협의”, “검토”, “처리 예정”처럼 실행 기준이 없는 문장은 보완한다.
7. 요구사항 반영
기준 요구사항이 프로세스·기능·정책 중 하나 이상에 모두 반영되어 있는지 확인한다.
작성 중 새로 발견한 사항은 요구사항 목록에도 추가한다.
