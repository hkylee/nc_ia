1. 핵심 신호

자동 납부 신청 / 자동 납부 변경 VoC는 고객이 업무를 시작하고, 조건을 이해하고, 결과를 확인하고, 실패를 복구하는 전체 흐름을 정책 기준으로 재설계해야 함을 보여준다.

Core signals
01

납부수단 변경 경로 불명확 및 ARS 의존 구조

" 빠르고 간단하게 처리하고 싶다 " → ARS· 전화 없이 앱에서 즉시 원스톱 처리하고 싶은 편의성 욕구

02

타인 명의 회선 · 카드 · 계좌 변경 시 인증 절차 복잡

" 가족 회선을 내가 직접 관리하고 싶다 " → 불필요한 동의 절차 없이 가족 회선을 통합 관리하고 싶은 자율성 욕구

03

자동납부 변경 · 신청 결과 불확실 및 처리 상태 확인 불가

" 변경이 제대로 됐는지 확인하고 싶다 " → 변경 완료 여부를 직접 눈으로 확인하고 싶은 통제감 욕구

04

가족 회선 일괄 변경 기능 부재 및 복합 업무 원스톱 처리 불가

" 가족 회선을 한 번에 처리하고 싶다 " · " 한 번의 통화로 모든 처리를 끝내고 싶다 " → 반복 작업 없이 원스톱 처리하고 싶은 효율성 욕구

05

카드 · 계좌 오류 사전 안내 부재 및 시스템 오류

" 카드 바꿨는데 자동으로 안 되나 " → 카드 변경 시 자동 연동되길 기대하는 편의성 욕구 + 미납 · 서비스 정지 공포

통합채널의 홈, 마이, 요금, 상품 영역에서 같은 업무 진입점을 제공하고, 현재 상태에서 가능한 다음 행동을 명확히 보여줘야 한다

권한

누가 직접 처리하거나 대신 처리할 수 있는가?

본인, 가족, 법정대리인, 대리인의 처리 가능 범위와 인증, 동의, 위임 조건을 정책 기준으로 분리해야 한다

사전 고지

무엇을 처리 전에 알려야 하는가?

요금, 혜택, 적용 시점, 제한 조건, 후속 영향을 업무 실행 전에 이해 가능한 문장과 비교 정보로 제공해야 한다

연계

어떤 업무를 한 번에 묶어야 하는가?

상담, 앱, BSS, 외부 기관에서 끊어진 처리를 고객 과업 기준의 단일 흐름으로 재구성해야 한다

상태·오류

실패와 오류를 어떻게 복구하는가?

접수, 검증, 반영, 실패, 취소 상태를 고객에게 보여주고 재시도, 대체 경로, 상담 전환 기준을 연결해야 한다

처리 가능 범위

언제 셀프 처리할 수 있고 언제 제한되는가?

평일, 야간, 주말, 공휴일, 청구 마감, 개통·해지 시점별 처리 가능 범위와 대체 행동을 정의해야 한다

상태

처리 상태를 어디까지 보여줘야 하는가?

자동 납부 신청 · 자동 납부 변경 업무의 접수, 진행, 완료, 실패, 취소 상태를 고객이 직접 확인하도록 정의해야 한다
