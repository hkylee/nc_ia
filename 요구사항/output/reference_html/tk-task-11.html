01

As-Is 문제

고객지원 기능은 많지만 문제 해결 흐름은 분산되어 있고, 검색 실패와 안내 이해 실패가 상담·매장 이탈로 이어진다.

  • 고객은 시작 메뉴를 직접 골라야 한다.
  • FAQ와 공지는 다음 행동으로 충분히 연결되지 않는다.
  • 상담과 매장 전환 시 이전 맥락이 이어지지 않는다.
02

전략 전환

고객지원 시작점을 문제 유형 허브로 통합하고, AI가 의도와 긴급도를 해석해 안내, 셀프 처리, 상담, 매장 경로를 추천한다.

  • 검색어와 고객 입력을 문제 유형으로 분류한다.
  • 공지·FAQ·셀프 처리·상담·매장을 한 흐름에서 제안한다.
  • 전환 시 확인 이력과 상태를 함께 전달한다.
03

To-Be 결과

고객은 어느 메뉴를 찾아야 할지 고민하지 않고 문제를 입력해 해결 경로를 받으며, 실패해도 맥락을 잃지 않고 다음 채널로 이동한다.

  • 셀프 해결률과 검색 성공 후 처리 전환율이 높아진다.
  • 상담 재설명과 오연결이 줄어든다.
  • 공지·장애 대응과 긴급 지원 경험이 안정된다.
Customer 문제 인지·입력·선택

Support Hub 검색·FAQ·공지·상담 통합

허브는 고객지원 기능을 메뉴가 아니라 문제 해결 단계와 긴급도 기준으로 배치한다.

AI Triage 의도 해석·경로 추천

AI는 문제 상황, 오류 코드, 업무명, 고객 상태를 해석해 안내와 실행 가능한 다음 행동을 제안한다.

Care Network 상담·매장·상태 연계

상담과 매장은 이전 맥락, 고객 입력, 처리 상태, 다음 행동을 이어받아 해결을 계속한다.

1. 이 문서가 다루는 과제

이 문서는 고객지원 기능을 상담, FAQ, 공지, 매장 조회의 병렬 메뉴가 아니라 문제 해결 허브로 재구성하는 기준을 정의한다.
고객은 문제 유형을 선택하거나 자연어로 입력하고, 시스템은 의도와 긴급도를 해석해 안내, 공지, 셀프 처리, 상담, 매장 예약, 상태 추적 중 적합한 경로를 제안한다.
상담과 오프라인 전환이 필요한 경우에도 검색어, 확인한 안내, 고객 입력, 처리 상태가 함께 이어져 반복 설명을 줄여야 한다.

통합채널 연결 관점

고객지원은 모든 과제의 실패와 예외가 모이는 마지막 접점이므로 각 업무의 상태, 실패 사유, 고객 입력 맥락을 받아 해결 경로로 전환해야 한다.

2. 과제 목적

  • 01

    분산된 온라인 고객지원 기능을 하나의 통합 고객지원 허브로 재구성한다.

  • 02

    고객이 스스로 문제를 정의하지 않아도 문제 유형, 검색, AI 입력을 통해 해결 경로를 찾도록 한다.

  • 03

    검색과 FAQ를 정보 조회에 머무르게 하지 않고 공지, 셀프 처리, 상담, 매장 예약으로 연결한다.

  • 04

    상담 전환이 필요할 때 고객이 이미 입력한 맥락과 확인한 안내가 상담원이나 매장으로 이어지게 한다.

  • 05

    공지와 장애 정보를 고객 회선, 서비스, 지역, 상품 기준의 영향도와 다음 행동으로 해석한다.

  • 06

    셀프 처리 실패, 미해결 이슈, 피드백, VoC를 고객지원 정책과 AI 품질 보정의 근거로 활용한다.

3. 문제 정의

현 상태

상담, 문의, FAQ, 이용안내, 공지, 매장찾기 기능이 메뉴별로 분산되어 있다.
고객은 문제를 해결하기 전에 어느 메뉴로 가야 하는지부터 판단해야 한다.

Pain Point

검색 결과가 없거나 안내를 이해하지 못하면 고객은 전화 상담이나 매장 방문으로 이동한다.
이동 과정에서 이전 검색어, 확인한 FAQ, 문의 내용이 이어지지 않아 반복 설명이 발생한다.

Root Cause

기능·메뉴 중심 IA와 고객 문제 유형 중심 분류 체계의 부재가 원인이다.
검색, FAQ, 공지, AI, 상담, 매장, 처리 상태가 하나의 오케스트레이션 기준으로 연결되어 있지 않다.

미해결 Risk

단순·반복 문의가 증가하고 셀프 처리율이 정체된다.
AI와 챗봇은 해결 시스템으로 신뢰받지 못하며, 온라인과 오프라인 고객지원은 계속 단절된다.

4. As-Is 문제점

01 진입점 분산
현상

고객지원 기능이 메뉴별로 흩어져 있다

문제점

고객은 상담, FAQ, 공지, 매장 중 어디에서 시작해야 하는지 직접 판단해야 하며 초기 진입부터 피로를 느낀다.

02 콘텐츠형 FAQ
현상

FAQ와 이용안내가 해결 허브가 아니라 콘텐츠 묶음에 머문다

문제점

고객은 안내를 읽은 뒤 실제 처리 화면이나 상담 경로로 자연스럽게 이동하지 못한다.

03 검색 실패
현상

검색은 찾기 단계에 머문다

문제점

검색 결과가 없거나 부정확하면 FAQ, 셀프 처리, 상담, 매장 같은 다음 행동으로 이어지지 않는다.

04 공지 해석 부족
현상

공지와 장애 정보가 내 문제 해결 정보로 해석되지 않는다

문제점

고객은 해당 공지가 내 회선, 서비스, 지역, 상품에 어떤 영향을 주는지 이해하기 어렵다.

05 다음 행동 부재
현상

문제 확인 후 다음 행동 제안이 약하다

문제점

고객은 해결 가능 여부, 필요한 준비사항, 처리 제한, 상담 필요 여부를 스스로 판단해야 한다.

06 매장 연결 약함
현상

매장과 오프라인 지원 정보가 단순 조회에 머문다

문제점

방문 예약, 방문 전 준비, 처리 가능 업무, AS·임대폰 지원과 충분히 연결되지 않는다.

07 셀프 범위 부족
현상

셀프 처리 가능 범위가 좁아 구조적으로 상담 전환이 발생한다

문제점

권한, 인증, 정책 제한이 사전에 안내되지 않아 처리 중단과 상담 의존이 반복된다.

08 상담 맥락 단절
현상

상담 경로가 불명확하고 이전 맥락 전달이 부족하다

문제점

고객은 상담 유형과 연결 방식을 다시 선택하고, 이미 확인한 내용도 반복 설명해야 한다.

09 정책 장벽
현상

정책·권한·인증 조건이 해결 흐름을 끊는다

문제점

미성년, 가족, 복합 업무, 위임 처리, 운영시간 제한 같은 조건이 늦게 드러나 고객이 막힌다.

10 AI 신뢰 부족
현상

AI와 챗봇이 해결 시스템으로 신뢰받지 못한다

문제점

답변 요약이나 단순 안내 중심이면 고객은 실제 문제 해결보다 상담 전환을 더 신뢰하게 된다.

11 이력 분산
현상

상담, 셀프 처리, 매장 이력이 통합되지 않는다

문제점

한 채널에서 진행한 이슈가 다른 채널로 넘어갈 때 이어지지 않아 고객지원 경험이 분절된다.

12 상태 공유 부족
현상

채널 이동 시 처리 상태가 공유되지 않는다

문제점

접수, 처리중, 예약, 방문 예정, 완료, 보류 같은 상태가 고객과 상담·매장 간에 일관되게 전달되지 않는다.

13 긴급 지원 취약
현상

주말·야간·긴급 상황의 지원 연결이 약하다

문제점

운영시간 외 해결 가능 업무와 접수 후 처리 업무, 긴급 상담·매장 대체 경로가 명확하지 않다.

5. To-Be 흐름

고객지원 흐름은 고객이 문제를 인지하는 순간부터 문제 유형 선택, AI·검색 기반 해석, 안내와 셀프 처리, 상담·매장 전환, 처리 상태 추적, 피드백·VoC 반영까지 끊기지 않아야 한다.

01
Issue Signal 문제 인지

고객은 고객지원 허브에서 현재 이슈, 최근 문의, 긴급 공지, 자주 쓰는 지원 업무를 확인한다.

02
Classify 문제 유형 분류

고객 입력, 검색어, 오류 코드, 업무명을 기준으로 문제 유형과 긴급도를 분류한다.

03
Triage AI·검색 해석

AI가 의도를 해석하고 관련 FAQ, 이용안내, 공지, 셀프 처리, 상담·매장 경로를 추천한다.

04
Resolve 셀프 해결 실행

고객은 조건, 준비사항, 인증·권한, 정책 제한을 확인한 뒤 가능한 업무를 바로 처리한다.

05
Handoff 상담·매장 전환

셀프 해결이 어렵거나 긴급·복합 문제인 경우 상담, 예약, 매장 방문으로 맥락을 전달한다.

06
Learn 상태 추적·보정

고객은 처리 상태와 다음 행동을 확인하고, 피드백·VoC는 지원 정책과 AI 품질 개선에 반영된다.

  • AI가 문제 유형, 긴급도, 상담 필요성, 셀프 처리 가능성을 분류한다.
  • 검색 결과가 없을 때 질문 재작성, 유사 FAQ, 공지, 상담, 매장, VoC 등록을 다음 행동으로 제공한다.
  • AI 답변은 안내 요약, 영향 해석, 조건 확인, 실행 CTA를 함께 제공한다.
기대 효과
  • 검색 실패가 곧 이탈이나 전화 전환으로 이어지는 비율이 줄어든다.
  • AI가 실제 해결 경로를 제안하는 시스템으로 인식된다.
03

공지·장애 영향도 해석

To-do

  • 고객 회선, 이용 서비스, 지역, 가입 상품 기준의 영향 여부를 설명한다.
  • 영향이 있는 경우 FAQ, 대체 행동, 상담, 매장, 예약, 알림 신청으로 연결한다.
기대 효과
  • 고객은 공지를 내 상황과 연결해 이해하고 필요한 행동을 바로 선택한다.
  • 장애·공지 관련 반복 문의와 불안이 줄어든다.
04

셀프 해결과 정책 조건 사전 안내

셀프 처리 가능성 사전 확인률셀프 처리 실패율정책 제한 관련 재문의율
To-do

  • 미성년, 가족, 대리, 복합 업무, 운영시간 제한처럼 막히기 쉬운 조건은 초기에 분기한다.
  • 셀프 불가 시 대체 경로, 상담 연결, 매장 예약, 접수 후 처리, 결과 알림을 제공한다.
기대 효과
  • 처리 중간에 막히는 경험이 줄어든다.
  • 상담 전환은 실패가 아니라 적합한 지원 경로로 정리된다.

  • 상담 유형과 연결 경로를 단순화하고 대기 시간, 처리 가능 업무, 준비사항을 사전에 보여준다.
  • 매장 예약은 문제 유형, 처리 가능 업무, 방문 전 준비, AS·임대폰 지원과 연결한다.
기대 효과
  • 고객의 반복 설명과 상담 오연결이 줄어든다.
  • 온라인에서 시작한 문제가 오프라인에서도 같은 맥락으로 이어진다.

To-do

  • 미해결 이슈 재진입, 상태 알림, 후속 행동, 처리 결과 확인을 고객지원 허브에서 제공한다.
  • 피드백과 VoC를 문제 유형, 실패 지점, 상담 전환 사유, 안내 품질 기준으로 분석한다.
기대 효과
  • 고객은 문제가 어디까지 처리되었는지 확인하고 다음 행동을 놓치지 않는다.
  • 운영자는 반복 실패 영역을 근거 기반으로 보정한다.

7. 주요 프로세스 및 기능 설명

01
Issue Entry

고객지원 허브 진입과 문제 인지

고객이 지원 메뉴를 고르는 대신 문제 자체를 입력하거나 선택하게 하는 시작점이다.

Outcome

지원 시작 피로 감소와 적합한 해결 경로 진입률 향상

  1. 01

    고객은 고객지원 허브에서 상담, FAQ, 공지, 매장, 예약, VoC를 문제 유형 기준으로 확인한다.

  2. 02

    최근 검색, 진행 중 문의, 긴급 공지, 미해결 이슈, 처리 필요 상태를 우선 확인한다.

  3. 03

    고객은 문제 유형을 선택하거나 자연어, 키워드, 오류 코드, 업무명으로 상황을 입력한다.

  4. 04

    시스템은 고객 상태와 입력 내용을 기준으로 문제 유형, 긴급도, 가능한 지원 경로를 1차 분류한다.

02
Triage Flow

AI·검색 기반 해결 경로 추천

검색과 AI가 답변 제공에 머물지 않고 다음 행동과 해결 경로를 추천한다.

Outcome

검색 실패 이탈 감소와 AI 지원 신뢰도 향상

  1. 01

    AI는 고객 입력을 해석해 FAQ, 이용안내, 공지, 셀프 처리, 상담, 매장 중 적합한 경로를 추천한다.

  2. 02

    검색 결과가 부족하면 질문 재작성, 유사 문제, 관련 공지, 대체 키워드, 상담·매장 경로를 제공한다.

  3. 03

    공지와 장애는 고객 회선, 서비스, 지역, 상품 기준의 영향 여부와 다음 행동으로 설명한다.

  4. 04

    AI 안내는 요약 답변, 근거, 필요한 조건, 바로 실행 가능한 CTA, 상담 전환 조건을 함께 제공한다.

03
Self Resolve

셀프 해결과 예외 분기

조건 확인, 인증, 준비사항, 정책 제한을 미리 안내해 처리 중단을 줄인다.

Outcome

셀프 처리 성공률 향상과 정책 제한성 재문의 감소

  1. 01

    고객은 추천된 안내를 확인한 뒤 셀프 처리 가능 여부와 필요한 인증·권한·준비사항을 확인한다.

  2. 02

    가능한 업무는 고객지원 허브에서 실행 경로로 연결하고 처리 결과와 다음 행동을 확인한다.

  3. 03

    미성년, 가족, 대리, 복합 업무, 운영시간 제한, 정책 제한으로 셀프 해결이 어려운 경우 대체 경로를 제공한다.

  4. 04

    처리 실패 시 실패 원인, 재시도, 다른 경로, 상담 연결, 매장 예약, 접수 후 처리 여부를 안내한다.

04
Care Handoff

상담·매장 전환과 상태 추적

전환이 필요한 문제도 고객 입력과 처리 상태를 유지한 채 다음 채널로 이어진다.

Outcome

반복 설명 감소와 온라인·오프라인 지원 연속성 확보

  1. 01

    상담 전환 시 검색어, AI 입력, 확인한 안내, 고객 선택값, 실패 사유, 첨부 정보, 희망 처리 방식을 정리한다.

  2. 02

    상담 유형, 대기 시간, 처리 가능 업무, 준비사항을 보여주고 적합한 상담 채널로 연결한다.

  3. 03

    매장 지원이 필요한 경우 문제 유형에 맞는 매장, AS, 임대폰, 방문 준비, 예약 경로를 제공한다.

  4. 04

    접수, 처리중, 예약, 방문 예정, 보류, 완료 상태를 추적하고 처리 결과, 후속 행동, 피드백·VoC를 관리한다.

8. 정책서 작성 시 남겨야 할 판단 기준

1고객지원은 상담, FAQ, 공지, 매장 메뉴 묶음이 아니라 문제 유형과 긴급도 기준의 해결 허브로 설계해야 한다.
2검색은 결과 목록 제공에서 끝나지 않고 FAQ, 공지, 셀프 처리, 상담, 매장, VoC 등록으로 이어지는 다음 행동을 가져야 한다.
3AI는 고객 발화와 오류 코드, 업무명을 의도·긴급도·셀프 가능성으로 분류하고 답변 근거와 실행 CTA를 함께 제공해야 한다.
4공지와 장애 정보는 고객 회선, 서비스, 지역, 상품 기준의 영향 여부와 대체 행동을 설명해야 한다.
5셀프 처리 전에는 준비사항, 인증·권한, 가족·미성년·대리 조건, 운영시간 제한, 정책상 불가 사유를 먼저 안내해야 한다.
6상담과 매장 전환 시 검색어, AI 입력, 확인한 안내, 실패 사유, 고객 선택값, 현재 상태를 전달하는 범위를 정책화해야 한다.
7문의, 상담, 예약, 처리중, 보류, 완료, 미해결 같은 고객지원 상태는 고객 화면, 상담, 매장, 알림에서 같은 기준으로 관리해야 한다.
8주말·야간·긴급 상황은 즉시 해결 가능, 접수 가능, 운영시간 이후 처리, 매장·상담 대체 경로를 구분해야 한다.
9피드백과 VoC는 문제 유형, 실패 지점, 안내 품질, 상담 전환 사유, 매장 처리 결과와 연결해 개선 루프로 관리해야 한다.
10고객지원은 각 업무에서 발생한 상태, 실패 사유, 고객 입력 맥락을 받아 상담·매장·셀프 해결 경로로 이어야 한다.
11반복 문의와 검색 실패 데이터는 AI, FAQ, 전시, 상품 지식, 주문·결제 정책 개선 근거로 관리해야 한다.
