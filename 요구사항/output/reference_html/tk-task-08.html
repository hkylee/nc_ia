2. 과제 목적

  • 01

    주문 완료 이후 발생하는 후속 업무를 고객이 완결적으로 수행할 수 있는 사후관리 체계를 구축한다.

  • 02

    분산된 사후 처리 구조를 정리해 상태 조회, 처리 과정, 결과 안내의 일관성을 확보한다.

  • 03

    주문내역, 배송조회, 교환·반품, 해지·환불·취소, 상품 연동, 인증, 셀프개통을 하나의 lifecycle로 관리한다.

  • 04

    고객이 현재 가능한 행동과 처리 절차를 직관적으로 파악하고 고객센터나 대리점으로 이관되지 않도록 한다.

  • 05

    클레임 처리 전 예상 환불금, 위약금, 회수 비용, 혜택·포인트 영향을 투명하게 안내한다.

3. 문제 정의

현 상태

주문 이후의 교환, 반품, 환불, 해지, 배송, 상품 연동 업무가 기능별로 분산되어 있다.
일부 후속 업무는 셀프처리가 불가능해 고객 흐름이 중단된다.

Pain Point

고객은 주문 이후 현재 가능한 행동과 처리 절차를 직관적으로 파악하기 어렵다.
셀프처리 불가 시 고객센터나 대리점으로 이관되어 이용 흐름이 끊긴다.

Root Cause

주문 이후 lifecycle 전반에 대한 통합 설계가 부족하다.
주문 상태, 배송 상태, 클레임 조건, 비용 영향, 상품 활성화 상태가 하나의 사후관리 기준으로 연결되어 있지 않다.

미해결 Risk

고객의 셀프 처리율이 낮아지고 상담 의존도가 증가한다.
예상치 못한 혜택 소멸, 위약금, 환불 지연이 고객 불만과 신뢰 저하로 이어진다.

4. As-Is 문제점

01 상태 분산
현상

주문 이후 상태 조회와 사후 업무 기능이 분산되어 있다

문제점

고객이 여러 경로를 이동해야 하며 처리 흐름을 이해하기 어렵고 셀프 처리율이 낮아진다.

02 영향 안내 부족
현상

해지·환불·취소 시 연관 서비스 영향 안내가 부족하다

문제점

결합, 혜택, 포인트 소멸과 같은 예상치 못한 영향으로 고객 불만이 발생한다.

03 셀프 범위 제한
현상

셀프 처리 가능한 사후 업무 범위가 제한적이다

문제점

이용 흐름이 중단되고 고객센터나 대리점으로 이관되어 고객 이탈로 이어진다.

04 개통 불편
현상

셀프개통 과정의 불편이 크다

문제점

개통 단계에서 고객이 오프라인 채널에 의존하고 디지털 사후관리 완결성이 낮아진다.

5. To-Be 흐름

사후관리 흐름은 주문 완료 상태 확인에서 시작해 배송·설치·방문 일정, 가능한 변경·취소·클레임, 비용·혜택 영향 확인, 접수, 처리 상태 추적, 상품 활성화까지 이어져야 한다.

01
Status 주문 상태 확인

고객은 주문 목록과 상세에서 주문, 배송, 설치·방문, 클레임, 상품 연동 상태를 통합 확인한다.

02
Action 가능 행동 판단

현재 상태에서 배송지 변경, 일정 변경, 취소, 교환, 반품, 해지, 환불, 셀프개통이 가능한지 확인한다.

03
Impact 영향·비용 확인

예상 환불금, 위약금, 회수 비용, 결합·혜택·포인트 소멸, 환불 일정과 계좌를 사전에 확인한다.

04
Request 사후 업무 접수

사유 입력, 회수 방식, 환불 계좌, 배송지·수령 방식·일정 변경, 제휴 계정 연동을 진행한다.

05
Track 처리 상태 추적

접수, 처리중, 회수중, 환불 예정, 완료, 불가, 보류 상태를 단계형으로 조회하고 결과를 확인한다.

06
Activate 활성화/이용 시작

셀프개통, 상품 연동, 이용 시작 가이드, 예외 복구, AI 문의 응대와 상담 연결로 실제 사용을 완료한다.

Feedback Loop

    상태별로 현재 가능한 다음 행동과 불가한 행동의 이유를 함께 안내한다.

  3. 03

    배송 현황, 설치·방문 일정, 주문 이행 과정, 클레임 처리 상태를 단계형으로 노출한다.

  4. 04

    주문, 배송, 클레임, 상품 연동 상태를 하나의 대시보드에서 통합 안내한다.

02
Delivery Care

배송관리와 일정 변경

배송과 방문 일정의 조회·변경 가능 여부를 상태 기준으로 안내하고 직접 변경하게 한다.

Outcome

배송·방문 관련 상담 의존 감소

  1. 01

    고객은 배송 상태를 조회하고 배송 단계별 진행 상황을 확인한다.

  2. 02

    허용 가능한 단계에서는 배송지, 수령 방식, 설치·방문 일정을 직접 변경한다.

  3. 03

    변경이 불가능한 경우 사유와 대체 경로, 상담 연결을 안내한다.

  4. 04

    변경 접수 후 결과와 변경된 일정, 후속 주의사항을 안내한다.

03
Claim Simulation

클레임 접수와 비용 영향 확인

클레임 접수 전 가능 조건과 비용·혜택 영향을 투명하게 보여준다.

Outcome

환불·위약금 불만과 접수 후 철회 감소

  1. 01

    교환, 반품, 해지, 환불, 취소 가능 여부와 조건을 주문 상태와 상품 유형 기준으로 판정한다.

  2. 02

    고객은 사유를 입력하고 회수 방식과 환불 계좌를 선택한다.

  3. 03

    처리 전 예상 환불금, 위약금, 회수 비용, 혜택·포인트·결합 영향 시뮬레이션을 확인한다.

  4. 04

    접수, 처리중, 회수중, 환불 예정, 완료, 보류, 불가 상태를 단계형으로 조회한다.

04
Activation Care

상품 연동·셀프개통·문제 해결

주문 후 실제 이용 시작을 방해하는 연동·개통 문제를 셀프로 해결하게 한다.

Outcome

셀프개통 성공률과 상품 활성화 완료율 향상

  1. 01

    제휴 계정 연동, 상품 활성화, 이용 시작 가이드를 주문 상세에서 이어서 제공한다.

  2. 02

    셀프개통 제약사항을 사전 확인하고 개통 단계별 가이드와 예외 대응 방법을 제공한다.

  3. 03

    개통이나 상품 연동 실패 시 AI 기반 문의 응대, 자가 복구, 재시도, 상담 연결을 제공한다.

  4. 04

    이용 시작이 완료되면 완료 상태와 후속 이용 안내를 제공한다.

8. 정책서 작성 시 남겨야 할 판단 기준

1주문 사후관리는 주문 목록, 주문 상세, 배송, 클레임, 상품 연동, 셀프개통을 별도 기능이 아니라 하나의 주문 lifecycle 상태로 관리해야 한다.
2주문 이후 고객에게 노출되는 상태는 접수, 배송중, 설치·방문 예정, 변경 가능, 클레임 가능, 처리중, 환불 예정, 완료, 보류, 불가처럼 후속 행동이 달라지는 단위로 정의해야 한다.
3배송지, 수령 방식, 설치·방문 일정 변경은 배송 단계와 상품 유형에 따라 허용 여부, 변경 가능 시점, 변경 실패 복구 기준을 가져야 한다.
4교환, 반품, 취소, 해지, 환불은 가능 조건, 사유, 회수 방식, 환불 계좌, 위약금, 회수 비용, 예상 환불금, 처리 일정 기준을 사전에 안내해야 한다.
5클레임 처리 전 결합, 혜택, 포인트, 쿠폰, 연계 서비스 소멸·변경 영향은 고객이 선택을 확정하기 전에 확인해야 한다.
6셀프개통과 상품 연동은 주문 이후 부가 안내가 아니라 실제 이용 시작을 완료하는 사후관리 핵심 기능으로 관리해야 한다.
7불가·예외·실패 상황은 단순 안내로 끝내지 않고 대체 경로, 재시도, AI 문제 해결, 상담 연결, 담당 부서 연결 기준을 함께 제공해야 한다.
8주문 이후 상태는 주문, 결제, 청구, 고객지원, 알림에서 같은 고객 언어와 예상 완료 시점으로 제공해야 한다.
9취소·반품·해지·환불은 비용, 혜택, 청구, 자산 상태 영향과 상담·매장 대체 경로를 함께 안내해야 한다.
