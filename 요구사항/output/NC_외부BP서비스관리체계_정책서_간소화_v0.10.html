1. 개요

가. 범위

• 본 정책서는 통합채널에서 고객이 외부 BP 서비스로 이동하기 전 기대 결과, 처리 주체, 권한·동의, 복귀 경로를 이해하고 안전하게 이용하도록 하는 기준을 정의한다.

• 포함 범위는 진입 전 안내, 호환 환경·설치·권한 사전 점검, 미니앱 공통 진입·종료, 외부 처리 결과·상태 동기화, 지연·보류 안내, 중단 후 이어하기, 실패 시 대체 경로 제공이다.

• 운영 범위는 제휴 서비스 마스터, 호출 방식·인증 방식, 전달 정보 항목, 노출 위치, 랜딩·복귀 규칙, 테스트 환경, 단계적 오픈, 차단·롤백, 장애 모니터링 기준을 포함한다.

• 핵심 정책 질문은 누가 외부 BP 경험을 책임지는가, 어떤 조건에서 진입을 허용하는가, 어떤 정보가 전달되는가, 실패하면 몇 번까지 재시도하고 어떤 대체 경로를 제공하는가, 결과와 이력을 어디에 남기는가이다.

• 대상 채널은 통합채널 앱·웹 FO이며, 외부 BP 서비스가 제휴 앱, 미니앱, 웹뷰, 인앱 브라우저, 딥링크, 앱링크, API 연계 중 하나로 연결되는 범위까지 포함한다.

• BSS와 채널 업무 시스템은 고객 상태, 로그인·인증 상태, 최소 고객 컨텍스트, 결과 수신·이력 저장 기준을 제공하거나 반영하는 연계 주체로 다룬다.

• 상세 요구사항 18건은 요구사항 Trace 기준으로 유즈케이스, 상태, 프로세스, 기능, 정책 또는 정책 항목 중 하나 이상에 매핑한다.

• 제외 범위는 제휴사 내부 화면 설계, 제휴사 API 필드 상세, 계약·정산 세부 조건, 제휴사 운영자 화면 설계, 외부 서비스의 자체 상품 정책이다.

나. 설계 원칙

• External Experience Ownership: 외부 BP로 이동하더라도 고객에게는 통합채널이 안내, 복귀, 실패 복구의 기준을 제공한다.

• Pre-check Before Handoff: 고객을 외부 서비스로 넘기기 전에 환경, 권한, 설치, 동의, 로그인 상태를 먼저 확인해 중복 입력, 중복 인증, 반복 안내를 줄인다.

• Minimum Context Transfer: 외부 연계에는 고객 과업 완료에 필요한 최소 정보만 전달하고 전달 목적과 동의 기준을 명확히 한다.

• Consistent Mini-app Experience: 제휴사별 화면 차이가 있어도 닫기, 뒤로가기, 도움말, 오류, 복귀 기준은 공통 경험으로 제공한다.

• Recoverable Failure: 외부 호출 실패, 권한 거부, 제휴사 장애, 처리 지연은 고객이 다음 행동을 선택할 수 있는 상태와 안내로 전환한다.

• Operational Kill-switch: 장애 또는 품질 이슈가 발견되면 제휴 서비스별 노출 차단, 우회, 롤백을 자동 처리 후보로 만들고 운영자 승인 후 일괄 적용할 수 있어야 한다.

TM-BPS-008대체 경로외부 서비스 이용이 불가능할 때 제공하는 웹 전환, 다시 시도, 상담 연결, 다른 제휴처 보기 같은 후속 행동이다.
TM-BPS-009제휴 서비스 마스터제휴사, 서비스 유형, 운영주체, 호출 방식, 노출 상태, 테스트 환경, 복귀 방식, 담당 조직을 관리하는 기준 정보다.
TM-BPS-010호출 방식외부 BP를 연결하는 iframe, webview, in-app browser, deeplink, app link, GET/POST, API 연계 방식이다.
TM-BPS-011단계적 오픈고객군, 채널, 앱 버전, 기간 기준으로 제휴 서비스 노출 범위를 점진적으로 확대하는 운영 방식이다.
TM-BPS-012운영 차단장애, 품질 이슈, 정책 충돌 발생 시 특정 제휴 서비스 노출과 호출을 중지하는 운영 상태다.

1) 상태 코드

상태 코드상태명정의대표 후속 처리
ST-BPS-001진입 전고객이 외부 BP 서비스 이동을 선택하기 전 상태다.
운영주체, 이동 이유, 전달 정보, 복귀 위치를 안내한다.

제한 사유와 앱 설치, 권한 허용, 웹 전환, 상담 연결 등 대체 경로를 안내한다.
ST-BPS-005외부 업무 진행 중고객에게는 외부 업무 진행 중으로 표시되고, 시스템 내부에서는 세션·권한·결과 수신 상태를 별도 기준으로 관리하는 상태다.
닫기, 뒤로가기, 도움말, 오류 처리와 복귀 기준을 유지한다.
ST-BPS-006외부 처리 보류제휴사 처리 지연 또는 추가 확인으로 즉시 완료되지 않은 상태다.
예상 처리 시점, 확인 경로, 완료 알림 여부를 안내한다.
ST-BPS-007외부 처리 완료외부 BP 서비스에서 고객 업무가 완료되어 결과 수신이 필요한 상태다.
완료 결과를 통합채널로 동기화하고 고객을 이전 화면 또는 다음 행동 화면으로 복귀시킨다.
ST-BPS-008복귀 완료고객 표시 상태는 복귀 완료이며, 시스템 내부 상태는 결과 수신·이력 저장·후속 행동 연결 완료 여부로 구분되는 상태다.
결과, 이력, 후속 행동을 표시하고 필요한 경우 다음 업무로 연결한다.
ST-BPS-009이어하기 가능이탈, 종료, 설치 전환, 오류 후에도 이전 진행 상태를 복원할 수 있는 상태다.
보관된 진행 상태와 유효 시간 기준을 확인하고 이어하기 진입을 제공한다.
ST-BPS-010대체 경로 필요외부 BP 호출, 인증 전달, 결과 수신, 복귀 처리 실패로 고객에게 재시도보다 대체 경로 안내가 필요한 상태다.
실패 유형에 따라 재시도, 웹 전환, 상담 연결, 다른 제휴처 보기 중 하나를 안내한다.
ST-BPS-014운영 기준 작성 중신규 또는 변경된 제휴 서비스의 마스터, 호출 방식, 인증 방식, 전달 정보 기준을 등록·수정하는 상태다.
운영 기준 입력값과 변경 이력을 저장하고 검증 대기 상태로 넘긴다.
ST-BPS-011운영 검증 대기신규 또는 변경된 제휴 서비스가 운영 반영 전 검증을 기다리는 상태다.
테스트 URL, 계정, 시나리오, 복귀 흐름을 검증하고 오픈 여부를 결정한다.
ST-BPS-012운영 오픈검증된 제휴 서비스가 고객군, 채널, 앱 버전, 기간 기준으로 노출되는 상태다.
호출 성공률, 지연, 실패 유형, 복귀 실패 지표를 모니터링한다.
ST-BPS-013운영 차단·롤백장애, 품질 이슈, 정책 충돌로 제휴 서비스 노출이나 호출을 중지한 상태다.
즉시 차단, 웹 fallback, 이전 연결 방식 롤백, 영향 고객 안내를 실행한다.

2) 상태 전이 기준

현재 상태전이 이벤트다음 상태처리 기준 및 후속 처리
진입 전외부 BP 서비스 진입 전 정보 확인사전 점검 중고객이 운영주체, 이동 이유, 전달 정보, 복귀 위치를 확인하고 계속 진행을 선택하면 사전 점검을 시작한다.
사전 점검 중호환 환경과 권한 사전 점검진입 가능지원 OS·앱 버전·로그인·필수 권한·동의 조건이 모두 충족되면 진입 가능으로 전환한다.
사전 점검 중호환 환경과 권한 사전 점검진입 제한설치 필요, 권한 거부, 지원 불가 환경, 동의 미충족 중 하나가 확인되면 제한 사유를 안내한다.
진입 가능외부 BP 서비스 이용 시작외부 업무 진행 중호출 방식 기준에 맞게 외부 BP가 열리고 고객 컨텍스트 전달이 성공하면 외부 업무 진행 중으로 전환한다.
외부 업무 진행 중외부 BP 지연·보류 확인 및 이어하기외부 처리 보류외부 처리 결과가 즉시 완료되지 않거나 제휴사가 보류 상태를 회신하면 예상 처리 시점과 확인 경로를 안내한다.
외부 업무 진행 중외부 처리 결과·상태 동기화외부 처리 완료외부 BP가 성공 결과와 고객 안내에 필요한 최소 결과 값을 회신하면 완료 상태로 전환한다.
외부 처리 완료외부 처리 결과·상태 동기화복귀 완료통합채널이 결과를 수신하고 이전 화면 또는 다음 행동 화면을 표시하면 복귀 완료로 전환한다.
외부 업무 진행 중외부 BP 지연·보류 확인 및 이어하기이어하기 가능앱 이탈, 종료, 설치 전환, 일시 오류가 발생해도 진행 상태 유효 시간이 남아 있으면 이어하기 가능으로 전환한다.
진입 가능외부 BP 실패 사유 확인 및 대체 경로 선택대체 경로 필요외부 호출 실패, 인증 전달 실패, 타임아웃, 제휴사 장애가 발생하면 실패 유형을 분류하고 고객에게 재시도 또는 대체 경로 선택을 요구한다.
대체 경로 필요외부 BP 실패 사유 확인 및 대체 경로 선택진입 제한동일 실패가 2회 이상 반복되거나 지원 불가 환경이면 재시도보다 대체 경로를 우선 안내한다.
운영 기준 작성 중제휴 서비스 연동 기준 관리운영 검증 대기서비스 마스터, 호출 방식, 인증 방식, 전달 정보 기준이 저장되고 변경 이력이 남으면 운영 검증 대기 상태로 전환한다.
운영 검증 대기제휴 서비스 노출·검증·오픈 관리운영 오픈테스트 계정, 진입, 이용, 결과 회신, 복귀 시나리오가 모두 통과하면 고객군·채널·기간 기준으로 오픈한다.
운영 오픈제휴 서비스 장애·차단·롤백 운영운영 차단·롤백호출 성공률 급락, 응답 지연, 복귀 실패, 상태 동기화 실패, 보안/동의 이슈가 발견되면 차단 또는 롤백한다.
운영 차단·롤백제휴 서비스 장애·차단·롤백 운영운영 검증 대기수정본이 등록되면 테스트 환경 검증을 다시 수행한 후 재오픈 여부를 결정한다.

3) 상태 전이 다이어그램

상태 전이 다이어그램은 대표 상태와 전이 이벤트를 개념적으로 보여주며, 상세 판정 기준은 상태 전이표를 따른다.

나. 정책 상세

1) 진입 전 안내 정책 (PG-BPS-ENT-001)

• 운영주체 안내 (PI-BPS-ENT-001)
- 외부 BP 이동 전 제휴사명, 운영주체, 이동 이유, 완료 후 복귀 위치를 한 화면에서 고지한다.
- 고객이 확인하지 않으면 외부 호출을 시작하지 않는다.
• 대체 경로 사전 안내 (PI-BPS-ENT-002)
- 진입 전 안내에는 이용 불가 시 웹 전환, 앱 설치, 권한 허용, 상담 연결 중 제공 가능한 대체 경로를 1개 이상 포함한다.

2) 동의·정보 전달 정책 (PG-BPS-CONS-001)

• 최소 전달 정보 (PI-BPS-CONS-001)
- 외부 BP에는 고객 과업 완료에 필요한 최소 컨텍스트만 전달한다.
- 전달 항목, 목적, 필수·선택 여부, 보관 주체는 진입 전 고지한다.
• 동의 거부 제한 (PI-BPS-CONS-002)
- 선택 동의를 거부해도 외부 BP 이용 자체가 불가능하지 않으면 기본 진입은 허용한다.
- 필수 동의 거부 시 제한 사유와 대체 경로를 안내한다.

3) 지원 환경·권한 점검 정책 (PG-BPS-ENV-001)

• 지원 환경 기준 (PI-BPS-ENV-001)
- 지원 OS, 브라우저, 앱 버전, 제휴 앱 설치 필요 여부, 필수 권한 조건 중 하나라도 미충족이면 진입 제한 상태로 전환한다.
• 권한 미충족 안내 (PI-BPS-ENV-002)
- 필수 권한이 거부된 경우 권한 허용 경로와 재시도 방법을 안내한다.
- 고객이 권한 허용을 거부하면 상담 연결 또는 웹 전환을 제공한다.

4) 인증·세션 연속성 정책 (PG-BPS-SESSION-001)

• 재인증 최소화 (PI-BPS-SESSION-001)
- 고객의 로그인과 본인 인증 상태가 유효하면 외부 BP 진입 시 재로그인·재인증을 요구하지 않는다.
- 동일 세션에서는 중복 로그인, 중복 인증, 중복 동의 안내를 반복하지 않고 필요한 경우에만 사유를 고지한다.
• 세션 만료 처리 (PI-BPS-SESSION-002)
- 외부 BP 이용 중 세션이 만료되면 고객을 대체 경로 필요 상태가 아니라 이어하기 가능 상태로 전환한다.
- 인증 후 이전 진행 상태가 유효하면 같은 단계로 복귀시킨다.

5) 미니앱 공통 경험 정책 (PG-BPS-MINI-001)

• 공통 종료 기준 (PI-BPS-MINI-001)
- 제휴 미니앱에는 닫기와 뒤로가기 동작을 제공한다.
- 닫기 선택 시 진행 중 업무가 있으면 중단 여부를 확인하고 이어하기 가능 여부를 안내한다.
• 도움말·오류 기준 (PI-BPS-MINI-002)
- 외부 BP 오류 화면에는 제휴사 문의만 표시하지 않고 통합채널 도움말, 상담 연결, 다시 시도 중 하나 이상의 다음 행동을 제공한다.

6) 결과·상태 동기화 정책 (PG-BPS-RESULT-001)

• 결과 수신 기준 (PI-BPS-RESULT-001)
- 외부 BP 완료, 실패, 보류, 중단 결과는 통합채널의 상태 코드로 매핑한다.
- 매핑되지 않은 결과 코드는 고객에게 일반 성공으로 표시하지 않고 운영 확인 필요 상태로 분류한다.
• 복귀 화면 기준 (PI-BPS-RESULT-002)
- 외부 BP 완료 후 고객은 이전 진입 화면, 결과 확인 화면, 다음 행동 화면 중 하나로 복귀한다.
- 복귀 위치는 제휴 서비스 마스터에 등록된 규칙을 우선한다.

7) 지연·보류 안내 정책 (PG-BPS-DELAY-001)

• 지연 판단 기준 (PI-BPS-DELAY-001)
- 제휴 서비스가 완료·실패 결과를 즉시 회신하지 않거나 등록된 처리 예상 시간을 초과하면 외부 처리 보류 상태로 전환한다.
• 완료 알림 기준 (PI-BPS-DELAY-002)
- 보류 상태에서는 예상 처리 시점, 확인 경로, 완료 알림 제공 여부를 안내한다.
- 완료 알림을 제공하지 못하면 고객이 직접 확인할 경로를 표시한다.

8) 중단 후 이어하기 정책 (PG-BPS-RESUME-001)

• 이어하기 유효 시간 (PI-BPS-RESUME-001)
- 중단 후 이어하기 상태는 제휴 서비스별 유효 시간 안에서만 제공한다.
- 유효 시간이 등록되지 않은 경우 당일 세션 안에서만 이어하기를 허용한다.
• 만료 안내 기준 (PI-BPS-RESUME-002)
- 이어하기 유효 시간이 만료되면 고객에게 처음부터 다시 진행해야 하는 사유와 이미 전달된 정보의 삭제 또는 보관 기준을 안내한다.

9) 실패·대체 경로 정책 (PG-BPS-FAIL-001)

• 재시도 제한 (PI-BPS-FAIL-001)
- 동일 외부 BP 호출 실패가 같은 세션에서 2회 반복되면 자동 재시도 대신 대체 경로 필요 상태로 전환하고 실패 사유와 대체 경로를 우선 제공한다.
- 같은 고객·서비스·세션의 중복 요청은 하나의 멱등 요청으로 묶어 중복 처리와 반복 고지를 방지한다.
• 대체 행동 기준 (PI-BPS-FAIL-002)
- 지원 불가 환경, 제휴 앱 미설치, 권한 거부, 제휴사 장애, 타임아웃은 서로 다른 대체 행동으로 안내한다.
- 최소 1개 이상의 후속 행동을 제공한다.

10) 제휴 서비스 마스터 정책 (PG-BPS-MASTER-001)

• 필수 등록 정보 (PI-BPS-MASTER-001)
- 제휴 서비스 마스터에는 제휴사명, 운영주체, 서비스 유형, 호출 방식, 지원 채널, 운영 상태, 테스트 환경, 복귀 방식, 담당 조직을 필수로 등록한다.
• 변경 이력 기준 (PI-BPS-MASTER-002)
- 운영 상태, 호출 방식, 전달 정보, 노출 조건, 복귀 규칙 변경은 변경 전후 값, 변경자, 적용 시각, 승인자를 이력으로 저장한다.

11) 호출 방식·파라미터 관리 정책 (PG-BPS-CALL-001)

• 호출 방식 승인 (PI-BPS-CALL-001)
- iframe, webview, in-app browser, deeplink, app link, GET/POST, API 연계 방식은 제휴 서비스별로 1개 이상의 주 호출 방식과 1개 이하의 fallback 방식을 등록한다.
• 전달 정보 검증 (PI-BPS-CALL-002)
- 호출 파라미터와 토큰은 운영 반영 전 테스트 환경에서 필수값, 만료, 위변조, 동의 버전 불일치 조건을 검증한다.

12) 노출·랜딩 규칙 정책 (PG-BPS-EXPOSURE-001)

• 노출 조건 (PI-BPS-EXPO-001)
- 제휴 서비스 노출은 고객군, 채널, 앱 버전, 기간, 고객 상태 조건을 모두 만족할 때만 허용한다.
- 조건이 충돌하면 노출하지 않는다.
• 복귀 규칙 (PI-BPS-EXPO-002)
- 종료·복귀 규칙은 제휴 서비스별로 이전 화면 복귀, 결과 화면 복귀, 다음 행동 화면 복귀 중 하나를 기본값으로 등록한다.

13) 테스트·검증 정책 (PG-BPS-TEST-001)

• 검증 필수 조건 (PI-BPS-TEST-001)
- 운영 반영 전 고객 진입, 외부 이용, 결과 회신, 중단 후 이어하기, 실패 대체 경로, 복귀 흐름을 모두 테스트한다.
• 테스트 이력 기준 (PI-BPS-TEST-002)
- 테스트 URL, 테스트 계정, 테스트 조건, 수행자, 수행 일시, 실패 케이스, 승인 결과는 운영 반영 전 이력으로 저장한다.

14) 단계적 오픈·차단·롤백 정책 (PG-BPS-OPEN-001)

• 단계적 오픈 기준 (PI-BPS-OPEN-001)
- 신규 제휴 서비스는 고객군, 채널, 앱 버전, 기간 중 하나 이상의 조건으로 단계적 오픈을 설정할 수 있어야 한다.
• 즉시 차단 기준 (PI-BPS-OPEN-002)
- 보안·동의 이슈, 연속 호출 실패, 복귀 실패 급증, 잘못된 고객 안내가 확인되면 운영자가 제휴 서비스 노출을 즉시 차단할 수 있다.

15) 장애 모니터링 정책 (PG-BPS-MON-001)

• 모니터링 지표 (PI-BPS-MON-001)
- 제휴 서비스별 호출 성공률, 실패 유형, 응답 지연, 타임아웃, 복귀 실패, 상태 동기화 실패를 운영 지표로 관리한다.
• 장애 기준 (PI-BPS-MON-002)
- 제휴사 장애 또는 통합채널 연동 장애가 확인되면 고객에게 책임 주체와 확인 경로를 안내하고 운영자는 차단·우회·롤백 중 하나를 선택한다.
- 동일 장애가 여러 제휴 서비스에 반복되면 영향 서비스 목록을 자동 처리 후보로 묶고 운영자 승인 후 일괄 차단 또는 일괄 복구한다.

최종 점검 기준

외부 BP 서비스 관리 체계 정책서 제출 전 점검
1. 범위 정합성
외부 BP 서비스 관리 체계가 외부 링크 모음이 아니라 진입 전 안내, 이용, 복귀, 실패 복구, 운영 관리 기준으로 작성되었는지 확인한다.
2. 고객 경험 연속성
고객이 외부 서비스로 이동해도 운영주체, 개인정보 전달, 기대 결과, 복귀 위치, 문제 해결 방법을 이해할 수 있는지 확인한다.
3. 사전 점검 완결성
지원 환경, 앱 설치, 권한, 로그인, 인증, 동의 조건이 진입 전에 검증되는지 확인한다.
4. 상태 전이 정합성
진입 가능, 진입 제한, 외부 업무 진행 중, 보류, 완료, 복귀, 이어하기, 대체 경로 필요, 운영 기준 작성, 운영 차단 상태가 고객 또는 운영자 후속 행동과 연결되는지 확인한다.
5. 프로세스-기능-정책 연결성
고객 진입·이용·복귀 프로세스와 운영 마스터·장애·롤백 프로세스가 기능과 정책 ID로 연결되는지 확인한다.
6. 개인정보·동의 기준
외부 연계에 전달되는 고객 정보 항목, 목적, 필수·선택 여부, 동의 거부 시 제한 범위가 정책으로 정의되는지 확인한다.
7. 운영 통제 가능성
호출 방식, 노출 조건, 테스트 환경, 단계적 오픈, 즉시 차단, 롤백, 변경 이력 기준이 운영 가능하게 작성되었는지 확인한다.
8. 요구사항 반영
상세 요구사항명과 상세 요구사항 설명 18건이 유즈케이스, 프로세스, 기능, 정책 중 하나 이상에 매핑되었는지 확인한다.
