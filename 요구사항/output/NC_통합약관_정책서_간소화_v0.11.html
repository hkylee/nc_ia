1. 개요

가. 범위

• 본 정책서는 통합채널에서 제공하는 통합 약관 업무의 처리 기준을 정의한다.

• 요구사항에서 확인된 핵심 판단축은 약관 원문·요약 열람, 필수·선택 동의 구분, 목적별·서비스별 동의, 거부권 보장, 개인정보 제공·위탁·쿠키 고지, 약관 개정·재동의, 철회·재동의 이력과 운영 증적을 포함한다.

• 대상 채널은 앱·웹 FO이며 고객이 직접 조회, 조건 확인, 처리 요청, 결과 확인을 수행하는 셀프 처리 흐름을 포함한다.

• BSS와 연계 시스템이 수행하는 가능 여부 검증, 상태 판정, 처리 결과 회신은 정책 판단 기준에 포함한다.

• 화면 UI 상세, API 필드, DB 컬럼, 운영자 화면 상세는 제외하되 정책 판단에 필요한 기준은 포함한다.

• 상세 화면 설계, API 필드, DB 컬럼, 테스트 케이스는 후속 화면 설계서와 기능 상세 명세서에서 상세화한다.

• 핵심 정책 질문은 누가 이 업무를 시작할 수 있는지, 어떤 조건에서 제한되는지, 중복 요청을 어떻게 처리하는지, 실패 시 어떤 상태와 후속 조치로 복구하는지를 기준으로 둔다.

• 고객 표시 상태는 고객이 이해해야 하는 진행·완료·제한 상태로, 시스템 내부 상태는 BSS·연계 시스템의 판정·저장·회신 상태로 구분한다.

• 개발·QA는 유즈케이스 ID, 프로세스 ID, 기능 ID, 정책/정책항목 ID를 기준으로 정상·제한·실패·복구 테스트 조건과 기대 결과를 도출한다.

나. 설계 원칙

• 고객 과업 중심: 통합 약관은 내부 시스템 단위가 아니라 고객이 완료하려는 목적을 기준으로 구성한다.

• 셀프 처리 우선: 고객이 앱·웹에서 직접 완료할 수 있는 업무는 상담 전환보다 셀프 처리 경로를 우선 제공한다.

• BSS 판단 통합: 고객 상태, 가입 정보, 요금, 혜택, 제한 조건처럼 BSS 판단이 필요한 정보는 프로세스와 정책에 포함한다.

• 영향도 사전 고지: 비용, 혜택, 약정, 포인트, 쿠폰, 회선, 주문 상태에 영향이 있으면 처리 전에 안내한다.

• 권한·보안 우선: 민감정보 노출 또는 변경 처리는 본인확인, 동의, 권한 기준을 충족한 경우에만 허용한다.

• 근거 기반 운영: 요구사항과 참고자료에서 확인한 기준은 프로세스, 기능, 정책 중 하나 이상으로 추적 가능하게 관리한다.

- 고객에게 제공하는 대상 정보는 업무 완료에 필요한 기준 정보, 상태 정보, 안내 정보로 제한한다.
• 노출 우선순위 (PI-ZBC-INF-001-02)
- 고객 조건에 직접 맞는 정보, 진행 중 업무, 최근 이용 정보, 공통 안내 순으로 노출 우선순위를 둔다.
• 노출 제한 (PI-ZBC-INF-001-03)
- 고객 권한, 약관 동의, 보안 기준, 기준일이 충족되지 않은 정보는 개인화 결과에서 제외한다.
• 기준일 표시 (PI-ZBC-INF-001-04)
- 요금, 혜택, 약정, 상품 조건처럼 변동 가능한 정보는 기준일 또는 적용 시점을 함께 안내한다.
• 약관 요약+상세 UI (PI-ZBC-RQCOV-016)
- 각 약관과 동의 항목은 1줄 요약과 상세 원문 링크 또는 펼침 구조로 제공한다.
- 고객은 요약에서 목적과 영향 범위를 먼저 이해하고 원문 확인이 필요한 경우 상세 원문을 확인할 수 있어야 한다.
• 필수·선택 약관 분리 노출 (PI-ZBC-RQCOV-026)
- 약관 동의 화면은 필수 항목과 선택 항목을 시각적으로 구분해 제공한다.
- 고객이 전체 동의와 개별 선택의 차이를 명확히 확인할 수 있게 항목 depth와 선택 상태를 표시한다.
• 약관 적용 대상 서비스/채널 범위 표시 (PI-ZBC-RQCOV-031)
- 각 약관과 동의가 적용되는 서비스, 채널, 기능 범위를 표시한다.
- 하나의 동의가 전체 서비스에 적용되는지 특정 기능에만 적용되는지 고객이 구분할 수 있어야 한다.

• 미동의/비활성 고객 기본 경험 보장 (PI-ZBC-RQCOV-002)
- 개인화 또는 선택 동의가 없어도 기본 홈, 기본 탐색, 필수 조회, 고객지원 진입은 동일 수준으로 제공한다.
- 미동의로 제한되는 기능은 개인화 추천, 마케팅 수신, 선택 알림 등 부가 기능으로 한정한다.
• 윤리·거부권 보장 (PI-ZBC-RQCOV-004)
- 선택 동의와 권한 요청은 목적별로 분리하고 고객이 거부해도 핵심 서비스 이용을 제한하지 않는다.
- 동의를 유도할 때는 동의 이익과 미동의 영향만 안내하고, 오인 유도 문구와 묶음 강제는 허용하지 않는다.
• 결제 동의/약관 (PI-ZBC-RQCOV-007)
- 결제 필수 약관은 결제 요청 전에 요약과 상세를 제공하고 최신 버전 동의가 완료된 경우에만 결제를 허용한다.
- 필수 동의 미완료 시 결제는 차단하고 동의 화면으로 연결한다.
• 약관 동의 상태 조회 (PI-ZBC-RQCOV-009)
- 고객은 필수·선택 동의의 현재 상태, 적용 서비스, 약관 버전, 마지막 변경 일시, 철회 여부를 통합 조회할 수 있다.
- 서비스별 약관 상태가 다르면 서비스명과 적용 범위를 함께 표시한다.
• 선택 동의 철회/재동의 (PI-ZBC-RQCOV-010)
- 선택 동의는 목적별·서비스별·채널별로 철회와 재동의를 허용한다.
- 철회 즉시 해당 목적의 처리와 수신을 중지하고, 철회 전 동의값과 변경 일시를 이력으로 저장한다.
• 약관 동의 분리 기준 (PI-ZBC-RQCOV-015)
- 필수 약관과 선택 동의는 목적별·서비스별로 분리한다.
- 통합 채널이라도 개인화, 마케팅, 선택 알림, 제3자 제공처럼 목적이 다른 동의를 하나로 묶어 강제하지 않는다.
• 선택 동의 거부권 보장 (PI-ZBC-RQCOV-017)
- 마케팅, 개인화, 선택 알림 동의는 거부해도 핵심 서비스 이용이 가능해야 한다.
- 거부로 제한되는 항목은 추천, 광고성 수신, 개인화 표시 등 선택 기능으로 한정해 안내한다.
• 동의 변경 즉시 반영 (PI-ZBC-RQCOV-018)
- 고객이 선택 동의를 변경하면 변경 결과를 즉시 반영한다.
- 반영 결과는 동의 항목, 이전 값, 변경 값, 처리 채널, 적용 시점과 함께 저장하고 고객에게 완료 여부를 안내한다.
• 약관/정책 변경 고지 및 재동의 안내 (PI-ZBC-RQCOV-020)
- 약관 변경 시 변경 요약, 시행일, 영향 범위, 재동의 필요 여부를 고객에게 사전 고지한다.
- 재동의가 필요한 경우 알림 또는 진입 화면에서 변경 요약 확인 후 재동의 단계로 연결한다.
• 약관 미동의 시 제한 범위 안내 (PI-ZBC-RQCOV-021)
- 필수 약관 미동의 시 제한되는 단계와 기능을 구체적으로 안내한다.
- 선택 동의 미완료 고객에게는 기본 조회와 필수 업무를 허용하고, 동의가 필요한 부가 기능만 비활성 처리한다.
• 법정대리인 동의 이력/철회/재동의 (PI-ZBC-RQCOV-022)
- 법정대리인 동의는 동의 버전, 일시, 채널, 대리인 확인 결과를 이력으로 저장한다.
- 철회, 만료, 재동의가 필요한 경우 미성년 고객의 이용 제한 범위와 후속 절차를 함께 안내한다.
• 선택 동의 재노출 조건 (PI-ZBC-RQCOV-023)
- 선택 동의 유도 배너는 고객이 최초 거부한 뒤 동일 목적을 반복 강요하지 않는다.
- 2회 이상 재노출은 고객 행동 맥락, 미동의 이후 경과 기간, 서비스 가치 설명이 충족될 때만 허용한다.
• 위치기반서비스·OS 위치 권한 분리 (PI-ZBC-RQCOV-024)
- 위치기반서비스 이용약관과 개인위치정보 동의, OS 위치 권한은 각각 별도 상태로 관리한다.
- 약관 동의가 있어도 OS 권한이 거부되면 위치 기반 기능은 제한하고 대체 UX를 안내한다.
• 권한·동의 요청 시점 최적화 (PI-ZBC-RQCOV-025)
- 푸시, 위치, 마케팅, 개인화 동의는 앱 최초 진입 시 일괄 요청하지 않는다.
- 고객이 해당 기능의 가치를 이해하는 시점에 요청하고, 거부 후 재요청 조건과 설정 이동 경로를 안내한다.
• 상품 선택 기반 통합 동의 (PI-ZBC-RQCOV-028)
- 주문 또는 가입을 위해 선택한 상품의 약관은 한 흐름에서 확인할 수 있게 구성한다.
- 단, 상품별 특약, 개인정보 제공, 선택 동의는 목적과 적용 범위를 분리해 표시한다.
• 동적 동의항목 구성 (PI-ZBC-RQCOV-029)
- 동의항목은 상품 구성, 가입 유형, 고객 유형, 이미 동의한 항목을 기준으로 동적으로 구성한다.
- 이미 동의한 최신 약관은 반복 수집하지 않고 미동의 또는 버전 변경 항목만 추가로 받는다.
• 초대 수락 시 동의 묶음 처리 (PI-ZBC-RQCOV-030)
- 초대 수락 흐름에서는 그룹 가입 동의, 권한 위임 성격의 동의, 서비스 약관 동의를 한 흐름에서 처리하되 각 동의 목적과 철회 가능성을 구분해 안내한다.
• 법정대리인·미성년 동의 분기 (PI-ZBC-RQCOV-033)
- 만 14세 미만 고객 또는 법정대리인 동의가 필요한 고객은 일반 고객과 다른 절차로 분기한다.
- 대리인 확인 결과, 제출 필요 정보, 가입 가능 조건을 화면과 이력에 남긴다.
• 마케팅·광고성 정보 수신 동의 채널별 관리 (PI-ZBC-RQCOV-034)
- SMS, MMS, 이메일, Push 등 광고성 정보 수신 동의는 채널별로 관리한다.
- 고객은 채널별 수신 상태와 변경 이력을 조회하고, 변경 즉시 해당 채널 수신 기준에 반영한다.
• 외국인 전용 약관·동의 분기 (PI-ZBC-RQCOV-035)
- 외국인 고객에게 별도 약관 또는 동의가 필요한 경우 고객 조건에 따라 자동 노출한다.
- 동의 이력에는 적용 조항, 버전, 타임스탬프, 처리 채널을 저장한다.
• SMS 링크 약관동의 상태 관리 (PI-ZBC-RQCOV-036)
- 유통망 또는 고객센터 가입과 연계되는 SMS 링크 약관동의는 링크 발송, 진입, 열람, 동의, 완료 상태를 구분해 관리한다.
- 링크 만료 또는 미완료 시 고객과 상담사가 같은 상태를 확인할 수 있어야 한다.
• 현재 동의 상태 통합 조회 (PI-ZBC-RQCOV-038)
- 고객은 현재 동의한 약관, 철회한 항목, 재동의가 필요한 항목을 한 화면에서 확인할 수 있다.
- 서비스별·채널별 동의 상태를 따로 찾지 않도록 통합 조회 기준을 제공한다.
• 부분 철회·채널별 철회 지원 (PI-ZBC-RQCOV-039)
- 선택 동의는 전체 철회뿐 아니라 목적별, 채널별, 수단별 부분 철회를 지원한다.
- 문자만 거부하고 이메일은 유지하는 등 세분화된 제어 결과를 즉시 반영한다.
• 동의 즉시 팝업 (PI-ZBC-RQCOV-044)
- 기기변경 완료 후 앱 진입처럼 필수 개인정보 동의가 즉시 필요한 시점에는 팝업으로 안내하고 바로 처리할 수 있게 한다.
- 팝업은 필수 동의 항목과 미동의 제한 범위를 함께 표시한다.
• 인증수단 약관 노출 방식 통합 (PI-ZBC-RQCOV-049)
- 통합 본인인증 화면에서 인증수단별 약관 동의 노출 방식은 채널 공통 기준을 따른다.
- 인증수단별 추가 약관이 있으면 필수 여부와 적용 범위를 별도로 표시한다.
• 최종약관동의 화면·체크박스 구조 통합 (PI-ZBC-RQCOV-051)
- 최종약관동의 화면과 체크박스 depth는 채널 공통 기준으로 적용한다.
- 필수 전체동의와 선택 개별동의의 관계가 고객에게 오해되지 않도록 구성한다.
• OMD/USIM 전용 약관 항목 분기 (PI-ZBC-RQCOV-052)
- 채널과 가입유형 파라미터에 따라 OMD 또는 USIM 전용 약관 항목 노출 여부를 분기한다.
- 해당 항목이 적용되지 않는 고객에게는 불필요한 약관을 노출하지 않는다.
• 타인 주민등록번호 도용 안내 분기 (PI-ZBC-RQCOV-053)
- 채널 또는 가입유형 기준으로 타인 주민등록번호 도용 관련 안내가 필요한 경우 해당 영역을 노출한다.
- 안내 확인 여부는 가입 또는 인증 이력과 함께 저장한다.

- 업무 처리에 필요한 최소 항목만 필수로 지정하고 선택 항목과 구분한다.

- 오류 항목, 수정 방법, 재시도 가능 여부를 안내한다.

- 비밀번호, 인증번호, 결제정보 등 민감정보는 저장과 화면 노출 범위를 제한하고 로그에서 마스킹한다.

7) 영향도 고지 정책 (PG-ZBC-IMP-001)

• 고지 시점 (PI-ZBC-IMP-001-02)
- 고객이 최종 처리 요청을 확정하기 전에 영향도, 제한 사유, 적용 시점, 철회 가능 여부를 안내한다.
• 확인 기준 (PI-ZBC-IMP-001-03)
- 고객이 필수 고지 항목을 확인한 경우에만 다음 단계 또는 최종 처리 요청을 허용한다.
• 고지 이력 (PI-ZBC-IMP-001-04)
- 고지 항목, 확인 여부, 확인 일시, 적용 기준은 처리 이력과 함께 저장한다.

8) 처리 요청 접수 정책 (PG-ZBC-REQ-001)

• 접수 조건 (PI-ZBC-REQ-001-01)
- 필수 입력, 인증·동의, 영향도 확인, 가능 여부 검증이 완료된 경우에만 처리 요청을 접수한다.
• 요청 상태값 (PI-ZBC-REQ-001-02)
- 접수, 진행 중, 완료, 실패, 보류, 제한으로 구분한다.
• 접수 실패 (PI-ZBC-REQ-001-03)
- 요청 접수 실패 시 고객에게 실패 사유와 재시도 가능 여부를 안내하고 완료 상태로 확정하지 않는다.
• 접수 이력 저장 항목 (PI-ZBC-REQ-001-04)
- 요청자, 접수 일시, 요청 내용, 처리 상태, 접수 실패 사유를 저장한다.

9) 중복 요청 제한 정책 (PG-ZBC-DUP-001)

• 중복 판단 기준 (PI-ZBC-DUP-001-01)
- 중복 판단 기준에서 이전 요청이 실패·만료·고객 취소로 종료된 경우에만 재요청을 허용한다.
- 진행 중 요청이 있으면 중복 요청으로 제한한다.
• 중복 제한 (PI-ZBC-DUP-001-02)
- 중복 요청은 추가 접수를 제한하고 기존 요청 상태와 예상 처리 기준을 안내한다.
• 재요청 허용 (PI-ZBC-DUP-001-03)
- 이전 요청이 실패, 취소, 만료 상태이면 업무별 재요청 가능 기준에 따라 다시 접수할 수 있다.
• 중복 이력 저장 항목 (PI-ZBC-DUP-001-04)
- 중복 제한 여부, 기존 요청 ID, 고객 안내 여부를 저장한다.

10) BSS 연계 판정 정책 (PG-ZBC-BSS-001)

• 조회 기준 (PI-ZBC-BSS-001-01)
- BSS 조회는 고객 상태, 계약, 청구, 혜택, 제한 조건처럼 처리 가능 여부에 영향을 주는 항목으로 제한한다.
• 판정 우선순위 (PI-ZBC-BSS-001-02)
- 채널 보유 정보와 BSS 결과가 다르면 BSS 최신 판정 결과를 우선 적용한다.
• 결과 반영 (PI-ZBC-BSS-001-03)
- BSS 성공 회신과 상태 변경 완료가 확인된 경우에만 처리 완료로 확정한다.
• 실패 처리 방식 (PI-ZBC-BSS-001-04)
- BSS 조회 실패 또는 처리 실패 시 개인화 처리와 상태 변경을 제한하고 공통 안내 또는 상담 경로를 제공한다.

11) 상태 전환 정책 (PG-ZBC-STAT-001)

• 전환 조건 (PI-ZBC-STAT-001-01)
- 상태 전환은 필수 검증과 연계 처리 결과가 충족된 경우에만 수행한다.
• 전환 실패 (PI-ZBC-STAT-001-02)
- 상태 전환 실패 시 완료로 확정하지 않고 보류, 실패, 운영 확인 필요 중 하나로 분류한다.
• 전환 고지 (PI-ZBC-STAT-001-03)
- 고객 이용 가능 여부나 혜택에 영향을 주는 상태 전환은 고객에게 결과와 적용 시점을 안내한다.
• 전환 이력 (PI-ZBC-STAT-001-04)
- 전환 전 상태, 전환 후 상태, 전환 사유, 처리자, 처리 시점은 감사 가능한 이력으로 저장한다.

12) 처리 결과·이력 정책 (PG-ZBC-RSLT-001)

• 결과 구분 (PI-ZBC-RSLT-001-01)
- 처리 결과는 성공, 실패, 보류, 제한, 운영 확인 필요로 구분하고 고객에게 다음 행동을 안내한다.
• 처리 결과·이력 반영 기준 (PI-ZBC-RSLT-001-02)
- 연계 시스템 성공 회신과 상태 변경 완료가 모두 확인된 경우에만 처리 완료로 확정한다.
• 처리 결과·이력 실패 안내 (PI-ZBC-RSLT-001-03)
- 실패 또는 제한 결과는 사유, 재시도 가능 여부, 상담 전환 가능 여부, 적용되지 않은 항목을 안내한다.
• 이력 저장 (PI-ZBC-RSLT-001-04)
- 요청자, 처리 일시, 결과, 실패 사유, 연계 응답, 고객 고지 여부는 감사 가능한 이력으로 저장한다.
• 주문 문서/약관 아카이브 (PI-ZBC-RQCOV-006)
- 주문·결제·가입 시 고객이 동의한 핵심 약관과 유의사항 요약은 주문 상세 또는 가입 이력에서 다시 열람할 수 있게 보관한다.
- 아카이브에는 동의 일시, 약관 버전, 주문 또는 가입 식별 기준을 함께 연결한다.
• 동의 이력 조회 (PI-ZBC-RQCOV-019)
- 필수·선택 동의의 현재 상태와 변경 이력은 일시, 채널, 주체, 약관 버전, 철회 여부 기준으로 조회할 수 있어야 한다.
- 이력은 고객 조회와 운영 감사 모두에 사용할 수 있게 보관한다.
• 동의 이력 증빙 조회 (PI-ZBC-RQCOV-040)
- 고객과 운영자는 동의 일시, 버전, 동의 방식, 채널, 철회 여부를 증빙 형태로 조회할 수 있다.
- 증빙은 원문 버전과 연결되어 분쟁 대응에 사용할 수 있어야 한다.
• 약관 변경 알림 후 재동의 연결 (PI-ZBC-RQCOV-041)
- 약관 변경 알림을 받은 고객은 별도 탐색 없이 변경 요약 확인과 재동의 단계로 연결되어야 한다.
- 알림에서 진입한 경우 필요한 행동과 미동의 영향이 즉시 보이도록 한다.

13) 후속 업무 연결 정책 (PG-ZBC-NEXT-001)

- 처리 완료, 실패, 제한, 보류 결과에 따라 다음 업무, 재시도, 상담, 관련 메뉴 중 하나 이상을 제공한다.

- 후속 업무로 이동할 때 고객 요청, 대상 정보, 처리 결과, 제한 사유를 필요한 범위에서 전달한다.
• 연결 이력 (PI-ZBC-NEXT-001-04)
- 후속 업무 선택, 진입 성공 여부, 실패 사유는 품질 관리와 운영 개선 이력으로 저장한다.

• 예외 안내 (PI-ZBC-ERR-001-04)
- 예외 발생 시 고객에게 처리 지연 사유, 대체 경로, 예상 후속 조치를 안내한다.
• 셀프 처리 불가 시 상담 fallback (PI-ZBC-RQCOV-042)
- 동의, 철회, 재동의 과정에서 고객이 스스로 해결할 수 없으면 현재 약관, 버전, 동의 상태, 실패 사유를 상담 문맥으로 전달한다.

- 알림 발송 실패 시 중요 업무는 재발송 또는 상담 안내 대상으로 분류한다.

16) 개인정보·로그 보호 정책 (PG-ZBC-PRV-001)

• 최소 수집 (PI-ZBC-PRV-001-01)
- 업무 처리와 품질 개선에 필요한 최소 정보만 수집하고 목적 외 활용을 제한한다.
• 개인정보·로그 마스킹 기준 (PI-ZBC-PRV-001-02)
- 주민등록번호, 계좌번호, 카드번호, 인증번호, 비밀번호 등 민감정보는 화면과 로그에서 마스킹한다.
• 개인정보·로그 보관 기간 (PI-ZBC-PRV-001-03)
- 처리 이력, 인증 이력, 고객 안내 이력은 목적과 법정 기준에 맞는 보관 기간을 적용한다.
• 개인정보·로그 열람 통제 (PI-ZBC-PRV-001-04)
- 운영자 열람은 권한, 사유, 이력 저장 기준을 충족한 경우에만 허용한다.
• 민감/개인정보 최소화 원칙 적용 (PI-ZBC-RQCOV-001)
- 개인화와 AI 적용에 필요한 정보는 목적 달성에 필요한 최소 항목으로 제한한다.
- 민감정보와 식별정보는 마스킹·비식별 처리하고, 선택 동의가 없으면 개인화 기능만 제한하며 기본 서비스 이용은 유지한다.
• 개인정보 제공 고지 (PI-ZBC-RQCOV-003)
- 개인정보 제공 고지는 제공 목적, 제공 항목, 수신자, 보관 기간, 철회 방법을 요약과 상세로 구분해 제공한다.
- 고객 확인이 필요한 고지는 확인 일시, 약관 버전, 처리 채널을 증적 기준으로 저장한다.
• 개인정보/쿠키 고지 (PI-ZBC-RQCOV-005)
- 웹 쿠키와 저장소 사용은 목적, 보관 기간, 거부 방법, 거부 시 제한되는 기능을 명확히 안내한다.
- 고객이 거부하면 추적·개인화 기능은 제한하되 필수 로그인과 기본 탐색은 유지한다.
• 결제/주문 단계 신뢰 고지 (PI-ZBC-RQCOV-008)
- 상품 상세, 결제 직전, 주문 완료 단계에는 청약철회, 환불·교환, 분쟁 처리, 구매안전, 개인정보 제공·위탁 고지를 필요한 수준으로 노출한다.
- 고객 확인이 필요한 고지는 확인 일시와 버전을 저장한다.
• 개인정보 처리 안내 (PI-ZBC-RQCOV-011)
- 개인정보 열람, 정정, 삭제, 처리정지 요청 방법을 안내하고 셀프 처리가 가능한 범위는 직접 처리 경로를 제공한다.
- 셀프 처리가 불가하면 상담 경로와 처리 기준을 함께 안내한다.
• 개인정보 제3자 제공/위탁 고지 접근 (PI-ZBC-RQCOV-012)
- 내 정보 또는 약관 메뉴에서 제3자 제공, 처리 위탁, 보관 기간 정보를 쉽게 접근할 수 있게 한다.
- 제3자 제공 동의는 고객이 셀프로 철회할 수 있게 하고 철회 후 제한되는 혜택 또는 기능을 안내한다.
• 데이터 다운로드 가능 범위 (PI-ZBC-RQCOV-013)
- 고객이 본인 데이터와 일부 이용내역을 다운로드하거나 요청할 수 있는 경로를 제공한다.
- 다운로드가 불가한 항목은 사유, 대체 조회 방법, 상담 경로를 안내한다.
• 약관/오픈소스 고지 (PI-ZBC-RQCOV-014)
- 앱과 웹에서 이용약관, 개인정보처리방침, 오픈소스 라이선스 고지를 조회할 수 있게 한다.
- 고지는 현행 버전과 이전 버전, 시행일, 다운로드 가능 여부를 함께 표시한다.

17) 운영 기준 정보 관리 정책 (PG-ZBC-OPR-001)

• 기준 정보 관리 (PI-ZBC-OPR-001-01)
- 운영자는 업무 기준 정보, 노출 조건, 제한 기준, 안내 문구, 적용 기간을 승인 상태로 관리한다.

- 변경 승인 기준에서 고객 피해 가능성과 증적 이력이 함께 확인된 경우에만 운영 보정을 허용한다.
- 보정 전후 값과 승인 주체를 변경 이력으로 저장한다.
• 배포 기준 (PI-ZBC-OPR-001-03)
- 배포의 우선순위는 고객 피해 가능성, 금전 영향, 상태 변경 여부 순으로 판단한다.
- 충돌 시 제한 기준을 먼저 적용하고 예외 사유를 기록한다.
• 변경 이력 (PI-ZBC-OPR-001-04)
- 변경자, 변경일, 변경 사유, 변경 전후 값, 승인자는 운영 이력으로 저장한다.
• 약관 원문·요약·서비스 매핑 관리 (PI-ZBC-RQCOV-046)
- 운영자는 약관 원문, 요약 문구, 변경 요약, 적용 대상 서비스·채널, 고객 노출 위치를 매핑해 관리한다.
- 고객 표시 설명과 실제 적용 범위가 어긋나면 배포를 제한한다.
• 약관 어드민 관리 기준 (PI-ZBC-RQCOV-050)
- 운영자는 어드민에서 약관 등록, 수정, 노출 위치, 버전, 승인 상태, 종료일을 관리할 수 있어야 한다.
- 권한이 없는 사용자는 약관 변경 기능에 접근할 수 없다.
• MMS·이메일 교부 동의 약관명 관리 (PI-ZBC-RQCOV-054)
- 운영자는 MMS와 이메일 교부 동의 약관명을 어드민에서 등록·수정·관리한다.
- 약관명 변경 시 고객 노출명, 동의 이력명, 증빙 출력명이 함께 정합성을 유지해야 한다.
• 약관관리 기능 접근 권한 (PI-ZBC-RQCOV-060)
- 약관 등록, 수정, 승인, 삭제, 배포, 증빙 조회 기능은 역할 기반 권한을 적용한다.
- 권한 없는 접근은 차단하고 접근 이력을 저장한다.
• 제휴카드 약관동의 대상 관리 (PI-ZBC-RQCOV-061)
- 제휴카드 약관동의 대상은 운영 여부, 노출 기간, 적용 상품, 고객군 기준으로 관리한다.
- 운영 중지 시 신규 노출은 중단하고 기존 동의 이력은 보관한다.
• 미사용 약관관리 메뉴 노출 제어 (PI-ZBC-RQCOV-062)
- 사용 여부가 확인되지 않거나 운영 계획이 없는 약관관리 메뉴는 노출하지 않는다.
- 메뉴 노출은 운영 상태, 권한, 적용 서비스 기준을 모두 충족해야 한다.
• 외부채널 약관관리 기준 일원화 (PI-ZBC-RQCOV-064)
- 외부 제휴처 약관 등록과 노출 설정은 단일 어드민 기준으로 관리한다.
- 외부채널에만 적용되는 약관도 원문 버전, 노출 위치, 동의 이력 기준은 통합 채널과 동일하게 유지한다.

- 종료일이 지나면 신규 노출과 신규 동의 대상에서 제외하고 이전 동의 증적은 보관한다.
• 약관 저장·삭제 참조 무결성 (PI-ZBC-RQCOV-056)
- 약관 저장과 삭제 시 적용 서비스, 동의 이력, 진행 중 업무, 외부채널 참조 관계를 검증한다.
- 참조 중인 약관은 삭제하지 않고 종료 또는 사용중지 상태로 전환한다.
• 약관 변경 이력 버전 관리 (PI-ZBC-RQCOV-057)
- 약관 수정 시 이전 버전을 보관하고 새 버전을 자동 증가 처리한다.
- 변경 이력에는 수정자, 승인자, 변경 사유, 시행일, 영향 서비스를 저장한다.
• 종료일 경과 자동 상태 전환 (PI-ZBC-RQCOV-058)
- 종료일이 지난 약관은 자동으로 종료 상태로 전환한다.
- 종료 상태 전환 후에는 신규 동의 대상에서 제외하고 기존 이력 조회와 증빙 출력만 허용한다.
• 선택 동의 항목 중복 검증 (PI-ZBC-RQCOV-059)
- 선택 동의 항목 ID와 정렬순서는 중복 없이 관리한다.
- 동일 목적의 선택 동의가 중복 등록되면 배포 전 차단하고 운영자에게 중복 항목을 안내한다.
• 약관 승인관리 체계 통합 (PI-ZBC-RQCOV-063)
- Seller 어드민과 TDS 어드민의 약관 승인 관리는 단일 승인 정책으로 통합한다.
- 승인 상태, 반려 사유, 재승인 요청, 배포 가능 여부를 공통 기준으로 관리한다.

19) 품질 관리 정책 (PG-ZBC-QUL-001)

• 품질 지표 (PI-ZBC-QUL-001-01)
- 완료율, 실패율, 재시도율, 상담 전환율, 처리 지연, 고객 피드백을 품질 지표로 관리한다.
• 품질 관리 모니터링 주기 (PI-ZBC-QUL-001-02)
- 품질 관리 모니터링 주기 기준에서 실패·보류·상담 전환이 같은 조건에서 3회 이상 반복되면 품질 모니터링 대상으로 등록한다.
- 반복 원인은 개선 과제로 관리한다.
• 개선 과제 (PI-ZBC-QUL-001-03)
- 반복 실패, 상담 전환 과다, 고객 불만 반복, 기준 정보 오류는 개선 과제로 등록한다.
• 효과 추적 (PI-ZBC-QUL-001-04)
- 운영 보정 후 완료율, 실패율, 상담 전환율, 고객 피드백 변화를 비교해 개선 효과를 확인한다.

20) 요구사항 반영 관리 정책 (PG-ZBC-REQMAP-001)

• 반영 기준 (PI-ZBC-REQMAP-001-01)
- 요구사항은 유즈케이스, 상태, 프로세스, 기능, 정책 중 하나 이상에 반영해야 한다.
• 미반영 기준 (PI-ZBC-REQMAP-001-02)
- BSS 반영은 결과 확정 후 원장 또는 업무 상태 변경이 필요한 경우에만 요청한다.
- 반영 실패 시 실패 상태와 재시도 가능 여부를 저장한다.
• 근거 부족 (PI-ZBC-REQMAP-001-03)
- 근거 부족의 외부 연계는 인증기관·결제기관·제휴 시스템 응답이 필요한 경우 수행한다.
- 응답 지연 시 보류 상태로 두고 재시도 기준을 남긴다.
• 추적 관리 (PI-ZBC-REQMAP-001-04)
- 추적 관리는 판정 결과가 허용·제한·보류·실패 중 하나로 확정되면 상태를 전환한다.
- 다음 상태와 후속 가능 여부를 함께 저장한다.

21) 연계 오류·장애 처리 정책 (PG-ZBC-ERR-002)

• 장애 구분 (PI-ZBC-ERR-002-01)
- BSS 장애, 연계 장애, 인증기관 장애, 채널 장애, 일시 지연을 구분해 처리 기준을 적용한다.
• 처리 제한 (PI-ZBC-ERR-002-02)
- 정확한 판정이 불가능한 장애 상황에서는 상태 변경과 비용 발생 처리를 제한한다.
• 재시도 기준 (PI-ZBC-ERR-002-03)
- 일시 장애는 제한 횟수 내 재시도를 허용하고 반복 장애는 상담 또는 운영 확인으로 전환한다.
• 장애 이력 (PI-ZBC-ERR-002-04)
- 장애 유형, 발생 시점, 고객 영향, 처리 결과는 운영 모니터링 이력으로 저장한다.

22) 데이터 보관·파기 정책 (PG-ZBC-DATA-001)

• 보관 대상 (PI-ZBC-DATA-001-01)
- 처리 이력, 인증 이력, 고지 확인, 연계 결과, 운영 변경 증적은 목적별 보관 대상으로 분류한다.
• 파기 대상 (PI-ZBC-DATA-001-02)
- 업무 목적이 종료되고 법정 보관 사유가 없는 개인정보와 임시 데이터는 파기 대상으로 분류한다.
• 보관 기간 (PI-ZBC-DATA-001-03)
- 보관 기간은 법정 보관 사유, 고객 분쟁 대응, 처리 이력 추적 필요성을 기준으로 정한다.
- 기간 만료 시 파기 대상으로 전환하고 파기 이력을 저장한다.
• 파기 이력 (PI-ZBC-DATA-001-04)
- 파기 대상, 파기 시점, 파기 방식, 처리자는 감사 가능한 이력으로 저장한다.
• 고객정보 저장범위 선택 및 보관 (PI-ZBC-RQCOV-027)
- 고객이 저장 범위를 선택한 추가 수집정보는 재사용 목적과 보관 기간을 함께 저장한다.
- 보관 기간이 만료되거나 고객이 철회하면 재사용 대상에서 제외한다.
• 고객·서비스·약관 데이터 분리 관리 (PI-ZBC-RQCOV-043)
- 고객 정보, 서비스 정보, 약관 정보, 동의 이력은 서비스별 종속 데이터가 아니라 통합 기준 데이터로 분리 관리한다.
- 서비스 추가 시 기존 고객정보와 약관 동의 상태를 재사용할 수 있어야 한다.

최종 점검 기준

통합 약관 정책서 제출 전 점검
1. 통합 약관 범위, 제외 범위, 후속 상세화 영역이 고객 과업 기준으로 명확한지 확인한다.
2. 고객이 업무를 시작해 완료하거나 제한 사유와 대체 경로를 확인할 수 있는지 확인한다.
3. 유즈케이스, 상태, 프로세스, 기능, 정책의 ID와 연결 관계가 누락 없이 이어지는지 확인한다.
4. BSS와 연계 시스템의 조회, 판정, 상태 변경, 결과 회신 기준이 프로세스와 정책에 반영됐는지 확인한다.
5. 모든 프로세스가 관련 기능과 실제 정책 그룹명에 연결되어 있는지 확인한다.
6. 모든 기능이 화면 단위가 아니라 처리 단위이며 세부 기능 구성이 있는지 확인한다.
7. 정책 상세가 기능 설명이 아니라 실제 기능 동작값, 허용 목록, 횟수, 시간, 제한 조건, 채널, 고지, 저장 기준으로 작성됐는지 확인한다.
8. 개인정보, 민감정보, 로그, 이력, 보관·파기 기준이 포함되어 있는지 확인한다.
9. 운영 기준 관리, 변경 이력, 품질 모니터링, 개선 과제 관리 기준이 포함되어 있는지 확인한다.
10. 요구사항 통합 list의 관련 요구사항이 유즈케이스, 상태, 프로세스, 기능, 정책 중 하나 이상에 반영됐는지 확인한다.
11. 참고자료에서 확인한 고객 불편, 채널 전략, IA, VoC, 운영 기준이 정책 판단 기준으로 재구성됐는지 확인한다.
12. 핵심 정책 질문(누가, 언제, 어떤 조건, 중복·실패·복구 기준)이 범위와 정책 상세에 반영되었는지 확인한다.
13. 고객 표시 상태와 시스템 내부 상태가 구분되어 있고 상태별 후속 행동이 테스트 가능하게 정의되었는지 확인한다.
14. 개발자가 기능 설계 항목으로 전환할 수 있도록 프로세스-기능-정책 연결과 ID 추적성이 유지되는지 확인한다.
15. QA가 정상·제한·실패·복구 테스트 조건과 기대 결과를 요구사항 매핑 기준으로 도출할 수 있는지 확인한다.
