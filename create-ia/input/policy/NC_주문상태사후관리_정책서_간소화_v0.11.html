1. 개요

가. 범위

• 본 정책서는 통합채널에서 주문 완료 이후 고객이 주문·배송·설치·픽업·교환·반품·상품 연동·증빙을 확인하고 다음 행동을 수행하는 기준을 정의한다.

• 포함 범위는 주문 내역 통합 조회, 배송유형 그룹핑, 배송지·수령 정보 변경, 설치·방문 일정, 배송 지연 복구, 교환·반품 접수, 제휴 계정 연동·인증, 주문 완료 후 서비스 관리와 증빙 제공을 포함한다.

• 고객은 현재 주문이 왜 이 상태인지, 언제 완료되는지, 무엇을 할 수 있는지, 실패하면 어떻게 복구하는지를 화면에서 이해할 수 있어야 한다.

• 주문 생성, 결제 승인·수납, 해지·환불 금액 산정, 상품 탐색은 인접 정책서에서 다루며 본 문서는 주문 이후 상태와 사후 행동 기준만 정의한다.

• API 필드, DB 컬럼, 배송사 전문, 운영자 화면 상세, 배치 설계는 후속 상세 설계에서 다루고 본 문서는 정책 판단과 개발·QA에 필요한 기준만 포함한다.

• 채널 업무 시스템은 주문 이후 현재 상태, 가능 행동, 제한 사유, 고객 안내를 책임지고 주문 사후 연계 시스템은 배송·설치·픽업·클레임·제휴 연동·BSS 권한 반영 결과를 책임진다.

• 고객 표시 상태는 진행 중, 고객 조치 필요, 배송·설치 대기, 교환·반품 가능, 연동 실패, 처리 완료로 안내하고 내부 상태는 배송사 응답, 설치 예약, 제휴 인증, 권한 활성화, 이력 저장으로 분리한다.

• 상세 요구사항 111건은 주문 내역 통합 조회, 배송·설치·픽업, 교환·반품, 상품 연동·인증, 구매후기, 알림·증빙, 실패 복구 기준으로 재구성해 유즈케이스·프로세스·기능·정책 중 하나 이상에 매핑했다.

나. 설계 원칙

• 사후 처리 허브화: 주문 이후 관리는 단순 상태 조회가 아니라 배송, 설치, 개통, 변경, 클레임, 연동, 증빙의 다음 행동을 연결하는 허브로 구성한다.

• 고객 예측 가능성: 현재 상태, 제한 사유, 예상 완료 시점, 고객 조치 필요 여부를 같은 흐름에서 안내한다.

• 상태 기반 통합: 배송사, 설치, 매장, 제휴, BSS의 서로 다른 상태를 고객이 이해 가능한 표준 상태와 다음 행동으로 변환한다.

• 셀프 복구 우선: 지연, 실패, 연동 오류, 클레임 불가 상황에서도 고객이 재시도, 재예약, 대체 경로, 상담 전환을 직접 선택할 수 있게 한다.

• BSS·연계 정합성: 고객 표시 상태와 BSS 원장, 외부 이행처 응답, 권한 활성화 결과가 불일치하면 완료로 확정하지 않는다.

• 이력 추적성: 상태 전이, 변경, 클레임, 인증, 제휴 연동, 운영 보정은 개발·QA가 추적할 수 있는 이력 기준으로 남긴다.

- 본 문서는 주문 이후 상태 조회, 배송·설치·픽업, 주문 후 변경 가능성, 교환·반품 접수, 상품 연동·인증, 증빙 조회를 포함한다.
• 제외 범위 (PI-FUL-SCOPE-002)
- 주문 생성, 결제 승인·수납, 해지·환불 금액 산정, 상품 탐색은 인접 정책서의 본문 범위로 두고 본 문서에서는 상태·후속 행동 연계 기준만 다룬다.
• 사후 처리 허브 (PI-FUL-SCOPE-003)
- 주문 이후 화면은 상태 조회만 제공하지 않고 현재 상태별 가능 행동, 제한 사유, 다음 행동을 1개 이상 안내해야 한다.
• 중복 경계 (PI-FUL-SCOPE-004)
- 외부 채널 주문과 제휴 주문은 통합 조회 대상이지만 변경·클레임 가능 범위는 판매주체와 이행주체 기준으로 제한한다.

2) 상태 라벨·다음 행동 정책 (PG-FUL-STATUS-001)

• 표시 상태 표준 (PI-FUL-STATUS-001)
- 고객 표시 상태는 접수, 준비, 진행, 예정, 완료, 지연/실패, 제한, 클레임 진행, 연동 필요, 운영 확인 중 하나의 축으로 분류한다.
• 다음 행동 노출 (PI-FUL-STATUS-002)
- 각 상태에는 조회, 변경, 재시도, 회수 예약, 상담 전환, 증빙 확인 중 고객이 실제 수행 가능한 행동을 최소 1개 이상 연결한다.
• 제한 사유 고지 (PI-FUL-STATUS-003)
- 변경 제한, 클레임 불가, 연동 실패 상태는 원인, 고객 조치 가능 여부, 상담 전환 기준을 같은 화면에서 고지한다.
• 상태 알림 (PI-FUL-STATUS-004)
- 출고, 배송 지연, 설치 일정 확정, 클레임 접수, 연동 실패처럼 고객 행동이 필요한 상태 변화는 알림 또는 화면 배지로 고지한다.
• 알림 신청·취소 관리 (PI-FUL-STATUS-005)
- 고객은 로그인 상태에서 지원금 변동, 입고 알림 신청 내역을 조회하고 알림 탭에서 신청을 취소할 수 있다.

3) 주문 내역 통합 조회 정책 (PG-FUL-ORDER-001)

• 목록 조회 (PI-FUL-ORDER-001)
- 주문 목록은 주문일, 주문번호, 상품군, 배송유형, 현재 상태, 다음 행동을 함께 제공하고 최근 6개월을 기본 조회 범위로 둔다.
• 주문 상세 (PI-FUL-ORDER-002)
- 주문 상세는 주문 단위별 상태, 배송·설치·픽업 정보, 결제·증빙 진입, 클레임 가능 여부, 외부 주문 제한을 구분해 제공한다.
• 비회원 주문 (PI-FUL-ORDER-003)
- 비회원 주문은 주문번호와 주문 시 입력한 연락처 검증을 모두 통과한 경우에만 조회를 허용한다.
• 외부 주문 조회 (PI-FUL-ORDER-004)
- 전문 채널·제휴사 주문은 통합 조회 대상에 포함하되 변경·클레임은 원 채널 가능 범위를 함께 표시한다.
• 예약 신청 현황·임시저장 주문서 (PI-FUL-ORDER-005)
- 예약 판매 상품은 예약 신청 현황, 접수 상태, 임시저장 주문서 존재 여부를 주문 내역에서 함께 표시하고 고객이 이어서 작성할 수 있는 진입 경로를 제공한다.
• 주문 단위별 분리 표시 (PI-FUL-ORDER-006)
- 설치형, 택배형, 픽업형, 즉시발급형 상품이 한 주문에 섞여 있으면 주문 상세에서 이행 단위별 카드로 분리 표시하고 상태, 변경, 문의 가능 범위를 단위별로 제공한다.
• 통합회원 구매내역 조회 (PI-FUL-ORDER-007)
- 통합회원 ID 기준 구매내역은 주문번호, 상품군, 결제·배송 상태를 함께 조회하되 비회원 또는 외부 주문은 별도 검증 경로를 통과한 경우에만 연결한다.
• 가입유형별 주문정보 팝업 (PI-FUL-ORDER-008)
- 나의쇼핑 주문정보 팝업은 가입유형에 따라 상품 정보, 가입자 정보, 가입 조건을 아코디언 구조로 구분해 제공한다.
• 인터넷/IPTV 주문정보 탭 (PI-FUL-ORDER-009)
- 인터넷/IPTV 주문정보 팝업은 가입 정보, 배송·혜택 정보, 납부 정보, 부가 정보 4개 탭으로 구성한다.
• 2nd 디바이스·모회선 주문번호 통합 조회 (PI-FUL-ORDER-010)
- 2nd 디바이스 주문은 모회선 주문번호와 연결해 주문완료 화면과 주문 내역에서 통합 조회할 수 있게 한다.

4) 주문 후 변경 가능성 정책 (PG-FUL-CHANGE-001)

• 변경 가능 상태 (PI-FUL-CHANGE-001)
- 배송지, 연락처, 요청사항, 수령 방식, 설치 일정 변경은 출고 지시, 픽업 준비, 설치 확정, 디지털 발급 요청 전 상태에서만 허용한다.
• 취소·변경 진입 (PI-FUL-CHANGE-002)
- 고객이 주문 취소 또는 변경을 선택하면 현재 상태 기준으로 가능 여부를 먼저 판정하고 가능할 때만 후속 절차로 진입시킨다.
• 변경 불가 안내 (PI-FUL-CHANGE-003)
- 변경이 불가하면 배송중, 설치 확정, 픽업 준비, 외부 발급 진행, 클레임 진행 중 같은 제한 사유와 대체 경로를 안내한다.
• 변경 재검증 (PI-FUL-CHANGE-004)
- 변경 접수 후 배송 가능 지역, 재고, 설치 슬롯, 제휴 연동 조건은 다시 검증하고 실패하면 기존 값 유지 여부를 고객에게 고지한다.

5) 배송지·수령 정보 정책 (PG-FUL-DELIVERYINFO-001)

• 배송지 입력 (PI-FUL-DELIVERYINFO-001)
- 배송지는 주소 검색, 상세주소, 기본 배송지 여부, 수령인, 연락처를 필수 값으로 구분하고 누락 시 접수를 제한한다.
• 연락처 검증 (PI-FUL-DELIVERYINFO-002)
- 배송 연락처는 숫자 형식, 자리 수, 중복 입력 여부를 검증한다.
- 주문 명의자와 수령인 정보가 불일치하거나 고가·민감 상품 배송인 경우 본인확인 또는 수령인 확인을 요구한다.
• 요청사항 제한 (PI-FUL-DELIVERYINFO-003)
- 배송·설치 요청사항은 사전 정의 옵션과 직접 입력을 허용하되 금칙어, 최대 길이, 개인정보 노출 가능성을 검사한다.
• 마스킹 표시 (PI-FUL-DELIVERYINFO-004)
- 저장된 배송지와 연락처는 목록에서 일부 마스킹하고 변경·삭제 시 추가 확인을 요구한다.

6) 설치·방문 일정 정책 (PG-FUL-SCHEDULE-001)

• 일정 선택 (PI-FUL-SCHEDULE-001)
- 설치·방문 일정은 가능한 일자와 시간대만 선택지로 제공하고 선택 불가 슬롯은 사유를 함께 표시한다.
• 일정 변경 (PI-FUL-SCHEDULE-002)
- 설치 기사 배정 전에는 일정 변경을 허용하고 배정 후에는 기사·고객 영향 기준에 따라 제한한다.
• 방문 실패 복구 (PI-FUL-SCHEDULE-003)
- 고객 부재, 주소 오류, 설치 환경 부적합으로 방문 실패가 발생하면 재예약 가능 기한과 상담 전환 기준을 안내한다.
• 일정 고지 (PI-FUL-SCHEDULE-004)
- 방문 전 알림에는 방문 예정일, 시간대, 준비물, 변경 가능 기한, 취소 제한 조건을 포함한다.

7) 매장 픽업·수령 방식 정책 (PG-FUL-PICKUP-001)

• 수령 방식 (PI-FUL-PICKUP-001)
- 택배, 매장 픽업, 설치 방문, 디지털 발급은 상품과 고객 주소, 재고, 외부 연동 조건 기준으로 허용 여부를 판정한다.
• 매장 픽업 (PI-FUL-PICKUP-002)
- 매장 픽업은 실시간 재고와 운영 시간 기준으로 가능 매장을 표시하고 위치 정보 미허용 시 지역·매장명 검색으로 대체한다.
• 픽업 제한 (PI-FUL-PICKUP-003)
- 픽업 준비 완료 후 지정 기간이 지나면 만료 또는 재예약 상태로 전이하고 고객에게 재방문 가능 여부를 안내한다.
• USIM 경로 (PI-FUL-PICKUP-004)
- USIM 또는 실물 매체 수령은 배송, 픽업, 개통 방식에 따라 고객 준비 사항과 수령 확인 기준을 분리해 표시한다.
• 바로픽업 매장 UI (PI-FUL-PICKUP-005)
- 단말 바로픽업 매장 선택은 대리점 재고 수에 따라 화면을 분기하지 않고 동일한 매장찾기 UI에서 재고 보유, 예약 가능, 픽업 가능 시간을 함께 표시한다.
• 위치기반 가능 매장 강조 (PI-FUL-PICKUP-006)
- 고객 위치 정보가 허용되면 가까운 매장 중 바로픽업 또는 수령 가능한 매장을 우선 표시하고, 위치 미허용 시 지역·매장명 검색 결과에서 가능 매장을 별도 배지로 구분한다.

8) 배송 진행·완료 정책 (PG-FUL-DELIVERY-001)

• 배송 조건 선확정 (PI-FUL-DELIVERY-001)
- 상품 상세·목록·주문 단계에서 배송 가능 지역, 방식, 예상 일정은 동일 기준으로 선확정해 표시한다.
• 배송 추적 (PI-FUL-DELIVERY-002)
- 배송사는 송장 또는 이행 식별자가 생성된 경우에만 추적 경로를 제공하고 미생성 상태는 준비 중으로 표시한다.
• 분할 배송 (PI-FUL-DELIVERY-003)
- 한 주문에 서로 다른 배송유형이나 재고 위치가 있으면 상품을 이행 그룹으로 묶고 그룹별 예상 도착일을 제공한다.
• 완료 기준 (PI-FUL-DELIVERY-004)
- 배송 완료, 픽업 완료, 설치 완료는 외부 연계 응답과 고객 표시 상태가 모두 확인된 경우에만 이행 완료로 확정한다.
• 택배사 연동 배송조회 (PI-FUL-DELIVERY-005)
- 운송장번호 또는 택배사 식별자가 생성된 배송 건은 택배사 연동 링크나 인앱 배송조회 경로를 제공하고, 미생성 상태는 준비 중 사유와 예상 갱신 시각을 안내한다.
• 합배송·묶음배송 (PI-FUL-DELIVERY-006)
- 동일 고객의 복수 상품은 배송지, 출고지, 상품군, 재고 위치가 모두 합배송 조건을 만족할 때만 묶음배송으로 표시하고, 불가한 경우 배송비와 예상 수령일을 이행 그룹별로 분리한다.

9) 지연·실패 복구 정책 (PG-FUL-DELAY-001)

• 지연 안내 (PI-FUL-DELAY-001)
- 예상 완료일을 초과하거나 외부 연계 지연이 감지되면 지연 사유, 새 예상 시점, 고객 조치 필요 여부를 고지한다.
• 재시도 기준 (PI-FUL-DELAY-002)
- 배송·설치·픽업 실패는 동일 사유 2회까지 재시도 또는 재예약을 허용하고 3회째는 상담 전환을 우선 안내한다.
• 상담 전환 (PI-FUL-DELAY-003)
- 고객 조치로 복구 불가하거나 외부 연계 상태가 불명확하면 상담사에게 주문번호, 상태, 실패 사유, 이전 시도 이력을 전달한다.
• 상태 불일치 (PI-FUL-DELAY-004)
- 고객 표시 상태와 BSS·배송사·제휴 응답이 다르면 운영 확인 상태로 전환하고 임의 완료 처리를 제한한다.

10) 재고·이행 가능성 정책 (PG-FUL-INVENTORY-001)

• 실시간 재고 (PI-FUL-INVENTORY-001)
- 단말, USIM, 매장 픽업 상품은 재고 보유, 예약 가능, 품절, 입고 예정 상태를 구분해 표시한다.
• 배송 가능 상품 (PI-FUL-INVENTORY-002)
- 배송 가능 여부는 상품군, 주소, 재고 위치, 설치 필요 여부, 외부 이행처 가능 상태를 함께 판단한다.
• 재고 변동 (PI-FUL-INVENTORY-003)
- 주문 후 재고가 소진되면 대체 수령 방식, 일정 변경, 상담 전환 중 하나를 다음 행동으로 제공한다.
• 장바구니 재검증 (PI-FUL-INVENTORY-004)
- 미로그인 선택 상품은 로그인 후 고객 주소와 재고 조건을 재검증하고 불가하면 사유와 대체 조건을 안내한다.
• 구매 가능 디바이스 목록 조회 (PI-FUL-INVENTORY-005)
- 구매 가능 디바이스 목록은 단말·USIM 재고, 픽업 가능 매장, 배송 가능 지역, 품절·입고 예정 상태를 동일 기준으로 표시하고 주문 단계의 가능 여부와 불일치하지 않아야 한다.

11) 교환·반품 접수 정책 (PG-FUL-CLAIM-001)

• 클레임 가능 조건 (PI-FUL-CLAIM-001)
- 교환·반품 접수는 수령 완료 후 상품군별 접수 가능 기간, 사용 여부, 포장 상태, 제휴 발급 여부를 만족해야 허용한다.
• 사유 입력 (PI-FUL-CLAIM-002)
- 클레임 사유는 오배송, 파손, 단순 변심, 설치 불가, 제휴 발급 오류처럼 표준 사유 코드로 선택하고 상세 설명을 선택 입력으로 받는다.
• 첨부 기준 (PI-FUL-CLAIM-003)
- 파손, 오배송, 설치 불가 등 증빙이 필요한 사유는 사진 또는 확인 자료 첨부를 필수로 요구한다.
• 불가 안내 (PI-FUL-CLAIM-004)
- 접수 기간 초과, 사용 완료, 외부 발급 완료, 위생·개봉 제한 상품이면 불가 사유와 상담 가능 여부를 안내한다.

12) 회수·교환 처리 정책 (PG-FUL-RETURN-001)

• 회수 방식 (PI-FUL-RETURN-001)
- 반품·교환 회수는 택배 회수, 매장 방문, 설치 기사 회수 중 상품군과 주소 조건에 맞는 방식만 제공한다.
• 회수 예약 (PI-FUL-RETURN-002)
- 회수 예약은 가능한 날짜와 연락처 확인을 필수로 하고 예약 변경은 회수 전 영업일 기준 1회까지 허용한다.
• 교환 처리 (PI-FUL-RETURN-003)
- 교환은 회수 접수와 대체 상품 출고 가능 조건이 모두 확인된 경우에만 진행 상태로 전환한다.
• 환불 예정 표시 (PI-FUL-RETURN-004)
- 환불 금액 산정은 환불/취소 정책서 기준을 따르되 본 문서에서는 회수·검수 완료 후 환불 예정 상태만 표시한다.
• 단말 반품·교환 기한 (PI-FUL-RETURN-005)
- 단말 상품의 구매확정 이후 취소 또는 교환은 수령 완료 기준 14일 이내 조건을 우선 안내하고, 기한 초과 시 클레임 불가 사유를 표시한다.

13) 외부·전문 채널 주문 정책 (PG-FUL-EXTERNAL-001)

• 전문 채널 주문 (PI-FUL-EXTERNAL-001)
- 전문 채널, 제휴사, 제3자 이행 주문은 통합 조회 대상이지만 판매주체 기준으로 변경·클레임 가능 범위를 제한한다.
• 원 채널 연계 (PI-FUL-EXTERNAL-002)
- 통합 채널에서 처리할 수 없는 변경, 클레임, 문의는 원 채널 또는 제휴사 경로로 연결하고 이관 사유를 표시한다.
• 외부 상태 수신 (PI-FUL-EXTERNAL-003)
- 외부 주문 상태는 수신 시각과 원천 시스템을 함께 저장하고 24시간 이상 미갱신이면 상태 지연으로 표시한다.
• 책임 경계 (PI-FUL-EXTERNAL-004)
- 제휴사 배송·설치·발급 오류는 고객 안내와 상담 접수는 제공하되 최종 판정은 이행주체 응답 기준으로 확정한다.

14) 디지털 발급·이용 정책 (PG-FUL-DIGITAL-001)

• 발급 대상 (PI-FUL-DIGITAL-001)
- 디지털 상품, 이용권, 쿠폰, 기프티콘은 결제·주문 완료 후 발급 대상 여부를 판정하고 발급 대기 상태를 표시한다.
• 사용 경로 (PI-FUL-DIGITAL-002)
- 발급 완료 후 사용 코드, 사용처, 유효기간, 재조회 경로, 사용 제한 조건을 고객에게 안내한다.
• 발급 실패 (PI-FUL-DIGITAL-003)
- 발급 실패는 동일 주문 기준 자동 재시도 1회 후 실패하면 운영 확인 또는 상담 전환으로 처리한다.
• 선물 상품 (PI-FUL-DIGITAL-004)
- 선물형 디지털 상품은 수신자 수락 또는 인증 완료 전까지 발급 완료로 확정하지 않는다.

15) 제휴 계정 연동 정책 (PG-FUL-LINK-001)

• 계정 연결 (PI-FUL-LINK-001)
- 제휴 계정 연결은 자동 연결을 우선 시도하고 불일치, 미가입, 권한 부족이면 수동 연결 경로를 제공한다.
• 재연동 (PI-FUL-LINK-002)
- 상품 상태 변경, 명의 변경, 제휴 계정 변경, 권한 만료가 발생하면 재연동 필요 상태로 전환한다.
• 연동 해제 (PI-FUL-LINK-003)
- 고객이 연동 해제를 요청하면 상품 이용 권한, 잔여 혜택, 외부 계정 영향도를 고지한 뒤 해제를 허용한다.
• 계정 불일치 (PI-FUL-LINK-004)
- 구매자, 실사용자, 제휴 계정 소유자가 일치하지 않으면 허용 가능한 위임·실사용자 인증 조건을 안내한다.

16) 연동·인증 강도 정책 (PG-FUL-AUTH-001)

• 이용 자격 (PI-FUL-AUTH-001)
- 상품 이용 자격은 고객 상태, 회선 상태, 가입 상태, 연령·명의 조건, 제휴 서비스 가입 여부를 함께 확인한다.
• 인증 수단 (PI-FUL-AUTH-002)
- 본인확인, 실사용자 인증, 법정대리인 동의, 제휴 계정 인증 중 상품별 필수 인증 수단을 구분한다.
• 위험 기반 강화 (PI-FUL-AUTH-003)
- 기기 변경, 명의 불일치, 고위험 상품, 반복 실패, 외부 계정 변경 시에는 Step-up 인증을 요구한다.
• 인증 재사용 (PI-FUL-AUTH-004)
- 동일 세션에서 완료한 인증은 10분 이내 동일 상품군 연동에만 재사용하고 범위가 달라지면 재인증한다.
• 명의자·실사용자·공유 이용 관계 인증 (PI-FUL-AUTH-005)
- 명의자와 실사용자, 가족 공유, 대표 회선 관계가 다른 상품은 기존 관계 정보와 완료된 인증을 우선 재사용하되 권한 범위가 달라지면 추가 동의 또는 Step-up 인증을 요구한다.

17) 동의·전자서명 정책 (PG-FUL-CONSENT-001)

• 동의 문서 (PI-FUL-CONSENT-001)
- 제휴 연동, 실사용자 인증, 위임, 전자서명은 문서명, 버전, 동의 시각, 동의 주체를 이력으로 저장한다.
• 전자서명 (PI-FUL-CONSENT-002)
- 전자서명은 계약 또는 제휴 권한 활성화에 필요한 경우에만 요구하고 서명 완료 전에는 권한 활성화를 제한한다.
• 대리 동의 (PI-FUL-CONSENT-003)
- 대리인 또는 법정대리인 동의가 필요한 경우 동의 주체, 대상 고객, 유효기간, 철회 가능 여부를 표시한다.
• 동의 철회 (PI-FUL-CONSENT-004)
- 동의 철회는 제휴 권한 회수, 상품 이용 제한, 이력 보관 영향도를 고지한 뒤 처리한다.

18) 권한·상태 동기화 정책 (PG-FUL-SYNC-001)

• 권한 활성화 (PI-FUL-SYNC-001)
- 제휴 계정과 상품 권한은 BSS 계약 상태와 제휴 응답이 모두 정상일 때 활성화한다.
• 상태 변경 영향 (PI-FUL-SYNC-002)
- 상품 정지, 해지, 명의 변경, 양도, 결제 실패가 발생하면 제휴 권한과 사용 가능 상태를 재판정한다.
• 동기화 실패 (PI-FUL-SYNC-003)
- BSS 상태와 제휴 상태가 불일치하면 고객 표시 상태는 운영 확인으로 두고 재처리 이력을 저장한다.
• 권한 회수 (PI-FUL-SYNC-004)
- 상품 이용 종료 또는 연동 해제 시 제휴 권한 회수 결과를 확인하고 실패하면 운영 확인으로 전환한다.

19) 주문 완료 후 서비스 관리 정책 (PG-FUL-SERVICE-001)

• 완료 화면 (PI-FUL-SERVICE-001)
- 주문 완료 화면은 주문유형, 배송 방식, 개통·설치·픽업 여부, 제휴 연동 필요 여부에 따라 다른 후속 액션을 제공한다.
• 첫 액션 체크리스트 (PI-FUL-SERVICE-002)
- 개통 확인, 제휴 계정 연결, 혜택 설정, 유심·번호 확인, 알림 설정은 미완료 항목으로 저장해 재진입 시 이어서 처리하게 한다.
• 서비스 상태 (PI-FUL-SERVICE-003)
- 개통, 활성화, 이용 가능, 제한, 연동 필요 상태는 주문 상세과 서비스 관리 화면에서 동일한 상태명으로 표시한다.
• 가입 후 연결 (PI-FUL-SERVICE-004)
- 주문·가입·연동 완료 후 고객은 해당 서비스 관리 화면으로 자동 연결되며 실패 시 수동 진입 경로를 제공한다.
• 바코드 이용권 첫 사용 (PI-FUL-SERVICE-005)
- 바코드 이용권 결제 완료 화면은 첫 번째 혜택 사용 버튼을 제공하고, 고객이 선택하면 사용 가능한 바코드 랜딩 화면으로 연결한다.
• 신규 가입자 온보딩 튜토리얼 (PI-FUL-SERVICE-006)
- 신규 가입자는 가입 완료 직후 T우주패스, 플러스 상품처럼 생소한 서비스 핵심용어와 이용 구조를 2~3단계 스텝 안내로 확인한다.
• 액세서리 주문완료 후속 동선 (PI-FUL-SERVICE-007)
- 액세서리 주문완료 화면은 주문 결과, 주문 상세내역 보기, 메인 이동, 배송 조회 진입 중 고객이 다음에 수행할 행동을 함께 제공한다.
• 구매후기 작성·조회 (PI-FUL-SERVICE-008)
- 구매후기는 개통완료일 기준 30일 동안 작성할 수 있고, 최근 5년 이내 작성한 구매후기를 조회할 수 있다.
• 셀프동의 URL 후속 동선 (PI-FUL-SERVICE-009)
- 셀프동의 URL 주문완료 화면은 가입 내역 확인, 배송 희망 시간 선택, 메인 이동 중 고객이 필요한 후속 동선을 제공한다.

20) 계약서·증빙 제공 정책 (PG-FUL-PROOF-001)

• 계약서 PDF (PI-FUL-PROOF-001)
- 고객은 계약서와 표준안내서를 PDF로 다운로드하거나 이메일로 발송할 수 있고 발송 결과를 이력으로 저장한다.

- 영수증, 거래내역, 확인서, 계약 상태 증빙은 주문 상세과 결제·서비스 이력에서 재조회할 수 있어야 한다.
• 발송 제한 (PI-FUL-PROOF-003)
- 이메일 발송은 본인확인 또는 주문 정보 검증이 완료된 고객에게만 허용하고 실패 시 재시도와 다운로드 대체 경로를 제공한다.
• 그룹 계약 증빙 (PI-FUL-PROOF-004)
- 그룹 상품 계약은 그룹 대표와 구성원 조회 권한을 구분해 계약 이력과 상태를 표시한다.
• 결제내역 필터·문서 다운로드 (PI-FUL-PROOF-005)
- 결제내역은 기간, 결제, 환불, 취소 유형으로 필터링해 조회할 수 있어야 하며 고객이 선택한 내역은 영수증 또는 거래 증빙 문서로 다운로드할 수 있게 한다.

21) BSS·연계 반영 정책 (PG-FUL-BSS-001)

• 원장 반영 (PI-FUL-BSS-001)
- 주문 상태, 계약 상태, 배송·픽업·설치 상태, 제휴 권한 상태는 BSS 또는 연계 원장 반영 결과를 기준으로 확정한다.
• 결과 회신 (PI-FUL-BSS-002)
- 주문 사후 연계 시스템은 성공, 보류, 실패, 재처리 필요, 외부 응답 지연 중 하나의 결과를 회신해야 한다.
• 상태 보류 (PI-FUL-BSS-003)
- BSS 반영 실패, 외부 응답 지연, 상태 불일치가 발생하면 고객 표시 상태를 운영 확인 또는 지연/실패로 전환한다.
• 멱등 처리 (PI-FUL-BSS-004)
- 동일 주문, 동일 클레임, 동일 연동 요청은 요청 식별자로 중복 처리 여부를 판정하고 중복 접수는 제한한다.

22) 이력·사유 코드 정책 (PG-FUL-HISTORY-001)

• 상태 이력 (PI-FUL-HISTORY-001)
- 상태 전이는 현재 상태, 이벤트, 다음 상태, 전이 시각, 전이 주체, 원천 시스템을 저장한다.

- 배송지, 연락처, 요청사항, 일정, 수령 방식 변경은 변경 전후 값과 승인 주체를 저장한다.

- 인증 수단, 성공·실패 결과, 재시도 횟수, 유효 시간, 동의 문서 버전은 연동 이력에 남긴다.
• 사유 코드 (PI-FUL-HISTORY-004)
- 클레임, 변경 제한, 지연, 연동 실패 사유는 표준 사유 코드와 고객 표시 문구를 함께 관리한다.

23) 운영 관리 정책 (PG-FUL-OPR-001)

• 상태 코드 운영 (PI-FUL-OPR-001)
- 운영자는 고객 표시 상태, 내부 상태, 외부 상태 매핑을 관리하고 변경 시 검수·승인·배포 이력을 남긴다.

- 배송 방식, 설치 가능 조건, 픽업 매장, 주소 변경 가능 시점, 요청사항 제한 기준은 운영자가 버전으로 관리한다.
• 클레임 정책 운영 (PI-FUL-OPR-003)
- 교환·반품 가능 기간, 비용, 회수 방식, 사유 코드, 불가 조건은 상품군별 기준으로 관리한다.
• 연동·인증 운영 (PI-FUL-OPR-004)
- 인증 수단, 인증 강도, 계정 매칭, 예외 승인, 재처리 기준은 상품군과 제휴처 기준으로 관리한다.
• 모니터링 (PI-FUL-OPR-005)

- 바로도착, 행복배송 등 외부 운영 의존 배송 방식은 TDS 내재화 또는 통합 운영 전환 가능 여부를 운영 확인 항목으로 관리하고, 고객 표시 상태는 기존 배송 상태와 동일 기준으로 유지한다.
• 개통신청내역 통합 운영 (PI-FUL-OPR-007)
- 운영자는 채널·가입 유형별로 분산된 개통신청내역 화면의 노출 항목과 데이터 처리 기준을 비교해 단일 화면 통합 전환 조건과 제외 범위를 관리한다.
• 모바일 주문관리 체계 통합 (PI-FUL-OPR-008)
- 운영자는 Seller 어드민과 TDS 어드민의 모바일 주문관리 기능 차이를 비교하고 통합 운영 대상, 이관 제외 대상, 전환 시점을 버전으로 관리한다.

24) 상담 전환 정책 (PG-FUL-CS-001)

• 상담 문맥 (PI-FUL-CS-001)
- 상담 전환 시 주문번호, 상품명, 상태, 실패 사유, 고객 입력, 이전 재시도, 외부 응답 시각을 상담사에게 전달한다.
• 복구 안내 (PI-FUL-CS-002)
- 상담사는 고객이 직접 재시도 가능한 항목과 운영 보정이 필요한 항목을 구분해 안내한다.
• 상담 후 이력 (PI-FUL-CS-003)
- 상담으로 변경, 재예약, 재연동, 클레임 보정이 이루어지면 처리 결과와 고객 안내 이력을 저장한다.
• 셀프 회귀 (PI-FUL-CS-004)
- 상담 종료 후 고객이 다시 앱/웹에서 이어서 처리할 수 있는 링크 또는 다음 행동을 제공한다.

4. 유즈케이스-프로세스 계위
유즈케이스는 고객 또는 운영자의 완결 업무 목표이고 각 Y 유즈케이스가 시작·판단·처리·결과·예외 프로세스로 분해되었는지 확인한다.
